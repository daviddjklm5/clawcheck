# clawcheck AGENTS.md

## 1. 项目概览
本项目用于采集 iERP 权限申请、组织、花名册等数据，并落库到 PostgreSQL，供后续规则判断、查询分析和数据治理使用。

当前主要技术栈：
- `Python`
- `Playwright`
- `PostgreSQL`
- `Node.js / npm`
- `React + Vite`（`webui`）

主要目录约定：
- `automation/config`
  - 配置文件目录
- `automation/sql`
  - 建表、迁移、修复 SQL
- `automation/db`
  - PostgreSQL 读写入口与字段映射
- `automation/flows`
  - 页面采集流程
- `webui`
  - 前端页面与工作台交互
- `方案计划文档`
  - 需求、方案、字段口径、发布步骤等文档

## 2. PostgreSQL 配置规范

### 2.1 配置来源
PostgreSQL 连接配置优先从以下位置读取：

1. `automation/config/settings.yaml`
2. 环境变量 `IERP_PG_*`

相关示例与生产配置文件：
- `automation/config/settings.example.yaml`
- `automation/config/settings.yaml`
- `automation/config/settings.prod.example.yaml`
- `automation/config/settings.prod.yaml`

### 2.2 当前配置项
项目当前识别以下数据库配置项：

- `db.host`
- `db.port`
- `db.dbname`
- `db.user`
- `db.password`
- `db.schema`
- `db.sslmode`

环境变量名称：
- `IERP_PG_HOST`
- `IERP_PG_PORT`
- `IERP_PG_DBNAME`
- `IERP_PG_USER`
- `IERP_PG_PASSWORD`
- `IERP_PG_SCHEMA`
- `IERP_PG_SSLMODE`

### 2.3 配置示例
以 `automation/config/settings.example.yaml` 为准，当前示例为：

```yaml
db:
  host: "127.0.0.1"
  port: 5432
  dbname: "ierp"
  user: "postgres"
  password: "your_password"
  schema: "public"
  sslmode: "prefer"
```

### 2.4 安全要求
- `AGENTS.md` 不记录真实生产密码
- 默认不在文档中记录真实账号口令
- 如需说明连接实例，只记录：
  - `host`
  - `port`
  - `dbname`
  - `schema`
- 真实密码仅保留在本地配置或安全凭据系统中

## 3. 数据库表名与字段名命名规范

### 3.1 表名规范
- 业务物理表统一使用中文表名
- 与页面、报表、业务语义保持一致
- 不新增新的英文业务表名

当前典型表名包括：
- `申请单基本信息`
- `申请单权限列表`
- `申请单审批记录`
- `申请表组织范围`
- `人员属性查询`
- `权限列表`
- `在职花名册表`
- `组织列表`
- `城市所属战区`
- `组织属性查询`

### 3.2 字段名规范
- 字段优先使用页面字段、报表表头或已确认业务字段的中文名
- PostgreSQL 中中文字段必须用双引号引用
- 派生字段、规则字段以对应方案文档定义为准
- 同义字段跨表允许不同名，不强行做全局统一翻译

示例：
- `document_no` -> `单据编号`
- `role_code` -> `角色编码`
- `role_name` -> `角色名称`
- `permission_level` -> `权限级别`
- `skip_org_scope_check` -> `不检查组织范围`

### 3.3 允许保留英文的边界
仅以下情况可评估保留英文物理字段：

1. 技术元数据字段
2. JSON 快照类字段
3. 已与外部系统形成稳定工程约定且改名收益极低的字段

典型可评估字段：
- `raw_payload`

除以上情况外，不应新增英文业务字段。

### 3.4 禁止事项
- 不新增中英文混搭的新字段体系
- 不长期维护同一业务字段的中英文双写版本
- 未经方案文档确认，不直接把英文技术名翻译为中文后入库

## 4. 当前核心表说明

### 4.1 申请单基本信息
用途：
- 存储权限申请单主表信息

主键：
- `单据编号`

典型字段：
- `工号`
- `权限对象`
- `申请理由`
- `单据状态`
- `申请日期`

### 4.2 申请单权限列表
用途：
- 存储单据中的角色申请明细

核心关系：
- 通过 `单据编号` 关联 `申请单基本信息`

典型字段：
- `明细行号`
- `申请类型`
- `角色名称`
- `角色编码`
- `组织范围数量`

### 4.3 申请单审批记录
用途：
- 存储单据审批轨迹

核心关系：
- 通过 `单据编号` 关联 `申请单基本信息`

### 4.4 申请表组织范围
用途：
- 存储单据内“角色-组织”展开后的组织范围明细

当前规则以 `013` 方案为准：
- 新增 `角色编码`
- 新增 `角色名称`
- 正常角色按 `1 个角色编码 + 1 个组织编码 = 1 行`
- `不检查组织范围 = TRUE` 时只保留一条 `组织编码 = NULL` 记录

核心业务键：
- 正常角色：`单据编号 + 角色编码 + 组织编码`
- 跳过组织范围角色：`单据编号 + 角色编码`

### 4.5 权限列表
用途：
- 存储角色主数据、权限级别与组织范围例外配置

关键字段：
- `角色编码`
- `角色名称`
- `权限级别`
- `不检查组织范围`

### 4.6 人员属性查询
用途：
- 存储基于最新 `在职花名册表` 全量刷新得到的人员属性与 HR 判定结果

刷新时机：
- `在职花名册表` 导入完成后同步全量刷新

主键：
- `工号`

典型字段：
- `姓名`
- `部门ID`
- `一级职能名称`
- `二级职能名称`
- `职位名称`
- `标准岗位名称`
- `组织路径名称`
- `万御城市营业部`
- `责任HR工号`
- `责任HR导入批次号`
- `申请人HR类型`

关联规则：
- 申请单侧如需使用 HR 类型、责任 HR、HR 判定依据等人员属性，统一通过 `申请单基本信息.工号 -> 人员属性查询.工号` 关联获取
- 不再把 HR 判定结果回写到 `申请单基本信息`

### 4.7 其他主数据与派生表
包括但不限于：
- `在职花名册表`
- `组织列表`
- `城市所属战区`
- `组织属性查询`

其详细字段口径、命名和规则，按对应方案文档执行。

## 5. 方案文档优先级
当代码、SQL、字段命名与方案文档存在冲突时，优先以已确认方案文档为准，再回改代码和 SQL。

执行契约文档优先原则：
- 已确认的需求文档、方案文档、发布步骤文档，视为本项目执行契约
- 实现变更前，先确认对应执行契约文档是否已有口径；若无，先补文档再落实现
- 改动提交前，必须同步更新受影响的执行契约文档，禁止代码已变更但文档口径仍滞后

当前重点参考：
- `方案计划文档/009权限列表主数据建设方案文档.md`
- `方案计划文档/011数据库字段中文化迁移方案文档.md`
- `方案计划文档/013申请表组织范围角色组织展开落库方案文档.md`

规则：
- 新增字段前，先确认是否已有方案口径
- 派生字段前，先确认是否已有正式中文字段名
- 结构改造前，先确认是否已有对应迁移方案或发布步骤

## 6. 开发与改库注意事项

### 6.1 改表前检查项
改动数据库结构前，至少检查以下位置：
- `automation/sql`
- `automation/db/postgres.py`
- `automation/flows`
- `方案计划文档`

### 6.2 改字段时必须同步检查
若修改字段名、字段粒度或字段口径，必须同步检查：
- 建表 SQL
- 迁移 SQL
- Python 入库映射
- 采集流输出结构
- 相关方案文档
- 校验 SQL 或验收脚本

### 6.3 唯一约束设计要求
- 先确认业务粒度，再设计唯一约束
- 不允许先按“看起来不重复”建唯一索引，后续再补业务解释
- 对明细表尤其注意：
  - 是否需要角色维度
  - 是否允许 `NULL`
  - 是否需要部分唯一索引

### 6.4 迁移原则
- 能无损迁移的字段再做历史迁移
- 无法无损补齐业务语义的旧数据，不强行 SQL 拼接恢复
- 必要时采用：
  - 结构升级
  - 清理旧口径数据
  - 重新采集或重刷历史单据

## 9. 开发环境统一口径（300/301）

### 9.1 Python 版本统一原则
- Windows 本地开发与 CI 的默认 Python 口径统一为 `3.12.x`
- `.venv-win` 统一视为仓库运行时虚拟环境，默认由 `automation/scripts/install_windows_env.ps1` 创建与维护
- 非兼容性验证场景下，不应使用系统 `3.10` 解释器执行本仓库正式脚本

### 9.2 `.venv-win` 重建前置动作
- 进行 Python 版本升级或 venv 重建前，先执行 `automation/scripts/stop_all.ps1` 释放运行中进程
- 若重建时报“访问被拒绝”或 `.pyd` 文件删除失败，先清理命令行中引用 `.venv-win\Scripts\python.exe` 的残留进程，再重试
- 重建完成后必须验证：
  - `.venv-win/pyvenv.cfg` 的 `version`
  - `.venv-win\Scripts\python.exe --version`

### 9.3 VS Code 解释器约定
- 本仓库在 VS Code 中应选择解释器：`.venv-win\Scripts\python.exe`
- 禁止在仓库任务中混用系统 Python 解释器（如 `C:\Program Files\Python310\python.exe`）
- 若解释器与预期不一致，优先通过 `Python: Select Interpreter` 显式切回 `.venv-win`

### 9.4 CI 版本一致性
- 涉及 Python 的 workflow 应与本地统一口径保持一致（默认 `3.12.x`）
- 如需新增多版本矩阵，必须在方案文档中明确“兼容性验证目的”，不得无说明引入版本分叉

### 9.5 Node 来源与 PATH 统一原则
- 本机 Node 建议仅保留一个主用来源，默认推荐 `C:\Program Files\nodejs\`
- 若使用 `CLAWCHECK_NODE_DIR`，不应同时在 PATH 前序保留另一套 Node 路径，避免 `node/npm` 来源漂移
- 修改 Node 安装、PATH 或环境变量后，必须重启终端与当前 Codex 会话再执行构建/启动

### 9.6 WebUI 前置检查口径
- 前端构建与开发启动前，统一先执行 `automation/scripts/preflight_webui_node.ps1`
- preflight 固定检查链路：`where node`、`where npm`、`node -v`、`npm -v`
- preflight 如提示多 Node 源（multiple node/npm locations），应先完成单来源收敛，再进入常规开发流

### 9.7 npm 可用性判定边界
- 不允许仅凭一次 `npm` 执行失败就下“缺少 npm”结论
- 判定顺序至少包含：PowerShell `npm -v`、`npm.cmd -v`、`cmd /c npm -v`
- `npm -v` 失败时，优先回退 `C:\Program Files\nodejs\npm.cmd` 再判定

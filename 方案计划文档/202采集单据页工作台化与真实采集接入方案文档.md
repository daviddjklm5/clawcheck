# 方案编号202：采集单据页工作台化与真实采集接入方案文档

## 1. 方案目标

在 `200` 可视化页面骨架与 `201` 工作台化交互原则基础上，对 `采集单据` 页面做一次正式落地改造。

本方案明确交付以下结果：

- `采集单据` 页面改为“工具栏 + 全宽列表 + 右侧详情抽屉”的工作台结构
- 页面不再使用 mock 摘要数据，改为读取 PostgreSQL 中的真实采集结果
- 页面提供真实 `采集单据` 任务启动入口，前端可直接触发权限申请单据采集
- 页面可展示当前采集任务状态、最近采集任务结果与单据级落库详情
- 后端统一由 FastAPI 承接采集触发与状态查询，前端不直接执行脚本命令

本方案不改 PostgreSQL 表结构、不改 `013` 组织范围展开口径、不改 `collect` 既有脚本采集语义，只改页面承载方式与 API 接线方式。

补充说明：

- 本方案解决的是“采集工作台如何正式接入真实采集链路”
- 自 `203` 方案起，采集成功后可按任务参数自动串联增量 `audit`，用于把新采集单据推进到 `处理单据` 工作台  
  详见：`方案计划文档/203新采集单据增量评估与处理工作台最新结果口径方案文档.md`

## 2. 背景与问题

截至当前版本，`采集单据` 页面仍存在以下问题：

1. 页面主体仍是 `200` 阶段的示意卡片与 mock 单据，不具备真实工作台价值
2. 单据详情与主列表常驻双栏展示，列表无法成为主要操作区
3. 页面没有正式“启动采集”入口，用户仍需回到命令行执行 `run.py collect`
4. 采集结果、任务状态、单据详情没有统一接入到同一页面闭环
5. 当前接口 `GET /api/documents/collect-dashboard` 仍返回 mock 数据，无法反映真实落库结果

因此，需要通过 `202` 方案把 `采集单据` 页从“示意页”升级为“真实采集工作台”。

## 3. 与现有方案关系

### 3.1 与 `200` 的关系

`200` 解决的是前端骨架、路由、API 壳与页面分类。

`202` 的作用是把 `采集单据` 从 `200` 的 mock 阶段推进到真实运行阶段，并同步回写 `200` 中“采集页仍主要使用 mock 摘要数据”的旧表述。

### 3.2 与 `201` 的关系

`201` 明确了工作台页面应遵循以下原则：

- 列表优先
- 明细按需打开
- 主体不被常驻详情区挤压
- 详情抽屉化而不是永久双栏化

`202` 直接复用这套交互原则到 `采集单据` 页，但页面职责仍然保持为“采集工作台”，不承接 `处理单据` 的审批语义。

补充口径：

- 采集页本身不承担风险判断展示
- 但可在任务编排层触发“采集成功后自动评估”，以保证新采集单据能进入 `处理单据`

### 3.3 与现有 `collect` 脚本的关系

本方案不重写 `automation/scripts/run.py collect` 的业务逻辑。

本次接入方式为：

- 前端调用 FastAPI
- FastAPI 在服务端封装并调度现有 `collect` runner
- 采集完成后刷新 PostgreSQL 与任务摘要
- 后续可按 `203` 方案在同一任务链中串联增量 `audit`

即：

- 前端不直接执行脚本
- 仍复用现有 Playwright 采集链路

## 4. 设计原则

### 4.1 采集页回归工作台定位

`采集单据` 页面要解决的是：

- 如何触发采集
- 当前采集结果落到了哪些表
- 某张单据当前是否补采完成
- 最近任务是否成功

因此页面主体必须围绕：

- 采集动作
- 单据列表
- 单据详情

展开，不再保留大面积示意卡片。

### 4.2 列表优先，详情抽屉化

参考 `201`，页面主体以 `采集单据列表` 作为唯一主工作区。

- 只有点击 `单据编号` 时打开详情抽屉
- 关闭抽屉后保留列表筛选、排序、分页状态
- 不再使用固定右栏常驻详情

### 4.3 任务入口集中，不分散到多个区域

采集触发入口统一收敛在页面顶部工具区，并支持两类场景：

- 按最新待办批量采集
- 按指定 `单据编号` 采集单张单据

详情抽屉中允许补充“重新采集当前单据”快捷入口，但它本质上仍调用同一后端采集接口。

### 4.4 API 统一承接脚本执行

采集是浏览器自动化长任务，前端不应直接执行本地脚本命令。

因此本方案要求：

- 前端只调用 HTTP API
- API 负责参数校验、单任务并发控制、状态维护与结果摘要落盘
- 采集执行仍复用 `run.py collect`

## 5. 范围与边界

### 5.1 本次纳入范围

- `采集单据` 页 UI 按 `201` 风格重构
- 采集页正式接入 PostgreSQL 真实查询
- 新增采集触发 API 与任务状态输出
- 新增单据详情抽屉接口
- 新增采集任务摘要持久化文件，供页面展示最近任务

### 5.2 本次不纳入范围

- 不改 `申请单基本信息 / 申请单权限列表 / 申请单审批记录 / 申请表组织范围` 表结构
- 不改 `013` 的角色-组织展开逻辑
- 不新增通用任务中心
- 不实现取消运行中的采集任务
- 不处理多用户并发任务调度

### 5.3 数据库边界

本方案原则上：

- **不新增数据库字段**
- **不新增 SQL 迁移**
- 所有采集工作台摘要均通过现有中文表实时查询得出

## 6. 页面方案

### 6.1 页面结构

`采集单据` 页统一为三层结构：

1. 页面说明与状态摘要
2. 工具栏
   - 单据查询
   - 采集状态筛选
   - 启动采集
   - 刷新
   - 当前任务摘要
3. 采集单据主 Grid
   - 占据页面主体
   - 默认每页显示 `20` 项

### 6.2 列表交互规则

- 仅点击 `单据编号` 时打开详情抽屉
- 点击其他单元格时不自动打开详情
- 主列表默认分页口径调整为 `20` 项
- 关闭抽屉后保留当前列表筛选、排序、分页状态

### 6.3 详情抽屉页签

详情抽屉统一包含以下页签：

1. `单据概览`
2. `落库情况`
3. `申请角色`
4. `目标组织`
5. `审批记录`

页签含义如下：

- 桌面端抽屉宽度沿用 `201` 大抽屉口径，并在原基础上扩大约 `40%`，优先保证多列表页签的横向可读性

- `单据概览`
  - 展示主表字段、申请人信息、采集次数、最近落库时间
- `落库情况`
  - 展示四张表的记录数、状态、最近更新时间与备注
- `申请角色`
  - 展示角色编码、角色名称、申请类型、组织范围数量
- `目标组织`
  - 去掉角色编码、角色名称
  - 按组织维度聚合展示组织编码、组织名称、组织单位、物理层级、组织流程层级分类、组织授权级别、聚合行数
  - 默认按物理层级降序显示，`聚合行数` 表示命中的角色-组织明细条数
- `审批记录`
  - 展示节点、审批人、审批动作、审批意见、审批时间

列表型页签默认展示契约：

- `落库情况` 默认每页显示 `20` 项
- `申请角色` 默认每页显示 `20` 项
- `目标组织` 默认每页显示 `20` 项
- `审批记录` 默认每页显示 `20` 项

### 6.4 工具栏执行能力

工具栏提供以下输入：

- `指定单据编号`
- `采集上限`
- `dry-run`

交互规则：

- 若填写 `单据编号`，默认仅采集该单据
- 若未填写 `单据编号`，按待办列表顺序采集 `limit` 张
- 运行中不允许再次启动新的采集任务
- 运行完成后支持手动刷新列表，也支持页面自动轮询任务状态

## 7. 接口与数据结构方案

### 7.1 接口拆分

后端接口按职责拆分为三类：

1. `采集工作台列表`
2. `单据详情抽屉`
3. `采集任务启动`

### 7.2 建议接口

- `GET /api/documents/collect-workbench`
  - 返回工作台统计、主列表、当前任务、最近任务
- `GET /api/documents/collect-workbench/{document_no}`
  - 返回单据详情抽屉数据
- `POST /api/documents/collect-workbench/run`
  - 启动新的采集任务

兼容要求：

- 原 `GET /api/documents/collect-dashboard` 可作为兼容别名保留，但正式前端应切换到 `collect-workbench`

### 7.3 任务状态约束

当前阶段采用“单任务串行”策略：

- 任一时刻仅允许一个采集任务处于 `queued / running`
- 若已有运行中任务，再次触发时返回错误
- 最近任务结果通过任务摘要文件保留，供页面展示

### 7.4 任务摘要文件

为支撑页面展示最近任务，本方案要求每次采集完成后输出一份摘要文件，至少包含：

- `任务ID`
- `状态`
- `started_at`
- `finished_at`
- `requested_document_no`
- `requested_limit`
- `dry_run`
- `requested_count`
- `success_count`
- `skipped_count`
- `failed_count`
- `dump_file`
- `message`

## 10. 验收标准

满足以下条件时，可认为 `202` 完成：

1. `采集单据` 页面主体改为“工具栏 + 全宽列表”，不再使用固定双栏详情
2. 只有点击 `单据编号` 时才会打开详情抽屉
3. 页面默认每页显示 `20` 项
4. 采集页不再读取 mock 数据，而是读取 PostgreSQL 真实结果
5. 页面可直接启动 `采集单据` 任务，无需用户回到命令行
6. 页面可展示当前运行中的采集任务状态
7. 页面可展示最近采集任务结果摘要
8. 详情抽屉可看到 `单据概览 / 落库情况 / 申请角色 / 目标组织 / 审批记录`
9. 单据详情中的四张表记录数与数据库真实结果一致
10. `200` 文档中的采集页现状描述已同步更新

## 11. 风险与注意事项

### 11.1 不要让采集页重新退回演示页

即使第一阶段任务执行仍复用现有 `run.py collect`，页面层也必须保证：

- 展示真实结果
- 可以真实触发执行
- 可以看到任务状态

不能继续停留在 mock 卡片阶段。

### 11.2 不要让详情抽屉再次变成常驻右栏

本次改造已明确复用 `201` 工作台原则，因此必须避免：

- 整行点击打开详情
- 常驻右侧详情区
- 主列表宽度长期被压缩

### 11.3 注意单据状态与“补采”状态不要混淆

页面中的采集状态，是页面根据四张表落库情况推导出的工作台状态，不等同于业务单据自身 `单据状态`。

## 12. 变更记录（2026-03-23）

- 采集单据工作台新增 `可见浏览器` 开关（前端开关状态为 `headed`）。
- `POST /api/documents/collect-workbench/run` 请求体新增 `headed?: boolean | null`。
- 后端执行 `run.py collect` 时按开关透传浏览器模式：
  - `headed=true` -> `--headed`
  - `headed=false` -> `--headless`
  - `headed` 为空 -> 继承 `settings.browser.headed`

# 方案编号105：处理单据页审批单据页签与批准写回方案文档

## 1. 方案目标

在现有 `200` 可视化页面与 `008` 单据打开能力基础上，为 `处理单据` 页面补齐“单据审批执行层”的第一阶段能力。

本方案第一阶段只交付以下内容：

- 在右侧详情抽屉新增 `审批单据` 页签，位置在 `风险总览` 之后
- 在该页签内提供 `审批意见` 输入框与 `批准` 按钮
- 点击 `批准` 后，将页面填写的审批意见带入 EHR
- 在 EHR 当前单据右侧 `任务处理` 页签中：
  - 将 `审批决策` 设为 `同意`
  - 将 `审批意见` 设为页面输入内容
  - 自动点击 EHR 实际执行按钮完成提交

本方案当前**不**包含以下能力：

- `驳回`
- `加签`
- 批量审批
- 审批结果落数据库新表
- 直接在浏览器前端页面内嵌 EHR

## 2. 背景与现状

### 2.1 与现有方案的关系

当前已落地方案中：

- `008` 已明确沉淀“按单据编号打开原始单据”的链路
- `200` 已建设 `处理单据` 页面骨架
- `103` / `104` 已提供风险评估与用户反馈展示

但 `200` 中对审批动作的定位仍停留在“预留”，尚未形成正式写回契约。

因此，`105` 的定位不是重做 `处理单据` 页面，而是在现有页面与现有 Playwright 打开单据能力上，新增一层**单据级审批执行契约**。

  - 已具备登录态恢复与 Playwright 运行入口

结论：

## 3. EHR 实页探测结论

### 3.1 探测对象

`2026-03-15` 已用单据 `RA-20260310-00019845` 做页面级 dry-run 探测，未执行真实提交。

### 3.2 已确认事实

打开该单据后，详情区可见以下页签：

- `任务处理`
- `审批记录`

切到 `任务处理` 后，已确认存在以下可用控件：

- 字段 `审批决策`
  - 当前值为 `同意`
- 字段 `下一步节点`
  - 当前值为 `结束`
- 字段 `审批意见`
  - 可见占位符为 `请输入审批意见。`
- EHR 实际执行按钮
  - 页面文案为 `提交`

已确认 `审批决策` 下拉可见选项至少包括：

- `同意`
- `驳回至已选节点`
- `不同意并终止`

### 3.3 关键技术结论

针对该单据的 `审批意见` 输入框，已验证：

- `Playwright locator.fill()` 对该组件**无效**
- `click + Ctrl+A + insert_text()` **有效**
- 直接设置 DOM `value` 并触发 `input/change` 事件 **有效**

因此本方案明确要求：

- 审批意见写入主路径使用**键盘输入法**
- DOM 事件写入作为兜底方案
- 不得把 `locator.fill()` 作为唯一实现

### 3.4 文案映射结论

本项目页面给用户展示的按钮文案可使用 `批准`，但 EHR 实际执行映射应固定为：

- 前端按钮：`批准`
- EHR 决策值：`同意`
- EHR 执行按钮：`提交`

也就是说，前端“批准”不是点击 EHR 上名为“批准”的按钮，而是执行：

1. `审批决策 = 同意`
2. 写入 `审批意见`
3. 点击 `提交`

### 3.5 待办列表兼容性结论

在同一次探测中，已确认：

- `待办任务` 列表本身可以正常打开
- 但旧实现中用于切换 `1000条/页` 的分页控件选择器已不稳定

因此本方案要求：

- 审批执行流进入待办列表时，先以 `#gridview` 可见和待办表头可读作为主判定
- 分页大小控件仅作为可选增强，不得作为必经阻塞点

## 4. 范围与边界

### 4.1 第一阶段范围

第一阶段仅支持：

- 单条单据手工触发
- 当前登录账号自己待处理的单据
- 单据级 `批准`
- 审批意见可手填
- 支持 dry-run 验证

### 4.2 第一阶段边界

以下能力暂不纳入 `105`：

- 批量审批
- 驳回原因模板
- 审批意见模板库
- 审批执行审计表
- 多浏览器并发审批队列
- 审批成功后自动重跑 `collect`

### 4.3 数据库边界

第一阶段建议：

- **不改 PostgreSQL 表结构**
- 审批执行留痕先写入 `automation/logs/approval_*.json`
- 失败时额外输出截图到 `automation/screenshots`

原因：

- 当前目标是先打通“页面审批可执行”
- 数据库审计表属于后续专项，不宜在第一阶段扩大范围

## 5. 前端设计

### 5.1 页签调整

当前 `ProcessDocumentsPage` 的第一页签文案为 `聚合反馈`，但从业务语义看，这一页实际就是“风险总览”。

按 `201` 最新落地口径，详情抽屉页签顺序调整为：

1. `单据概览`
2. `风险总览`
3. `审批单据`
4. `原始低分明细`
5. `申请角色`
6. `目标组织`
7. `审批记录`

说明：

- `单据概览` 用于展示基础事实字段，避免与风险摘要混放
- `风险总览` 继续使用现有 `104` 聚合反馈内容
- `审批单据` 插在其后，符合用户处理动作顺序

### 5.2 审批单据页签结构

`审批单据` 页签建议包含：

- 单据基础信息摘要
  - `申请理由`
  - `单据编号`
  - `权限对象`
  - `当前总结论`
  - `建议动作`

补充展示约束：

- `申请理由` 显示在 `单据编号`、`权限对象` 之前
- `申请理由` 作为长文本信息块，占据两列宽度；移动端仍按单列自然换行
- EHR 映射说明
  - `本次将执行：审批决策 = 同意，EHR 按钮 = 提交`
- 多行输入框 `审批意见`
- 主按钮 `批准`
- 次级按钮 `验证连通性（dry-run）`
- 执行结果区
  - `执行中`
  - `成功`
  - `失败原因`
  - `日志文件`

### 5.3 前端交互规则

- 未选中单据时：
  - `审批单据` 页签可见，但按钮禁用
- 正在执行时：
  - `批准` 按钮禁用
  - 输入框禁用
  - 展示执行进度
- 执行成功后：
  - 提示 `已提交至 EHR，请刷新处理单据列表确认状态`
- 执行失败后：
  - 展示后端返回的错误信息与日志文件名

## 7. 自动化执行流设计

### 7.1 新增流对象

建议新增文件：

- `automation/flows/document_approval_flow.py`

职责：

- 接收 `document_no`
- 复用现有登录态与打开单据能力
- 切换到 `任务处理`
- 写入审批意见
- 执行 `同意 -> 提交`
- 返回执行结果

### 7.2 主流程

建议流程如下：

1. 复用 `run.py` 现有配置与登录态恢复逻辑
2. 打开首页并确认已登录
3. 进入 `待办任务`
4. 按 `document_no` 打开目标单据
5. 校验 `单据编号` 与入参一致
6. 切换到 `任务处理`
7. 校验存在：
   - `审批决策`
   - `审批意见`
   - `提交`
8. 读取当前 `审批决策` 值与可选项
9. 将 `审批决策` 置为 `同意`
10. 将页面输入的 `approvalOpinion` 写入 EHR `审批意见`
11. 读回输入框值，确认与目标意见完全一致
12. 若 `dryRun = true`：
    - 记录成功
    - 不点击 `提交`
13. 若 `dryRun = false`：
    - 点击 `提交`
14. 执行后做结果校验
15. 输出日志与截图

### 7.3 审批意见写入策略

写入顺序固定为：

1. 聚焦审批意见框
2. `Ctrl+A`
3. `insert_text(approvalOpinion)`
4. 读取输入框值校验
5. 若校验失败，再走 JS 事件兜底：
   - `value = approvalOpinion`
   - 触发 `input`
   - 触发 `change`
6. 再次读回校验

若两种方式都无法写入成功：

- 直接失败
- 禁止继续点击 `提交`

### 7.4 审批决策选择策略

建议逻辑：

- 若当前值已经是 `同意`：
  - 不强制再次切换
- 若当前值不是 `同意`：
  - 打开下拉
  - 选择 `同意`
  - 读回确认

若下拉中不存在 `同意`：

- 直接失败
- 返回 `当前页面不支持批准动作`

### 7.5 提交结果判定

点击 `提交` 后，满足任一条件即可判定成功：

1. 当前单据页不再存在 `任务处理` 可编辑区
2. 页面弹出成功类提示
3. 返回待办列表后该单据不再位于当前账号待办中
4. 重新打开单据时 `审批记录` 已追加本次审批意见

第一阶段推荐主判定顺序：

1. 成功提示 / 页面状态变化
2. 回读 `审批记录`
3. 待办列表中单据消失

## 8. 风险控制与失败处理

### 8.1 前置校验

执行前必须校验：

- 当前单据详情页 `单据编号` 与请求值一致
- `任务处理` 页签存在
- `审批决策` 字段存在
- `审批意见` 字段存在
- `提交` 按钮存在且可点击

任一条件不满足时：

- 不执行提交
- 返回明确错误

### 8.2 防误操作

第一阶段建议新增两层保护：

1. 前端显示明确提示：
   - `本操作会真实写回 EHR`
2. 后端只允许：
   - `action = approve`
   - `document_no` 单条执行

### 8.3 失败留痕

失败时必须输出：

- 执行日志 JSON
- 当前页截图
- 失败阶段
- 控件读回信息
- 当前 URL

日志文件建议命名：

- `automation/logs/approval_YYYYMMDD_HHMMSS_<document_no>.json`

## 11. 最终结论

基于 `2026-03-15` 对单据 `RA-20260310-00019845` 的页面探测结果，本方案结论如下：

1. `处理单据工作台 -> 右侧详情抽屉 -> 审批单据 -> EHR 任务处理` 这条链路**可以打通**
2. EHR 中“批准”业务语义实际对应 `审批决策 = 同意 + 点击提交`
3. 审批意见框可写，但必须采用**键盘输入或 JS 事件兜底**，不能只用 `fill()`
4. 第一阶段无需改库，可先以前后端接口 + Playwright 执行流 + 本地日志留痕方式落地

因此，`105` 可以进入实现阶段，且建议按“`dry-run` 先行、`批准` 后开”的顺序推进。

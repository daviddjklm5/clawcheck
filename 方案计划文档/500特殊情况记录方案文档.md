# 方案编号500：特殊情况记录方案文档

## 1. 目标

本方案用于统一记录本项目内已确认、但不适合直接并入通用规则的各类特殊情况。

这里的 `特殊情况` 包括但不限于：

- 特定单据的业务例外
- 特定组织、角色、人员类别的例外口径
- 现有规则在个别场景下的解释例外
- 在正式通用化前，需先以白名单或案例方式沉淀的专项口径

本方案的定位是：

1. 为特殊情况提供统一登记入口
2. 为后续规则实现、展示说明、人工复核提供依据
3. 避免把一次性、局部性、未完成泛化建模的例外直接散落到多个方案文档中

## 2. 使用边界

### 2.1 适用场景

本方案适用于以下情况：

1. 已有业务确认结论
2. 现有通用规则直接套用会产生误判、误解或不完整表达
3. 短期内尚不适合立刻改造成完整基础模型或通用规则
4. 需要先按“专项例外”方式沉淀，并给后续实现提供执行依据

### 2.2 不适用场景

以下内容不应直接写入本方案主文：

1. 长期稳定且已完全通用化的正式规则
2. 仅是实现细节、但不构成业务例外的技术问题
3. 尚未确认、仅凭猜测的业务判断
4. 需要完整独立设计的大型专项需求

若某类特殊情况后续已沉淀为完整正式规则，应从 `500` 迁出到对应正式方案，`500` 仅保留索引或历史说明。

## 3. 记录原则

### 3.1 先确认，再入文档

写入 `500` 的特殊情况，必须至少具备以下信息：

1. 具体样例或适用对象
2. 当前业务确认结论
3. 例外点相对于通用规则的差异
4. 当前生效边界

### 3.2 只记录规则和边界，不堆临时聊天结论

每条特殊情况应尽量回答清楚：

1. 为什么它是特殊情况
2. 与哪个通用规则或通用理解发生冲突
3. 当前应如何处理
4. 暂时不处理到什么程度

### 3.3 保守生效

`500` 中的特殊情况默认采用保守生效原则：

1. 仅对已确认样例或已确认对象生效
2. 未经补充确认，不得擅自向同类名称、同类组织、同类角色自动扩展
3. 如需扩大适用范围，应先补文档，再改实现

## 6. 特殊情况明细

### 6.1 `500-001` 矩阵组织导致的跨组织单位申请表象例外

#### 6.1.1 背景

当前 `103申请表审批风险与信任度评估实现方案文档` 中，`申请的组织` 维度对以下场景按跨组织单位规则处理：

- `申请人所属组织单位 != 所申请组织所属组织单位`

该规则对绝大多数普通组织有效，但在矩阵组织场景下，可能出现“页面显示跨组织单位、实际又属于申请人所在业务单元”的情况。

此时若仍直接按普通跨组织单位申请理解，存在误判风险。

#### 6.1.2 核心定义

本条特殊情况中的 `矩阵组织` 指：

- 某目标组织在管理归属上挂在一个中心或条线下
- 但在业务归属上同时明确属于另一个组织单位或业务单元
- 因此仅凭单一 `组织单位` 字段，无法完整表达其真实归属

#### 6.1.3 已确认样例

样例单据：

- 单据编号：`RA-20260409-00021095`
- 申请人所属组织单位：`丹田`
- 申请组织范围涉及组织单位：`财务与资金管理中心`
- 目标组织：`丹田财务管理部`
- 目标组织数量：`1`

业务确认结论：

- `丹田财务管理部` 同时属于 `财务与资金管理中心`，又属于 `丹田`
- 该场景属于矩阵组织的一种
- 因此不应按普通 `跨组织单位申请` 直接解释

建议记录文案：

```text
申请人属“丹田”，申请组织范围涉“财务与资金管理中心”：丹田财务管理部，共 1 个组织。
该组织经确认属于矩阵组织，同时归属“财务与资金管理中心”与“丹田”，不按普通跨组织单位申请直接解释。
```

#### 6.1.4 执行口径

对命中本条特殊情况的组织范围，执行以下原则：

1. `是否跨组织单位` 不能只看当前单值组织单位字段
2. 需优先看是否已有业务确认其为矩阵组织
3. 若已确认属于矩阵组织，则默认：
   - 不直接输出普通 `跨组织单位申请` 风险结论
   - 不直接使用普通跨组织单位摘要文案
   - 应改为输出“矩阵组织特殊情况”说明，或至少在人工复核时标记为例外

#### 6.1.5 当前边界

在底层尚未形成稳定自动识别能力前，本条特殊情况采用保守口径：

1. 仅对 `已在方案文档中确认` 的组织或样例生效
2. 未经业务确认的新组织，不得因为名称相似就自动套用矩阵组织例外
3. 如后续要扩充样例清单，应先补充本方案或其子文档，再改实现

| --- | --- | --- | --- |
| `丹田` | `财务与资金管理中心` | `丹田财务管理部` | 同时归属 `财务与资金管理中心` 与 `丹田` |

### 6.2 `500-002` 审批记录同名人员导致工号无法唯一回填例外

#### 6.2.1 背景

权限申请单采集审批记录时，审批人工号当前按以下顺序补齐：

1. 若页面审批人文本为 `姓名(工号)` 或 `姓名（工号）`，直接取括号内工号
2. 若页面审批人文本不带工号，则按 `审批人 = 在职花名册表.姓名` 精确匹配
3. 只有当姓名在花名册中唯一对应一个 `人员编号` 时，才回填到 `申请单审批记录.工号`
4. 若姓名未命中或命中多个不同 `人员编号`，则 `申请单审批记录.工号` 保持为空

该规则是为了避免同名人员被错误回填。

#### 6.2.2 已确认样例

处理单据审批记录中存在以下页面文本形态：

```text
审批 冉艳丽|人事运营 同意
```

该文本只包含：

- 审批人姓名：`冉艳丽`
- 审批人岗位或职位：`人事运营`

页面文本未展示括号工号，因此不能通过第一优先级直接提取工号。

当前 `在职花名册表` 中按姓名 `冉艳丽` 精确匹配到两个不同人员编号：

| 姓名 | 人员编号 | 部门 | 职位名称 |
| --- | --- | --- | --- |
| `冉艳丽` | `01029659` | `上海徐汇人事交付服务组` | `人事运营` |
| `冉艳丽` | `05281553` | `Q深圳燕罗街道项目组` | `保洁` |

由于同名人员存在两个不同 `人员编号`，现有唯一匹配 SQL 不返回结果，因此新增或重采时会出现 `审批人 = 冉艳丽` 但 `工号` 为空的情况。

#### 6.2.3 原因结论

`冉艳丽` 工号带不出来的直接原因不是审批记录解析失败，而是审批人工号补齐链路主动拒绝非唯一姓名匹配：

1. 页面审批记录没有展示 `冉艳丽(01029659)` 这类括号工号
2. 页面只展示 `冉艳丽|人事运营`
3. 花名册中存在两个姓名均为 `冉艳丽` 的人员
4. 现有代码要求 `COUNT(DISTINCT 人员编号) = 1` 才允许按姓名回填
5. 因此该姓名在当前花名册下无法自动唯一映射，工号保持为空

历史上部分 `冉艳丽` 审批记录已带出 `01029659`，不代表当前规则已能稳定唯一识别。更可能是当时花名册快照、采集时点或页面数据条件下未形成当前同名冲突；当前以最新花名册唯一匹配结果为准。

#### 6.2.4 执行口径

本条特殊情况确认以下人工口径：

1. 当审批记录文本为 `冉艳丽|人事运营` 时，业务上对应人员为 `01029659 冉艳丽`
2. 该人员当前花名册信息为：
   - 部门：`上海徐汇人事交付服务组`
   - 职位名称：`人事运营`
3. 同名的 `05281553 冉艳丽` 当前职位为 `保洁`，不应作为该审批记录的审批人工号

#### 6.2.5 当前边界

本条特殊情况当前仅作为文档特例记录，不代表已完成代码自动回填。

后续若要工程化落地，建议采用保守白名单方式：

1. 仅当 `审批人 = 冉艳丽`
2. 且 `审批人组织或职位 = 人事运营`
3. 且页面未直接提供括号工号
4. 才允许回填 `工号 = 01029659`

不得仅凭 `审批人 = 冉艳丽` 无条件回填，避免误伤同名人员。

### 6.3 `500-003` 远程共享未接入时属地同事可开通权限例外

#### 6.3.1 背景

在部分权限申请场景中，系统或人工判断可能会倾向于认为：

1. 相关账户应由远程共享统一承接
2. 非远程共享人员不应申请该类权限

但该判断并不适用于所有属地实际执行场景。

当某个具体账户尚未接入远程共享、且实际业务由属地团队承担时，仅因申请人“不属于远程共享”而直接判定为不应开通，会与实际业务执行口径冲突。

#### 6.3.2 已确认样例

样例单据：

- 单据编号：`RA-20260415-00021351`

业务确认事实：

1. 申请人不是远程共享人员
2. 申请人所属为 `祥盈企服` 的 `广州公共服务中心`
3. 涉及账户为：`深圳万科祥盈管理服务有限公司广州分公司` 的 `社保公积金账户`
4. 该账户当前由属地操作
5. 经确认该账户远程共享尚未接入
6. 因此可将相应权限开给属地同事，以便开展日常工作

#### 6.3.3 原因结论

本条特殊情况的核心不是“申请人属于远程共享”，而是：

1. 目标账户当前尚未纳入远程共享处理范围
2. 实际业务仍由属地团队承接
3. 因而属地同事申请对应权限具备明确业务必要性

因此，对该单据不能仅凭“申请人不是远程共享”就直接否定权限合理性。

#### 6.3.4 执行口径

对本条特殊情况，当前确认以下执行口径：

1. 若申请对象为 `深圳万科祥盈管理服务有限公司广州分公司社保公积金账户`
2. 且该账户经确认 `远程未接入`
3. 且实际由 `祥盈企服广州公共服务中心` 属地操作
4. 则属地同事申请相应权限可视为合理业务申请

建议记录文案：

```text
申请人不属于远程共享，而属于祥盈企服广州公共服务中心。因深圳万科祥盈管理服务有限公司广州分公司社保公积金账户当前由属地操作，且经确认该账户远程未接入，故可开通相应权限给属地同事，以便开展日常工作。
```

#### 6.3.5 当前边界

本条特殊情况当前仅对已确认样例生效，不自动泛化到全部“社保公积金”或全部“属地操作”场景。

执行边界如下：

1. 仅适用于已确认 `远程未接入` 的具体账户
2. 仅适用于已确认实际由属地团队承接的场景
3. 不得仅凭申请人属于 `祥盈企服` 或 `广州公共服务中心` 就默认全部同类权限均可开通
4. 若后续远程共享已正式接入该账户，则本条例外应重新复核，不当然长期生效

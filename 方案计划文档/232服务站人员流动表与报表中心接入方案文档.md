# 方案编号232：服务站人员流动表与报表中心接入方案文档
## 1. 目标
本方案用于新增正式报表 `服务站人员流动表`，并同步定义其在 `报表中心` 中的模块归属、参数口径、取数规则、导出行为、默认下载路径、另存为路径规则与实施落点。

本方案落地后，需要支持：

1. 前端新增一级页面 `报表中心`
2. `报表中心` 下新增模块 `服务站分析`
3. `服务站分析` 模块下第一个报表为 `服务站人员流动表`
4. 报表支持填写：
   - `开始日期`
   - `结束日期`
   - `另存为路径`
5. 报表支持页面查看与 `Excel` 导出
6. 报表能输出：
   - 服务站目标岗位人员期初到期末的去向
   - 期末目标岗位人员的来源
   - 各战区、各目标子域的人数变化
   - 离职明细
   - 转其他 HR 岗位明细

## 2. 业务范围
### 2.1 报表对象
目标岗位按 `人员属性查询历史.HR子域` 定义，仅包含：

- `服务站-人事运营`
- `服务站-招聘`

统计范围例外：

- 报表层排除 `部门ID = 55005710`
- 对应组织为 `北京航空人事交付服务组`
- 该排除仅作用于 `服务站人员流动表` 的查询与导出口径
- 不回写、不修改 `人员属性查询` / `人员属性查询历史` 中既有 `HR子域`
- 不改变 `010申请单申请人HR与疑似HR识别方案文档.md` 中 `人事交付服务组 -> 属地服务站 -> 服务站-人事运营/服务站-招聘` 的底层识别规则

### 2.2 时间口径
本报表是“两期历史快照之间”的点对点流动分析，不是按月连续轨迹表。

即：

- `开始日期` 代表期初快照
- `结束日期` 代表期末快照
- 报表仅比较这两个日期在 `人员属性查询历史` 中的人员状态差异
- 不对中间未采集日期做插值、推断或连续轨迹拼接

### 2.3 人员主键
人员唯一身份统一按 `工号` 判定。

### 2.4 离职口径
若某人在 `开始日期` 命中目标岗位，但在 `结束日期` 的 `人员属性查询历史` 中按 `工号` 找不到，则判定为：

- `离职`

本方案不再额外区分“调出系统外”“花名册漏数”等子状态，统一按业务确认口径执行。

## 3. 关键设计原则
### 3.1 历史战区必须使用历史快照，不允许用当前组织口径反挂
当前 `人员属性查询历史` 已保存历史 `组织单位`，但尚未保存历史 `所属战区`。

如果直接用当前 `组织属性查询` 按 `部门ID -> 行政组织编码` 回挂历史战区，则：

- `开始日期` 的战区会受当前组织口径影响
- 一旦组织归属或战区映射变化，历史战区会被“回写成当前值”
- 报表中的“各战区流动”会失真

因此本方案明确要求：

1. `服务站人员流动表` 正式实施前，必须先把 `所属战区` 纳入历史快照口径
2. 报表起止两端的战区，均应取自各自日期的历史快照值
3. 不接受“期初用当前组织表反查战区”的临时正式实现

### 3.2 组织单位以历史快照为准
`人员属性查询历史` 已包含 `组织单位`，报表中的期初/期末组织单位统一直接使用历史快照字段，不再反查当前组织表覆盖。

### 3.3 报表默认服务于“专题分析 + 可导出复核”
本报表不是单一数字卡片，而是完整的复核报表。第一版必须同时提供：

- 页面摘要
- 页面明细
- Excel 导出
- 默认导出路径
- 用户自定义另存为路径

### 3.4 统计范围例外优先在报表层收口
若业务仅要求“某个组织不纳入 `服务站人员流动表` 统计范围”，默认应优先在报表层增加排除条件，不直接修改底层 `HR子域` 识别规则。

原因：

1. `HR子域` 是 `人员属性查询` / `人员属性查询历史` 的基础派生字段
2. 直接改底层规则会影响其他依赖 `服务站-人事运营` / `服务站-招聘` 或 `属地服务站` 的模块与方案文档
3. 本次 `55005710` 需求经确认仅作用于 `报表中心 -> 服务站人员流动表`

## 4. 数据模型调整
### 4.1 必要字段补充

- 本方案新增 `所属战区` 后，实施时需同步更新 `014人员属性查询历史与花名册查询日期方案文档.md`

### 4.2 字段来源
`所属战区` 的生成口径：

- `人员属性查询.部门ID -> 组织属性查询.行政组织编码`
- 取 `组织属性查询.所属战区`

刷新链路要求：

1. 在职花名册导入完成
2. 刷新 `人员属性查询` 时，同时补齐 `所属战区`
3. 写入 `人员属性查询历史` 时，把当次 `所属战区` 一并快照落库

### 4.3 不新增独立流动结果表
第一版不新增新的持久化结果表，如：

- `服务站人员流动结果表`

统一按“实时读取两期历史快照并计算”的方式生成报表结果。

原因：

1. 流动结果天然依赖 `开始日期 + 结束日期` 参数
2. 预先固化所有日期组合收益低、维护成本高
3. 当前业务更需要“可核对、可导出”的专题报表，而不是额外派生持久层

## 6. 参数规则
### 6.1 参数定义
- `开始日期`
  - 类型：`date`
  - 必填
- `结束日期`
  - 类型：`date`
  - 必填
- `另存为路径`
  - 类型：`string`
  - 选填
  - 表示报表导出的目标落盘路径
  - 为当前应用所在机器的本地文件系统路径，不是浏览器上传控件

### 6.2 默认值
为降低首次使用门槛，第一版建议默认：

- `结束日期`：`人员属性查询历史` 中最新 `生效日期`
- `开始日期`：小于 `结束日期` 的最近一个可用 `生效日期`
- `另存为路径`：默认留空，表示走系统默认导出目录

若历史表中不足两个可用日期：

- 页面提示“可用历史快照不足，无法生成流动报表”
- 不允许执行查询

### 6.3 校验规则
执行查询前必须校验：

1. `开始日期`、`结束日期` 都存在于 `人员属性查询历史.生效日期`
2. `开始日期 < 结束日期`
3. 两端快照数据都存在
4. `另存为路径` 若填写，必须可解析为合法本地路径
5. `另存为路径` 若填写到文件级，后缀必须为 `.xlsx`

不满足时直接提示，不进入取数执行。

### 6.4 导出路径规则
本方案明确区分：

1. 默认导出路径
2. 用户指定另存为路径

#### 6.4.1 默认导出路径
当用户未填写 `另存为路径` 时，报表默认导出到：

- `settings.runtime.logs_dir/report_exports/`

对应当前仓库默认建议绝对路径：

- `C:\Users\Administrator\source\repos\clawcheck\automation\logs\report_exports\`

默认文件名规则：

- `服务站人员流动表_{开始日期}_to_{结束日期}_{时间戳}.xlsx`

说明：

- 该口径与当前仓库其他导出脚本默认写入 `runtime.logs_dir` 的约定一致
- 默认目录固定落在 `runtime.logs_dir` 下的 `report_exports` 子目录
- 若默认目录不存在，后端应自动创建

#### 6.4.2 另存为路径
页面新增输入项：

- `另存为路径`

填写规则：

1. 支持绝对路径，例如：
   - `D:\报表\服务站人员流动表_2025-12-31_to_2026-03-30.xlsx`
2. 支持仓库相对路径，例如：
   - `automation\output\服务站流动\服务站人员流动表.xlsx`
3. 若填写的是目录路径而不是文件路径，则系统在该目录下按默认文件名自动生成文件
4. 若填写的是 `.xlsx` 文件路径，则系统直接按该文件名落盘
5. 若父目录不存在，后端应自动创建父目录
6. 若目标文件已存在，第一版默认直接覆盖，不额外弹出二次确认
7. 最终实际落盘路径必须在接口返回值和页面结果区明确展示

#### 6.4.3 页面交互要求
导出成功后，页面必须展示：

- `导出文件路径`

并至少提供：

1. `复制路径`
2. `打开所在目录`

若当前页面运行环境不便直接调起系统“文件另存为”原生弹窗，则以 `另存为路径` 输入框作为第一版正式能力，不把浏览器原生下载弹窗作为契约前提。

## 7. 计算规则
### 7.1 起点样本
起点样本定义为：

- `人员属性查询历史.生效日期 = 开始日期`
- `HR子域 in ('服务站-人事运营', '服务站-招聘')`
- 排除 `部门ID = 55005710`

### 7.2 终点样本
终点样本定义为：

- `人员属性查询历史.生效日期 = 结束日期`
- `HR子域 in ('服务站-人事运营', '服务站-招聘')`
- 排除 `部门ID = 55005710`

### 7.3 去向分类
对每个“起点样本人员”按 `工号` 关联期末快照，分类规则如下：

1. `结束日期` 找不到该 `工号`
   - `离职`
2. `结束日期` 仍在目标岗位，且：
   - 同 `HR子域`、同 `所属战区`
     - `留任原子域原战区`
   - 同 `HR子域`、不同 `所属战区`
     - `留任原子域跨战区`
   - 不同 `HR子域`、同 `所属战区`
     - `目标岗位内转子域(同战区)`
   - 不同 `HR子域`、不同 `所属战区`
     - `目标岗位内转子域并跨战区`
3. `结束日期` 不在目标岗位，但 `申请人HR类型` 仍属于 `H*`
   - `转其他HR岗位`
4. `结束日期` 不在目标岗位，且 `申请人HR类型` 不属于 `H*`
   - `转非HR岗位`

### 7.4 来源分类
对每个“终点样本人员”按 `工号` 关联期初快照，分类规则如下：

1. 期初就在目标岗位，且同 `HR子域`、同 `所属战区`
   - `来自原子域原战区`
2. 期初就在目标岗位，且同 `HR子域`、不同 `所属战区`
   - `来自原子域跨战区`
3. 期初就在目标岗位，但来自另一目标子域
   - `来自另一目标子域`
4. 期初不在目标岗位，但 `申请人HR类型` 属于 `H*`
   - `来自其他HR岗位`
5. 期初不在目标岗位，且 `申请人HR类型` 不属于 `H*`
   - `来自非HR岗位`
6. 期初按 `工号` 找不到
   - `期初不在目标岗位`

### 7.5 战区变化统计
战区汇总需分别统计：

1. `服务站-人事运营` 期初人数、期末人数、增减
2. `服务站-招聘` 期初人数、期末人数、增减
3. 合计期初人数、期末人数、增减
4. 该战区内：
   - 离职人数
   - 转其他 HR 人数
   - 其他 HR 转入人数
   - `服务站-人事运营 -> 服务站-招聘` 人数
   - `服务站-招聘 -> 服务站-人事运营` 人数

# 402 对话页面重复内容专项解决方案

## 1. 问题现象

在 `http://127.0.0.1:5173/chat-workspace` 页面中，同一条 Assistant 回复会被重复追加多次，表现为大段文本连续重复（如“我现在直接查仓库...”循环出现）。

## 2. 影响范围

- Web 对话页 `chat-workspace`
- SSE 流式消息消费链路（前端 `EventSource` + 后端 `/api/chat/sessions/{id}/stream`）
- 用户体验：回复可读性下降、消息体积膨胀、滚动与渲染压力上升

## 3. 根因分析

### 3.1 前端去重缺失（主因）

- 文件：`webui/src/pages/ChatWorkspacePage.tsx`
- 现状：收到 SSE 事件后直接按 `token` 事件拼接文本，但未在消费前判断 `payload.seq` 是否已处理。
- 结果：当连接重连后服务端重放历史事件时，旧事件被二次消费，导致重复拼接。

### 3.2 SSE 断线续传位点未使用（主因）

- 文件：`automation/api/routers/chat.py`
- 现状：接口只读取 `afterSeq` 查询参数，未处理浏览器自动重连时携带的 `Last-Event-ID`。
- 结果：浏览器重连后即使已收到较新事件，服务端仍可能从较早位点重新推送历史事件。

### 3.3 服务端空闲断流触发重连（放大器）

- 文件：`automation/chat/service.py`
- 现状：会话空闲超过 30 秒后 `stream_events` 返回，浏览器自动重连。
- 结果：在 3.1/3.2 存在时，空闲断流会持续触发“历史事件重放 + 重复消费”。

## 4. 已实施修复（热修复）

### 4.1 前端按 seq 幂等消费

- 文件：`webui/src/pages/ChatWorkspacePage.tsx`
- 改动：新增 `payload.seq <= lastSeqRef.current` 直接忽略逻辑，仅消费严格递增事件。
- 目的：即使收到重放事件，也不会再次拼接。

### 4.2 后端支持 Last-Event-ID 续传

- 文件：`automation/api/routers/chat.py`
- 改动：新增请求头 `Last-Event-ID` 解析，`resume_after_seq = max(afterSeq, Last-Event-ID)`。
- 目的：浏览器自动重连后按最新位点续传，避免从旧位点重放。

### 4.3 测试补充

- 文件：`tests/test_chat_router.py`
- 新增断言：
  - 默认 `stream` 调用 after_seq 为 0；
  - 带 `Last-Event-ID` 时，after_seq 正确使用该位点。

## 5. 验收结果

- 本地页面联调：可稳定返回“页面联调成功”，未再出现同段重复追加。
- 自动化测试：`154 passed`（全量通过）。

## 6. 发布与回滚建议

### 6.1 发布步骤

1. 部署后端（含 `Last-Event-ID` 续传逻辑）。
2. 发布前端（含 `seq` 幂等消费逻辑）。
3. 在 `chat-workspace` 连续执行 3 轮“发送 -> 等待完成 -> 空闲 30 秒后再次发送”验证无重复。

### 6.2 回滚策略

- 如出现异常，可先回滚前端至上一个稳定版本（最小化用户可见影响）。
- 后端续传逻辑可独立保留，不影响现有会话写库结构。

## 7. 后续增强（建议）

1. 在前端增加 `done` 后会话详情兜底拉取，进一步消除 token 丢失/乱序风险。
2. 在后端 SSE 增加心跳帧，降低空闲断流重连频率。
3. 增加“重复事件计数”监控指标，用于持续观测异常重放。

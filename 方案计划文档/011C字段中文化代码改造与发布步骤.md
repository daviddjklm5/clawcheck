# 方案编号011C：字段中文化代码改造与发布步骤

## 1. 目标

本清单用于承接 `011A` 与 `011B/011D`，把“数据库物理改列”落成可执行发布步骤。

本清单不再讨论是否改列，只回答 3 个问题：

1. 哪些代码必须一起改
2. 正确发布顺序是什么
3. 哪些步骤失败时必须中止并回滚

## 2. 当前结论

当前仓库**还不能直接执行** `012_rename_columns_to_cn_fixed_schema.sql` 后就发布应用，原因是代码仍大量硬编码英文列名。

最关键的硬编码入口包括：

- `automation/db/postgres.py`
- `automation/utils/roster_excel.py`
- `automation/utils/organization_list_excel.py`
- `automation/sql/006_organization_attribute_query.sql`
- `automation/sql/001_permission_apply_collect.sql`
- `automation/sql/002_active_roster.sql`
- `automation/sql/003_organization_list.sql`
- `automation/sql/009_permission_catalog.sql`

因此正确顺序不是：

- 先改库
- 再慢慢修代码

而是：

- 先完成代码切换分支
- 再在停机窗口切库
- 再发布新代码

## 4. 推荐发布顺序

### 4.1 发布前准备

1. 冻结字段设计变更
2. 冻结与这 9 张表相关的功能开发
3. 在目标库执行实际字段盘点
4. 在目标库执行依赖对象盘点
5. 在预发或影子库演练：
   - 前向 SQL
   - 应用发布
   - 全链路回归
   - 回滚 SQL

### 4.2 应用改造阶段

1. 基于 `011A` 完成代码分支改造
2. 统一把仓库内固定字段切到中文列名
3. 保证新代码只依赖中文列名，不再混用英文列名
4. 在测试库先建中文列版本并跑回归

### 4.3 停机窗口正式切换

正确顺序：

1. 停应用写入
2. 备份数据库
3. 执行 `automation/sql/012_rename_columns_to_cn_fixed_schema.sql`
4. 发布新代码
5. 运行最小回归链路
6. 运行全量刷新链路
7. 执行验收 SQL

不建议的顺序：

1. 先发布新代码再改库
2. 先改库并继续跑旧代码

这两种都会导致线上即时报错。

## 5. 验收顺序

建议按依赖顺序验收：

1. `权限列表`
2. `申请单基本信息` 系列 4 张表
3. `在职花名册表`
4. `组织列表`
5. `城市所属战区`
6. `组织属性查询`

原因：

- `组织属性查询` 依赖 `组织列表` + `城市所属战区`
- `权限列表` 相对独立，适合作为先行校验项

## 6. 必跑链路

发布后至少执行：

1. 权限目录初始化
2. 权限申请采集写库
3. 在职花名册导入
4. 组织列表导入
5. `SELECT refresh_组织属性查询();`

若库内已部署依赖 `组织属性查询` 的视图（例如低分反馈相关视图），在执行第 4、5 步前应先执行：

6. `automation/sql/026_fix_organization_attribute_query_refresh_dependency.sql`

## 7. 中止条件

出现以下任一情况，必须停止继续发布并评估回滚：

1. 任一 `INSERT`/`DELETE`/`SELECT` 报列不存在
2. `refresh_组织属性查询()` 执行失败
3. `组织列表` 导入失败
4. `在职花名册表` 主键冲突异常明显增加
5. 任一核心表行数与预期差异异常

若失败信息包含以下特征：

- `cannot drop table "组织属性查询__old" because other objects depend on it`
- 依赖对象指向 `申请单低分明细富化视图` 或 `申请单低分反馈预聚合视图`

则判定为“组织属性查询刷新换表策略与依赖视图冲突”，处理顺序应为：

1. 执行 `automation/sql/026_fix_organization_attribute_query_refresh_dependency.sql`
2. 重新执行 `SELECT refresh_组织属性查询();`
3. 再重跑 `组织列表` 导入任务

## 8. 回滚顺序

若需回滚，推荐顺序：

1. 停应用写入
2. 执行 `automation/sql/013_rename_columns_to_en_fixed_schema_rollback.sql`
3. 回退应用代码到英文列名版本
4. 重新执行最小回归
5. 再决定是否恢复业务写入

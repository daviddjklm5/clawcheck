# 方案编号203：新采集单据增量评估与处理工作台最新结果口径方案文档

## 1. 方案目标

在现有 `103 / 201 / 202` 基础上，补齐“采集 -> 评估 -> 处理工作台”之间的闭环，解决新采集单据无法及时进入 `处理单据` 页的问题。

本方案明确交付以下结果：

- `audit` 支持按单张单据、按一组单据编号执行增量评估
- `采集单据` 成功落库后，可按任务参数自动衔接增量 `audit`
- `处理单据` 工作台改为“按单据取最新评估结果”口径，不再只看“最新评估批次”
- `评估分析` 页面继续保留批次视角，用于查看某一批次的分布、日志与结果
- 不新增业务表、不改评分规则，只调整任务编排、查询口径与 API/UI 契约

## 2. 背景与问题

当前链路存在两个核心断点：

1. `采集单据` 与 `audit` 是两条独立链路  
   新单据即使已经写入：
   - `申请单基本信息`
   - `申请单权限列表`
   - `申请单审批记录`
   - `申请表组织范围`

   只要未执行 `audit`，就不会出现在 `申请单风险信任评估`，因此 `处理单据` 页无法检索到。

2. `处理单据` 当前按“最新评估批次”取数  
   这会带来两个问题：
   - 新采集单据如果在最新批次之后才落库，必须重跑整批 `audit` 才能进入处理页
   - 若只对一张单据跑一次 `audit --document-no`，会生成新的小批次，并把处理页主列表切成只剩这一个小批次中的单据

因此，需要把“工作台视角”和“批次分析视角”正式拆开：

- `处理单据` 看“每张单据当前最新的评估结果”
- `评估分析` 看“某一批次的整体情况”

## 3. 与现有方案关系

### 3.1 与 `103` 的关系

`103` 定义评分规则、结果表与 `audit` 基础动作。

`203` 不改 `103` 的评分逻辑和结果表结构，只补充：

- `audit` 的输入方式
- `audit` 与 `collect` 的编排关系
- 工作台查询口径

### 3.2 与 `201` 的关系

`201` 解决的是 `处理单据` 页的信息架构与交互方式。

`203` 补充的是 `201` 未明确的数据口径：

- `处理单据` 页不是“最新批次工作台”
- 而是“逐单最新结果工作台”

### 3.3 与 `202` 的关系

`202` 解决的是 `采集单据` 页的真实采集接入。

`203` 在此基础上补齐：

- `采集成功后如何进入评估`
- 采集页如何把增量评估衔接到处理页

## 4. 设计原则

### 4.1 工作台按单据，分析台按批次

- `处理单据` 页服务于操作员逐单处理，必须使用“单据当前最新评估结果”
- `评估分析` 页服务于批次复盘，继续保留“按评估批次查看”

两者不允许再共用“最新批次”这一口径。

### 4.2 增量评估复用现有引擎

增量 `audit` 仍复用：

- `PostgresRiskTrustStore.fetch_document_bundles()`
- `RiskTrustEvaluator`
- `write_assessment_results()`

不引入第二套评估逻辑。

### 4.3 历史批次保留，工作台只取最新快照

- `申请单风险信任评估` 与 `申请单风险信任评估明细` 继续保留批次历史
- `处理单据` 查询层通过“每张单据取最新一条评估结果”形成工作台快照
- 不新增“当前结果表”或英文影子表

### 4.4 无需新增 SQL 迁移

本方案优先通过：

- Python 查询层
- CLI 参数
- FastAPI 任务编排
- 前端页面交互

完成改造。

原则上：

- 不新增数据库字段
- 不新增 SQL 迁移
- 不新增新的中文业务表

## 5. 范围与边界

### 5.1 本次纳入范围

- `采集单据` 任务支持“采集成功后自动触发增量 audit”
- `处理单据` 列表、统计、详情默认按“逐单最新评估结果”查询
- FastAPI 新增正式 `audit` 启动入口与任务摘要输出
- 前端补齐采集后增量评估入口与处理页最新结果口径

### 5.2 本次不纳入范围

- 不改 `申请单风险信任评估*` 两张表结构

- 不实现长任务取消
- 不改 `评估分析` 页的批次分析职责

## 6. 目标口径

### 6.1 `audit` 输入口径

`audit` 统一支持以下输入方式：

1. 单张单据
   - `document_no`
2. 多张单据
   - `document_nos[]`
3. 批量扫描
   - `limit`

规则：

- 若传入 `document_nos[]`，则只评估这些单据
- 若仅传入 `document_no`，视为 `document_nos[]` 的单元素场景
- 若未传入单据编号，则按 `limit` 从采集主表读取候选单据
- 多张单据场景必须去重，且保留输入顺序

### 6.2 `采集 -> audit` 编排口径

`采集单据` 任务新增 `auto_audit` 语义：

- `dry_run = TRUE` 时，禁止自动触发 `audit`
- `dry_run = FALSE` 且 `success_count > 0` 时：
  - 若 `auto_audit = TRUE`，对本次成功采集的单据集合触发一次增量 `audit`
  - 若 `auto_audit = FALSE`，仅完成采集，不衔接评估

建议默认值：

- `auto_audit = TRUE`

原因：

- 用户默认预期是“采集完成后，这张单据应可进入处理页”

### 6.3 `处理单据` 工作台口径

`处理单据` 页面主列表、统计卡与默认详情，统一改为：

- 以 `单据编号` 为主键
- 从 `申请单风险信任评估` 中为每张单据取最新一条结果
- “最新”定义为：
  - `评估时间` 倒序
  - 同时以 `评估ID` 倒序作为并列打破条件

由此得到的结果集，称为“逐单最新评估结果快照”。

该快照用于：

- `处理单据` 主 Grid
- `处理单据` 顶部统计
- `GET /process-workbench/{document_no}` 默认详情

### 6.4 `评估分析` 页面口径

`评估分析` 页面继续保留批次视角：

- 默认查看最近落库批次
- 统计某一批次中的 `总结论 / 信任分 / 低分维度 / 规则分布`
- 最近执行日志继续按批次展示

说明：

- 增量 `audit` 会产生更小的评估批次，这是允许的
- 该变化不影响 `处理单据` 工作台，因为工作台已切换到逐单最新结果口径

## 7. 接口契约调整

### 7.1 `采集单据` 启动接口

`POST /api/documents/collect-workbench/run`

新增入参：

- `autoAudit`

返回结果需补充：

- 本次是否请求自动评估
- 若已触发自动评估，返回关联 `auditTaskId` 或关联摘要信息

### 7.2 `audit` 启动接口

新增正式接口：

- `POST /api/documents/process-workbench/audit`

建议入参：

- `documentNo`
- `documentNos`
- `limit`
- `dryRun`

建议返回：

- `taskId`
- `status`
- `requestedDocumentNos`
- `requestedLimit`
- `dryRun`
- `message`

### 7.3 `处理单据` 查询接口

以下接口需切换为“逐单最新评估结果”口径：

- `GET /api/documents/process-workbench`
- `GET /api/documents/process-workbench/{document_no}`

兼容规则：

- 若详情接口显式传入 `assessment_batch_no`，仍优先返回该批次结果
- 若未传入，则返回该单据的最新结果

## 8. 后端实现要求

### 8.1 `run.py audit`

需要补齐：

- 单张与多张单据的统一参数解析
- 多张单据场景下的去重与顺序保留
- 输出 `audit_*.json`
- 非 `dry-run` 时写回两张评估表

### 8.2 `PostgresRiskTrustStore`

需要补齐两类能力：

1. `fetch_document_bundles()` 支持按一组 `document_nos`
2. `fetch_process_workbench()` / `fetch_process_document_detail()` 支持按“逐单最新评估结果”查询

推荐实现方式：

- 在查询层使用 `ROW_NUMBER() OVER (PARTITION BY 单据编号 ORDER BY 评估时间 DESC, 评估ID DESC)` 或等价写法
- 再筛选 `row_number = 1`

### 8.3 任务编排

建议新增独立的 `audit` 任务封装，与现有 `collect` 任务保持一致的摘要输出方式。

最低要求：

- 可单独启动 `audit`
- 可被 `collect` 成功后串联触发
- 有状态、日志文件、摘要文件

## 9. 前端实现要求

### 9.1 `采集单据`

页面需补充：

- `自动评估` 开关，默认开启
- 若本次采集已串联评估，明确展示关联评估任务状态
- 单据详情中允许手动发起“仅评估当前单据”

### 9.2 `处理单据`

页面需遵循以下口径：

- 搜索范围是“逐单最新评估结果快照”
- 不能再因为某次单张 `audit` 而丢失其他已评估单据
- 打开详情时，默认读取该单据最新结果

## 10. 验收标准

至少满足以下验收项：

1. 对新采集单据执行 `collect` 且 `auto_audit = TRUE` 后，该单据可进入 `处理单据` 页
2. 对单张单据执行增量 `audit` 后，`处理单据` 页原有其他单据不消失
3. `处理单据` 详情默认显示该单据最新评估结果
4. `评估分析` 页面仍可查看批次级分布与执行日志
5. 全链路不新增数据库字段、不新增新业务表

## 12. `105B` 后补口径

`203` 明确的是：

- `处理单据` 取“逐单最新评估结果”

`105B` 在此基础上补充一层新的工作台状态维度：

- `待办处理状态`

因此当前正式口径应表述为：

- `处理单据` 的基础数据源仍是“逐单最新评估结果”
- 但页面展示与审批动作可再叠加“当前账号最近一次待办同步结果”

补充规则：

1. `collect` 成功时，单据回写 `待办处理状态 = 待处理`
2. `同步待办状态` 可把已不在当前账号 EHR 待办中的单据更新为 `已处理`
3. 单据若后续重新进入待办，可再切回 `待处理`
4. 该状态不改变 `203` 已定义的“逐单最新评估结果”主口径，只补充工作台可操作性

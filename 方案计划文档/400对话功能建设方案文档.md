# 方案编号400：对话功能建设方案文档（Web优先，IDE可选扩展）

## 1. 方案目标

在现有 `clawcheck` 项目中建设“类 Codex 对话能力”，但执行策略明确为：

- 第一优先：在现有 `webui` 内落地对话功能（主入口）
- 第二优先：后续按需提供 IDE 扩展入口（可选，不阻塞主线）

本方案交付目标是“先形成可运营、可审计、可回滚”的对话闭环，再评估 IDE 投入收益。

## 2. 关键结论

### 2.1 是否必须 IDE 形态

结论：不必须。

原因：

- 本项目当前主用户场景是“任务运行、单据处理、结果查询”，并非“边写代码边对话”
- 仓库已具备 `FastAPI + React/Vite`，Web 形态落地成本显著低于 IDE 扩展
- IDE 形态会额外引入扩展打包、版本兼容、企业终端安装策略等维护成本

### 2.2 推荐路线

采用“后端能力统一、前端入口分层”的路线：

1. 先落地 WebUI 对话页（MVP）
2. 后端接口设计保持与 IDE 场景兼容
3. 后续若研发场景明确，再新增薄 IDE 扩展复用同一套 API

## 3. 背景与约束

### 3.1 背景

- 项目已存在稳定 API 层：`automation/api`
- 项目已存在稳定前端层：`webui`
- 已具备任务执行、日志、配置摘要等基础能力

### 3.2 硬约束

- 对话执行器采用开源 `codex cli`
- 云端模型服务走外部 API Key 配置
- 密钥不入库，不在文档写明文密钥
- 新增数据库对象遵循中文表名/字段名规范

## 4. 方案范围

### 4.1 本次纳入范围（MVP）

- WebUI 新增对话页面与会话管理
- FastAPI 新增对话网关 API
- 新增 `codex cli` 适配层（启动、取消、超时、输出规整）
- 新增模型服务配置装载与脱敏摘要
- 新增最小落库：会话、消息、执行日志

### 4.2 本次不纳入范围

- IDE 扩展正式发布
- 多租户权限体系
- 复杂审批动作自动执行
- 向量检索与知识库体系

## 5. 总体架构

采用三层主架构 + 一层可选入口：

1. WebUI 层（主入口，React）
2. API 网关层（FastAPI）
3. 执行与模型层（Codex CLI + 外部模型服务）
4. IDE 扩展层（可选入口，后续复用 API）

主链路：

1. 用户在 WebUI 输入问题
2. WebUI 调用 `/api/chat/*`
3. API 网关调用 `codex cli`
4. `codex cli` 访问外部模型服务（API Key）
5. 网关通过 SSE/轮询回传执行状态与输出
6. 关键数据落库以便审计与追溯

## 7. 配置方案（外部模型Key）

### 7.1 配置来源优先级

1. 环境变量（生产推荐）
2. `automation/config/settings.yaml`（仅非敏感项）

### 7.2 配置项建议

`settings.yaml` 建议新增：

```yaml
ai:
  provider: "openai_compatible"
  base_url: "https://api.openai.com/v1"
  model: "gpt-5-mini"
  timeout_seconds: 120
  max_output_tokens: 4000
  api_key_env: "CLAWCHECK_AI_API_KEY"
```

环境变量：

- `CLAWCHECK_AI_PROVIDER`
- `CLAWCHECK_AI_BASE_URL`
- `CLAWCHECK_AI_MODEL`
- `CLAWCHECK_AI_TIMEOUT_SECONDS`
- `CLAWCHECK_AI_MAX_OUTPUT_TOKENS`
- `CLAWCHECK_AI_API_KEY`

要求：

- `api_key` 仅从环境变量读取
- 配置摘要接口不回显密钥

## 8. API契约（MVP）

统一前缀：`/api/chat`

### 8.1 会话接口

- `POST /sessions`：创建会话
- `GET /sessions`：查询会话列表（分页）
- `GET /sessions/{会话ID}`：查询会话详情

### 8.2 消息接口

- `POST /sessions/{会话ID}/messages`：提交消息并触发执行
- `POST /sessions/{会话ID}/cancel`：取消执行
- `GET /sessions/{会话ID}/stream`：SSE 订阅执行事件

### 8.3 配置与健康接口

- `GET /config-summary`：脱敏配置摘要
- `GET /health`：对话能力健康检查

事件类型建议：

- `status`（queued/running/completed/failed/canceled）
- `token`（增量文本）
- `tool_call`（命令执行开始）
- `tool_result`（命令执行结果）
- `done`（结束事件）

## 9. 数据库设计（中文命名）

新增表：

1. `对话会话`
2. `对话消息`
3. `对话执行日志`

字段建议：

`对话会话`：

- `会话ID`（主键）
- `会话标题`
- `创建时间`
- `最后活跃时间`
- `工作目录`
- `模型提供方`
- `模型名称`
- `状态`

`对话消息`：

- `消息ID`（主键）
- `会话ID`（外键）
- `角色`
- `消息内容`
- `创建时间`
- `token数量`

`对话执行日志`：

- `日志ID`（主键）
- `会话ID`（外键）
- `消息ID`（可空）
- `事件类型`
- `事件摘要`
- `退出码`（可空）
- `创建时间`

## 10. WebUI交互方案

页面结构：

- 左侧会话列表
- 中间消息流（流式增量）
- 底部输入区（发送/停止/重试）
- 右侧参数抽屉（模型、目录、执行状态）

关键交互：

- 提交后立即展示 `running`
- 支持取消当前执行
- 支持失败重试
- 对命令执行日志默认折叠，避免主视图噪音

## 11. 安全与治理

- 密钥永不入库
- 日志统一脱敏（`Authorization`、`api_key`、`token`）
- 限制工作目录范围与危险命令
- 保留会话级审计记录

## 12. 分期计划

### 12.1 Phase 1（Web MVP，1周）

- 后端 `chat` router + `codex_runner` 打通
- WebUI 单会话页面上线
- API Key 配置、健康检查、错误分级

### 12.2 Phase 2（Web增强，1周）

- 多会话管理、历史查询
- 执行日志落库与检索
- 取消/超时/重试策略完善

### 12.3 Phase 3（IDE评估，0.5周）

- 是否进入 IDE 形态评审
- 输出评审结论与投入产出比
- 若通过，进入 `401` 专项方案（IDE 扩展）

## 13. 发布与回滚

### 13.1 发布步骤

1. 执行 SQL：`030_chat_conversation.sql`
2. 发布 FastAPI（含 `/api/chat`）
3. 发布 WebUI（含对话工作台）
4. 注入 `CLAWCHECK_AI_*` 环境变量
5. 执行联调与验收脚本

### 13.2 回滚步骤

1. 前端隐藏对话入口
2. 回滚后端服务版本
3. 暂停 `chat` 路由
4. 保留已落库审计数据，不做破坏性删除

## 14. 验收标准

- Web端可完成连续 10 轮对话
- 正常网络下首包时间小于 3 秒
- 取消执行可生效
- 缺少Key、Key无效、网络失败可区分提示
- 数据库可按会话追溯消息与执行日志
- 日志中无明文密钥

## 15. 风险与应对

- 风险：CLI 在不同主机环境行为差异  
  应对：上线前执行 `health` 预检并落日志

- 风险：外部模型限流影响体验  
  应对：429 退避重试 + 前端明确状态

- 风险：日志泄露敏感信息  
  应对：统一脱敏中间件 + CI 检查

# 方案编号220A：定时采集后自动评估补充方案文档

## 1. 背景

`220定时采集单据实现方案文档` 已完成定时采集、按完成时间起算、daemon 热生效、跨进程互斥等能力，但原始边界未把“采集成功后立即评估”纳入正式链路。

当前实际现象为：

1. 定时任务按计划完成 `collect`。
2. 单据已成功写入 PostgreSQL，并进入 `处理单据` 工作台。
3. 因未自动触发评估，工作台中的单据仍显示 `待评估`。
4. 业务侧需要再手工执行评估，导致“定时采集 -> 处理单据”链路不闭环。

## 2. 问题归因

当前存在两条不同入口：

1. 手工采集入口：`collect_workbench -> start_collect_task -> run_audit_now`
2. 定时采集入口：`windows_task_daemon -> run_windows_task.ps1 -> run.py collect`

手工入口具备 `autoAudit` 语义，但 `220` 的定时入口直接调用 `run.py collect`，没有携带任何“采集后自动评估”参数，因此评估能力被旁路掉了。

这不是偶发 bug，而是 `220` 原方案边界导致的能力缺口。

## 3. 目标

本次补充目标如下：

1. 定时采集成功后，默认自动对“本轮成功写库的单据”执行增量评估。
2. 评估成功后，单据进入 `处理单据` 工作台时直接具备最新评估结果。
3. 该能力可在 `常规配置 -> 定时采集单据` 中显式开关。
4. 继续复用正式 `collect` / `audit` 规则口径，不新增第二套评估实现。

## 4. 方案设计

### 4.1 配置口径

`定时采集单据` 新增配置项：

1. `启用定时采集`
2. `采集频率（分钟）`
3. `采集后自动评估`

单一事实来源仍为：

- `automation/config/windows_task_daemon.local.json`

其中 `collect.args` 补充：

```json
[
  "-Action",
  "collect",
  "-Headless",
  "-AutoAudit",
  "-Limit",
  "100"
]
```

说明：

1. `-AutoAudit` 为定时采集后的自动评估开关。
2. UI 保存时通过维护 `collect.args` 中是否包含 `-AutoAudit` 落盘，不引入第二份配置源。

### 4.2 执行链路

补充后的正式链路为：

1. `windows_task_daemon.py` 读取 `collect` 任务配置。
2. 到期后执行：
   - `powershell.exe -File automation/scripts/run_windows_task.ps1 -Action collect -Headless -AutoAudit -Limit 100`
3. `run_windows_task.ps1` 将 `-AutoAudit` 转换为：
   - `run.py collect --auto-audit`
4. `run.py collect` 在本轮采集成功写库后，立即对本轮成功单据执行增量评估。

### 4.3 自动评估触发条件

仅当同时满足以下条件时触发自动评估：

1. `--auto-audit = true`
2. 本轮不是 `dry-run`
3. 本轮存在成功写库单据

以下情况不触发：

1. 本轮无待办
2. 本轮只有跳过单据、没有成功写库单据
3. `dry-run`
4. 明确关闭 `采集后自动评估`

### 4.4 自动评估范围

自动评估只处理：

- 本轮成功采集并成功写库的 `document_no`

不改为全量批次评估，避免：

1. 放大定时任务耗时
2. 将与本轮无关的旧单据重复评估
3. 混淆“本轮采集结果”和“历史全量评估结果”

### 4.5 失败口径

若采集成功但自动评估失败：

1. 最近一次定时任务结果中必须明确体现“增量评估失败/部分失败”
2. 该批次应视为链路未完整闭环
3. 工作台最近结果与日志文件中应保留失败信息，便于排障

## 5. 影响文件

本次补充至少影响以下文件：

1. `automation/scripts/run.py`
2. `automation/scripts/run_windows_task.ps1`
3. `automation/utils/collect_schedule.py`
4. `automation/api/routers/settings.py`
5. `automation/config/windows_task_daemon.example.json`
6. `automation/config/windows_task_daemon.local.json`
7. `webui/src/pages/RuntimeSettingsPage.tsx`
8. `webui/src/types/dashboard.ts`
9. `tests/test_collect_schedule.py`
10. `tests/test_settings_router.py`

## 6. 验收标准

满足以下条件可认为本补充完成：

1. 定时采集配置页可查看和保存 `采集后自动评估` 开关。
2. 现有 `collect` 定时任务配置默认包含 `-AutoAudit`。
3. 定时采集成功后，对本轮成功写库单据自动生成评估结果。
4. `处理单据` 工作台不再因为定时采集链路缺少评估而大量停留在 `待评估`。
5. 若自动评估失败，最近结果与日志中可直接看到失败信息。

## 7. 结论

`220A` 的核心不是新增独立评估任务，而是把 `220` 原有的定时采集链路补成“采集完成即可闭环进入处理单据”的正式链路。

# 方案编号013：申请表组织范围角色组织展开落库方案文档

## 1. 方案目标
对 `申请表组织范围` 的字段设计与落库方式进行专项调整，使该表能够准确表达“**单据内某个角色对应哪些组织编码**”。

本方案目标：
- 为 `申请表组织范围` 新增 `角色编码`、`角色名称`
- 将落库粒度从“单据-组织”调整为“单据-角色-组织”
- 明确 `不检查组织范围 = TRUE` 时的落库表达方式
- 为后续组织范围校验、规则引擎、问题排查提供可直接查询的明细结构

本方案仅覆盖 `申请表组织范围` 的字段与落库口径，不重新定义 `申请单权限列表`、`权限列表` 的总体职责。

## 2. 背景与问题判断

### 2.1 当前表结构不足
当前 `申请表组织范围` 表结构仅包含：
- `单据编号`
- `组织编码`
- `记录创建时间`

该结构只能表达“某张单据出现过哪些组织编码”，不能表达：
- 某个 `角色编码` 对应哪些 `组织编码`
- 同一张单据内两个不同角色是否共用了同一个组织编码
- 某个角色是否本身就属于“不检查组织范围”

### 2.2 当前入库方式存在信息丢失
现有采集流中：
- `申请单权限列表` 已采集 `角色编码`、`角色名称`
- 但 `申请表组织范围` 入库时只保留整张单据聚合后的 `organization_codes`

这会导致两个直接问题：

1. 角色与组织的对应关系丢失
   - 当前只能知道“单据里有哪些组织”，不能知道“哪个角色对应哪个组织”

2. 约束口径错误
   - 当前唯一约束是 `("单据编号", "组织编码")`
   - 若同一张单据中两个不同角色都关联同一个组织编码，现状会被错误合并成一行

### 2.3 与 `009权限列表` 现有口径未打通
`009` 已在 `权限列表` 中定义字段：
- `角色编码`
- `角色名称`
- `不检查组织范围`

但 `申请表组织范围` 目前无法承接该口径，导致：
- “角色不检查组织范围”这一规则无法在事实表层清晰表达
- 后续规则判断仍需绕回多张表拼接，查询成本高

### 2.4 问题结论
因此，本方案判断：
- `申请表组织范围` 必须升级为“角色组织展开表”
- `组织编码` 必须允许为 `NULL`
- 唯一性必须以 `角色编码` 为核心，而不能只以 `组织编码` 为核心

## 3. 方案结论

### 3.1 最终选型
本方案建议保留表名：
- `申请表组织范围`

但将其业务语义明确调整为：
- 一张按 `单据编号 + 角色编码 + 组织编码` 展开的角色组织明细表

其核心落库规则如下：

1. 正常角色
   - `1 个角色编码 + 1 个组织编码 = 1 行`

2. 不检查组织范围角色
   - `1 个角色编码 = 1 行`
   - 此时 `组织编码 = NULL`

3. 同一单据内相同角色重复出现时
   - 按 `单据编号 + 角色编码 + 组织编码` 去重
   - 不重复制造噪音行

### 3.2 不采用的方案
本方案不建议继续使用以下方式：

- 方案一：仅保留 `单据编号 + 组织编码`
  - 原因：无法表达角色与组织的对应关系

- 方案二：把多个组织编码聚合成数组或逗号串存一行
  - 原因：不利于查询、去重、关联和规则判断

- 方案三：`不检查组织范围 = TRUE` 时仍展开全部组织编码
  - 原因：会产生大量无实际判断价值的噪音数据

## 4. 表结构设计

### 4.1 建议字段
建议 `申请表组织范围` 调整为以下结构：

- `组织范围ID`
  - 主键，`BIGSERIAL`
- `单据编号`
  - 关联 `申请单基本信息.单据编号`
- `角色编码`
  - 角色唯一识别码，作为本表核心业务键之一
- `角色名称`
  - 角色名称快照，便于直接查询和排查
- `组织编码`
  - 组织编码；当角色 `不检查组织范围 = TRUE` 时允许为 `NULL`
- `记录创建时间`
  - 默认 `NOW()`

建议字段可空性如下：
- `单据编号`：`NOT NULL`
- `角色编码`：`NOT NULL`
- `角色名称`：`NOT NULL`
- `组织编码`：`NULL`
- `记录创建时间`：`NOT NULL`

建议 DDL 形态如下：

```sql
CREATE TABLE IF NOT EXISTS "申请表组织范围" (
    "组织范围ID" BIGSERIAL PRIMARY KEY,
    "单据编号" VARCHAR(64) NOT NULL REFERENCES "申请单基本信息"("单据编号") ON DELETE CASCADE,
    "角色编码" VARCHAR(128) NOT NULL,
    "角色名称" VARCHAR(255) NOT NULL,
    "组织编码" VARCHAR(64) NULL,
    "记录创建时间" TIMESTAMP NOT NULL DEFAULT NOW()
);
```

### 4.2 字段口径
- `角色编码`
  - 取自 `申请单权限列表` 当前明细行的 `角色编码`
  - 本表不以 `角色名称` 作为主关联键

- `角色名称`
  - 取自 `申请单权限列表` 当前明细行的 `角色名称`
  - 作为申请当时快照保留，不用于主键判重

- `组织编码`
  - 正常角色时，取该角色组织详情中采集到的每个组织编码
  - `不检查组织范围 = TRUE` 时固定落 `NULL`

### 4.3 建议约束与索引
不再使用原唯一约束：
- `UNIQUE ("单据编号", "组织编码")`

建议改为以下约束：

1. 非空组织编码唯一索引
```sql
CREATE UNIQUE INDEX uq_apply_form_org_scope_doc_role_org
ON "申请表组织范围" ("单据编号", "角色编码", "组织编码")
WHERE "组织编码" IS NOT NULL;
```

2. 空组织编码唯一索引
```sql
CREATE UNIQUE INDEX uq_apply_form_org_scope_doc_role_null_org
ON "申请表组织范围" ("单据编号", "角色编码")
WHERE "组织编码" IS NULL;
```

3. 普通查询索引
```sql
CREATE INDEX idx_apply_form_org_scope_document_no
ON "申请表组织范围" ("单据编号");

CREATE INDEX idx_apply_form_org_scope_role_code
ON "申请表组织范围" ("角色编码");
```

这样设计的原因：
- 允许同一组织编码在同一单据下被不同角色分别持有
- 允许 `组织编码 = NULL` 表达“不检查组织范围”
- 同时保证同一角色不会重复写入多条 `NULL` 记录

## 5. 落库规则设计

### 5.1 正常角色落库规则
当某角色 `不检查组织范围 = FALSE` 时：
- 采集该角色明细行对应的全部组织编码
- 每个组织编码单独写一行

示例：

角色：
- `角色编码 = HR001`
- `角色名称 = HRBP查看`

组织：
- `ORG1001`
- `ORG1002`
- `ORG1003`

则落库 `3` 行：

| 单据编号 | 角色编码 | 角色名称 | 组织编码 |
| --- | --- | --- | --- |
| DOC001 | HR001 | HRBP查看 | ORG1001 |
| DOC001 | HR001 | HRBP查看 | ORG1002 |
| DOC001 | HR001 | HRBP查看 | ORG1003 |

### 5.2 不检查组织范围角色落库规则
当某角色在 `权限列表` 中满足：
- `不检查组织范围 = TRUE`

则落库规则为：
- 只保留该角色一条记录
- `组织编码 = NULL`
- 不再展开该角色的组织详情明细

示例：

| 单据编号 | 角色编码 | 角色名称 | 组织编码 |
| --- | --- | --- | --- |
| DOC001 | RLHMD001 | 某远程角色 | NULL |

该行的业务语义是：
- 此角色存在于该单据中
- 但它不参与组织范围限制
- 因此无需保留其具体组织列表

### 5.3 重复数据处理规则
同一单据内若出现以下重复情况，统一去重：

- 同一 `角色编码` 下重复出现同一 `组织编码`
- 同一 `角色编码` 被重复判断为 `不检查组织范围 = TRUE`

去重后的正式业务键为：
- 正常角色：`单据编号 + 角色编码 + 组织编码`
- 跳过组织范围角色：`单据编号 + 角色编码`

### 5.4 空组织编码使用边界
`组织编码 = NULL` 仅允许用于以下场景：
- 该角色 `不检查组织范围 = TRUE`

以下场景不允许用 `NULL` 兜底：
- 页面采集失败
- 角色本应检查组织范围，但组织详情为空
- 角色编码缺失

原因：
- `NULL` 在本方案中有明确业务含义，不能混用成“异常兜底值”

### 5.5 异常场景处理建议
- `角色编码` 为空
  - 不写入 `申请表组织范围`
  - 记录异常日志，待人工排查

- 角色需要检查组织范围，但未采集到任何组织编码
  - 不写入 `NULL` 行
  - 记录告警日志
  - 由后续重采或人工核查处理

- 同一 `角色编码` 在同一单据中出现不同 `角色名称`
  - 以当前明细行快照原文写入
  - 若最终同键冲突，保留首个非空值并记录日志

## 6. 采集与入库改造设计

### 6.1 采集层改造点
当前 `automation/flows/permission_collect_flow.py` 中：
- `extract_all_organization_codes()` 返回的是整单聚合后的 `list[str]`

本方案建议改为按角色返回明细结构，例如：

```json
[
  {
    "role_code": "HR001",
    "role_name": "HRBP查看",
    "organization_codes": ["ORG1001", "ORG1002"]
  },
  {
    "role_code": "RLHMD001",
    "role_name": "某远程角色",
    "organization_codes": []
  }
]
```

也可以直接展开为扁平行：

```json
[
  {
    "role_code": "HR001",
    "role_name": "HRBP查看",
    "org_code": "ORG1001"
  },
  {
    "role_code": "HR001",
    "role_name": "HRBP查看",
    "org_code": "ORG1002"
  }
]
```

两种方式都可行，但建议优先采用“**按角色聚合、入库前再展开**”：
- 采集侧更贴近页面结构
- 更容易在入库时结合 `权限列表` 判断是否跳过组织范围

本次实现补充约束如下：
- 组织范围采集以明细行顺序逐行执行，返回结构命名为 `role_organization_scopes`
- 明细 grid 在整单采集过程中复用同一个已识别的权限申请明细 grid，不再为下一行重新打开单据
- 每次点击 `查看详情(n)` 后，在当前单据页读取组织详情 grid，读取完成后通过 `Esc` 关闭详情框，再继续下一行
- 对 `不检查组织范围 = TRUE` 的角色，即使页面明细中仍显示 `查看详情(n)`，采集层也按业务规则直接跳过组织详情读取，保留空 `organization_codes`
- 典型例外角色包括：`RLHMD001`、`DTX009`、`qbireport(QBI报表权限申请)`；其中 `qbireport` 页面无行政组织详情时也按“不检查组织范围”处理
- `organization_codes = []` 仅表示“采集层未展开组织列表”；真正写库时再按本方案转换为单条 `组织编码 = NULL` 记录

### 6.2 入库层改造点
当前 `automation/db/postgres.py` 中：
- `_write_document()` 仅按 `organization_codes` 写入 `单据编号 + 组织编码`

本方案建议改为：

1. 先按 `角色编码` 维度遍历权限明细
2. 通过 `权限列表.角色编码` 判断该角色是否 `不检查组织范围`
3. 根据判断结果分支写入：
   - `FALSE`：展开组织编码逐行写入
   - `TRUE`：仅写入一条 `组织编码 = NULL` 记录

### 6.3 推荐判断顺序
推荐以下判断顺序：

1. 读取当前单据 `permission_details`
2. 对每条明细取出：
   - `角色编码`
   - `角色名称`
3. 查询 `权限列表` 中该角色的 `不检查组织范围`
4. 若为 `TRUE`
   - 不写组织明细
   - 只写一条 `NULL` 组织编码行
5. 若为 `FALSE`
   - 写入该角色对应的全部组织编码明细

页面交互补充：
- 角色组织详情在当前页面内弹出，不是新标签页
- 行与行之间的切换依赖关闭当前详情框，而不是重新进入单据
- 关闭动作统一使用键盘 `Esc`

### 6.4 与现有“先删后插”策略的关系
`003` 已明确 `申请表组织范围` 采用：
- 按单据先删后插

本方案继续沿用该策略，不调整刷新模式。

刷新逻辑仍为：
- 先按 `单据编号` 删除旧数据
- 再按本方案新口径整单重写

这样可以避免：
- 历史结构残留
- 角色组织映射变更后的脏数据

## 7. 历史数据迁移策略

### 7.1 不能直接从旧表无损迁移
旧表只有：
- `单据编号`
- `组织编码`

没有：
- `角色编码`
- `角色名称`
- 角色与组织的对应关系

因此旧数据无法仅靠 SQL 无损补齐到新结构。

### 7.2 建议迁移方式
建议采用：
- 调整表结构
- 清空旧口径数据
- 对需要保留的历史单据执行重新采集

也就是说：
- 本方案是“**结构升级 + 单据重刷**”
- 不是“旧表直接在线字段补齐”

### 7.3 SQL 编号建议
由于仓库中 `automation/sql/013_rename_columns_to_en_fixed_schema_rollback.sql` 已存在，后续正式实施 SQL 建议另行编号，避免混淆，例如：

## 8. 验证建议

### 8.1 建议校验 SQL
```sql
SELECT COUNT(*)
FROM "申请表组织范围";

SELECT "单据编号", "角色编码", "角色名称", "组织编码"
FROM "申请表组织范围"
WHERE "单据编号" = 'DOC001'
ORDER BY "角色编码", "组织编码" NULLS FIRST;

SELECT "单据编号", "角色编码", COUNT(*)
FROM "申请表组织范围"
WHERE "组织编码" IS NULL
GROUP BY "单据编号", "角色编码"
HAVING COUNT(*) > 1;

SELECT "单据编号", "角色编码", "组织编码", COUNT(*)
FROM "申请表组织范围"
WHERE "组织编码" IS NOT NULL
GROUP BY "单据编号", "角色编码", "组织编码"
HAVING COUNT(*) > 1;
```

### 8.2 验收标准
满足以下条件可视为方案完成：
- `申请表组织范围` 成功新增 `角色编码`、`角色名称`
- 同一角色下多个组织编码被正确拆为多行
- 同一组织编码可在不同角色下分别保留，不被错误合并
- `不检查组织范围 = TRUE` 的角色只保留一条 `组织编码 = NULL` 记录
- `不检查组织范围 = TRUE` 的角色即使页面展示 `查看详情(n)`，采集层也不会误采组织列表
- 组织详情采集完成后可通过 `Esc` 关闭，并继续下一条角色的组织采集
- 非跳过角色不会因为异常情况被错误写成 `NULL` 组织编码

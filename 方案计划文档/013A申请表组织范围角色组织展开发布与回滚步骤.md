# 方案编号013A：申请表组织范围角色组织展开发布与回滚步骤

## 1. 目标
本清单用于承接 `013申请表组织范围角色组织展开落库方案文档`，把“`申请表组织范围` 从单据-组织粒度升级为单据-角色-组织粒度”的设计落成可执行发布步骤。

本清单重点回答 4 个问题：

1. 哪些代码和 SQL 必须一起发布
2. 正确发布顺序是什么
3. 哪些情况必须中止
4. 出现问题时如何回滚

## 2. 当前结论
`013` 不是单纯加两列，而是一次**明细粒度重构**。

本次变更同时涉及：
- 表结构调整
- 唯一约束调整
- 页面采集结构调整
- PostgreSQL 入库逻辑调整
- 历史数据处理策略调整

因此不能只执行 `014_apply_form_org_scope_role_org_refactor.sql` 后继续跑旧代码，也不能只发新代码不改库。

正确顺序必须是：
- 先准备回滚点
- 再执行表结构迁移
- 再发布新代码
- 再执行最小回归与验收 SQL

## 3. 必改文件清单

### 3.1 数据库结构
必须同步：
- `automation/sql/001_permission_apply_collect.sql`
- `automation/sql/010_permission_apply_collect_migrate_basic_info.sql`
- `automation/sql/014_apply_form_org_scope_role_org_refactor.sql`

### 3.2 采集流
必须同步：
- `automation/flows/permission_collect_flow.py`

必须完成的改造点：
- 不再只输出整单聚合的 `organization_codes`
- 需要输出按角色保留的 `role_organization_scopes`

### 3.3 入库层
必须同步：
- `automation/db/postgres.py`

必须完成的改造点：
- 识别 `申请表组织范围` 新字段
- 按 `角色编码 + 组织编码` 写入明细
- 结合 `权限列表.不检查组织范围` 生成 `组织编码 = NULL` 记录

### 3.4 文档与验收
至少同步：
- `方案计划文档/013申请表组织范围角色组织展开落库方案文档.md`
- `方案计划文档/013A申请表组织范围角色组织展开发布与回滚步骤.md`
- `automation/sql/015_apply_form_org_scope_role_org_acceptance.sql`

## 4. 发布前准备

### 4.1 冻结范围
发布窗口前建议冻结以下范围：
- 权限申请采集链路
- `申请表组织范围` 相关字段设计
- `权限列表` 中 `不检查组织范围` 规则

### 4.2 执行前盘点
正式执行前至少确认：

1. 当前库中 `申请表组织范围` 列结构
2. 当前唯一约束是否仍为 `("单据编号", "组织编码")`
3. 当前应用代码是否已切到 `013` 版本
4. `权限列表` 已存在且可查询 `角色编码`、`角色名称`、`不检查组织范围`

### 4.3 备份要求
本次必须先备份以下对象：
- `申请表组织范围`
- `申请单权限列表`
- `权限列表`

原因：
- `014` 会把旧结构下无法无损补齐角色维度的数据清理掉
- 一旦迁移后验收失败，回滚依赖备份或库级恢复

## 5. 正式发布顺序

### 5.1 发布前停写
1. 暂停权限申请采集任务
2. 暂停任何会写入 `申请表组织范围` 的自动化流程

### 5.2 数据备份
1. 备份目标库
2. 至少单独备份：
   - `申请表组织范围`
   - `申请单权限列表`
   - `权限列表`

### 5.3 执行数据库迁移
执行：
- `automation/sql/014_apply_form_org_scope_role_org_refactor.sql`

该步骤完成后应满足：
- 新增 `角色编码`
- 新增 `角色名称`
- `组织编码` 允许 `NULL`
- 旧唯一约束被移除
- 新部分唯一索引创建成功

### 5.4 发布应用代码
发布包含以下改造的代码版本：
- `automation/flows/permission_collect_flow.py`
- `automation/db/postgres.py`
- `automation/sql/001_permission_apply_collect.sql`
- `automation/sql/010_permission_apply_collect_migrate_basic_info.sql`

### 5.5 最小回归
发布后至少执行一张权限申请单据的完整采集，并检查：

1. `申请单权限列表` 正常写入
2. `申请表组织范围` 正常写入
3. 正常角色被按组织编码拆成多行
4. `不检查组织范围 = TRUE` 的角色只写一条 `组织编码 = NULL` 记录

### 5.6 验收 SQL
执行：
- `automation/sql/015_apply_form_org_scope_role_org_acceptance.sql`

## 6. 中止条件
出现以下任一情况，必须停止继续发布并评估回滚：

1. `014` 执行报错
2. 新索引未创建成功
3. 采集后 `申请表组织范围` 仍无 `角色编码` 或 `角色名称`
4. `不检查组织范围 = TRUE` 的角色被错误展开为多条组织编码
5. 正常角色被错误写成 `组织编码 = NULL`
6. `申请单权限列表` 与 `申请表组织范围` 角色数差异明显异常

## 7. 回滚原则
本次**不建议提供“反向 SQL 直接回旧结构”的正式回滚路径**，原因如下：

1. 旧结构只支持 `单据编号 + 组织编码`
2. 新结构已引入 `角色编码`、`角色名称`
3. 迁移过程中旧结构下无法无损补齐角色维度的数据会被清理
4. `组织编码 = NULL` 在新结构中有明确业务语义，旧结构无法表达

因此本次回滚以：
- 数据库备份恢复
- 代码版本回退

为准，而不是依赖反向 SQL 强行恢复旧表结构。

## 8. 回滚顺序
若需回滚，推荐顺序：

1. 停止应用写入
2. 回退应用代码到 `013` 之前版本
3. 从发布前备份恢复：
   - `申请表组织范围`
   - `申请单权限列表`
   - `权限列表`
4. 执行最小回归，确认旧链路恢复正常
5. 再决定是否恢复业务写入

## 9. 验收顺序
建议按以下顺序验收：

1. 表结构验收
2. 索引与唯一性验收
3. 单据级样本验收
4. `不检查组织范围` 例外角色验收
5. 重复数据验收

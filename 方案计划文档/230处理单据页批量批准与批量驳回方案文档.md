# 230 处理单据页批量批准与批量驳回方案文档

## 1. 目标

在 `处理单据` 页面补充批量批准与批量驳回能力，满足以下业务要求：

1. 列表第一列增加勾选框。
2. 支持多选待办单据。
3. 在 `重置筛选` 按钮后增加 `批准`、`驳回` 按钮。
4. 批量批准时，对勾选单据一次完成全部批准，审批意见统一写入 `通过`。
5. 批量驳回时，点击 `驳回` 后先二次确认；确认后一次完成全部驳回，审批意见默认使用系统 `生成驳回意见` 产出的值。
6. 批量批准、批量驳回均必须在同一个 EHR 浏览器窗口会话中顺序完成，不允许按单据重复拉起浏览器。

## 2. 页面交互口径

### 2.1 列表勾选

- `处理单据` 列表开启 `checkboxSelection`，勾选框固定在第一列。
- 仅允许勾选满足以下条件的单据：
  - `待办处理状态 = 待处理`
  - `hasAssessment = true`
- 不满足条件的行保留展示，但不允许进入批量操作。

### 2.2 批量按钮

- 在列表工具栏中，`重置筛选` 后追加：
  - `批准`
  - `驳回`
- 按钮文案可带出当前勾选数量，例如：
  - `批准（3）`
  - `驳回（3）`
- 未勾选单据时按钮禁用。
- 批量执行期间：
  - 批量按钮禁用
  - 单单审批按钮禁用
  - 待办同步按钮禁用

### 2.3 批量批准

- 点击 `批准` 后直接调用批量审批接口。
- 后端进入正式审批前，先执行一次 `同步待办状态`，用于把工作台待办口径与当前账号 EHR 待办重新对齐。
- 前端不再要求逐条填写审批意见。
- 后端统一按每张单据写入 `通过`。

### 2.4 批量驳回

- 点击 `驳回` 后先弹出二次确认框。
- 确认框展示：
  - 勾选单据数量
  - 本次涉及的单据编号
  - 明确提示“将在同一个 EHR 浏览器会话中依次处理”
- 用户确认后调用批量审批接口。
- 后端逐单据读取该单据的风险总览摘要，按 `生成驳回意见` 的口径自动生成审批意见。

### 2.5 结果反馈

- 页面展示批量执行结果提示。
- 至少展示：
  - 总数
  - 成功数
  - 待确认数（继承 `105C` 的“提交动作已发出但未拿到强成功回执”口径）
  - 失败数
  - 批量日志文件路径
- 若存在失败单据，展示失败单据编号与失败原因摘要。
- 若存在 `submitted_pending_confirmation` 单据，页面应以警告态提示“请先不要重复点击”，并允许用户后续通过 `同步待办状态` 复核，而不是把这类单据直接当作普通失败重试。

## 3. 接口与后端执行口径

### 3.1 接口

- 复用现有接口：
  - `POST /documents/process-workbench/approval/batch`

请求体：

```json
{
  "action": "approve | reject",
  "documentNos": ["RA-20260323-00020321", "RA-20260323-00020322"],
  "dryRun": false,
  "headed": true
}
```

### 3.2 审批意见口径

- `approve`
  - 固定返回 `通过`
- `reject`
  - 由后端读取单据详情中的 `feedbackOverview.feedbackGroups.summaryLines`
  - 按现有 `生成驳回意见` 口径拼接为多行文本

### 3.3 浏览器会话要求

- 批量审批开始前，后端先执行一次全量 `同步待办状态`；若同步失败，则本次批量审批整体失败，不进入后续 EHR 审批会话。
- 批量审批入口必须只申请一次审批浏览器会话。
- 在同一个 `browser/context/page` 生命周期中依次处理 `documentNos`。
- 全部单据处理完成后统一释放会话。
- 单据间允许页面复用、登录态复用、EHR 首页复用。

### 3.4 成功确认继承口径

- `230` 批量审批沿用 `105C` 的提交后成功确认口径。
- 若单据返回 `submitted_pending_confirmation`，表示：
  - 提交动作大概率已经发出
  - 当前未拿到强成功回执
  - 前端不应鼓励用户立即重复点击同一批单据
- 因此批量结果统计应拆分为：
  - `成功`
  - `待确认`
  - `失败`
- 页面仅自动保留“硬失败”单据的勾选状态，`待确认` 单据由用户后续查看日志或执行 `同步待办状态` 复核。

当前实现入口：

- `automation/api/process_dashboard.py`
  - `approve_process_documents_batch`
  - `_run_logged_document_approval_in_active_session`

- 增加批量驳回按钮
- 增加批量驳回确认弹窗

### 4.2 后端

## 5. 验收标准

### 5.1 批量批准

1. 在 `处理单据` 页面可勾选多条待处理单据。
2. 点击 `批准` 后，后端会先执行一次 `同步待办状态`。
3. 若同步成功，只弹出一个 EHR 浏览器窗口。
4. 同一窗口内依次完成全部勾选单据的批准。
5. 每条单据审批意见为 `通过`。
6. 页面返回成功/部分成功/失败结果与日志路径。

### 5.2 批量驳回

1. 在 `处理单据` 页面可勾选多条待处理单据。
2. 点击 `驳回` 时出现二次确认框。
3. 确认后，只弹出一个 EHR 浏览器窗口。
4. 同一窗口内依次完成全部勾选单据的驳回。
5. 每条单据审批意见默认采用 `生成驳回意见` 的产出。
6. 页面返回成功/部分成功/失败结果与日志路径。

## 6. 风险与边界

- 若审批前 `同步待办状态` 失败，本次批量审批直接失败，避免在待办口径失真的前提下继续提交。
- 未完成评估的单据无法生成驳回意见，不纳入批量操作范围。
- 若单据最近一次待办同步状态已为 `已处理 / 已驳回`，后端会按单据失败返回，不影响其他单据继续执行。
- 若批量过程中部分单据失败，页面保留失败结果，允许用户按失败原因继续处理。
- 若批量过程中命中 `submitted_pending_confirmation`，应按“待确认”处理，不应被视为可立即重试的普通失败，否则会放大重复提交风险。

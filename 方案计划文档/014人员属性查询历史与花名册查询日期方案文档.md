# 方案编号014：人员属性查询历史与花名册查询日期方案文档

## 1. 目标
本方案用于补齐两项能力：

1. 新增 `人员属性查询历史`，按 EHR 在职花名册的 `查询日期` 保留 `人员属性查询` 的历史快照。
2. 在“同步主数据 -> 同步在职花名册”中新增 `查询日期` 输入能力，并把该日期真正传递到 EHR 查询条件中。

本方案要求做到：

- `同步在职花名册` 成功导入后，自动刷新 `人员属性查询`；
- 刷新完成后，按 `生效日期 = 查询日期` 覆盖写入 `人员属性查询历史`；
- 同一 `生效日期` 重跑时，先删后插，不保留同日期旧快照；
- WebUI、FastAPI、CLI、Playwright 流程统一支持 `queryDate / --query-date`；
- 真实连通 EHR 并验证历史日期查询可行。

## 2. 新增表设计

### 2.1 表名
- `人员属性查询历史`

### 2.2 字段规则
- 表结构以 `人员属性查询` 为基础；
- 额外新增一列 `生效日期`；
- `人员属性查询` 与 `人员属性查询历史` 同步纳入 `所属战区`；
- `生效日期` 取 EHR 在职花名册查询条件中的 `查询日期`；
- `花名册查询日期` 继续保留在快照数据中，口径与 `生效日期` 一致。

补充口径：

- 新跑花名册刷新时，`所属战区` 优先取 `组织属性查询.所属战区`；
- 对本方案落地前已经存在、但尚未保存 `所属战区` 的旧快照，允许基于历史 `组织路径名称` 中的 `xx人力资源与行政服务部` 片段做一次性回填；
- 不把“按当前组织表回查历史战区”作为正式报表实现口径。

### 2.3 主键与索引
- 主键：`生效日期 + 工号`
- 常用索引：
  - `生效日期`
  - `工号`
  - `申请人HR类型`
  - `部门ID`
  - `是否责任HR`

### 2.4 同日期覆盖规则
- 每次写入 `人员属性查询历史` 前，先删除 `生效日期 = 本次查询日期` 的全部历史行；
- 删除完成后，再从最新 `人员属性查询` 全量插入该日期快照；
- 不做同日期增量 merge。

## 3. 运行链路调整

### 3.1 同步时序
`同步在职花名册` 的正式链路调整为：

1. EHR 花名册页面按指定 `查询日期` 查询并导出；
2. 下载文件导入 `在职花名册表`；
3. 基于最新 `在职花名册表` 全量刷新 `人员属性查询`；
4. 以 `生效日期 = 查询日期` 覆盖写入 `人员属性查询历史`。

- `automation/scripts/run.py`

## 4. EHR 查询日期交互口径

### 4.1 实测页面结构
在正式环境 `https://hr.onewo.com/ierp/?formId=home_page` 中，花名册查询日期字段实测为：

- 容器：`div#date`
- 可编辑输入框：`div#date input`

### 4.2 失败交互结论
在 `2026-03-30` 的第一次实测中，直接对输入框执行 `fill("2026-03-29")` 虽然页面输入值会变化，但查询结果仍回到当天 `2026-03-30`。  
结论：该控件不是普通文本框，单纯 `fill` 不足以把历史日期提交到服务端表单状态。

### 4.3 可行交互结论
在 `2026-03-30` 的后续实测中，以下交互可以稳定生效：

1. 点击 `div#date input`，进入日历态；
2. 使用 `PageUp / PageDown` 切换月份；
3. 使用方向键切换具体日期；
4. 按 `Enter` 确认；
5. 点击 `查询` 后，以页面摘要中的 `查询日期` 回读校验。

因此正式实现统一采用“日历态键盘选日并确认”的方案，而不再使用纯 `fill`。

## 5. 同步主数据页调整

### 5.1 页面参数
“同步主数据 -> 同步在职花名册”新增：

- `查询日期`
  - 控件类型：日期输入框
  - 默认值：当天
  - 支持手工修改

### 5.2 参数传递链路
- WebUI 请求字段：`queryDate`
- FastAPI 入参字段：`queryDate`
- CLI 入参：`--query-date YYYY-MM-DD`
- Playwright 流程入参：`query_date`

### 5.3 最近任务展示
最近任务列表保留本次执行使用的 `queryDate`，便于回溯历史快照来源。

## 8. 风险与注意事项

### 8.1 导出耗时
历史日期花名册导出可能明显超过 30 分钟，后台导出必须保留重试与任务跟踪逻辑，不能只依赖浏览器直接下载事件。

### 8.2 日期控件不可退回为纯 fill
若后续维护中把日期设置逻辑改回普通 `fill`，历史日期查询会重新失效。

### 8.3 文档优先级
后续如需继续扩展 `人员属性查询历史` 的字段、索引或刷新口径，应优先更新本方案和 `003`、`204`，再改代码与 SQL。

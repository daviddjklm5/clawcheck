# 方案编号103：申请表审批风险与信任度评估实现方案文档

## 1. 方案目标

基于附件 `方案计划文档/申请表审批风险与信任度评估.xlsx` 中定义的“风险与信任度评估模型”，在现有 `Python + Playwright + PostgreSQL` 项目基础上，建设一套可落库、可回放、可解释、可逐步演进的“申请表是否建议通过审批”的评估能力。

本方案目标不是直接改造 iERP 页面自动审批，而是先在本项目内形成稳定的“评估层”：

- 对每张权限申请单输出 `风险/信任分`
- 对每个申请角色输出可解释的评分明细
- 输出建议动作：`拒绝 / 人工干预 / 仅关注 / 可信任`
- 为后续自动审核规则引擎、审批路由、历史复盘和模型校准提供统一基础

### 2.3 当前最大缺口

当前项目仍存在一个关键缺口：

- 当前 `43` 张申请单最新审批动作全部为 `待审核`
- 即：当前数据库里没有“最终通过 / 最终驳回”的完结标签
- 当前采集入口仍以“待办任务”链路为主，并未形成“已办 / 历史完结单据”稳定采集链路

这意味着：

1. 现在可以落地“规则化风险与信任评估”
2. 现在不能直接训练“监督式通过概率模型”
3. 若后续要做真实通过率校准，必须补齐“已完结单据采集”

## 3. 附件模型解读

### 3.1 模型结构

附件 `情况` sheet 实际定义了 5 个评分维度：

1. `申请人的角色判断`
2. `申请人所属的组织流程层级分类`
3. `审批人判断`
4. `申请的权限`
5. `申请的组织`

每个维度按情形输出一个分值，分值范围为：

- `0`
- `0.5`
- `1`
- `1.5`
- `2`
- `2.5`

### 3.2 模型输出含义

附件给出的最终解释区间如下：

- `0`：`拒绝`
- `0.5 ~ 1`：`需要人工干预`
- `1.5 ~ 2.0`：`仅关注`
- `2.5`：`可信任`

### 3.3 聚合方式判断

附件没有给出明确公式单元格，但给了两个关键说明：

- `申请多个角色，取最低分`
- `一个角色命中多个条件，总是取分数较低者`

基于该设计，本方案判断该模型应按“最低分优先”的保守逻辑落地：

1. 单个维度命中多条条件：取最低分
2. 单个角色涉及多个组织：取最低分
3. 单据申请多个角色：取最低分
4. 单据最终信任分：取各维度最低分

即：

- 该模型本质上不是“平均分模型”
- 而是“信任下限模型”或“风险短板模型”

这样可以自然解释附件中的输出区间：

- 最低分为 `0`：直接拒绝
- 最低分为 `0.5` 或 `1`：人工干预
- 最低分为 `1.5` 或 `2`：仅关注
- 只有全部关键维度都达到 `2.5` 时，才进入 `可信任`

### 3.4 审批链评分规则的实现判断

附件对审批链维度写了“从最后一次提交开始”，并要求：

- 排除 `平台运营组对接人`
- 提交人不能算作审批人判断

当前已确认：

- `平台运营组对接人` 的识别口径为：
  - `申请单审批记录.节点名称 = 平台运营组对接人`
  - 命中该节点名称的环节及其对应审批人，统一不参与审批链评分

因此审批链评分不能简单看全量审批记录，而应先做“审批轮次切分”：

1. 以 `审批动作 = 提交` 作为轮次切分点
2. 取最后一次 `提交` 之后的审批记录作为当前有效审批轮
3. 在该轮里剔除：
   - 提交人本人
   - 平台运营组对接人
4. 再判断是否存在 `战区人行部门` 类型审批人

说明：

- 附件中 `审批人判断` 的第 `0` 分与 `0.5` 分两条规则存在一定口径重叠
- 本方案建议在第一版实现中，按“全历史未覆盖战区人行部门 = 0；历史有覆盖但当前轮未覆盖 = 0.5”的方式落地
- 审批链评分应以审批人的人员/组织属性为主，不以节点名称的业务语义替代审批人属性判断
- 原因是审批人存在发起人手动选择场景，节点名称只能说明流程位置，不能天然证明审批人具备该组织身份
- 这是根据附件语义做出的实现性推断，正式上线前应由业务再确认一次

## 5. 评分规则落地方案

### 5.1 维度一：申请人的角色判断

直接使用 `人员属性查询.申请人HR类型` 映射：

- `H1/H2/H3 -> 2.5`
- `HY -> 1.5`
- `HX -> 1`

若 `申请人HR类型 = NULL`：

- 不建议直接按 `HX` 处理
- 应进入 `人工干预`
- 原因记为 `申请人未匹配人员属性`

### 5.2 维度二：申请人所属的组织流程层级分类

直接使用 `组织属性查询.组织流程层级分类` 映射：

- `战区人行部门 -> 2.5`
- `属地服务站 -> 2.5`
- `人事远程交付中心 -> 2.5`
- `BG人行中心与学社 -> 1`
- `属地组织 -> 2.5`
- `万物云本部 -> 1`
- `业务单元本部 -> 2.0`
- `NULL -> 0.5`
- `蝶发人行部 -> 1`

### 5.3 维度三：审批人判断

本维度应新增审批 facts：

- `最后一次提交时间`
- `当前审批轮有效审批人集合`
- `当前审批轮是否存在战区人行部门审批人`
- `全历史是否存在战区人行部门审批人`
- `当前审批轮是否全部为提交人本人`
- `是否命中被排除审批节点`
- `被排除审批节点名称列表`

第一版建议实现以下评分：

- `从最后一次提交开始，所有有效审批人与提交人相同 -> 0`
- `申请人组织流程层级分类 in (属地组织, NULL)` 且 `权限对象所属组织单元 != 祥盈企服` 且全历史无战区人行部门审批人 -> `0`
- `申请人组织流程层级分类 in (属地组织, NULL)` 且 `权限对象所属组织单元 != 祥盈企服` 且历史有战区人行部门、但当前轮无 -> `0.5`
- 其他情形 -> `2.5`

说明：

- 该维度是本模型里最依赖解释性加工的部分
- `平台运营组对接人` 当前已确认按 `申请单审批记录.节点名称 = 平台运营组对接人` 识别
- 该节点属于流程排除口径，可继续配置化维护
- 但 `战区权限对接人`、`属地人力资源部负责人` 等节点名称只作为辅助观察字段，不应直接替代审批人所属组织分类参与评分
- 自 `2026-03-19` 起，`权限对象所属组织单元 = 祥盈企服` 场景不再触发两条“缺少战区人行部门审批”规则（`APPROVAL_LOCAL_WITHOUT_WARZONE_HISTORY` / `APPROVAL_LOCAL_WITHOUT_WARZONE_CURRENT_ROUND`）

### 5.4 维度四：申请的权限

对每个角色使用 `权限列表.权限级别` 打分：

- `申请的权限不在 权限列表 主数据中 -> 0.5`
- `W类 -> 0`
- `A类-远程` 且申请人不是 `人事远程交付中心 -> 0`
- `A类-远程` 且申请人是 `人事远程交付中心 -> 2.5`
- `S1类-限定` 且申请人不是 `BG人行中心与学社 / 人事远程交付中心 -> 0`
- `S1类-限定` 且申请人是 `BG人行中心与学社 / 人事远程交付中心 -> 0.5`
- `S2类-限定` 且申请人是 `属地服务站 / 属地组织 / NULL -> 0`
- `S2类-限定` 且申请人不是 `属地服务站 / 属地组织 / NULL -> 0.5`
- `B1类-涉薪 / B2类-涉档案绩效 / C类-常规` 不再在 `申请的权限` 维度单独评估风险，统一按 `2.5` 作为中性跳过分处理

说明：

- 附件对 `S1/S2/B1/B2/C` 已给出分段口径
- 当 `角色编码` 或 `角色名称` 未匹配到 `权限列表` 时：
  - 按 `0.5` 计分
  - 建议动作进入 `人工干预`
  - 干预说明应明确为：`权限主数据缺失，需管理员更新权限列表后复核`
- `B级及以下` 角色仍继续参与 `申请的组织` 维度评分，不影响组织授权级别、跨组织单位等组织侧规则命中
- 单个角色命中多条条件时，必须取最低分

### 5.5 维度五：申请的组织

对每个角色-组织组合打分：

- `申请一级授权 -> 0.5`
- `申请二级授权 -> 1`
- `申请二级授权` 且 `申请组织所属组织单位` 命中战区代表处名单 -> `按三级授权口径计分 = 1.5`
- `申请三级授权 -> 1.5`
- `申请四级授权 -> 默认不单独评估组织风险，按 2.5 中性跳过`
- `若同时命中“跨组织单位申请”规则，则优先按跨组织规则评分（高于四级授权跳过策略）`
- `未命中授权等级 -> 2`
- `当申请人组织流程层级分类 = 属地组织，且申请组织编码对应的所属公司 = 权限对象所属公司 -> 2.5`
- `不检查组织范围 = TRUE` 且 `组织编码 = NULL -> 2.5`
- `申请一级授权，但申请人组织流程层级分类不在 (BG人行中心与学社, 人事远程交付中心) -> 0`
- `申请二级授权，但申请人组织流程层级分类不在 (BG人行中心与学社, 人事远程交付中心, 蝶发人行部, 业务单元本部) -> 0`
- `申请人所属组织单位 != 所申请组织所属组织单位`：
  - 若申请人分类是 `人事远程交付中心 / BG人行中心与学社 / 蝶发人行部 / 战区人行部门 / 属地服务站 -> 不进行跨组织单位申请的风险评估`
  - 若申请人分类是 `属地组织 / NULL -> 0.5`
  - 若申请人分类是 `万物云本部 -> 2.5`
  - 其他情形 -> `1`

说明：

- 该维度必须基于 `申请表组织范围`
- 不能只在单据级看“组织范围数量”
- 必须做到 `单据编号 + 角色编码 + 组织编码` 粒度判断
- 对 `人事远程交付中心 / BG人行中心与学社 / 蝶发人行部 / 战区人行部门 / 属地服务站` 这 5 类申请人组织流程层级分类：
  - 已确认其岗位工作中存在稳定的跨组织单位申请场景
  - 因此当仅命中“申请人所属组织单位 != 所申请组织所属组织单位”时，不单独输出跨组织单位风险
  - 仍需继续执行本维度下的授权等级、不检查组织范围等其他规则
- 自 `2026-03-19` 起，跨组织单位规则优先级高于四级授权跳过策略：
  - `四级授权` 仅在“非跨组织单位”场景下按 `2.5` 跳过
  - 只要命中跨组织单位规则，按跨组织规则的对应分值输出
- 对 `不检查组织范围 = TRUE` 的角色，组织编码为 `NULL` 时：
  - 已确认按 `2.5` 计分
  - 该场景不再归入 `未命中授权等级 = 2`
- 自 `2026-03-19` 起，战区代表处按二级授权降档为三级授权评分。当前名单：
  - `安徽战区代表处`、`福州战区代表处`、`厦门战区代表处`、`山东战区代表处`、`杭州战区代表处`
  - `东北战区代表处`、`鄂豫战区代表处`、`湘赣战区代表处`、`南京战区代表处`、`西北战区代表处`
  - `深圳战区代表处`、`津晋战区代表处`、`京冀战区代表处`、`福建战区代表处`、`川云战区代表处`
  - `广州战区代表处`、`琼桂战区代表处`、`佛山战区代表处`、`苏州战区代表处`、`上海战区代表处`、`渝贵战区代表处`
  - 配置入口：`automation/config/rules/risk_trust_constants.yaml` `target_org_l2_representative_office_org_units`

### 6.2 明细级输出

建议新增一张明细表：`申请单风险信任评估明细`

粒度建议为：

- `单据编号 + 角色编码 + 组织编码（允许 NULL） + 维度编码`

建议字段：

- `单据编号`
- `角色编码`
- `角色名称`
- `组织编码`
- `维度名称`
- `命中规则编码`
- `命中规则说明`
- `维度得分`
- `明细结论`
- `是否低分明细`
- `建议干预动作`
- `证据摘要`
- `证据快照`
- `评估批次号`
- `评估时间`

说明：

- 文档级结果负责“最终建议”
- 明细级结果负责“为什么是这个建议”
- 单据级必须输出 1 条 `总结论`
- 当任一维度、任一角色、任一角色-组织组合的得分 `<= 1` 时，必须额外输出“低分明细结论”
- “低分明细结论”不得只写分数，至少要说明：
  - 命中的角色
  - 命中的组织（如有）
  - 命中的维度
  - 命中的规则
  - 为什么低分
  - 需要谁干预、干预什么
- 典型示例：
  - `角色编码 DTX005 未命中权限列表主数据，维度=申请的权限，得分=0.5，需管理员补充权限列表后复核`
  - `角色编码 RLRZ001 申请一级授权，但申请人不属于允许的一线组织分类，维度=申请的组织，得分=0，建议拒绝`
- 这两张表是后续规则回放、抽样核查、业务复核的基础

### 6.3 评估结论输出规则

评估结论建议分两层输出：

1. 单据级总结论
2. 低分命中明细结论

单据级总结论的生成方式建议为：

- 先对所有角色、所有组织、所有维度取最低分
- 再按最低分映射 `总结论 + 建议动作`
- 总结论建议固定输出以下 4 档：
  - `拒绝`
  - `人工干预`
  - `仅关注`
  - `可信任`

低分命中明细结论的生成方式建议为：

- 所有 `<= 1` 分的命中都单独输出 1 条明细结论
- 若同一角色命中多条 `<= 1` 分规则，全部保留，不做吞并
- 单据级的 `低分明细结论` 字段可汇总为多条摘要文本，便于查询层直接查看

因此，本方案不是只输出一个分数，而是同时输出：

- `总结论`
- `建议动作`
- `最终信任分`
- `低分明细结论`

### 7.2 模块职责

## 8. 数据读取与计算链路

### 8.1 输入装配

每张单据装配为一个 `RawDocumentBundle`：

- `申请单基本信息`
- `申请单权限列表`
- `申请单审批记录`
- `申请表组织范围`
- `人员属性查询` 中申请人信息
- `权限列表` 中角色属性
- `组织属性查询` 中申请人组织、审批人组织、目标组织属性

### 8.2 Facts 计算

建议至少计算以下 facts：

- `applicant.hr_type`
- `applicant.process_level_category`
- `applicant.org_unit_name`
- `details[].role.permission_level`
- `details[].role.skip_org_scope_check`
- `details[].targets[].org_auth_level`
- `details[].targets[].org_unit_name`
- `approval.last_submit_at`
- `approval.effective_approvers`
- `approval.has_warzone_hr_in_current_round`
- `approval.has_warzone_hr_in_history`

### 8.3 评分执行

执行顺序建议：

1. 先算申请人维度
2. 再算角色维度
3. 再算角色-组织维度
4. 最后归并审批链维度
5. 单角色取最低分
6. 多角色整单再取最低分

### 8.4 最终动作映射

建议固定为：

- `0 -> 拒绝`
- `0.5 / 1 -> 人工干预`
- `1.5 / 2 -> 仅关注`
- `2.5 -> 可信任`

说明：

- 第一阶段输出的是“审批建议”，不是页面自动回写
- 页面自动驳回/自动通过必须等历史验证稳定后再谈

## 10. 分阶段实施路径

### 10.1 第一阶段：评分模型 MVP

目标：

- 完成文档级和明细级评分落库
- 输出建议动作
- 能解释每个低分是怎么来的

范围：

- 不做页面回写
- 不做通用 CEL 引擎
- 不做管理后台

### 10.2 第二阶段：历史回放与规则稳定

目标：

- 对现有待办单据批量回放
- 让业务核对“建议动作是否合理”
- 对附件中模糊口径做最终确认

重点：

- 审批链 `0` 分与 `0.5` 分的边界
- `NULL` 组织等级的业务解释

### 10.3 第三阶段：补采完结单据

目标：

- 新增“已办 / 历史完结单据”采集能力
- 把最终通过、最终驳回结果拉回数据库

没有这一阶段，就无法做以下事情：

- 评估模型命中准确率统计
- 评分阈值校准
- 监督式模型训练
- 自动通过策略上线

### 10.4 第四阶段：监督式校准或混合模型

当完结单据标签足够后，再考虑：

- 用历史结果对规则分进行校准
- 建立 `通过概率`
- 形成“规则分 + 概率分”的混合判断

本阶段不建议提前启动。

原因：

- 当前真实完结标签为 `0`
- 现在做监督学习只会制造伪模型

## 13. 最终结论

本项目实现“申请表是否建议通过审批”的最佳落地路径，不是直接上机器学习，也不是继续只停留在采集入库，而是：

1. 以附件 Excel 为准，先落地一个配置化、可解释的“风险与信任评估引擎”
2. 复用当前已成熟的 `人员属性查询 / 权限列表 / 组织属性查询 / 申请表组织范围`
3. 新增文档级与明细级评估结果表，形成稳定评估层
4. 先服务于“待办申请单的风险判断与人工审批辅助”
5. 再通过补采“已完结单据”，进入历史校准与监督式模型阶段

换句话说：

- 当前项目已经完成了“数据底座”
- 下一步真正要做的是“评估层”
- 而不是继续只做采集，也不是现在就跳到黑盒预测模型

# 文档编号101：权限自动审核规则配置规范

## 1. 目标
本规范用于定义本项目权限自动审核规则的配置标准，统一规则文件结构、字段命名、版本管理、条件表达、动作定义、文案模板、常量引用、审批节点配置与校验要求。

本规范与 `方案计划文档/100技术栈选择规范.md` 配套使用，重点回答以下问题：
- 规则文件如何组织
- 规则字段如何命名
- `YAML` 与 `CEL` 分别承载什么
- 复杂条件如何表达
- 审批动作如何配置
- 规则如何发布、灰度、回滚与追溯

## 2. 适用范围
本规范适用于以下配置文件及后续扩展文件：
- `automation/config/rules/audit_rules.yaml`
- `automation/config/rules/message_templates.yaml`
- `automation/config/rules/constants.yaml`
- `automation/config/rules/approval_matrix.yaml`

本规范适用于以下规则类型：
- 资格准入规则
- 组织范围规则
- 角色白名单/黑名单规则
- 字段取值合法性规则
- 审批链追加规则
- 风险提示规则
- 人工复核规则

## 3. 配置设计总原则

### 3.1 原则一：结构化优先
规则必须采用结构化配置，不允许在 YAML 中直接嵌入任意 Python 代码。

### 3.2 原则二：元数据与条件分离
- 规则元数据、优先级、动作、文案引用由 YAML 承载
- 复杂判断条件由 `when_cel` 承载
- 事实计算必须由 Python 完成，不得放入 YAML

### 3.3 原则三：规则与文案分离
- 规则中只引用 `message_code`
- 文案正文单独放在模板文件中维护
- 同一条文案允许被多条规则复用

### 3.4 原则四：规则与审批人分离
- 规则中只能配置审批角色、审批来源、审批节点语义
- 具体审批人由 Python 在运行期根据审批矩阵与主数据解析

### 3.5 原则五：可灰度、可回滚、可追溯
所有规则必须具备版本号、启停状态、生效日期、创建人/更新时间等元数据，保证可审计、可回放、可追溯。

## 4. 配置文件组织规范

### 4.1 推荐目录结构

```text
automation/
  config/
    rules/
      audit_rules.yaml
      message_templates.yaml
      constants.yaml
      approval_matrix.yaml
```

### 4.2 文件职责划分
- `audit_rules.yaml`
  - 存放审核规则主体
- `message_templates.yaml`
  - 存放驳回、提示、人工复核等文案模板
- `constants.yaml`
  - 存放角色白名单、组织白名单、关键词集合等可复用常量
- `approval_matrix.yaml`
  - 存放审批角色解析策略与组织映射策略

### 4.3 单文件容量建议
- 单个规则文件建议控制在 `100~300` 条规则以内
- 超过后按业务域拆分，例如：
  - `audit_rules_role.yaml`
  - `audit_rules_org.yaml`
  - `audit_rules_salary.yaml`
  - `audit_rules_approval.yaml`

## 5. 顶层结构规范

### 5.1 audit_rules.yaml 顶层结构

标准结构如下：

```yaml
version: "2026-03-12"
package: "permission_audit"
description: "权限自动审核规则包"
enabled: true
effective_from: "2026-03-12 00:00:00"
effective_to: null
owner: "HR数字化产品组"

imports:
  - "constants.yaml"
  - "message_templates.yaml"

defaults:
  hit_policy: "first_match_continue"
  unknown_fact_policy: "manual_review"
  timezone: "Asia/Shanghai"

rules:
  - id: "R001"
    name: "属地HR不可申请远程角色"
    enabled: true
    scope: "detail"
    category: "资格准入"
    severity: "high"
    priority: 100
    tags: ["属地HR", "远程角色", "资格"]
    when_cel: "applicant.hr_mode == 'local' && detail.role.is_remote_role"
    then:
      action: "reject"
      message_code: "REJ_LOCAL_NO_REMOTE"
```

### 5.2 顶层字段说明
- `version`
  - 规则包版本号，建议使用日期或语义版本
- `package`
  - 规则包名称
- `description`
  - 规则包用途说明
- `enabled`
  - 规则包总开关
- `effective_from`
  - 生效开始时间
- `effective_to`
  - 生效结束时间，可为空
- `owner`
  - 责任团队或责任人
- `imports`
  - 外部配置依赖文件
- `defaults`
  - 包级默认策略
- `rules`
  - 规则定义列表

## 6. 单条规则结构规范

### 6.1 标准结构

```yaml
- id: "R201"
  name: "战区HR跨组织追加审批"
  enabled: true
  scope: "document"
  category: "审批链"
  severity: "medium"
  priority: 80
  tags: ["跨组织", "战区HR", "追加审批"]
  owners: ["HR权限治理组"]
  effective_from: "2026-03-12 00:00:00"
  effective_to: null
  stop_on_hit: false
  when_cel: >
    applicant.duty_tags.exists(t, t in ["战区人力资源部HR", "人事交付服务组"]) &&
    document.is_cross_org
  then:
    action: "require_approvals"
    approvals:
      - approver_role: "战区权限对接人"
        org_source: "target_warzone"
      - approver_role: "战区人力资源部负责人"
        org_source: "target_warzone"
  explain:
    title: "战区HR跨组织申请需追加审批"
    summary: "命中跨组织申请规则，需追加战区权限对接人与战区人力资源部负责人审批"
```

### 6.2 规则字段定义
- `id`
  - 规则唯一编号，必须全局唯一
- `name`
  - 规则名称，建议体现业务语义
- `enabled`
  - 规则开关
- `scope`
  - 作用域，见第 7 节
- `category`
  - 业务类别，例如资格准入、组织范围、审批链、字段合法性
- `severity`
  - 风险等级，建议枚举：`low`、`medium`、`high`、`critical`
- `priority`
  - 执行优先级，数值越大优先级越高
- `tags`
  - 标签，便于分组、检索、灰度、报表统计
- `owners`
  - 规则责任团队/责任人
- `effective_from`
  - 单条规则生效时间
- `effective_to`
  - 单条规则失效时间
- `stop_on_hit`
  - 命中后是否停止后续同作用域规则执行
- `when_cel`
  - 复杂条件表达式
- `then`
  - 动作定义
- `explain`
  - 解释信息，用于命中展示、日志与回放说明

## 7. 作用域规范

### 7.1 允许的 scope 枚举
- `document`
- `detail`
- `approval`

### 7.2 含义说明
- `document`
  - 对整张申请单判断一次
- `detail`
  - 对每条权限申请明细分别判断
- `approval`
  - 对审批链、审批节点完整性进行判断

### 7.3 推荐使用方式
- 单据整体资格、跨组织、非 HR 准入类规则使用 `document`
- 角色、组织、调薪原因等明细维度规则使用 `detail`
- 审批节点追加与审批完整性检查使用 `approval`

## 8. 条件表达规范

### 8.1 统一字段
复杂条件统一使用 `when_cel` 表达，禁止在规则中混用 `when` 条件树与 `when_cel`。

### 8.2 表达式书写原则
- 优先引用标准 facts，不直接引用原始字段文本
- 表达式必须保持无副作用
- 不允许出现数据库操作、外部调用、时间睡眠等逻辑
- 超长表达式建议使用 YAML 多行文本 `>` 书写

### 8.3 允许使用的上下文对象
CEL 运行上下文建议统一暴露以下对象：
- `document`
- `applicant`
- `details`
- `detail`
- `approval`
- `constants`
- `context`

说明：
- `detail` 仅在 `scope=detail` 时有效
- `approval` 仅在 `scope=approval` 时有效
- `constants` 用于白名单、黑名单、关键字集合等常量引用

### 8.4 表达式示例

#### 示例 1：属地 HR 不可申请远程角色

```yaml
when_cel: "applicant.hr_mode == 'local' && detail.role.is_remote_role"
```

#### 示例 2：万御组织仅允许申请 6 个白名单角色

```yaml
when_cel: >
  detail.target_org.line == "万御" &&
  !(detail.role.code in constants.wy_allowed_role_codes)
```

#### 示例 3：非集约 HR 跨组织默认驳回

```yaml
when_cel: >
  document.is_cross_org &&
  !applicant.duty_tags.exists(t, t in ["一站式集约HR", "战区人力资源部HR", "人事交付服务组"])
```

#### 示例 4：存在任一万御非白名单角色且申请人非例外人群

```yaml
when_cel: >
  details.exists(d,
    d.target_org.line == "万御" &&
    !(d.role.code in constants.wy_allowed_role_codes)
  ) &&
  !applicant.special_tags.exists(t, t in ["BG人行COE", "远程主管"])
```

## 9. 动作配置规范

### 9.1 标准动作枚举
`then.action` 只允许以下取值：
- `reject`
- `warning`
- `manual_review`
- `require_approvals`

### 9.2 reject 动作

```yaml
then:
  action: "reject"
  message_code: "REJ_LOCAL_NO_REMOTE"
```

要求：
- 必须配置 `message_code`
- 文案正文不得直接写在规则里

### 9.3 warning 动作

```yaml
then:
  action: "warning"
  message_code: "WARN_LOAD_EXISTING_SCOPE_FIRST"
```

要求：
- 用于提醒，不直接拦截
- 可与 `require_approvals` 同时命中

### 9.4 manual_review 动作

```yaml
then:
  action: "manual_review"
  message_code: "MR_FACTS_INCOMPLETE"
  params:
    reason: "岗位职责无法准确识别"
```

要求：
- 用于 facts 缺失、条件不充分或命中高风险例外场景
- 必须给出人工复核原因

### 9.5 require_approvals 动作

```yaml
then:
  action: "require_approvals"
  approvals:
    - approver_role: "战区权限对接人"
      org_source: "target_warzone"
    - approver_role: "战区人力资源部负责人"
      org_source: "target_warzone"
```

要求：
- 只能配置审批角色，不允许直接配置审批人姓名
- `org_source` 由审批解析层解释到具体组织上下文

## 10. 审批配置规范

### 10.1 规则中的 approvals 结构
- `approver_role`
  - 审批角色名
- `org_source`
  - 组织来源，例如 `applicant_org`、`target_org`、`target_warzone`
- `required`
  - 是否必需，默认 `true`
- `order_hint`
  - 节点顺序提示，最终顺序由引擎统一归并

示例：

```yaml
approvals:
  - approver_role: "万科物业区域人力支持专家"
    org_source: "target_org"
    required: true
    order_hint: 30
  - approver_role: "万御认证中心负责人"
    org_source: "target_org"
    required: true
    order_hint: 40
```

### 10.2 禁止配置
规则文件中禁止直接出现：
- 具体审批人姓名
- 工号
- 手机号
- 固定邮箱

如确需固定到具体人员，必须在审批矩阵或例外映射表中维护，并留痕版本。

## 11. 文案模板规范

### 11.1 message_templates.yaml 结构

```yaml
version: "2026-03-12"
messages:
  REJ_LOCAL_NO_REMOTE:
    type: "reject"
    text: "仅远程共享可以申请，属地HR不可申请，请删除"
  REJ_OLD_ROLE:
    type: "reject"
    text: "为ehr1.0系统的老角色，须取消该类权限角色（申请类型选择“取消角色”），并按照权限操作指引中的新角色进行新增申请"
```

### 11.2 规范要求
- `message_code` 必须全局唯一
- `type` 必须与规则动作兼容
- 模板中可使用参数占位符，例如 `${role_name}`、`${org_name}`
- 同一模板允许被多条规则复用

## 12. 常量规范

### 12.1 constants.yaml 结构

```yaml
version: "2026-03-12"
constants:
  wy_allowed_role_codes:
    - "ROLE_WY_PAYROLL"
    - "ROLE_WY_ARCHIVE_EXPORT"
    - "ROLE_WY_SALARY_DETAIL"
    - "ROLE_WY_TURNOVER_REPORT"
    - "ROLE_WY_COMP_ROSTER"
    - "ROLE_WY_PERF_BP"

  special_exempt_tags:
    - "BG人行COE"
    - "远程主管"
```

### 12.2 使用要求
- 常量命名统一使用小写蛇形命名或小写下划线命名
- 角色编码集合优先使用编码，不用角色中文名做主判断键
- 白名单/黑名单/例外名单必须单独维护，不得散落在规则正文中

## 13. 命名规范

### 13.1 规则编号
建议规则编号格式：
- `R001`、`R002`：基础规则
- `R1xx`：资格准入类
- `R2xx`：组织与范围类
- `R3xx`：字段合法性类
- `R4xx`：审批链类
- `R9xx`：人工复核与兜底类

### 13.2 message_code
建议格式：
- `REJ_...`
- `WARN_...`
- `MR_...`

### 13.3 tags
标签建议体现以下维度：
- 组织线：`万御`、`万物云`、`BG人行`
- 岗位线：`属地HR`、`远程共享`、`BP`
- 风险线：`跨组织`、`大权限`、`老角色`

## 14. 优先级与冲突规范

### 14.1 优先级规则
- 数值越大，优先级越高
- 建议范围：`0~999`

### 14.2 推荐分层
- `900~999`：兜底与阻断类
- `700~899`：资格准入类
- `500~699`：组织与范围类
- `300~499`：字段合法性类
- `100~299`：审批链补充类
- `0~99`：提示类

### 14.3 冲突处理原则
- `reject` 高于 `manual_review`
- `manual_review` 高于 `require_approvals`
- `require_approvals` 高于 `warning`
- 多条审批规则命中时，审批节点取并集去重
- 白名单规则不得无边界覆盖全局规则

## 15. 校验规范

### 15.1 语法校验
规则发布前必须通过：
- YAML 语法校验
- Pydantic 模型校验
- `when_cel` 编译校验

### 15.2 语义校验
规则发布前还必须通过：
- `id` 唯一性校验
- `message_code` 引用存在性校验
- 常量引用存在性校验
- scope 与上下文变量兼容性校验
- 动作与字段完整性校验

### 15.3 示例
- `scope=detail` 时允许使用 `detail`
- `scope=document` 时禁止直接引用 `detail.role.code`
- `action=reject` 时必须有 `message_code`
- `action=require_approvals` 时必须有 `approvals`

## 16. 版本管理规范

### 16.1 版本字段
每个规则包必须包含：
- `version`
- `effective_from`
- `effective_to`

### 16.2 管理原则
- 新增规则或修改规则必须提升版本
- 已发布规则禁止无版本覆盖修改
- 运行结果必须带 `rule_version`

### 16.3 回滚原则
- 回滚以规则包版本为单位
- 回滚后不得删除旧版本审计日志

## 17. 灰度发布规范

### 17.1 灰度字段建议
可在规则中增加：

```yaml
rollout:
  mode: "shadow"
  sample_rate: 1.0
  include_tags: ["万御", "跨组织"]
```

### 17.2 模式说明
- `shadow`
  - 只记录结果，不执行拦截
- `enforce`
  - 正式执行拦截/追加审批

### 17.3 使用原则
- 新规则首次上线默认使用 `shadow`
- 通过历史回放和人工一致性验证后，再切换为 `enforce`

## 18. 典型规则示例

### 18.1 属地 HR 不可申请远程角色

```yaml
- id: "R101"
  name: "属地HR不可申请远程角色"
  enabled: true
  scope: "detail"
  category: "资格准入"
  severity: "high"
  priority: 820
  tags: ["属地HR", "远程角色"]
  when_cel: "applicant.hr_mode == 'local' && detail.role.is_remote_role"
  then:
    action: "reject"
    message_code: "REJ_LOCAL_NO_REMOTE"
```

### 18.2 定薪申请调薪原因仅允许入职定薪

```yaml
- id: "R301"
  name: "定薪申请调薪原因仅允许入职定薪"
  enabled: true
  scope: "detail"
  category: "字段合法性"
  severity: "high"
  priority: 620
  tags: ["定薪申请", "调薪原因"]
  when_cel: >
    detail.role.group == "定薪申请" &&
    detail.role.is_local_role &&
    detail.salary_change_reason != "入职定薪"
  then:
    action: "reject"
    message_code: "REJ_SALARY_ENTRY_ONLY"
```

### 18.3 万御组织申请非白名单角色驳回并追加审批

```yaml
- id: "R221"
  name: "万御组织非白名单角色驳回"
  enabled: true
  scope: "detail"
  category: "组织范围"
  severity: "critical"
  priority: 780
  tags: ["万御", "白名单角色"]
  when_cel: >
    detail.target_org.line == "万御" &&
    !(detail.role.code in constants.wy_allowed_role_codes) &&
    !applicant.duty_tags.exists(t, t in ["一站式集约HR"])
  then:
    action: "reject"
    message_code: "REJ_WY_ROLE_NOT_ALLOWED"
```

### 18.4 战区 HR 跨组织申请追加审批

```yaml
- id: "R401"
  name: "战区HR跨组织申请追加审批"
  enabled: true
  scope: "document"
  category: "审批链"
  severity: "medium"
  priority: 260
  tags: ["跨组织", "战区HR", "审批链"]
  when_cel: >
    applicant.duty_tags.exists(t, t in ["战区人力资源部HR", "人事交付服务组"]) &&

      - approver_role: "战区权限对接人"
        org_source: "target_warzone"

        org_source: "target_warzone"
```

## 19. 禁止事项
以下写法在规则配置中禁止出现：
- 在 `when_cel` 中拼接 SQL
- 在 YAML 中直接写 Python 函数名并期望动态执行
- 使用审批人姓名代替审批角色
- 使用角色中文名替代角色编码作为唯一主判断键
- 将自由文本正则作为主要准入依据
- 在未配置文案模板的情况下直接写死驳回语句

## 20. 结论
本项目规则配置规范最终要求如下：
- `YAML` 负责规则元数据、动作、审批角色与版本控制
- `CEL` 负责复杂条件表达
- 文案模板、常量、审批矩阵独立维护
- 配置必须先校验、后发布、再执行

该规范的目标不是让 YAML 变成“可执行代码”，而是让规则配置保持结构化、可评审、可版本化，同时将复杂条件交由 CEL 表达，将业务副作用与主数据解析留在 Python 层，形成稳定可维护的工程边界。

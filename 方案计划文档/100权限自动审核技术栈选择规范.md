# 文档编号100：权限自动审核技术栈选择规范

## 1. 目标
为本项目“全新申请单自动化、自动化审核”能力建立统一的技术栈选择规范，明确规则配置、复杂条件表达、事实计算、审批解析、结果落库与审计追踪的职责边界，避免业务规则散落在页面自动化流程与数据库写入逻辑中。

本规范的目标是形成一套可长期演进、可审计、可灰度、可回放的审核技术栈标准，支撑以下场景：
- 权限申请单自动审核
- 动态追加审批节点
- 驳回原因模板化输出
- 规则版本化发布与回放验证
- 历史单据复核与命中解释

## 2. 适用范围
本规范适用于以下目录及后续新增相关模块：
- `automation/scripts/`
- `automation/flows/`
- `automation/db/`
- `automation/config/`
- `automation/rules/`（建议新增）

本规范适用于以下业务对象：
- `申请单基本信息`
- `申请单权限列表`
- `申请单审批记录`
- `申请表组织范围`
- `在职花名册表`
- `组织列表`
- 后续新增审核规则、规则命中日志、审批矩阵、角色目录、岗位职责映射等表

## 3. 技术选型总原则

### 3.1 总体原则
本项目审核能力统一采用以下职责划分：
- `YAML` 管规则框架
- `CEL` 管复杂条件
- `Python` 管 facts 和动作执行

即：
- `YAML` 负责规则元数据、优先级、作用域、动作、文案模板引用
- `CEL` 只负责复杂判断条件表达
- `Python` 负责事实计算、主数据查询、审批解析、动作执行、日志落库、审计留痕

### 3.2 选择原则
技术栈选择必须同时满足以下要求：
- **动态性**：规则调整尽量不改业务代码
- **可读性**：规则评审时，业务与技术均可理解
- **可靠性**：结果可解释、可回放、可追溯
- **安全性**：禁止直接执行不受控脚本表达式
- **可替换性**：第三方表达式引擎需封装适配层，避免直接耦合
- **低侵入性**：尽量复用现有 `Python + Playwright + PostgreSQL` 主体架构

## 4. 技术栈选择结论

### 4.1 运行时与主语言
- 主语言统一选择：`Python 3.12`
- 页面自动化继续沿用现有：`Playwright`
- 数据库存储继续沿用现有：`PostgreSQL`
- 数据库驱动继续沿用现有：`psycopg 3`

选择原因：
- 项目现有自动化、Excel 导入、PG 写入均已基于 Python 实现
- 无需引入第二主语言，能最大化复用现有代码与运行环境
- Python 适合承担事实计算、主数据整合、审批派生、审计落库等复杂宿主逻辑

### 4.2 规则配置载体
- 规则配置载体统一选择：`YAML`
- 配置文件建议目录：`automation/config/rules/`
- 发布态可同步写入 PostgreSQL，运行时以“已发布版本”为准

选择原因：
- YAML 适合承载规则元数据、动作、文案引用、优先级、启停状态
- 比纯 JSON 更适合业务评审与人工维护
- 已与项目现有 `PyYAML` 依赖保持一致，接入成本最低

### 4.3 复杂条件表达引擎
- 复杂条件表达统一采用：`CEL`
- Python 侧接入原则：**必须通过适配层封装，不得在业务模块中直接依赖第三方 CEL API**
- 适配层建议模块：`automation/rules/cel_adapter.py`

选择原因：
- CEL 适合表达复杂布尔组合、集合判断、存在性判断、例外条件、白名单/黑名单等逻辑
- 相比持续扩展自研 YAML 条件树操作符，CEL 在复杂规则表达上更紧凑、更稳定
- CEL 为受限表达式语言，安全性显著优于直接使用 Python `eval`

### 4.4 配置校验与模型约束
- 规则配置模型统一采用：`Pydantic v2`
- 规则文件加载后必须先完成模型校验，再进入编译与执行阶段
- 所有规则文件必须具备显式版本号、规则编号、启停标志、优先级与动作定义

选择原因：
- YAML 本身仅是文本格式，必须增加结构化校验，避免错误配置直接进入运行态
- Pydantic 适合对配置对象、facts 对象、动作对象做强校验和类型约束
- 可为后续管理后台、规则导入接口、JSON Schema 导出提供统一模型基础

### 4.5 日志、审计与可追溯能力
- 运行日志继续采用现有：`logging`
- 审计记录统一落 PostgreSQL
- 所有审核结果必须记录：
  - `rule_version`
  - `rule_id`
  - `document_no`
  - `hit_scope`
  - `facts_snapshot`
  - `decision`
  - `message_code`
  - `message_text`
  - `required_approvals`
  - `executed_at`

选择原因：
- 审核系统的核心不是“算出结果”，而是“结果可解释、可回放、可复核”
- 结构化审计日志是灰度、纠偏、追责、版本比对的基础

## 5. 职责边界规范

### 5.1 YAML 负责内容
YAML 只允许承载以下内容：
- 规则基础元数据：`id`、`name`、`enabled`、`scope`、`priority`
- 条件表达：`when_cel`
- 动作定义：`reject`、`warning`、`manual_review`、`require_approvals`
- 文案模板引用：`message_code`
- 审批角色定义：`approver_role`、`org_source`
- 常量引用：白名单、黑名单、可复用角色集合

YAML 不允许承载以下内容：
- 数据库查询语句
- 页面元素定位逻辑
- 具体审批人查找逻辑
- 复杂 Python 函数调用
- 不受控脚本片段

### 5.2 CEL 负责内容
CEL 只允许负责以下内容：
- 多条件组合判断
- 列表存在性判断
- 集合成员判断
- 白名单/黑名单判断
- 例外场景排除判断
- 字段非空、枚举、匹配判断

CEL 不允许负责以下内容：
- 数据库写入
- 外部 HTTP 调用
- 审批节点回写
- 日志打印
- 审批人查询
- 业务副作用执行

### 5.3 Python 负责内容
Python 统一负责以下内容：
- 从现有采集表装配审核输入对象
- 聚合 `申请单基本信息`、`申请单权限列表`、`申请单审批记录`、`申请表组织范围`
- 关联 `在职花名册表`、`组织列表` 形成标准 facts
- 角色目录、岗位职责映射、组织归属、白名单等主数据加载
- 审批链解析、审批角色到审批人的映射
- 规则加载、编译、执行与结果合并
- 驳回文案渲染、日志落库、结果回放与差异比对

## 6. 标准模块分层规范
建议新增如下目录：

```text
automation/
  config/
    rules/
      audit_rules.yaml
      message_templates.yaml
  rules/
    __init__.py
    models.py
    loader.py
    validator.py
    cel_adapter.py
    facts.py
    engine.py
    actions.py
    approvals.py
    explain.py
    repositories.py
```

模块职责如下：
- `models.py`：规则配置模型、facts 模型、命中结果模型
- `loader.py`：加载 YAML 并转换为模型对象
- `validator.py`：配置合法性校验、引用检查、版本检查
- `cel_adapter.py`：封装 CEL 编译与求值接口
- `facts.py`：从数据库与主数据计算标准 facts
- `engine.py`：规则排序、执行、命中合并、优先级裁决
- `actions.py`：驳回、警告、人工复核、追加审批等动作构造
- `approvals.py`：审批角色解析到审批人
- `explain.py`：规则命中说明、证据输出、回放对比说明
- `repositories.py`：数据库读取与审计结果落库

## 7. 规则配置规范

### 7.1 推荐规则结构
标准规则结构定义如下：

```yaml
version: "2026-03-12"
constants:
  wy_allowed_role_codes:
    - "ROLE_WY_PAYROLL"
    - "ROLE_WY_ARCHIVE_EXPORT"

rules:
  - id: "R001"
    name: "属地HR不可申请远程角色"
    enabled: true
    scope: "detail"
    priority: 100
    when_cel: "applicant.hr_mode == 'local' && detail.role.is_remote_role"
    then:
      action: "reject"
      message_code: "REJ_LOCAL_NO_REMOTE"
```

### 7.2 必填字段规范
每条规则必须包含：
- `id`
- `name`
- `enabled`
- `scope`
- `priority`
- `when_cel`
- `then.action`

以下字段按动作类型决定是否必填：
- `then.message_code`
- `then.approvals`
- `then.params`

### 7.3 scope 规范
规则作用域只允许以下枚举：
- `document`
- `detail`
- `approval`

说明：
- `document`：整单级别判断
- `detail`：逐明细判断
- `approval`：审批链完整性或追加审批判断

## 8. Facts 计算规范

### 8.1 Facts 设计原则
Facts 是审核引擎唯一可信输入，必须先标准化、后判断。

规则不得直接依赖原始脏文本字段，必须优先依赖标准 facts，例如：
- `applicant.hr_mode`
- `applicant.duty_tags`
- `applicant.special_tags`
- `document.is_cross_org`
- `detail.role.is_remote_role`
- `detail.role.is_legacy`
- `detail.role.group`
- `detail.target_org.line`

### 8.2 Facts 来源规范
Facts 来源按优先级排序如下：
1. 权限申请单采集数据
2. 组织主数据表 `组织列表`
3. 在职人员主数据表 `在职花名册表`
4. 角色目录表
5. 岗位职责映射表
6. 白名单、特批、例外映射表

### 8.3 Facts 计算约束
- Facts 计算必须幂等
- Facts 计算函数不得产生业务副作用
- Facts 缺失时必须显式输出缺失标志，不得默认当作“通过”
- 无法判定的场景统一进入 `manual_review`

## 9. 动作执行规范

### 9.1 标准动作枚举
审核引擎动作只允许以下枚举：
- `reject`
- `warning`
- `manual_review`
- `require_approvals`

### 9.2 动作优先级
动作冲突时按以下优先级处理：
- `reject` > `manual_review` > `require_approvals` > `warning`

### 9.3 执行约束
- 任一明细命中 `reject`，整单默认驳回
- 多条审批规则命中时，审批节点取并集去重
- 白名单仅允许覆盖被明确声明的规则，不允许全局绕过
- 动作执行层不得再回头修改 facts 与规则条件

## 10. 第三方库接入规范

### 10.1 已固定沿用库
以下库属于当前项目既有基础能力，继续沿用：
- `playwright`
- `psycopg`
- `PyYAML`
- `openpyxl`
- `xlrd`

### 10.2 新增推荐库
本规范建议新增以下依赖：
- `pydantic`：规则模型与配置校验
- `pytest`：规则引擎单元测试与回放测试
- `pytest-cov`：覆盖率统计
- `freezegun` 或等价时间冻结工具：固定时间相关测试

### 10.3 CEL 接入规范
截至 `2026-03-12`，优先采用**具备官方来源或官方宣布支持的 Python CEL 实现**，但业务代码不得直接绑定具体三方实现，必须满足：
- 引擎接入点只出现在 `cel_adapter.py`
- 业务侧只依赖统一接口，如：
  - `compile_expression(expr: str)`
  - `evaluate_expression(program, context: dict)`
- 若后续替换 CEL 实现，不得影响业务规则文件结构与调用方模块

建议新增以下表：
- `audit_rule_package`
- `audit_rule_definition`
- `audit_message_template`
- `权限列表`
- `audit_position_duty_map`
- `audit_approval_matrix`
- `audit_exception_whitelist`
- `audit_decision_log`
- `audit_rule_hit_log`

存储规范如下：
- 规则定义保留发布版本与生效时间

- 审计表必须支持按 `document_no`、`rule_version`、`executed_at` 查询
- 规则与文案分离存储

## 12. 测试与发布规范

### 12.1 测试分层
测试必须至少分为三层：
- **配置校验测试**：校验 YAML 合法性、引用完整性、枚举正确性
- **规则单元测试**：对单条规则输入 facts，验证命中与动作输出
- **历史回放测试**：对历史单据重放，比较人工结果与自动结果偏差

### 12.2 发布流程
规则发布必须经过以下步骤：
1. YAML 语法检查
2. Pydantic 模型校验
3. CEL 编译检查
4. 规则单测通过
5. 历史回放通过
6. 灰度发布
7. 正式发布

### 12.3 灰度规则
- 新规则初次上线默认采用“建议结果”模式
- 灰度期内仅记录命中结果，不直接触发自动驳回
- 通过人工审核一致性验证后，再切换为正式拦截

## 13. 禁止事项
以下实现方式在本项目中禁止采用：
- 将业务审核规则硬编码在 `flows/` 页面流程中
- 在数据库写入逻辑中夹带审核判断
- 使用 Python `eval`、`exec` 执行规则表达式
- 将审批人姓名直接写死在规则文件中
- 用自由文本正则替代结构化 facts 作为主要判断依据
- 在未记录命中证据的情况下直接自动驳回

## 14. 落地顺序建议

### 14.1 第一阶段
- 建立 `automation/rules/` 模块骨架
- 新增规则 YAML、文案模板 YAML
- 建立 facts 计算层
- 实现纯本地评估与日志落库

### 14.2 第二阶段
- 引入 CEL 适配层
- 完成复杂条件迁移
- 增加审批矩阵与审批人解析

### 14.3 第三阶段
- 接入历史回放与差异分析
- 接入灰度发布
- 接入规则管理后台或发布脚本

## 15. 结论
本项目审核能力的技术栈选择规范最终确定为：
- **YAML 管规则框架**
- **CEL 管复杂条件**
- **Python 管 facts 和动作执行**
- **PostgreSQL 管规则版本、主数据、审计结果与回放记录**

该方案在本项目现有 `Python + Playwright + PostgreSQL` 架构上改造成本最低，同时兼顾复杂规则表达能力、配置化程度、运行安全性与长期可维护性，是当前阶段最平衡、最适合落地的选择。

# 方案编号404：对话工作台单据审批能力建设方案文档

## 1. 背景

`403` 已将 `chat-workspace` 从通用聊天入口升级为“可查询正式工作台实时数据”的对话入口，但当前能力仍停留在**读**：

1. 助手可以查询 `处理单据工作台`、`采集工作台`、单据详情等正式数据。
2. 助手不能直接发起“批准/驳回”这类真实写动作。
3. 用户若想在对话中完成审批，仍需切回 `处理单据` 页面手工执行。

与此同时，项目内其实已经存在完整的正式审批链路：

1. 前端 `webui/src/pages/ProcessDocumentsPage.tsx`
   - 已支持 `批准`、`驳回`、`dry-run`。
2. 前端 `webui/src/services/api.ts`
   - 已定义 `approveProcessDocument(documentNo, { action, approvalOpinion, dryRun })`。
3. 后端 `automation/api/routers/documents.py`
   - 已暴露：
     - `POST /api/documents/process-workbench/{document_no}/approval`
     - `POST /api/documents/process-dashboard/{document_no}/approval`
4. 后端 `automation/api/process_dashboard.py`
   - `approve_process_document()` 已统一接管审批动作。
5. 自动化流 `automation/flows/document_approval_flow.py`
   - `execute_action()` 已支持：
     - `approve -> 同意`
     - `reject -> 驳回至已选节点`
     - `dry-run`

因此，`404` 的目标不是新造审批实现，而是把**现有正式审批能力**安全接入 `chat-workspace`。

---

## 2. 现状确认

### 2.1 对话链路当前仍是读能力优先

当前 `automation/chat/tool_registry.py` 中仅注册了 3 个正式工具：

1. `get_process_workbench`
2. `get_process_document_detail`
3. `get_collect_workbench`

它们全部是只读工具。

当前 `automation/chat/router_models.py` 的 `RouterDecision.route` 也仅包含：

1. `tool_first`
2. `direct_answer`
3. `general_chat`

这意味着当前路由模型**没有“审批写动作”分支**。

### 2.2 审批正式口径已经稳定

截至 `2026-03-22`，审批能力已有以下稳定工程事实：

1. `automation/api/process_dashboard.py`
   - `approve` 成功后将待办处理状态落为 `已处理`
   - `reject` 成功后将待办处理状态落为 `已驳回`
   - 当单据最近一次待办同步结果已是 `已处理/已驳回` 时，会直接拒绝再次执行
2. `automation/flows/document_approval_flow.py`
   - `dry-run` 会完成页面连通、决策切换、意见写入与回滚，但**不点击提交**
3. 审批动作的正式请求契约已固定为：

```json
{
  "action": "approve" | "reject",
  "approvalOpinion": "审批意见",
  "dryRun": false
}
```

### 2.3 已有方案文档可直接复用

本方案与以下执行契约直接相关：

1. `403`：对话工作台实时数据直答能力建设方案
2. `105`：处理单据页审批单据页签与批准写回方案
3. `208`：处理单据页驳回功能实现方案
4. `109`：风险总览风险事项与审批意见提示词标准化方案

`404` 的定位是：

1. 复用 `105/208` 的正式审批执行链路
2. 复用 `109` 的审批意见口径作为后续增强参考
3. 在 `403` 的对话路由框架上新增“高风险写动作安全编排”

---

## 3. 方案目标

`404` 目标是让助手在 `chat-workspace` 中具备**单据级审批辅助与执行能力**，同时满足以下约束：

1. 支持单据级 `批准`、`驳回` 与 `dry-run`。
2. 明确区分“助手建议”与“助手真实执行”。
3. 真实执行必须经过**二次确认**，不得首轮消息直接落提交。
4. 必须复用现有正式审批服务，不新增第二条并行审批链路。
5. 必须保留日志、截图、执行状态、来源接口等审计信息。
6. 对缺失参数、单据状态异常、审批意见缺失等情况，必须先追问或拒绝，不得猜测执行。

---

## 4. 范围与边界

### 4.1 纳入范围

第一阶段纳入以下能力：

1. 单条单据审批意图识别
2. 单条单据 `批准/驳回` 的审批计划生成
3. 基于正式数据源的审批前校验
4. 基于 `109` 口径的驳回意见建议稿生成
5. `dry-run` 连通性验证
6. 用户确认后的真实提交
7. 审批执行结果在对话中回显

### 4.2 明确排除

以下能力不纳入 `404` 第一阶段：

1. 批量审批
2. 多单据自然语言模糊解析
3. 未提供单据编号时的自动猜测
4. 未提供审批意见时的自动真实提交
5. `加签`、`转办`、`不同意并终止`
6. 助手自行决定审批结论后直接提交
7. 绕开现有 `process-workbench approval API` 直接操控 Playwright

### 4.3 审批意见边界

第一阶段建议采用保守口径：

1. `approve/reject` 真实执行前，必须有明确 `approvalOpinion`
2. 若用户未提供审批意见，助手只可：
   - 对 `approve` 场景继续追问用户补齐
   - 对 `reject` 场景可基于 `109` 生成驳回意见建议稿
3. 自动生成的驳回意见仅作为建议稿，不得视为用户最终确认输入
4. 真实提交时，以用户最终明确输入或确认采用的审批意见为准

---

## 5. 方案原则

本方案采用以下原则：

1. **正式链路复用原则**
   - 对话侧不重写审批逻辑，统一复用 `approve_process_document()`
2. **读写分离原则**
   - 查询类问题继续走 `403` 的通用工具路由
   - 审批类问题走独立的“审批计划 -> 二次确认 -> 执行”链路
3. **二次确认原则**
   - 用户第一次表述审批意图时，助手只能生成计划，不能直接提交
4. **高风险工具不直暴露原则**
   - 不把真实写动作直接放入通用 `tool_registry` 让模型自由调用
5. **缺参不猜原则**
   - `documentNo`、`action`、`approvalOpinion` 任一缺失时，不得执行
6. **可审计原则**
   - 每次审批计划、确认、执行、失败都必须有留痕

---

## 6. 总体设计

### 6.1 分层架构

推荐将对话审批拆成四层：

1. 路由层
   - 识别当前是否为审批意图
2. 计划层
   - 基于正式数据源生成“待确认审批计划”
3. 确认层
   - 要求用户显式确认或 dry-run
4. 执行层
   - 复用正式审批 API 发起真实动作

对应职责如下：

1. `automation/chat/router_models.py`
   - 扩展审批意图结构化输出
2. `automation/chat/service.py`
   - 负责编排审批计划、确认与执行
3. `automation/chat/approval_plan_store.py`
   - 保存待确认审批计划
4. `automation/api/process_dashboard.py`
   - 继续作为正式审批执行入口

### 6.2 为什么不把审批写动作直接注册成通用 Tool

原因如下：

1. 当前通用 Tool 体系默认面向查询，不带提交确认门禁
2. `approve/reject` 是真实业务写动作，不能让模型在 `tool_first` 分支里直接落提交
3. 审批需要“先校验 -> 再回显 -> 再确认 -> 再执行”，不是单次 Tool Call 能安全覆盖的链路

因此建议：

1. `tool_registry.py` 继续仅保留只读正式工具
2. 审批由 `ChatService` 的专用分支接管
3. 真正的写入动作只允许在“待确认审批计划”上执行

## 7. 路由扩展设计

### 7.1 `RouterDecision` 扩展

建议在 `automation/chat/router_models.py` 中新增审批请求模型：

```json
{
  "route": "approval_prepare",
  "selectedSkills": ["clawcheck-project"],
  "selectedReferences": ["process-workbench", "process-approval"],
  "toolCalls": [],
  "answerMode": "templated",
  "confidence": 0.93,
  "missingInputs": [],
  "clarificationQuestion": "",
  "reason": "用户正在要求助手发起单据审批动作。",
  "approvalRequest": {
    "documentNo": "RA-20260310-00019845",
    "action": "approve",
    "approvalOpinion": "同意，按当前职责范围处理。",
    "dryRun": false
  }
}
```

建议新增字段：

1. `route`
   - 新增枚举值：`approval_prepare`
2. `approvalRequest`
   - `documentNo`
   - `action`
   - `approvalOpinion`
   - `dryRun`

### 7.2 路由约束

路由层必须遵守：

1. 审批意图命中时，只能输出 `approval_prepare`
2. `approval_prepare` 不得包含真实写工具调用
3. 若审批关键参数缺失：
   - `missingInputs` 必须填充
   - `approvalRequest` 只允许填已明确值
   - `toolCalls` 为空

### 7.3 Prompt 约束

建议在 `automation/chat/router_prompt.py` 中补充规则：

1. 当用户要求“批准/驳回/审批/验证审批连通性”时，优先考虑 `approval_prepare`
2. 不允许把审批类请求降级成普通 `tool_first`
3. 不允许虚构 `documentNo`
4. 不允许在缺少审批意见时默认补“同意”并直接执行

## 8. 审批计划设计

### 8.1 审批计划的定位

审批计划不是最终执行，而是对一次审批请求的**待确认快照**。

它的作用是：

1. 向用户回显将要执行的对象和动作
2. 固化本轮审批参数
3. 为后续 `dry-run` 与真实提交提供受控入口

### 8.2 审批计划生成流程

当路由结果为 `approval_prepare` 时，`ChatService` 应执行：

1. 校验 `approvalRequest`
2. 调用只读正式数据源拉取单据详情：
   - `get_process_document_detail(documentNo)`
3. 校验单据当前是否允许审批：
   - 单据存在
   - 工作台详情可查询
   - `待办处理状态` 不是 `已处理/已驳回`
4. 若 `action=reject` 且用户未提供 `approvalOpinion`
   - 基于 `109` 与单据 `风险总览` 生成驳回意见建议稿
5. 生成审批计划
6. 回显给用户确认

### 8.3 审批计划内容

建议计划对象至少包含：

```json
{
  "planId": "approval-plan-20260322-001",
  "sessionId": "chat-session-id",
  "documentNo": "RA-20260310-00019845",
  "action": "approve",
  "approvalOpinion": "同意，按当前职责范围处理。",
  "approvalOpinionSource": "user_input | suggested_from_109",
  "requestedDryRun": false,
  "status": "pending_confirmation",
  "snapshot": {
    "todoProcessStatus": "待处理",
    "documentStatus": "审批中",
    "riskSummary": "风险总览摘要",
    "assessmentBatchNo": "..."
  },
  "confirmCommand": "确认审批计划 approval-plan-20260322-001",
  "dryRunCommand": "验证审批计划 approval-plan-20260322-001",
  "cancelCommand": "取消审批计划 approval-plan-20260322-001",
  "createdAt": "2026-03-22 10:30:00",
  "expiresAt": "2026-03-22 10:40:00"
}
```

### 8.4 审批计划存储

第一阶段建议不改数据库结构，采用运行时文件持久化：

1. 目录：`automation/runtime/chat/approval-plans/`
2. 单文件：`{sessionId}_{planId}.json`

原因：

1. 与 `403` 的运行时目录设计一致
2. 不引入新的 PostgreSQL DDL
3. 足以支撑当前单实例对话工作台

计划状态建议包含：

1. `pending_confirmation`
2. `dry_run_succeeded`
3. `submitted`
4. `failed`
5. `canceled`
6. `expired`

## 9. 二次确认与执行设计

### 9.1 第一阶段确认方式

第一阶段建议采用**精确确认命令**，不强依赖前端新增确认按钮。

助手在生成审批计划后，回复示例：

1. `确认审批计划 <planId>`：发起真实提交
2. `验证审批计划 <planId>`：先执行 dry-run
3. `取消审批计划 <planId>`：放弃本次执行

这是第一阶段唯一允许的确认硬编码语法，作用仅限：

1. 待确认计划执行
2. 待确认计划取消

它不替代路由模型，也不用于一般意图识别。

补充约束：

1. 若当前计划中的 `approvalOpinionSource = suggested_from_109`
   - 助手回显时必须明确标注“当前为建议稿”
2. 用户可直接补发新的审批意见文本覆盖建议稿
3. 覆盖后再执行 `确认审批计划 <planId>` 时，以覆盖后的用户输入为准

### 9.2 为什么第一阶段不用前端按钮

原因如下：

1. 当前 `ChatWorkspacePage.tsx` 只承载文本消息与 SSE 事件
2. 第一阶段先用精确确认命令可最小化前端改造
3. 能先打通后端执行与安全门禁

第二阶段再补：

1. `确认提交`
2. `仅做 dry-run`
3. `取消计划`

按钮式交互。

### 9.3 执行前再校验

用户输入 `确认审批计划 <planId>` 或 `验证审批计划 <planId>` 后，系统不得直接执行，必须先再次校验：

1. 计划是否存在
2. 计划是否属于当前 session
3. 计划是否未过期
4. 计划状态是否仍为可执行
5. 单据当前详情是否仍允许审批
6. 当前 session 是否已有执行中的 run

若再校验失败，应直接返回拒绝信息并终止本次执行。

### 9.4 真实执行入口

执行层统一复用现有正式审批服务：

1. 内部函数：`automation/api/process_dashboard.py -> approve_process_document()`
2. 或等价 HTTP 路径：
   - `POST /api/documents/process-workbench/{document_no}/approval`

禁止：

1. 在 chat 层直接 new 一套 Playwright 审批逻辑
2. 在 chat 层绕过 `approve_process_document()` 直接操作 `DocumentApprovalFlow`

### 9.5 `dry-run` 执行口径

当用户执行 `验证审批计划 <planId>` 时：

1. 使用计划中的 `documentNo/action/approvalOpinion`
2. 调用正式审批服务，强制 `dryRun=true`
3. 返回 `confirmationType=dry_run`
4. 保留计划为“可继续真实提交”状态，直到：
   - 用户确认真实提交
   - 或计划过期/取消

### 9.6 真实提交口径

当用户执行 `确认审批计划 <planId>` 时：

1. 使用计划中的 `documentNo/action/approvalOpinion`
2. 调用正式审批服务，强制 `dryRun=false`
3. 成功后将计划状态更新为 `submitted`
4. 对话中回显：
   - `action`
   - `status`
   - `confirmationType`
   - `message`
   - `logFile`
   - `screenshotFile`

## 10. 审批意见口径

### 10.1 第一阶段口径

第一阶段采用保守规则：

1. 用户已明确给出审批意见时，按用户意见进入审批计划
2. `approve` 场景下，用户未给出审批意见时：
   - 助手只追问，不提交
3. `reject` 场景下，用户未给出审批意见时：
   - 助手可基于 `109` 与 `风险总览` 生成驳回意见建议稿
   - 建议稿进入审批计划，但必须标记为 `suggested_from_109`
   - 用户可接受、修改或重写
4. 最终真实提交时，一律以用户最终输入为准，不以系统建议稿直接替代用户确认

### 10.2 第二阶段增强

在不改变第一阶段安全门禁前提下，第二阶段继续增强：

1. 优化 `109` 建议稿的可读性与去重逻辑
2. 支持对建议稿做结构化展示与一键采纳
3. 允许用户在聊天侧针对建议稿做定向修改，而不是整段重写

## 11. ChatService 改造点

### 11.1 建议新增模块

建议新增：

1. `automation/chat/approval_models.py`
2. `automation/chat/approval_plan_store.py`

建议修改：

1. `automation/chat/router_models.py`
2. `automation/chat/router_prompt.py`
3. `automation/chat/service.py`
4. `automation/chat/skills/clawcheck-project/references/process-approval.md`

### 11.2 `service.py` 新职责

`ChatService` 建议新增以下编排能力：

1. `prepare_approval_plan()`
2. `execute_pending_approval_plan()`
3. `cancel_pending_approval_plan()`
4. `try_handle_approval_confirmation_command()`

### 11.3 执行顺序

建议顺序固定为：

1. 收到用户消息
2. 先检查是否为精确确认命令
3. 若是确认命令：
   - 直接进入待确认计划执行链路
4. 若不是确认命令：
   - 进入常规路由模型
5. 若路由结果为 `approval_prepare`
   - 生成审批计划
6. 若路由结果为读能力
   - 继续走 `403`

这样可保证：

1. 确认动作不依赖模型二次理解
2. 一般问题仍由模型路由
3. 审批动作的最后一步执行边界明确

## 18. 风险控制清单

为避免 `404` 落地后出现误触发、重复提交、跨会话误执行或建议稿误提交，第一阶段补充以下控制要求。

### 18.1 风险一：审批写动作被误当作普通 Tool 直接执行

风险描述：

1. 若把 `approve/reject` 直接注册进通用 `tool_registry`，模型可能绕过二次确认直接触发真实写动作
2. `403` 当前 Tool 体系默认面向只读查询，不具备提交门禁

控制方案：

1. `tool_registry.py` 继续只保留只读正式工具，不注册真实审批写 Tool
2. `router_models.py` 新增 `approval_prepare` 路由，但该路由 `toolCalls` 必须为空
3. `service.py` 后端校验新增硬约束：
   - 只有“待确认审批计划执行分支”允许调用正式审批 API
   - 普通 `tool_first/direct_answer/general_chat` 分支不得触发 `approve_process_document()`
4. `CLAWCHECK_CHAT_APPROVAL_DRY_RUN_ONLY=true` 时，即使收到 `确认审批计划` 命令，也必须降级为拒绝真实提交

验收标准：

1. 路由命中审批意图时，不出现任何真实写 Tool 名称
2. 首轮用户消息不会直接触发正式审批 API
3. 灰度阶段所有确认命令均不可绕过 `dry-run only` 限制

### 18.2 风险二：审批计划状态不严导致重复提交或跨会话误执行

风险描述：

1. 同一 `planId` 被多次确认，可能造成重复执行
2. 不同会话引用同一 `planId`，可能造成跨会话误执行
3. 计划过期后继续执行，可能导致快照与真实单据状态脱节

控制方案：

1. 审批计划持久化文件名固定为：`{sessionId}_{planId}.json`
2. 计划读取时必须同时校验：
   - 当前会话 `sessionId`
   - `planId`
   - `status`
   - `expiresAt`
3. 第一阶段状态机固定为：
   - `pending_confirmation`
   - `dry_run_succeeded`
   - `submit_requested`
   - `submitted`
   - `failed`
   - `canceled`
   - `expired`
4. 状态流转只允许单向前进：
   - `pending_confirmation -> dry_run_succeeded`
   - `pending_confirmation -> submit_requested`
   - `pending_confirmation -> canceled`
   - `dry_run_succeeded -> submit_requested`
   - `dry_run_succeeded -> canceled`
   - `submit_requested -> submitted | failed`
   - 任意未终态 -> expired
5. `submit_requested/submitted/canceled/expired` 状态的计划不得再次执行
6. 执行前必须重新查询正式单据状态：
   - 若 `待办处理状态` 已是 `已处理/已驳回`
   - 或单据详情已不可审批
   - 则本次计划转为 `failed` 并拒绝执行
7. `approval_plan_store.py` 必须提供基于文件锁或“读出状态后原子改写”的并发保护，避免双确认并发穿透

验收标准：

1. 相同 `planId` 重复确认，只有第一次允许进入执行
2. 非当前 `sessionId` 的会话无法执行其他会话计划
3. 过期计划返回明确错误，不进入正式审批 API

### 18.3 风险三：前端卡片/按钮过早介入导致改动面过大

风险描述：

1. 若第一阶段同时引入按钮、卡片、表单覆盖、结构化回显，联调面会同时扩散到前后端与路由事件协议
2. 会延长 `404` 从“后端闭环”到“可灰度上线”的路径

控制方案：

1. 第一阶段保持文本型确认命令，不新增前端确认按钮
2. `ChatWorkspacePage.tsx` 仅补充状态文案：
   - `approval_preparing`
   - `approval_confirmation_required`
   - `approval_dry_running`
   - `approval_submitting`
3. 第一阶段不变更 `chat` HTTP API 结构；继续复用现有：
   - `POST /api/chat/sessions/{session_id}/messages`
   - `GET /api/chat/sessions/{session_id}/stream`
4. 结构化计划卡片、按钮交互统一延后到 Phase 2

验收标准：

1. Phase 1 前端改造文件范围收敛在 `ChatWorkspacePage.tsx`、`types/chat.ts`
2. 无需新增 chat 专用确认 API 即可完成闭环

### 18.4 风险四：`109` 建议稿被误当成最终审批意见直接提交

风险描述：

1. `109` 输出的风险提示词本质是建议稿输入，不是最终审批意见
2. 若系统直接拿建议稿执行驳回，会突破“用户最终确认”边界

控制方案：

1. 计划对象新增 `approvalOpinionSource`
   - `user_input`
   - `suggested_from_109`
   - `user_override_after_suggestion`
2. 当 `approvalOpinionSource = suggested_from_109` 时：
   - 助手回显中必须明确标注“当前为建议稿，需用户确认或改写”
   - `确认审批计划 <planId>` 不得直接触发真实提交
3. 用户必须以新的自然语言消息补发审批意见，或明确发送覆盖命令后，计划才可进入可提交状态
4. 真实提交时，正式请求中的 `approvalOpinion` 只能取：
   - 用户原始输入
   - 或用户覆盖后的最终文本
   - 不得直接取纯建议稿

验收标准：

1. `reject` 且无审批意见时，系统只生成建议稿计划，不会真实提交
2. 用户未覆盖建议稿时，`确认审批计划` 返回阻断提示
3. 用户覆盖后再确认，提交文本等于覆盖后的最终文本

## 23. 回滚细则补充

在第 17 节基础上，若线上出现异常，按以下顺序收敛：

1. 立即设置 `CLAWCHECK_CHAT_APPROVAL_ENABLED=false`
2. 保留 `403` 实时查询能力，不回滚 chat 基础框架
3. 若仅真实提交异常但 `dry-run` 正常：
   - 设置 `CLAWCHECK_CHAT_APPROVAL_DRY_RUN_ONLY=true`
   - 暂停真实提交，保留验证链路
4. 若建议稿生成异常：
   - 关闭建议稿分支
   - 驳回场景改为强制用户手填审批意见
5. 保留：
   - 审批计划文件
   - `approval_*.json`
   - 失败截图
   - chat SSE 执行日志
6. 复盘完成前，不直接删除运行时计划文件

## 24. 结论

补充上述控制后，`404` 的第一阶段可以收敛为一条更可执行的正式链路：

1. 模型只负责识别审批意图与缺参
2. 后端只负责生成审批计划、执行确认门禁、调用正式审批 API
3. 前端第一阶段只负责文本与状态回显
4. 建议稿只作为建议，不作为最终提交依据

按本完善方案落地后，`chat-workspace` 将在不破坏 `403/105/208/109` 既有口径的前提下，形成：

1. 可灰度
2. 可审计
3. 可回滚
4. 可逐阶段增强

# 方案编号301：开发环境标准化与本地一键启停治理方案文档

## 1. 方案目标

本方案用于解决 `clawcheck` 当前本地开发环境中存在的解释器版本不一致、前后端启动入口分散、服务状态不可统一观测、最小质量门禁缺失、运行产物忽略规则不完整等问题。

本方案交付后，应达到以下目标：

- Windows 本地开发环境以单一 Python 版本口径运行，默认要求 `Python 3.12.x`
- `.venv-win` 的创建、校验、重建规则明确，避免出现“文档写 3.12，实际跑 3.10”的隐性偏差
- Node.js 与 npm 的安装位置、发现顺序、PATH 策略明确，避免出现“已安装但终端不可直接使用”的状态
- 本地开发支持统一入口进行 `start / stop / restart / status`
- 前端开发态提供独立脚本入口，不再要求手工进入 `webui` 目录执行命令
- API、任务守护进程、PostgreSQL、Vite 的 PID、端口、健康检查地址、日志路径有统一约定
- 仓库具备最小质量门禁，至少覆盖 Python 语法可编译、基础测试、前端构建
- 开发运行产生的日志、PID、错误输出文件不再污染 Git 工作区

## 2. 背景与当前问题

截至 `2026-03-21`，仓库与当前本机环境已经确认存在以下问题：

### 2.1 Python 版本与虚拟环境口径不一致

- `automation/README.md` 当前要求本地使用 `Python 3.12.x`
- `automation/scripts/install_windows_env.ps1` 默认参数也为 `3.12`
- 当前本机 `.venv-win/pyvenv.cfg` 实际显示为 `3.10.11`
- 现有安装脚本仅在 venv 不存在时创建，不会在版本不一致时自动重建

这会导致：

- 解释器链路不透明
- 排障时容易误判运行进程的真实 Python 版本
- 新同事按文档安装后与已有环境行为不一致

### 2.2 本地启动入口分散

当前仓库虽已有以下单点脚本：

- `automation/scripts/start_api.ps1`
- `automation/scripts/start_task_daemon.ps1`
- `automation/scripts/start_postgres_user_mode.ps1`
- `automation/scripts/build_webui.ps1`

但仍缺少统一的开发启动与管理入口：

- 不存在 `start_all.ps1`
- 不存在 `stop_all.ps1`
- 不存在 `restart_all.ps1`
- 不存在 `status_all.ps1`
- 不存在 `start_webui_dev.ps1`

这导致：

- 开发启动顺序依赖个人记忆
- 服务重启时需要手工拼装命令
- 排障成本高

### 2.3 Node.js / npm 安装可用性口径不清

当前仓库在前端构建侧已经支持：

- `build_webui.ps1` 通过 `-NodeDir`
- `build_webui.ps1` 通过环境变量 `CLAWCHECK_NODE_DIR`
- `install_node_windows.ps1` 将 Node 安装到用户目录并写入 `CLAWCHECK_NODE_DIR`

但当前仍存在一类典型问题：

- Node / npm 已安装
- `CLAWCHECK_NODE_DIR` 已存在
- 但 `node`、`npm` 不在系统 PATH 中
- 用户直接在终端执行 `npm install`、`npm run dev` 会失败

这会导致：

- README 中手工命令与真实机器状态不一致
- 用户误判为“npm 未安装”或“Node 安装损坏”
- 前端开发依赖个人记忆手工补 PATH

### 2.4 服务状态检查未标准化

当前已有部分能力：

- FastAPI 提供 `/api/health`
- PostgreSQL 已有 `probe_postgres.ps1`
- Windows 任务守护进程已有独立日志目录约定

但整体仍未形成统一规范：

- `probe_postgres.ps1` 当前存在 PowerShell 参数命名冲突，脚本入口本身不可稳定执行
- 启动脚本默认以前台进程运行，缺少统一 PID 文件
- API、daemon、Vite、PostgreSQL 缺少统一状态查询输出格式
- 日志路径虽部分存在，但没有统一“服务名 -> 日志文件 -> PID 文件 -> 端口 -> 健康检查地址”的清单
- 服务存活、端口存活、业务健康三者未被统一表达

### 2.5 最小质量门禁缺失

当前开发依赖与质量门禁较弱：

- `automation/requirements-dev.txt` 目前主要只有 `pytest`、`httpx`
- `webui/package.json` 目前仅体现前端构建校验
- 仓库当前未发现 CI workflow 目录

这意味着：

- lint、编译、测试、前端构建没有统一执行入口
- 合并前缺少自动化兜底
- 本地运行成功不代表代码具备基础可发布性

### 2.6 运行产物忽略规则不完整

当前 `.gitignore` 已忽略：

- `.venv-win/`
- `automation/logs/*`
- `webui/node_modules/`
- `webui/dist/`

但未忽略以下前端开发产物：

- `webui/dev.log`
- `webui/dev.err.log`
- `webui/dev.pid`

这会使开发态运行结果直接进入工作区，增加噪音并干扰变更审查。

## 3. 与既有方案关系

### 3.1 与 `300` 方案的关系

`300本项目Windows原生部署替代WSL运行方案文档` 已明确：

- 正式运行以 Windows 原生链路为准
- 不再以 `uvicorn --reload`、`npm run dev`、Docker + WSL2 作为正式运行方式

`301` 不改变该结论，而是在其基础上补齐“开发环境如何可重复安装、可一键启停、可快速验活”的工程化能力。

### 3.2 与业务方案的关系

本方案仅治理开发与运维辅助链路，不改变以下内容：

- 业务表中文命名规范
- PostgreSQL 表结构口径
- Playwright 采集流程业务逻辑
- FastAPI 业务接口返回结构

## 4. 设计原则

### 4.1 优先标准化，再补便利性

先统一版本、目录、日志、PID、状态口径，再提供一键脚本。避免先做一个“能跑”的总控脚本，后续继续承载不一致环境。

### 4.2 开发入口统一，但不改写现有业务脚本

已有 `start_api.ps1`、`start_task_daemon.ps1`、`run_*.ps1` 等脚本继续保留。新的总控脚本在其之上编排，不重写业务执行逻辑。

### 4.3 正式运行与本地开发分层

需要明确区分两类场景：

- 正式运行：以 FastAPI 托管 `webui/dist` 为主
- 本地开发：允许使用 Vite dev，但必须通过标准脚本启动和管理

### 4.4 优先保证脚本可发现 Node，不强依赖用户手工改 PATH

当前项目已经存在 `CLAWCHECK_NODE_DIR` 这一工程约定，因此后续脚本设计应优先保证：

- 即使 `node`、`npm` 不在 PATH，脚本仍可正常工作
- 文档明确区分“命令行直接可用”和“脚本可用”是两个层次
- 是否写入 PATH 应作为可选增强，而不是唯一依赖路径

### 4.5 状态输出必须可机器读取，也可人工排障

所有服务状态脚本应至少同时满足：

- 人工阅读时可快速定位问题
- 后续如需接入脚本编排或 CI，可输出结构化字段

### 4.6 不为了开发便利引入新的部署复杂度

本方案不优先引入：

- Docker 化开发链路
- WSL 依赖回流
- 新的进程管理器
- 新的后端任务平台

## 5. 范围与边界

### 5.1 本次纳入范围

- Python 版本与 `.venv-win` 校验策略
- Node 定位策略与前端开发启动入口
- 本地 `start / stop / restart / status` 总控脚本
- 日志目录、PID 文件、端口、健康检查 URL 规范
- 最小质量门禁脚本与 CI 落地
- `.gitignore` 补齐开发运行产物忽略规则
- README 对开发环境章节的同步更新

### 5.2 本次不纳入范围

- 业务数据库结构改造
- 中文字段或表名调整
- Playwright 业务流程重写
- 引入 Docker Desktop 作为主开发环境
- 引入 WSL 作为正式运行前提
- 大规模前端工程体系升级

## 6. 目标设计

### 6.1 Python 与虚拟环境策略

约定如下：

- 仓库默认开发 Python 版本统一为 `3.12.x`
- 默认虚拟环境目录仍为 `.venv-win`
- `install_windows_env.ps1` 在发现以下任一情况时默认重建 venv：
  - venv 不存在
  - venv 中 Python 主次版本不等于请求版本
  - venv 中 `python.exe` 不可执行
- 脚本应输出：
  - 当前请求 Python 版本
  - 实际命中的 Python 可执行文件
  - 现有 venv 版本
  - 是否执行重建

建议增加参数：

- `-ForceRecreateVenv`
- `-SkipRecreateOnVersionMismatch`

默认行为以“版本不一致自动重建”为准，只有显式传参时才允许跳过。

### 6.2 Node 与前端开发策略

约定如下：

- 正式构建继续使用 `build_webui.ps1`
- 本地开发新增 `start_webui_dev.ps1`
- `install_node_windows.ps1` 负责安装 Node，并继续维护 `CLAWCHECK_NODE_DIR`
- `start_webui_dev.ps1` 的 Node 定位顺序与 `build_webui.ps1` 保持一致：
  1. 传入 `-NodeDir`
  2. 读取 `CLAWCHECK_NODE_DIR`
  3. 尝试 PATH 中的 `npm`

同时补充以下口径：

- README 不再把“直接执行 `npm install && npm run dev`”作为默认前端开发入口
- 如用户希望在终端直接使用 `node` / `npm`，可提供可选脚本将 Node 安装目录追加到用户 PATH
- 即使未写入 PATH，仓库脚本也必须保持可用
- `status_all.ps1` 应能显示当前 Node 来源：
  - `NodeDir 参数`
  - `CLAWCHECK_NODE_DIR`
  - `PATH`

脚本应负责：

- 自动补齐 PATH
- 如 `node_modules` 不存在，则执行安装
- 启动 Vite dev
- 写出 PID、stdout、stderr 日志
- 输出 `http://127.0.0.1:5173`

建议补充一个可选能力：

- `install_node_windows.ps1` 新增参数控制是否同时写入用户 PATH
- 或新增单独脚本用于把 `CLAWCHECK_NODE_DIR` 追加到用户 PATH

默认要求仍是：

- 不依赖 PATH 也能完成仓库内全部前端相关脚本执行

### 6.3 统一服务目录规范

建议新增统一运行状态目录，例如：

- `automation/runtime/dev/`

目录下统一存放：

- `api.pid`
- `task_daemon.pid`
- `webui_dev.pid`
- `api.stdout.log`
- `api.stderr.log`
- `task_daemon.stdout.log`
- `task_daemon.stderr.log`
- `webui_dev.stdout.log`
- `webui_dev.stderr.log`
- `status.json`

PostgreSQL 如继续使用其自身日志目录，可在统一状态文件中记录其真实日志路径，而不强行搬迁日志位置。

### 6.4 统一启停脚本设计

新增以下脚本：

- `automation/scripts/start_all.ps1`
- `automation/scripts/stop_all.ps1`
- `automation/scripts/restart_all.ps1`
- `automation/scripts/status_all.ps1`
- `automation/scripts/start_webui_dev.ps1`

建议职责如下：

#### `start_all.ps1`

- 按顺序启动 PostgreSQL、API、task daemon、Vite dev
- 支持按参数关闭某一子服务启动
- 启动后输出统一摘要

#### `stop_all.ps1`

- 按依赖反向停止服务
- 优先使用 PID 文件停止
- PID 失效时尝试按端口或进程命令行补救识别

#### `restart_all.ps1`

- 复用 `stop_all.ps1 + start_all.ps1`

#### `status_all.ps1`

- 输出每个服务的：
  - 运行状态
  - PID
  - 端口
  - 健康检查 URL
  - 日志路径
  - 最近一次检查时间
- 对 API 执行 `/api/health`
- 对 Vite 检查 `5173`
- 对 PostgreSQL 检查配置端口与探测结果
- 对 task daemon 至少检查 PID 与进程存活状态

同时要求：

- 修复 `probe_postgres.ps1` 的 PowerShell 参数命名冲突
- `status_all.ps1` 对 PostgreSQL 的检查优先复用修复后的 `probe_postgres.ps1` 或其等价底层实现
- PostgreSQL 探测失败时，应区分“数据库不可达”和“探测脚本自身异常”

### 6.5 服务状态输出标准

所有服务统一按下列字段表达：

- `service`
- `enabled`
- `running`
- `pid`
- `port`
- `healthUrl`
- `healthStatus`
- `logStdout`
- `logStderr`
- `detail`
- `checkedAt`

其中：

- `running` 表示进程是否存在
- `healthStatus` 表示服务是否可用，不等同于进程存活
- `detail` 用于放置错误摘要、端口占用、PID 文件缺失等信息

### 6.6 最小质量门禁

本方案要求新增统一质量脚本，例如：

- `automation/scripts/check_quality.ps1`

最小执行集合为：

1. `ruff check automation tests`
2. `python -m compileall automation`
3. `python -m pytest -q`
4. `npm run build`

为支撑该门禁，需要：

- 在 `automation/requirements-dev.txt` 中补充 `ruff`
- README 中补充“本地提交前检查”
- 新增最小 CI workflow，在 Windows 环境执行上述检查

### 6.7 Git 忽略规则

本方案要求至少补充忽略：

- `webui/dev.log`
- `webui/dev.err.log`
- `webui/dev.pid`
- 统一运行状态目录下的 PID 与日志文件

同时需要处理已被跟踪的历史开发产物，将其移出版本控制。

## 8. 验收标准

### 8.1 Python 环境验收

- 在已有 `.venv-win` 为 `3.10.x`、请求版本为 `3.12` 时，执行安装脚本会默认重建
- 重建后 `.venv-win/pyvenv.cfg` 显示为 `3.12.x`
- 脚本输出能明确说明“因版本不一致而重建”

### 8.2 前端开发入口验收

- 在 `node` 不在 PATH 但 `CLAWCHECK_NODE_DIR` 存在时，`start_webui_dev.ps1` 可正常启动
- 启动后能输出访问地址、PID、日志路径
- 停止脚本可清理对应进程与 PID 文件
- 在 `node`、`npm` 不在 PATH 时，仓库内构建脚本与开发脚本仍可正常执行
- 文档需明确说明当前机器若想直接执行 `npm` 命令，应满足的 PATH 前置条件

### 8.3 一键启停验收

- `start_all.ps1` 可在本机一次性拉起核心开发服务
- `status_all.ps1` 可输出每个服务的 PID、端口、日志、健康状态
- `restart_all.ps1` 可完成可重复重启
- `stop_all.ps1` 可稳定清理启动产生的进程
- `probe_postgres.ps1` 可独立成功执行，不再因 PowerShell 内建变量冲突失败

### 8.4 质量门禁验收

- 本地单命令即可执行完整最小质量检查
- CI 自动执行同一套检查
- 检查失败时能明确指出失败步骤

### 8.5 Git 工作区验收

- 前端开发运行后不再新增未忽略的日志或 PID 文件
- 历史被跟踪的开发运行产物已从版本控制中移除

## 9. 风险与应对

### 9.1 Python 3.12 未安装

风险：

- 默认重建策略会因本机缺少 `3.12` 而失败

应对：

- 安装脚本需明确报错当前缺少指定版本
- README 需提供安装前检查命令
- 必要时保留显式跳过重建参数，但不得作为默认路径

### 9.2 端口已被占用

风险：

- 一键启动时 API 或 Vite 端口可能被已有进程占用

应对：

- 启动前先执行端口占用检查
- 错误输出中明确冲突 PID 和命令行信息

### 9.3 PostgreSQL 探测脚本本身失效

风险：

- 数据库实际正常，但探测脚本因 PowerShell 参数冲突报错，导致误判为数据库异常

应对：

- 修复 `probe_postgres.ps1` 参数命名，避免使用与 PowerShell 内建只读变量冲突的参数名
- 在 `status_all.ps1` 中区分“探测脚本执行失败”和“数据库连接失败”
- 保留底层 Python 探测入口作为兜底

### 9.4 Node 安装存在但终端不可直接使用

风险：

- 用户看到 Node 已安装，但终端执行 `npm` 仍失败，误判为安装异常

应对：

- 文档中明确区分“脚本内可用”与“终端直输可用”
- 前端脚本统一自行补 PATH
- 如需终端直输体验，提供可选 PATH 写入能力

### 9.5 PID 文件失真

风险：

- 异常退出后 PID 文件可能残留

应对：

- 停止与状态检查脚本不能只信任 PID 文件
- 必须同时校验进程是否仍存在且命令行是否匹配

### 9.6 CI 与本地环境不一致

风险：

- 本地通过但 CI 失败，或反之

应对：

- CI 明确使用 Windows runner
- 质量脚本尽量复用本地命令，避免 CI 单独拼装逻辑

## 10. 发布与回滚原则

本方案主要涉及脚本、文档、忽略规则与 CI，不涉及业务表结构迁移。

发布原则：

- 先落文档与脚本，再更新 README
- 先保证本地可重复执行，再接入 CI

回滚原则：

- 如一键总控脚本存在问题，可临时退回已有单点脚本执行
- 如质量门禁影响紧急修复，可临时放宽 CI 必选项，但不得回退 Python 版本统一原则

## 11. 建议实施顺序

建议按以下顺序落地：

1. 修复 `.gitignore` 与已跟踪开发产物
2. 改造 `install_windows_env.ps1`，补版本不一致默认重建
3. 修复 `probe_postgres.ps1`，恢复 PostgreSQL 标准探测入口
4. 明确 Node / npm PATH 策略，并补可选 PATH 写入能力
5. 新增 `start_webui_dev.ps1`
6. 新增 `status_all.ps1`
7. 新增 `start_all.ps1 / stop_all.ps1 / restart_all.ps1`
8. 新增 `check_quality.ps1`
9. 补 CI workflow
10. 回写 README

## 12. 结论

当前项目的高收益改进点不是更换承载方式，而是把现有 Windows 原生开发链路补齐为“可重复安装、可一键启停、可统一验活、可最小质量校验”的工程化状态。

`301` 的核心任务不是引入新技术栈，而是消除当前开发环境中的不确定性，使后续 API、自动化采集、前端工作台和本地数据库调试都能建立在一致、稳定、可观测的基础之上。

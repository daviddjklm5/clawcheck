# 方案编号004：组织快速维护列表下载入库技术设计文档

## 1. 设计目标
在现有自动化工程基础上，新增“组织快速维护列表下载入库”能力，复用 003 方案已有的：
- 正式环境登录与登录态复用
- Playwright 下载监听能力
- Excel 解析与 PostgreSQL 入库模式

新增能力聚焦于：
- 菜单进入 `组织快速维护`
- 精准点击左侧顶级节点 `万物云`
- 激活 `列表包含所有下级`
- 从 `更多` 中执行 `引出数据（按列表）`
- 将导出 Excel 解析并写入表 `组织列表`

## 2. 与 003 方案的主要差异

### 2.1 页面差异
003 针对页面：
- `在职人员花名册`

004 针对页面：
- `组织快速维护`

### 2.2 查询条件差异
003 的关键动作是设置报表条件：
- `报表显示方案`
- `任职类型`

004 的关键动作改为页面内组织上下文切换：
- 左侧树节点：`万物云`
- 开关条件：`列表包含所有下级 = 开`

### 2.3 导出交互差异
003 的导出入口是工具栏主按钮：
- `引出`

004 的导出入口是二级菜单：
- `更多 -> 引出数据（按列表）`

003 实测后台按钮文案为：
- `转后台执行`

004 实测导出进度按钮文案为：
- `转入后台`

### 2.4 入库目标差异
003 入库目标表：
- `在职花名册表`

004 入库目标表：
- `组织列表`

## 3. 页面自动化流程设计

### 3.1 总流程
1. 打开首页。
2. 校验登录态，若失效则重新登录。
3. 点击左上角菜单展开 `最近使用`。
4. 点击 `组织快速维护`。
5. 在左侧组织树点击顶级节点 `万物云`。
6. 判断 `列表包含所有下级` 是否已激活；若未激活，则点击打开。
7. 等待右侧列表刷新完成。
8. 点击 `更多 -> 引出数据（按列表）`。
9. 在 `操作确认` 弹窗点击 `确定`。
10. 监听浏览器下载事件并保存 Excel。
11. 解析 Excel。
12. 非 `--dry-run` 时写入 PostgreSQL `组织列表`。

### 3.2 新增 Flow 设计
建议新增：
- `automation/flows/organization_quick_maintain_flow.py`

建议新增核心方法：
- `open_org_quick_maintain_page()`
- `select_top_org_node(node_name: str = "万物云")`
- `ensure_include_children_enabled()`
- `wait_org_list_ready()`
- `export_org_list(downloads_dir: Path, timeout_sec: int)`

这样可与 `automation/flows/active_roster_flow.py` 的职责边界保持一致。

### 3.3 页面入口定位策略
入口策略继续复用 003 的左上角展开逻辑：
1. 打开首页
2. 若页面未出现 `最近使用`，点击左上角区域展开菜单
3. 点击 `组织快速维护`

页面成功打开后的确认标志建议同时满足：
- 页面文本包含 `组织快速维护列表`
- 页面文本包含 `行政组织维护`
- 工具栏可见 `更多`

### 3.4 左侧顶级节点定位策略
注意：页面右侧表格中同样会出现文本 `万物云`，不能全局点击。

定位原则：
- 必须限制在左侧组织树区域内匹配 `万物云`
- 优先点击树节点容器，而不是右侧表格单元格

实测可用结构特征：
- 左侧树选中节点 class 包含：`kd-cq-tree-treenode-selected`
- 顶级节点文本在树节点 `a.kd-cq-tree-treenode-text` 中

建议策略：
1. 先在左侧树容器内查找首个包含 `万物云` 的树节点
2. 点击该树节点容器
3. 点击后验证其节点 class 进入选中态；若已选中也允许重复点击

### 3.5 “列表包含所有下级”开关策略
实测该控件真实结构为：
- 字段容器：`#chkincludechild`
- 隐藏输入：`#chkincludechild input[type="checkbox"]`
- 可点击开关：`#chkincludechild ._1HdIQduJ`

关键结论：
- 它不是展示 `是/否` 文本的普通单选框
- 自动化应直接读取 checkbox 的 `checked` 状态

处理策略：
1. 读取 `#chkincludechild input[type="checkbox"]` 的 `checked`
2. 若为 `False`，点击 `#chkincludechild ._1HdIQduJ`
3. 再次校验 `checked == True`

### 3.6 列表加载完成判定策略
实测中，切换开关后未必总能看到明显的 `正在加载...` 标识，因此不能仅依赖加载文案。

建议采用“开关状态 + 总条数变化 + 稳定轮询”三重判定：
1. 记录切换前列表总数 `before_count`
2. 点击开关后轮询页面文本中的 `共N条`
3. 条件满足时认为加载完成：
   - checkbox 已为 `True`
   - 页面不存在 `正在加载...` / `（加载中）`；或这些文案已消失
   - 当前 `after_count > before_count`
   - `after_count` 连续 2~3 次轮询保持不变

> 说明：`2026-03-11` 实测中，`万物云 + 包含所有下级` 的结果从 `37` 条变为 `32020` 条，但实现中不能写死该数字。

### 3.7 导出入口策略
实测 `更多` 按钮为：
- `#baritemap1`

展开后目标菜单项为：
- `引出数据（按列表）`

必须避免误点：
- `引出数据（按引出模板）`

建议导出步骤：
1. 点击 `#baritemap1`
2. 等待下拉菜单出现
3. 精确点击 `引出数据（按列表）`
4. 等待 `操作确认` 弹窗出现
5. 点击弹窗中的 `确定`

### 3.8 下载等待策略
实测点击 `确定` 后，页面会出现：
- 标题：`引出中`
- 提示：`正在引出组织快速维护，完成后将自动下载`
- 按钮：`转入后台`

推荐实现策略：
1. 在点击 `确定` 前注册浏览器 `download` 事件监听
2. 点击 `确定` 后优先等待直接下载事件
3. 若短时间内无下载事件，兼容识别 `引出中` 弹窗
4. 对后台按钮文案扩展兼容：
   - `转后台执行`
   - `转后台运行`
   - `转入后台`
5. 若后续实装时发现必须进入后台结果页，再复用 003 已有的后台任务轮询/下载逻辑做兜底

> 说明：本次实测中，不点击 `转入后台` 也在约 `27 秒` 后直接收到了下载事件。因此 004 的首选策略仍应是“监听浏览器下载事件”，后台任务流作为兼容兜底而不是唯一主路径。

### 3.9 下载文件命名策略
实测下载文件名示例：
- `引出列表_组织快速维护_0311222405.xlsx`

建议继续沿用 003 的安全命名策略：
- 保留浏览器建议文件名主体
- 前缀加时间戳
- 统一保存到 `automation/downloads/`

## 4. Excel 解析设计

### 4.1 解析目标
导出 Excel 的核心字段与页面表头一致，至少应解析以下列：
- `行政组织编码`
- `行政组织名称`
- `行政组织类型`
- `上级行政组织`
- `上级行政组织编码`
- `所属公司`
- `行政组织层级`
- `所在城市`
- `物理层级`
- `部门子分类.编码`
- `部门子分类.名称`
- `部门分类.编码`
- `部门分类.名称`
- `组织长名称`

### 4.2 解析实现建议
建议新增：
- `automation/utils/organization_list_excel.py`

建议提供函数：
- `parse_organization_list_workbook(path: str | Path) -> dict[str, Any]`

返回结构建议包含：
- `headers`
- `rows`
- `row_count`
- `source_file`
- `exported_at`（若文件中可解析到）

### 4.3 表头规范化策略
建议对表头执行统一规范化：
- 去首尾空白
- 全角/半角括号统一
- 连续空格折叠
- 保留原始中文名称，不强制英文化

原因：
- 该报表属于业务字段报表，中文表头就是业务主键语义
- 与页面和用户口径保持一致，更利于排障和复核

## 5. 数据库写入设计

### 5.1 建表建议
建议新增 SQL：
- `automation/sql/003_organization_list.sql`

建议目标主表：
- `组织列表`

建议字段至少包括：
- `org_code`
- `org_name`
- `org_type`
- `parent_org_name`
- `parent_org_code`
- `company_name`
- `org_level`
- `city_name`
- `physical_level`
- `dept_subcategory_code`
- `dept_subcategory_name`
- `dept_category_code`
- `dept_category_name`
- `org_full_name`
- `source_root_org`
- `include_all_children`
- `import_batch_no`
- `source_file_name`
- `created_at`
- `updated_at`

### 5.2 写入模式
建议采用“全量快照覆盖”而不是单纯 UPSERT：
1. 将当次导出数据写入临时集合
2. 在事务内清空正式表
3. 批量插入本次全量数据

这样设计的原因：
- 本需求导出的本质是某一时点的完整组织清单快照
- 组织停用/删除场景下，若只做 UPSERT，会残留历史脏数据

### 5.3 唯一性建议
若表中保留唯一约束，建议以以下组合控制：
- `org_code`

原因：
- `行政组织编码` 在业务上具有天然唯一性
- 全量覆盖场景下也便于二次校验数据质量

## 6. CLI 与配置设计

### 6.1 CLI 动作建议
建议在 `automation/scripts/run.py` 中新增动作：
- `orglist`

参数风格可对齐 003 的 `roster`：
- `--input-file`
- `--skip-export`
- `--skip-import`
- `--downloads-dir`
- `--download-timeout-minutes`

### 6.2 导入模式建议
支持两种模式：
1. 全流程模式：打开页面 -> 导出 -> 下载 -> 入库
2. 文件直导模式：仅对已有 Excel 文件执行解析与入库

这样便于：
- 页面导出与入库分开排障
- 针对历史文件重跑导入

## 8. 异常与兜底设计

### 8.1 登录态失效
- 自动判断登录态
- 若失效则重新登录并刷新 `storage_state`

### 8.2 左侧树误点风险
- 限定树区域点击 `万物云`
- 不允许使用全局 `text=万物云` 直接点击

### 8.3 开关未真正生效
- 必须校验 checkbox `checked == True`
- 必须校验列表总条数已发生变化并稳定

### 8.4 导出菜单误点风险
- 必须精确点击 `引出数据（按列表）`
- 不能模糊匹配 `引出数据`

### 8.5 下载等待不稳定
- 优先监听浏览器 download 事件
- 后台按钮文案兼容 `转入后台`
- 必要时再复用 003 的后台任务轮询逻辑做兜底

## 9. 当前实现边界
- 本设计基于 `2026-03-11` 的正式环境页面结构
- 当前仅覆盖 `万物云 + 列表包含所有下级 = 开` 的导出场景
- 若后续需要支持其他组织节点或不同过滤条件，应将根组织和开关策略参数化

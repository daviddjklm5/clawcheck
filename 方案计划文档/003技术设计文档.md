# 设计编号003：权限申请待办单据采集入库技术设计文档

## 1. 设计目标
基于现有 `automation/` 下的 `Python + Playwright` 自动化框架，新增“权限申请待办单据采集入库”能力，实现以下目标：
- 从 iERP 首页进入“我的消息 -> 待办任务”
- 识别 `单据=权限申请` 的待办记录
- 进入详情页后采集基本信息、权限申请明细、审批记录、行政组织详情组织编码
- 将数据落库至 PostgreSQL
- 支持 `--dry-run` 本地 JSON 落盘验证，降低联调风险

## 2. 技术方案概览

### 2.1 运行入口
沿用现有入口文件：`automation/scripts/run.py`

新增动作：
- `collect`

示例命令：
```bash
python automation/scripts/run.py collect --dry-run --document-no QX-260311-00000223 --headless
python automation/scripts/run.py collect --document-no QX-260311-00000223 --headless
```

### 2.2 核心模块划分
- `automation/scripts/run.py`
  - 解析 `collect` 命令参数
  - 处理登录态与运行参数
  - 触发采集流程
  - 负责 JSON 落盘与 PG 写库
- `automation/flows/permission_collect_flow.py`
  - 负责页面导航与数据采集
  - 负责明细表、审批记录、组织编码滚动抓取
- `automation/db/postgres.py`
  - 负责 PostgreSQL 连接与写入逻辑
- `automation/sql/001_permission_apply_collect.sql`
  - 负责建表 DDL

## 3. 页面自动化流程设计

### 3.1 总流程
1. 打开首页。
2. 校验登录态，若失效则重新登录。
3. 点击“我的消息 -> 待办任务”。
4. 读取待办列表 grid。
5. 过滤 `单据 = 权限申请`。
6. 根据 `--document-no` 或 `--limit` 确定采集范围。
7. 进入每条目标单据详情。
8. 采集基本信息。
9. 采集权限申请明细。
10. 对每条明细点击“行政组织详情”。
11. 滚动采集组织编码并去重。
12. 切换“审批记录”页签并采集审批流。
13. 输出 JSON。
14. 非 `--dry-run` 时写入 PG。

### 3.2 待办列表定位策略
待办列表页使用 grid 表头识别，而不是依赖动态页面 ID。
目标表头：
- `单据`
- `单据编号`
- `发起人`
- `主题`
- `状态`

进入详情时，点击主题列中的链接文本：
- `单据编号：{document_no}`

### 3.3 基本信息采集策略
基本信息区域使用字段容器 `#fs_baseinfo` 作为根节点。
每个字段容器按以下规则解析：
1. 读取字段标题 `.kd-cq-field-title-wrap span`
2. 优先读取 `input/textarea` 的 `value`
3. 若无，则读取下拉选中值 `.kd-cq-combo-selected`
4. 仍无值时，读取 `.kd-cq-field-value-wrap` 文本

该策略避免了依赖输入框固定顺序，较“按 input 序号取值”更稳。

### 3.4 权限申请明细采集策略
权限申请明细区域是标准 grid，按表头匹配：
- `申请类型`
- `角色名称`
- `角色编码`
- `行政组织`
- `行政组织详情`

实现策略：
- 通用读取 `.kd-table-container`
- 匹配包含目标表头的 grid
- 从 `tbody tr -> td` 提取每行数据
- 以表头映射为字典，避免写死列号

### 3.5 审批记录采集策略
审批记录区域不是标准表格，而是时间轴卡片结构。
已确认可见容器为：
- `#tabpageap_approvalrecord`

每条审批记录卡片解析规则：
- 节点与审批人信息：`h4`
- 时间：`p.zPG-NFNf`
- 审批意见：`p._3aIXYPkW`（当 display 非 none 时）

字符串拆解规则：
- 从 `h4` 中提取：`节点名称 + 审批人|组织职位 + 审批动作`
- 若结构异常，则保留 `raw_text` 兜底

### 3.6 行政组织详情采集策略
行政组织详情是虚拟滚动 grid，不是一次性把全部行渲染到 DOM 中。
因此不能只取 `tbody tr` 当前可见行，否则会漏抓。

处理策略：
1. 点击明细行中的 `行政组织详情` 链接。
2. 匹配包含以下表头的组织详情 grid：
   - `组织编码`
   - `组织名称`
   - `所属公司`
   - `组织长名称`
3. 定位其滚动容器：`.kd-table-body.kd-horizontal-scroll-container`
4. 循环执行：
   - 收集当前可见 `tbody tr` 中的 `组织编码`
   - 下滚 `scrollTop`
   - 再次收集并去重
5. 终止条件：
   - 已采集数量达到 `查看详情(n)` 中的期望数量；或
   - 滚动到底部且连续多轮无新增编码

### 3.7 去重策略
去重分两层：
- 采集层：Python `set()` 去重
- 数据库层：`organization_code(document_no, org_code)` 唯一约束再次防重

## 4. 数据库写入设计

### 4.1 写入模式
采用“主表 UPSERT + 子表按单据清空重写”的方式：
- `basic_info`：`ON CONFLICT(document_no) DO UPDATE`
- `permission_apply_detail`：先删后插
- `approval_record`：先删后插
- `organization_code`：先删后插，同时保留唯一约束

### 4.2 这样设计的原因
- 同一单据可能被重复采集
- 页面状态可能变化（如审批记录新增）
- 清空后重写最简单、可预测、便于追平最新页面数据

### 4.3 表设计说明
建表 SQL 文件：`automation/sql/001_permission_apply_collect.sql`

核心表：
- `basic_info`
- `permission_apply_detail`
- `approval_record`
- `organization_code`

关联方式：
- 均通过 `document_no` 关联

在 `automation/config/settings.example.yaml` 中新增：

  dbname: "ierp"

  password: "your_password"

- `IERP_PG_PORT`
- `IERP_PG_DBNAME`
- `IERP_PG_USER`
- `IERP_PG_PASSWORD`

- `IERP_PG_SSLMODE`

## 6. 参数设计
新增 `collect` 相关参数：
- `--document-no`：只采集指定单据
- `--limit`：限制采集条数
- `--dry-run`：仅采集并输出 JSON，不写 PG
- `--dump-json`：指定 JSON 输出路径
- `--headless`：强制无头模式运行

## 7. 异常与兜底设计

### 7.1 登录态失效
- 自动判断登录态
- 若失效则重新登录并刷新 `storage_state`

### 7.2 页面结构波动
- 表格采集按“表头集合”识别，不依赖动态 pageId
- 基本信息按字段标题解析，不依赖输入框固定序号

### 7.3 审批记录结构异常
- 拆解失败时保留 `raw_text`
- 避免因为单条解析异常导致整单失败

### 7.4 组织编码滚动不完整
- 通过 `查看详情(n)` 识别期望数量
- 滚动过程中持续去重
- 达到期望数量优先终止
- 若未达到期望数量但已无新增，记录日志，便于后续排障

## 8. 输出与验证设计

### 8.2 联调建议
建议联调顺序：
1. 执行 `collect --dry-run`，确认 JSON 结构正确
2. 执行 SQL 建表
3. 配置 PG 连接
4. 执行 `collect` 写库
5. 使用 SQL 验证四表数据是否按 `document_no` 关联完整

## 9. 当前实现边界
- 当前实现按页面现状完成，适用于已核实的 iERP 页面结构
- 若后续系统升级导致 DOM 结构变化，需要重新校准字段选择器和审批记录卡片结构
- 当前默认采集“待办任务”中的权限申请单据，不覆盖“已办”“在办”等列表

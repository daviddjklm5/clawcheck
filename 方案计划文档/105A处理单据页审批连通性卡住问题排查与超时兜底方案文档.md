# 方案编号105A：处理单据页审批连通性卡住问题排查与超时兜底方案文档

## 1. 方案背景

`105` 已为 `处理单据 -> 单据详情抽屉 -> 审批单据` 页签补齐：

- `批准`
- `验证连通性（dry-run）`
- EHR `任务处理` 页签中的 `审批决策 = 同意`
- `审批意见` 写入与 `提交`

但在 `2026-03-16` 的现场使用中，出现了以下反馈：

- 打开 `http://127.0.0.1:5173/process-documents?...&documentNo=RA-20260310-00019845&tab=approvalAction`
- 点击 `单据详情` 后，在 `审批单据` 页签点击 `验证连通性`
- 页面长时间停留在弹出的 EHR 浏览器窗口中
- 前端按钮持续显示 `验证中...`
- 使用者无法判断当前是：
  - 正在执行
  - 执行已失败
  - 还是已经卡在某一步

因此需要在 `105` 已落地能力之上，补一份针对“审批连通性卡住”场景的排查与兜底文档。

## 2. 问题定义

本次问题的业务定义不是“审批链路完全不可用”，而是：

- 当审批链路执行时间异常拉长时
- 前端缺少超时退出与明确提示
- 后端缺少执行中的实时日志刷新
- 导致使用者主观感受为“卡死在 EHR 弹出窗”

该问题会直接影响：

- 现场操作判断
- 故障定位效率
- 是否需要人工接管浏览器

## 3. `2026-03-16` 复核结论

### 3.1 复核范围

针对样例单据 `RA-20260310-00019845`，已分别执行以下三类复核：

1. 直接调用 Python 审批服务入口  
   `automation.api.process_dashboard.approve_process_document(..., dry_run=True)`
2. 通过 FastAPI 接口调用  
   `POST /api/documents/process-workbench/{document_no}/approval`
3. 通过本地 Web UI 实际点击  
   `审批单据 -> 验证连通性（dry-run）`

### 3.2 当前结论

截至 `2026-03-16` 本地复核结果：

- 上述三条链路均已成功返回
- 对样例单据未稳定复现“永久卡死”
- 单次执行耗时约 `8 ~ 13` 秒

因此，本次不将问题定性为：

- `DocumentApprovalFlow` 已整体失效
- `审批单据` 页签选择器失效
- `EHR` 单据已无法打开

当前可确认的是：

- 该问题具备“偶发卡住体感”
- 但未形成稳定可复现的单一根因
- 工程上更需要先补足“可观测性 + 超时兜底”

### 3.3 已观察到的工程风险

复核过程中已确认以下风险点：

1. 前端审批请求原先没有超时控制  
   只要后端连接不返回，按钮就会持续显示 `验证中...`
2. 后端审批日志原先主要在执行结束后一次性落盘  
   如果执行中途卡住，`approval_*.json` 无法及时反映当前走到哪一步
3. 历史日志中已出现过页面上下文异常关闭情形  
   例如 `approval_20260316_092218_RA-20260313-00019961.json` 中出现 `TargetClosedError`

结论：

- 即使审批流本身大多数时候可执行
- 只要缺少超时退出和过程日志
- 使用者依旧会认为“系统卡死”

## 4. 方案目标

`105A` 的目标不是重写审批流，而是补齐以下两类故障兜底能力：

1. 让前端在审批执行时间过长时明确结束等待并提示下一步操作
2. 让后端在审批执行过程中持续写入日志，便于查看当前卡在哪一步

本方案不改变以下业务契约：

- `approve` 仍是唯一开放动作
- `dry-run` 仍不点击 EHR `提交`
- `审批决策 = 同意` 的映射不变
- PostgreSQL 不新增结构

## 5. 解决方案

### 5.1 后端：审批执行日志改为实时刷新

改动文件：

- `automation/api/process_dashboard.py`

处理原则：

- 在审批请求一开始就生成 `approval_*.json`
- `status = running` 时先写出初始日志
- 后续每追加一个阶段事件，立即刷新日志文件

本次新增的关键执行事件包括：

- `browser_context_ready`
- `open_home_page`
- `home_ready`
- `reuse_existing_login_state`
- `approval_flow_started`
- `approval_flow_finished`

这意味着当现场再次出现“卡在 EHR 弹出窗”时，可以直接打开最新日志查看最后停留的阶段，而不需要等接口返回。

### 5.2 前端：审批请求增加超时兜底

改动文件：

- `webui/src/services/api.ts`

处理原则：

- 为通用 `request()` 增加可选超时控制
- 审批接口单独配置 `120000ms`
- 超时后主动中断请求，并返回明确错误信息

前端超时提示口径为：

- 审批请求超过 `120` 秒未返回
- 后端可能卡在 EHR 窗口
- 引导用户检查：
  - 弹出的 EHR 浏览器
  - `automation/logs` 下最新 `approval_*.json`

### 5.3 文档口径

本方案落地后，`审批单据` 页签的交互补充约束为：

- 正常情况下继续等待后端返回
- 若审批请求超过 `120` 秒：
  - 不再无限展示 `验证中...`
  - 前端需给出明确错误提示
  - 操作员按日志与 EHR 实页继续排查

## 6. 排查步骤约定

后续若再次出现“验证连通性卡住”，排查顺序统一为：

1. 查看前端是否已出现 `120` 秒超时提示
2. 打开 `automation/logs` 下最新 `approval_*.json`
3. 查看 `events` 中最后一条记录停在哪个阶段
4. 若日志停在 `approval_flow_started` 之后但没有 `approval_flow_finished`：
   - 说明审批流进入了 Playwright 执行阶段
   - 需结合 EHR 浏览器当前页面判断是否等待人工干预
5. 若同时存在 `approval_error_*.png`：
   - 结合截图看页面是否被关闭、跳登录、弹出异常提示

## 9. 最终结论

`105A` 的结论如下：

1. `2026-03-16` 对样例单据的复核未稳定复现永久卡死
2. 当前更准确的问题定义是“审批执行时缺少超时退出与过程可观测性”
3. 本次优先补齐：
   - 后端审批日志实时刷新
   - 前端审批请求 `120` 秒超时兜底
4. 后续若再次出现现场卡住，应先按本方案的日志与 EHR 页面联合排查，而不是直接将问题归因于审批流完全失效

## 10. `105B` 后补口径

`2026-03-16` 之后已确认，现场出现的部分“验证连通性失败”并不属于 `105A` 定义的“执行卡住”，而是另一类场景：

- 单据已经不在当前账号 EHR 待办中
- 审批流仍尝试从待办列表打开它
- 因而报出 `Todo document link not ready`

因此自 `105B` 起，排查顺序需先分流：

1. 若前端已提示该单据最近一次待办同步结果为 `已处理`
   - 优先按“待办状态不同步”处理
   - 先执行 `同步待办状态`
2. 若单据仍为 `待处理`，但审批执行时间过长或前端超时
   - 再按本方案的“卡住排查”路径处理

也就是说：

- `105A` 继续负责“真的卡住了怎么办”
- `105B` 负责“其实已经不在待办里了，不该再硬打开”

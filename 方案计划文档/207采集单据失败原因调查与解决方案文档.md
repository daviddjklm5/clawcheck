# 方案编号207：采集单据部分失败（RA-20260318-00020169）原因调查与解决方案文档

## 1. 问题背景

2026-03-21 16:23:43 触发采集任务（taskId=`6d619a461c43`），任务最终状态为 `partial`：

1. 成功 7 张
2. 跳过 7 张
3. 失败 1 张
4. 增量评估已完成，批次号 `audit_20260321_163037`

本次用户反馈的故障 dump：

- `automation/logs/collect_20260321_162343_6d619a461c43.json`
- `automation/logs/collect_20260321_162343_6d619a461c43_failed.json`
- `automation/logs/run_20260321_162343.log`

## 2. 故障现象与证据

失败单据为：

- `RA-20260318-00020169`

失败信息（failed dump）：

- `RuntimeError: Organization detail grid not found after clicking detail link`

关键日志证据（`run_20260321_162343.log`）：

1. 16:28:43 开始采集 `RA-20260318-00020169`
2. 16:29:26 已完成到 detail row 17 的组织编码采集
3. 16:29:34 触发重试 attempt 2（会话重置回待办）
4. 16:29:42 触发重试 attempt 3（再次重置）
5. 16:30:35 最终失败：`Organization detail grid not found after clicking detail link`

说明：

1. 当前重试配置 `retries=2`，总尝试次数为 3（`retry_call` 逻辑）
2. 三次都落到同一类异常，属于可复现的流程缺陷，不是单次网络抖动

## 3. 根因定位

### 3.1 异常抛出位置

异常来自 `automation/flows/permission_collect_flow.py`：

1. `extract_role_organization_scopes()` 对每条明细行点击“查看详情”
2. `_extract_codes_from_org_grid()` 在点击后读取组织明细表格
3. 若 `_get_org_grid_snapshot()` 返回空，则抛出：
   - `RuntimeError("Organization detail grid not found after clicking detail link")`

### 3.2 根因判断

根因为“权限明细虚拟滚动行提取存在脏行/伪行，且组织明细采集缺少防御性判定”，导致流程把无效行当作有效行处理：

1. 明细抽取依赖虚拟滚动快照，存在混入“非业务有效行”的可能
2. 当前过滤条件对脏行拦截不充分（例如仅含部分渲染信息但不具备有效角色语义的行）
3. 进入组织详情采集时，代码对“点击后无弹层/无组织表格”直接抛错，导致整张单据失败

同类旁证：

1. `run_20260321_094950.log` 曾出现 `Permission detail row not ready ... line_no=''`，表明明细行集合中混入了无效行
2. 说明本问题是同一链路的稳定缺陷，而非本单据独有数据异常

## 4. 影响评估

1. 单据级影响：单张单据在最后阶段失败，整单不入库（本次即该单据失败）
2. 批次级影响：任务整体仍可完成并触发增量评估，但成功数减少
3. 运维影响：需要人工补采失败单据，否则数据缺口进入后续审核链路

## 5. 解决方案

## 5.1 改造目标

在不降低数据质量前提下，消除“脏行导致整单失败”问题，并保留可追溯性。

## 5.2 代码改造点

改造文件：`automation/flows/permission_collect_flow.py`

1. 强化明细行有效性过滤（源头治理）
   - 明细行判定不再把 `#`（行号）单独视为业务有效
   - 至少满足 `角色编码/角色名称/申请类型` 中任一有效，再进入组织采集
   - 对 `line_no=''` 且关键业务字段为空的行直接丢弃并记录 warning

2. 增加组织详情点击后的防御逻辑（过程治理）
   - 当 `expected_count == 0` 时不点击详情，直接记空组织列表
   - 点击后若未出现组织表格，先做一次“重新聚焦行 + 重点位点击”重试
   - 重试仍失败时，不立刻整单失败：记录“行级失败”，继续采集其余行

3. 行级失败与单据失败解耦（结果治理）
   - 新增 `failed_org_scope_rows` 结构（单据内行级异常）
   - 仅当“基础信息/审批记录/明细表整体不可读”时判定单据失败
   - 行级失败进入日志和 dump，供后续补采或人工核对

4. 日志增强（可观测治理）
   - 组织详情异常时输出：`document_no`、`line_no`、`role_code`、`role_name`、`detail_text`、`expected_count`
   - 明确区分：`row_skipped`、`row_retrying`、`row_failed_nonblocking`

## 5.3 临时处置（未发版前）

先对失败单据做定向补采（仅补这一张）：


若仍失败，保留当次日志与 failed dump，按本方案实施代码修复后再次补采。

## 8. 结论

本次 `2026-03-21 16:23:43` 采集任务的“1 张失败”并非数据库或评估链路问题，核心是采集流程对虚拟滚动明细脏行缺少足够防御，导致组织详情弹层未出现时直接将整单判失败。  
按本方案改造后，可将该类故障从“整单失败”降级为“可追踪的行级异常”，并显著提高采集稳定性与可运维性。

本节用于对本方案关键判断做逐项复核，结论分为“已确认 / 部分确认”。

     - `skippedCount = 7`

   - 实现：`automation/utils/retry.py` 使用 `for attempt in range(1, retries + 2)`，即总尝试次数 = `retries + 1 = 3`

   - `_is_empty_detail_row()` 当前将 `#`（行号）纳入 `business_fields` 判空条件  
   - 实测（使用 `.venv-win` 解释器）：
     - 仅 `#` 非空时，`_is_empty_detail_row(...)` 返回 `False`（不会被过滤）
     - 全空时返回 `True`

### 9.2 部分确认项

1. “本次失败由脏行直接触发”属于高概率结论，但现有日志证据尚未做到 100% 直接指认

   - 旁证：同日另一次任务已有 `line_no=''` 的明细行异常（`run_20260321_094950.log`）  
   - 但因为当前日志未记录“失败时对应的 row_idx/role_code/detail_text”，无法仅凭现有日志直接还原该次失败行的业务字段

2. “expected_count=0 处理缺口”属于代码级已确认风险点，但是否本次直接触发原因未最终证实  
   - 代码当前对 `expected_count == 0` 无特殊分支，会继续点击详情并强依赖组织表格出现

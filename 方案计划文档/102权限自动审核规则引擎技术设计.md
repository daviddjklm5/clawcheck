# 文档编号102：权限自动审核规则引擎技术设计

## 1. 设计目标
在现有 `Python + Playwright + PostgreSQL` 自动化项目基础上，新增一套可配置、可解释、可回放、可灰度的权限自动审核规则引擎。

规则引擎设计目标如下：
- 从现有采集数据中构建标准 facts
- 加载并校验 YAML 规则包
- 编译并执行 CEL 条件
- 输出统一审核结果与审批追加建议
- 记录规则命中证据、执行日志与审计结果
- 支持历史回放、版本对比与灰度发布

## 2. 设计范围
本设计覆盖以下内容：
- 规则引擎模块划分
- 核心对象模型
- facts 计算流程
- 规则加载与校验流程
- CEL 编译与执行流程
- 动作执行与结果归并流程
- 审计落库与回放设计

本设计暂不覆盖以下内容：
- Web 管理后台
- 页面回写审批动作
- 外部工作流系统改造
- 非权限申请类型单据审核

## 3. 现状与设计基线
当前仓库已具备以下基础能力：
- `automation/flows/permission_collect_flow.py`
  - 负责权限申请待办单据采集
- `automation/db/postgres.py`
  - 负责 PostgreSQL 写入与读取基础设施
- `automation/sql/001_permission_apply_collect.sql`
  - 已有权限申请单相关基础表
- `automation/sql/002_active_roster.sql`
  - 已有在职花名册表
- `automation/sql/003_organization_list.sql`
  - 已有组织列表表

因此，规则引擎不需要重新建设“数据采集能力”，而应建立在现有采集入库之上，作为新增的审核决策层。

## 4. 总体架构设计

### 4.1 总体分层

规则引擎建议分为六层：
1. 输入装配层
2. facts 计算层
3. 规则加载与校验层
4. 条件执行层
5. 动作归并与审批派生层
6. 审计与回放层

### 4.2 分层职责

#### 4.2.1 输入装配层
职责：
- 从 `申请单基本信息`、`申请单权限列表`、`申请单审批记录`、`organization_code` 装配单据原始对象
- 读取 `在职花名册表`、`组织列表` 及后续主数据表

输出：
- `RawDocumentBundle`

#### 4.2.2 facts 计算层
职责：
- 清洗和标准化原始字段
- 计算 `is_cross_org`、`hr_mode`、`duty_tags`、`is_remote_role`、`is_legacy_role` 等标准 facts

输出：
- `FactsContext`

#### 4.2.3 规则加载与校验层
职责：
- 加载 `YAML` 规则包、文案模板、常量、审批矩阵
- 完成模型校验、引用校验、CEL 编译校验

输出：
- `CompiledRulePackage`

#### 4.2.4 条件执行层
职责：
- 基于 `FactsContext` 执行 `document/detail/approval` 不同作用域规则
- 输出命中结果与命中证据

输出：
- `RuleHit[]`

#### 4.2.5 动作归并与审批派生层
职责：
- 合并多条命中结果
- 应用动作优先级
- 去重审批节点
- 将审批角色解析为审批人建议

输出：
- `AuditDecision`

#### 4.2.6 审计与回放层
职责：
- 记录执行结果、命中证据、版本号
- 支持历史单据重放与差异对比

输出：
- `AuditExecutionLog`

## 5. 模块设计

### 5.1 建议目录

```text
automation/
  rules/
    __init__.py
    models.py
    loader.py
    validator.py
    cel_adapter.py
    facts.py
    engine.py
    actions.py
    approvals.py
    explain.py
    repositories.py
```

### 5.2 模块职责
- `models.py`
  - 定义原始输入对象、facts 对象、规则对象、命中对象、决策对象
- `loader.py`
  - 负责 YAML 解析与配置装配
- `validator.py`
  - 负责规则结构校验、引用校验与兼容性校验
- `cel_adapter.py`
  - 负责 CEL 表达式编译、缓存与执行
- `facts.py`
  - 负责 facts 计算、标准化与缺失标志处理
- `engine.py`
  - 负责调度执行、规则排序、分 scope 执行与命中合并
- `actions.py`
  - 负责标准动作构造、冲突处理与结果裁决
- `approvals.py`
  - 负责审批角色到审批人候选的解析
- `explain.py`
  - 负责命中说明、证据格式化与差异输出
- `repositories.py`
  - 负责数据库读写与审计落库

## 6. 核心对象模型设计

### 6.1 原始输入对象

```python
class RawDocumentBundle:
    basic_info: dict
    permission_details: list[dict]
    approval_records: list[dict]
    organization_codes: list[str]
    roster_profile: dict | None
    organization_profiles: list[dict]
    extra_masterdata: dict
```

说明：
- 用于承接数据库读取结果
- 不直接提供给规则层使用

### 6.2 FactsContext

```python
class FactsContext:
    document: dict
    applicant: dict
    details: list[dict]
    approvals: list[dict]
    constants: dict
    context: dict
    missing_facts: list[str]
```

说明：
- 这是规则引擎唯一可信输入
- `missing_facts` 用于触发人工复核或解释提示

### 6.3 RuleDefinition

```python
class RuleDefinition:
    id: str
    name: str
    enabled: bool
    scope: str
    category: str
    severity: str
    priority: int
    tags: list[str]
    stop_on_hit: bool
    when_cel: str
    then: dict
    explain: dict | None
```

### 6.4 RuleHit

```python
class RuleHit:
    rule_id: str
    rule_name: str
    scope: str
    action: str
    message_code: str | None
    approvals: list[dict]
    hit_target: str
    evidence: dict
```

说明：
- `hit_target` 用于标记命中对象，如文档、某条明细、某个审批节点

### 6.5 AuditDecision

```python
class AuditDecision:
    result: str
    final_action: str
    messages: list[str]
    message_codes: list[str]
    required_approvals: list[dict]
    manual_review_reason: str | None
    hits: list[RuleHit]
```

## 7. Facts 计算设计

### 7.1 Facts 计算目标
将数据库中的原始字段转成稳定、低歧义、可复用的标准事实字段。

### 7.2 输入来源
- `申请单基本信息`
- `申请单权限列表`
- `申请单审批记录`
- `organization_code`
- `在职花名册表`
- `组织列表`
- 后续新增：
  - `权限列表`
  - `audit_position_duty_map`
  - `audit_exception_whitelist`

### 7.3 标准 facts 示例

#### document 维度
- `document.document_no`
- `document.is_cross_org`
- `document.target_org_lines`
- `document.requested_role_codes`
- `document.requested_role_count`
- `document.reason_tags`

#### applicant 维度
- `applicant.employee_no`
- `applicant.position_name`
- `applicant.hr_mode`
- `applicant.duty_tags`
- `applicant.special_tags`
- `applicant.is_hr_tag`
- `applicant.is_part_time_hr`

#### detail 维度
- `detail.apply_type`
- `detail.role.code`
- `detail.role.name`
- `detail.role.group`
- `detail.role.is_remote_role`
- `detail.role.is_local_role`
- `detail.role.is_legacy`
- `detail.role.is_high_privilege`
- `detail.target_org.code`
- `detail.target_org.line`
- `detail.salary_change_reason`

### 7.4 关键计算逻辑

#### 7.4.1 跨组织判断
判断建议：
- 申请人归属组织与申请目标组织不一致时，标记 `document.is_cross_org = true`
- 如有多条明细，则以任一明细跨组织为整单跨组织

#### 7.4.2 岗位职责识别
优先级建议：
1. 例外白名单
2. 显式职责映射表
3. 职位名称关键字映射
4. 默认规则

输出：
- `applicant.duty_tags`

#### 7.4.3 角色属性识别
从角色目录主数据计算：
- 是否远程角色
- 是否属地角色
- 是否老角色
- 是否高权限
- 所属角色组

#### 7.4.4 事实缺失处理
当关键信息缺失时：
- 在 `missing_facts` 中记录
- 默认不直接放行
- 按包级策略或规则级策略进入 `manual_review`

## 8. 规则加载与校验设计

### 8.1 加载流程
1. 读取 `audit_rules.yaml`
2. 读取 `message_templates.yaml`
3. 读取 `constants.yaml`
4. 读取 `approval_matrix.yaml`
5. 通过 `Pydantic` 转换为配置模型
6. 执行结构校验与引用校验
7. 对每条 `when_cel` 进行编译检查
8. 生成 `CompiledRulePackage`

### 8.2 校验内容
- 规则编号唯一性
- message_code 存在性
- 常量引用存在性
- scope 与上下文变量兼容性
- 动作与字段匹配性
- 生效时间合法性

### 8.3 编译缓存
建议对 `when_cel` 编译结果做内存缓存：
- key：`rule_version + rule_id + when_cel_hash`
- value：已编译表达式对象

目的：
- 降低重复执行开销
- 保证同一版本规则执行一致性

## 9. 规则执行设计

### 9.1 执行入口
建议新增统一入口服务：

```python
evaluate_document(document_no: str, rule_version: str | None = None) -> AuditDecision
```

### 9.2 执行顺序
建议执行顺序：
1. 加载单据数据
2. 构建 `FactsContext`
3. 加载规则包
4. 按优先级排序规则
5. 先执行 `document` 规则
6. 再执行 `detail` 规则
7. 最后执行 `approval` 规则
8. 归并命中结果
9. 输出最终决策
10. 落库审计结果

### 9.3 scope 执行策略

#### document 规则
- 对整单执行一次
- 上下文中提供 `document`、`applicant`、`details`

#### detail 规则
- 对每条明细循环执行
- 每次执行时将当前明细注入 `detail`
- 命中结果需记录对应明细标识

#### approval 规则
- 对当前审批链或待追加审批上下文执行

### 9.4 stop_on_hit 策略
- 若规则 `stop_on_hit = true`，则在同 scope 内命中后停止后续低优先级规则
- 默认建议为 `false`
- 阻断类、明确互斥类规则可开启 `true`

## 10. 动作归并设计

### 10.1 动作优先级
统一优先级如下：
- `reject`
- `manual_review`
- `require_approvals`
- `warning`

### 10.2 归并原则
- 任一命中 `reject`，整单最终动作默认为 `reject`
- 若无 `reject` 但有 `manual_review`，则最终结果为 `manual_review`
- 若命中多条 `require_approvals`，审批节点取并集去重
- `warning` 作为附加提示保留

### 10.3 文案归并
- 按优先级与命中顺序输出 `message_code`
- 再根据模板渲染最终文本
- 同文案模板允许去重输出

### 10.4 审批节点归并
去重键建议为：
- `approver_role`
- `org_source`
- `resolved_org_code`

## 11. 审批派生设计

### 11.1 设计目标
将规则中的“审批角色要求”解析为“具体审批建议节点”。

### 11.2 输入
- `required_approvals`
- `approval_matrix`
- 组织主数据
- 当前申请目标组织信息

### 11.3 输出

```python
[
    {
        "approver_role": "战区人力资源部负责人",
        "resolved_org_code": "A51",
        "resolved_persons": [
            {"employee_no": "05000001", "name": "蔡蟒生"}
        ],
    }
]
```

### 11.4 设计原则
- 规则不写死审批人姓名
- 审批人变更只更新审批矩阵或主数据
- 如无法唯一解析审批人，应进入 `manual_review`

## 13. 回放设计

### 13.1 回放目标
用于验证：
- 新规则上线前效果
- 旧规则版本与新版本差异
- 自动审核与人工审核一致性

### 13.2 回放入口
建议新增 CLI：

```bash
python automation/scripts/run.py audit-replay --from-date 2026-03-01 --to-date 2026-03-12 --rule-version 2026-03-12
```

### 13.3 回放输出
- 总单量
- 命中规则统计
- 驳回率变化
- 审批追加率变化
- 与人工审核结果不一致样本清单

## 14. 灰度发布设计

### 14.1 模式设计
- `shadow`
  - 只执行规则，记录结果，不影响实际业务流
- `enforce`
  - 正式执行驳回/追加审批

### 14.2 灰度策略
- 按规则包灰度
- 按标签灰度
- 按组织灰度
- 按时间窗口灰度

### 14.3 风险控制
- 新规则默认 `shadow`
- 通过历史回放验证后再切换 `enforce`

## 15. 性能设计

### 15.1 目标
在单据审核场景中，规则引擎应保证单张单据审核时间可控，避免明显拖慢整体自动化流程。

### 15.2 关键优化点
- 规则包加载缓存
- CEL 编译缓存
- 主数据按批次预加载
- facts 计算函数幂等复用
- 审批矩阵查询结果缓存

### 15.3 不建议的优化方式
- 不建议为追求性能牺牲命中证据留痕
- 不建议将复杂逻辑提前写死到 SQL 中取代 facts 计算

## 16. 异常与兜底设计

### 16.1 配置异常
如遇以下情况，应拒绝发布或拒绝执行：
- YAML 结构不合法
- `when_cel` 编译失败
- message_code 缺失
- scope 不兼容

### 16.2 运行异常
如遇以下情况，应进入 `manual_review`：
- 关键主数据缺失
- 审批人无法解析
- facts 计算结果不完整
- 规则执行上下文关键字段为空

### 16.3 兜底原则
- 宁可人工复核，不可静默放行
- 所有异常必须留痕

- CEL 编译测试

## 18. 运行入口设计

### 18.1 CLI 建议
建议在 `automation/scripts/run.py` 中新增动作：
- `audit-eval`
- `audit-replay`
- `audit-validate-rules`

示例：

```bash
python automation/scripts/run.py audit-eval --document-no QX-260311-00000223
python automation/scripts/run.py audit-validate-rules --rules automation/config/rules/audit_rules.yaml
```

### 18.2 与现有流程衔接
建议衔接顺序：
1. `collect` 采集单据并入库
2. `audit-eval` 从库中读单据并执行规则
3. 输出审核建议与审批追加建议
4. 后续如需自动回写审批系统，再单独设计执行层

## 19. 实施顺序建议

### 19.1 第一阶段：最小可用版本
- 建立 `automation/rules/` 目录

## 20. 结论
本规则引擎技术设计坚持以下架构原则：
- 使用现有 `Python + PostgreSQL` 作为宿主执行底座
- 使用 `YAML` 管理规则结构与动作
- 使用 `CEL` 表达复杂条件
- 使用 facts 标准化屏蔽原始字段差异
- 使用结构化审计日志保证可解释、可回放、可灰度

该设计既能兼容当前项目的自动化采集能力，又能为后续审批自动化、规则平台化、历史回放与治理分析提供稳定基础。

## 3. 标准关联口径

### 3.1 统一关联键
申请单侧统一使用：
- `申请单基本信息.工号`

人员属性侧统一使用：
- `人员属性查询.工号`

禁止继续使用：
- `申请单基本信息` 上已经删除的 HR 判定列
- 直接假设申请单主表自带 `申请人HR类型`、`是否责任HR` 等字段

### 3.2 标准查询模板
后续报表、规则 SQL、临时分析 SQL，统一参考以下写法：

```sql
SELECT
    b."单据编号",
    b."工号",
    b."权限对象",
    b."申请理由",
    p."申请人HR类型",
    p."是否责任HR",
    p."是否HR人员",
    p."是否疑似HR人员",
    p."HR主判定依据",
    p."HR主判定值",
    p."HR子域",
    p."HR判定原因"
FROM "申请单基本信息" b
LEFT JOIN "人员属性查询" p
    ON BTRIM(b."工号") = BTRIM(p."工号");
```

说明：
- 使用 `LEFT JOIN`，避免申请单因人员属性缺失被过滤
- 两侧统一 `BTRIM`，减少工号两端空格导致的误差
- 若未命中 `人员属性查询`，应按“人员属性未命中”处理，而不是回退读取主表旧列

## 4. 下游改造要求

### 4.1 报表类 SQL
凡是需要以下字段的报表：
- `申请人HR类型`
- `是否责任HR`
- `是否HR人员`
- `是否疑似HR人员`
- `HR主判定依据`
- `HR主判定值`
- `HR子域`
- `HR判定原因`

一律改为：
- 从 `人员属性查询` 读取
- 通过 `申请单基本信息.工号 -> 人员属性查询.工号` 关联

### 4.2 规则引擎输入装配
若后续规则引擎需要申请人 HR 属性：
- 原始申请单对象仍来自 `申请单基本信息`
- HR 派生属性需作为扩展属性，由 `人员属性查询` 关联补齐

### 4.3 临时分析脚本
分析脚本、Notebook、手工 SQL 也应遵守同一口径：
- 主表负责单据信息
- `人员属性查询` 负责人维度属性

## 5. 排查方法
建议后续每次发布前都执行一次全文检索，重点检查以下关键词：


若搜索结果中的文件不是：

则应继续确认是否误用了旧口径。

## 6. 完成标准
满足以下条件时，可认为下游依赖改造完成：

1. 运行态代码不再读取 `申请单基本信息` 旧 HR 列
2. 新增报表或规则 SQL 统一通过 `人员属性查询` 取数
3. 执行 `020_person_attributes_acceptance.sql` 时，旧 HR 列盘点结果持续为 `0`
4. 文档侧明确写清标准关联方式，避免后续回退到旧口径

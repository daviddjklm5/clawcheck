# 方案编号011：九张表字段中文化迁移方案文档

## 1. 方案结论

结论分两层：

- **业务命名目标本身合理**
  - 你提出“优先使用报表/页面/已确认表格中的中文字段名，避免自行翻译”这一原则是正确的。
  - 这一原则尤其适合本仓库当前场景，因为同一个英文技术名在不同表中实际对应的业务含义并不完全相同。
  - 例如：
    - `employee_no` 在 `申请单基本信息` 中应落为页面字段 `工号`
    - `employee_no` 在 `在职花名册表` 中应落为报表字段 `人员编号`
    - `company_name` 在不同表中也分别对应 `公司`、`所属公司`

- **但“直接一次性把 9 张表全部物理字段改成中文”在当前代码基线下不合理**
  - 当前仓库 9 张表合计约 `230` 个字段。
  - 仓库内对这些字段的直接引用约 `564` 处。
  - 其中不仅包含建表 SQL，还包含：
    - Python 入库逻辑
    - 解析器字段映射
    - 刷新存储过程
    - 历史迁移 SQL
    - 方案文档示例 SQL
  - PostgreSQL 使用中文字段名后，几乎所有 SQL 都必须改为带双引号的写法，改动面会明显扩大。

因此，本方案建议采用：

- **命名治理先行**
- **代码收敛后再切库**
- **分两阶段发布，一次正式切换**

不建议“当天梭哈式直接改库再修代码”。

## 3. 命名规则评估与补充

你提出的 3 条规则方向正确，但还不够闭环。建议收敛为以下 **5 条正式规则**。

### 3.1 规则一：报表/页面原始字段优先

若字段直接来自以下来源，则字段名以来源原文为准：

- 下载报表表头
- 系统页面字段标题
- 你已确认并提供的表格字段名

例如：

- `document_no` -> `单据编号`
- `employee_no` -> `工号` 或 `人员编号`
- `position_name` -> `职位` 或 `职位名称`

这里允许同一个英文旧名在不同表中映射为不同中文名。

### 3.2 规则二：方案派生字段以方案口径为准

若字段并非原始采集字段，而是结果字段、规则字段、派生字段，则以对应方案文档中的中文定义为准。

本次重点包括：

- `组织属性查询`
- `权限列表`

例如：

- `war_zone` 应按方案口径落为 `所属战区`
- `org_auth_level` 应按方案口径落为 `组织授权级别`

但需注意：

- 当前部分派生字段在既有方案中只有英文物理名，没有最终确认的中文字段名。
- 这类字段必须先补齐“字段中文名裁决表”，再执行物理迁移。

### 3.3 规则三：不适合作为字段的字符统一规范化

建议统一采用以下规范：

- 空格删除
- `.`、`/`、`-`、中英文括号统一替换为 `_`
- 连续多个 `_` 合并为一个
- 首尾 `_` 去掉

示例：

- `流程层级.名称` -> `流程层级_名称`
- `部门子分类.编码` -> `部门子分类_编码`
- `具体岗位(新)` -> `具体岗位_新`
- `岗位备注（物业）` -> `岗位备注_物业`

### 3.4 规则四：仅以下情况允许保留英文

你原规则中的“字段本身就是英文名，则保持”，建议缩窄为以下 3 类，否则会导致中文化目标落空：

1. 上游权威来源字段名本身就是英文或代码名
2. 技术元数据字段已与外部系统形成稳定约定，改名收益极低
3. JSON 快照类字段更适合保留英文技术语义

建议优先视为“可保留英文候选”的字段类型：

- 纯代码类：如确无正式中文名的特殊编码字段
- 技术审计类：如 `raw_payload`
- 极强工程语义字段：确有兼容必要时才保留

但以下字段**不应因为当前是英文就直接保留**：

- `org_code`
- `company_name`
- `department_name`
- `city_name`
- `permission_level`

因为它们在当前业务中都已有明确中文来源。

### 3.5 规则五：同义字段跨表允许不同名，不强求全局统一

本次目标是“忠实反映来源”，不是“把所有表做成全局统一数据仓库字段字典”。

因此允许出现：

- `工号`
- `人员编号`
- `组织编码`
- `行政组织编码`

分别在不同表中成立。

这不是缺陷，而是对来源口径的保留。

## 4. 表分类与迁移策略

建议先按来源把 9 张表分为三类。

### 4.1 第一类：原始采集型表

这类表优先按页面/报表字段名改：

- `申请单基本信息`
- `申请单权限列表`
- `申请单审批记录`
- `申请表组织范围`
- `在职花名册表`
- `组织列表`

### 4.2 第二类：字典维护型表

这类表按业务口径改：

- `城市所属战区`

建议目标字段倾向为：

- `city_name` -> `城市名称`
- `war_zone` -> `所属战区`

### 4.3 第三类：方案派生型表

这类表必须先完成字段中文名裁决，再实施改名：

- `组织属性查询`
- `权限列表`

原因：

- 它们包含大量非原始采集字段
- 这些字段直接服务于规则、判断、分类、结果展示
- 若仅做“英文转中文”直译，极易造成二义性

## 5. 推荐实施路线

推荐路线为：**两阶段发布，一次切换，不做长期双写**。

### 5.1 阶段 A：命名治理与代码收敛

本阶段不改生产库物理字段，只做准备工作。

交付内容：

1. 产出 `字段映射台账`
   - 至少包含：
     - 表名
     - 旧字段名
     - 新字段名
     - 来源类型
     - 来源依据
     - 是否保留英文
     - 是否需要人工裁决

2. 收敛代码中的散落字段名
   - 重点改造：
     - `automation/db/postgres.py`
     - `automation/utils/roster_excel.py`
     - `automation/utils/organization_list_excel.py`
     - `automation/sql/006_organization_attribute_query.sql`
   - 目标：
     - 业务代码不再直接散写大量列名
     - 列名改由集中映射表生成或引用

3. 补齐方案派生字段中文名
   - 尤其是：
     - `组织属性查询`
     - `权限列表`

4. 冻结字段新增
   - 在迁移窗口前，不再接受新的字段设计变更

### 5.2 阶段 B：正式切换

本阶段在停机窗口内完成物理改名。

建议执行顺序：

1. 备份生产库
2. 执行字段重命名 SQL
3. 同步修改索引、约束、存储过程、迁移脚本
4. 发布新代码
5. 重新跑全量导入/刷新链路
6. 执行验收 SQL

不建议在本次迁移中采用 230 列级别的长期双写方案，原因：

- 复杂度过高
- 易出现双列不同步
- 清理成本极高

## 6. 详细实施分解

### 6.1 第一步：建立字段映射台账

这是本方案的核心前置物，不完成不能动库。

建议台账结构：

- `表名`
- `旧字段名`
- `新字段名`
- `字段类别`
  - 原始采集
  - 方案派生
  - 技术元数据
- `命名依据`
  - 页面
  - 报表
  - 方案文档
  - 保留英文
- `依据位置`
  - 文档名
  - 页面字段名
  - Excel 表头
- `是否需裁决`
- `备注`

### 6.2 第二步：优先改造代码入口层

建议把本仓库所有与数据库列名直接耦合的位置先收敛到两层：

1. `解析层`
   - 负责“页面/报表字段 -> 标准业务键”
2. `落库层`
   - 负责“标准业务键 -> 当前物理字段名”

这样切库时，只需要替换落库层映射，而不是全仓逐点硬改。

### 6.3 第三步：生成正式迁移脚本

建议新增一组专门的字段改名 SQL，例如：

- `automation/sql/012_rename_columns_to_cn_permission.sql`
- `automation/sql/013_rename_columns_to_cn_roster.sql`
- `automation/sql/014_rename_columns_to_cn_org.sql`

脚本内容至少覆盖：

- `ALTER TABLE ... RENAME COLUMN ...`
- 依赖索引重建
- 依赖函数/过程重建
- 依赖示例 SQL 同步修订

### 6.4 第四步：一次全链路重跑

字段重命名完成后，必须至少重跑以下链路：

1. 权限申请采集入库
2. 在职花名册下载入库
3. 组织快速维护下载入库
4. `refresh_组织属性查询()`
5. `权限列表` 初始化或校验

### 6.5 第五步：文档与查询示例同步

仓库内已有多份方案文档引用旧英文字段。

本次切换后，以下文档至少需要同步：

- `002`
- `003`
- `004`
- `005`
- `007`
- `009`
- `010`
- `100`

否则后续查询示例会持续误导。

## 7. 风险评估

### 7.1 高风险点

1. `组织属性查询` 风险最高
   - 该表不仅字段多，还依赖函数、递归 CTE、刷新过程、索引和其他方案口径。

2. `在职花名册表` 风险次高
   - 单表字段 `107` 个，且报表字段多、别名多、历史兼容列多。

3. PostgreSQL 中文字段名要求全量加引号
   - 任一漏改 SQL 都可能在运行时失败。

4. 文档与代码可能短期口径不一致
   - 若先改库后补文档，后续人工查询很容易继续用旧字段名。

### 7.2 主要收益

1. 业务可读性显著提升
2. 报表字段与数据库字段对齐
3. 减少“自行翻译”带来的语义漂移
4. 后续给非研发人员出 SQL 时更直观

## 8. 回滚方案

本次必须准备**反向回滚脚本**，且与正向脚本同批交付。

建议回滚策略：

1. 切换前全库备份
2. 生成逆向 `RENAME COLUMN` 脚本
3. 保留切换前应用版本
4. 若验收失败，在同一窗口内回滚数据库与应用

不建议只备份数据、不准备逆向脚本。

## 9. 验收标准

满足以下条件，才视为 `011` 完成：

1. 9 张表字段名均符合已确认映射台账
2. 允许保留英文的字段均有书面依据
3. 三条主链路可成功执行：
   - 权限申请采集
   - 花名册导入
   - 组织列表导入
4. `组织属性查询` 可成功刷新
5. `权限列表` 可成功初始化或查询
6. 切换前后关键行数一致
7. 切换前后关键业务抽样结果一致
8. 仓库内示例 SQL 已同步为新字段名

## 10. 最终建议

本次迁移**可以做**，但前提是按以下口径执行：

- 不把它当成“简单改列名”
- 把它当成“数据库命名体系重构”
- 先出字段映射台账，再动库
- 先收敛代码，再做一次切换

如果你要我给这件事下最终判断：

- **命名原则合理**
- **直接一把改完不合理**
- **按本方案的两阶段路线推进，合理且可落地**

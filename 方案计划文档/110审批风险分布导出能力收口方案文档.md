# 110 审批风险分布导出能力收口方案文档

## 1. 背景

当前审批风险分布导出能力已经可以产出 Excel，但存在两个工程问题：

1. 导出脚本同时承担参数解析、配置装载、数据库取数、反馈聚合、工作簿写出等职责，难以单测。
2. 导出相关查询与数据组装逻辑散落在脚本内部，后续若被 API、任务脚本或其他入口复用，容易再次复制逻辑。

另外，现阶段仓库内存在“中文路径 + PowerShell 控制台编码”读取体验不稳定的问题。本次不直接回改已有方案文档，避免误读既有文档内容后引入二次破坏；改为新增本专项方案文档，仅描述本次导出能力收口方案。

## 2. 目标

本次仅做导出能力工程收口，不改动既有评估口径：

1. 将审批风险分布导出能力收口到 `automation/reporting` 模块。
2. 保留独立脚本入口，供本地手工导出和后续任务接入。
3. 让“取数 + 聚合 + 导出”链路可被单元测试覆盖。
4. 不调整数据库表结构，不新增数据库字段。

## 3. 范围

涉及目录：

- `automation/reporting`
- `automation/scripts/export_audit_distribution_report.py`
- `tests/test_audit_distribution_report.py`

不在本次范围：

- 既有低分反馈聚合规则调整
- 风险评估表结构调整
- WebUI 导出入口改造
- 历史方案文档回写修订

## 4. 方案

### 4.1 模块边界

新增/明确以下分层：

1. `automation/scripts/export_audit_distribution_report.py`
   - 只保留参数解析
   - 只保留配置读取
   - 只保留日志输出
   - 调用 reporting 模块执行导出

2. `automation/reporting/audit_distribution.py`
   - 负责批次解析
   - 负责查询评估主表、评估明细表、审批记录表
   - 负责把低分反馈概览组装成导出页签所需结构
   - 负责渲染工作簿

### 4.2 数据对象

在 reporting 模块内统一构造导出载荷对象，至少包含：

- `batch_no`
- `assessment_version`
- `summary_rows`
- `detail_rows`
- `approval_rows`
- `ignored_node_names`
- `document_feedback_rows`

这样脚本和测试都只围绕同一份标准导出载荷工作。

### 4.3 测试策略

测试分两层：

1. 纯函数层
   - 统计分布函数
   - 反馈行组装函数
   - 导出载荷构造函数

2. 假数据链路层
   - 用 fake store / fake connection / fake cursor 模拟数据库返回
   - 验证 reporting 模块可完成“取数 -> 组装 -> 导出文件”

## 5. 执行原则

1. 不改动既有评分规则口径。
2. 不新增数据库字段和表。
3. 不在脚本里继续扩散查询 SQL 和聚合逻辑。
4. 后续若 API 也要复用该导出能力，优先直接复用 reporting 模块，不再复制脚本逻辑。

## 6. 验收口径

满足以下条件视为完成：

1. 独立脚本仍可执行审批风险分布导出。
2. workbook 结构与原有导出结果保持兼容。
3. `tests/test_audit_distribution_report.py` 覆盖导出载荷构造与 fake store 导出链路。
4. 脚本内不再保留大段取数与反馈组装逻辑。

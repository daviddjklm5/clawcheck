# 方案编号403：对话工作台实时数据直答能力建设方案文档

## 1. 问题背景

2026-03-21 在 `http://127.0.0.1:5173/chat-workspace` 中出现如下问题：

1. 用户询问“介绍一下本项目”时，助手可以基于仓库知识正常回答。
2. 用户继续询问“单据处理模块有多少条待办”并明确要求“检查本项目文档和 API，通过调用 API 获取”时，助手没有直接给出结果。
3. 助手转而回复“当前会话无法访问运行环境去实际调用 API”，并让用户自己定位接口、手工 `curl`。

用户期望的是：

1. 对于本项目已经存在正式数据口径、正式接口、正式查询函数的问题，助手应直接回答结果。
2. 对实时问题，助手应优先调用本项目自己的后端能力，而不是把“找接口和调接口”的工作转交给用户。

本方案用于明确根因，并给出对话工作台的改造路线。

---

## 2. 现状确认

### 2.1 “单据处理模块待办数”已有正式口径

当前项目已经存在完整的数据链路：

1. 前端 `webui/src/services/api.ts`
   - 已定义 `getProcessWorkbench()`
2. 后端 `automation/api/routers/documents.py`
   - 已暴露 `GET /api/documents/process-workbench`
3. 后端 `automation/api/process_dashboard.py`
   - `get_process_workbench()` 已接入 PostgreSQL 实时查询
4. 数据库侧 `automation/db/postgres.py`
   - 已在 `stats` 中输出标签为 `待处理单据` 的正式统计口径

结论：

1. 问题不在于没有数据源。
2. 问题在于对话链路没有把这条正式能力接入为“可稳定调用”的对话能力。

### 2.2 当前 `chat-workspace` 仍是通用对话执行链路

当前链路为：

1. `webui/src/pages/ChatWorkspacePage.tsx`
2. `automation/api/routers/chat.py`
3. `automation/chat/service.py`
4. `automation/chat/codex_runner.py`
5. `codex exec --json --sandbox read-only`

当前 `ChatService` 会把用户消息直接交给通用模型执行，没有先做项目级的业务路由和上下文装配。

---

## 3. 根因分析

### 3.1 缺少“项目 skill 上下文”

当前对话 prompt 很轻，只包含：

1. 这是 clawcheck 项目
2. 默认用简洁中文
3. 回答最新问题

但没有告诉模型：

1. 本项目有哪些正式工作台
2. 哪些问题应优先走实时查询
3. 实时查询的权威来源是什么
4. 遇到结果冲突时应以哪个系统口径为准

因此模型只能做泛化推理，无法稳定理解“这是一个应优先调内部能力的问题”。

### 3.2 缺少“模型路由阶段”

当前系统没有单独的意图判定阶段。

也就是说，系统没有先让模型回答下面这个问题：

1. 这是开放式聊天问题，还是项目内结构化实时查询问题
2. 如果是查询问题，应调用哪个后端能力
3. 是否需要先读 skill 文件中的特定上下文

因此当前所有请求都直接落入同一种通用执行模式，无法实现稳定的项目内能力编排。

### 3.3 缺少“有界工具注册”

当前对话系统没有把可调用的后端能力以明确工具目录暴露给模型。

结果是模型只能：

1. 自己猜接口
2. 自己猜参数
3. 自己猜字段
4. 自己猜是否允许 shell / curl / 网络调用

这会把本来确定的问题变成不稳定的探索型问题。

### 3.4 当前设计过度依赖模型临场发挥

本问题不是创作问题，而是一个结构化业务查询问题：

1. 目标对象明确：单据处理工作台
2. 统计口径明确：待处理单据
3. 数据来源明确：`process-workbench`
4. 结果字段明确：`stats`

这种问题不适合靠“模型自由发挥 + 临时查仓库”解决，必须改造成：

1. 模型负责理解用户意图
2. 后端负责执行有界工具
3. 模型负责组织最终表述

---

## 4. 方案原则

本方案采用以下原则：

1. 不采用“关键词/正则 -> 函数”的硬编码触发表。
2. 任何意图判断都由大模型完成。
3. 项目上下文通过 skill 文件提供，而不是把大量业务知识硬塞进代码 prompt。
4. 后端只暴露有界、可审计、可测试的正式工具给模型选择。
5. 对极确定的问题，工具执行后可由后端直接模板化回答，不强依赖第二次大模型生成。

这五条原则同时满足：

1. 避免规则表笨拙、难扩展
2. 保留模型语义理解能力
3. 保持后端工具边界可控

---

## 5. 推荐架构

### 5.1 三层分工

推荐分层如下：

1. 后端对话服务负责模型路由、skill 装配、工具执行、审计记录
2. 大模型负责意图判断、skill 选择、工具选择、最终答复组织
3. 正式业务函数继续由现有后端模块提供，不在聊天层重复造逻辑

### 5.2 Skill 作为项目上下文载体

建议新增 chat 专用 skill 目录，例如：

1. `automation/chat/skills/clawcheck-project/SKILL.md`
2. `automation/chat/skills/clawcheck-project/references/process-workbench.md`
3. `automation/chat/skills/clawcheck-project/references/collect-workbench.md`
4. `automation/chat/skills/clawcheck-project/references/answer-policy.md`

职责划分：

1. `SKILL.md`
   - 只放高层规则
   - 说明什么问题属于实时查询
   - 说明正式数据源优先级
   - 说明何时必须调用工具
2. `references/process-workbench.md`
   - 说明处理工作台的正式接口、字段口径、常见问法、返回结构
3. `references/collect-workbench.md`
   - 说明采集工作台相关正式口径
4. `references/answer-policy.md`
   - 说明失败兜底、时间语义、来源表述要求

这样做的好处是：

1. 项目知识不再散落在 prompt 拼接逻辑里
2. skill 可持续演进
3. 模型可按需加载，不必每轮全量注入

### 5.2.1 Skill 文件如何管理、索引和维护

本方案中，skill 不是零散 Markdown 文件，而是受控目录资源。

管理原则如下：

1. `automation/chat/skills/` 是唯一 skill 根目录
2. 每个 skill 必须独占一个子目录
3. 子目录名必须与 skill 名一致，使用小写加连字符
4. `SKILL.md` 是每个 skill 的唯一入口文件
5. `references/`、`assets/` 为可选子目录
6. 不允许用多个散落文件共同充当一个 skill 的入口

推荐目录结构如下：

```text
automation/chat/skills/
  clawcheck-project/
    SKILL.md
    references/
      process-workbench.md
      collect-workbench.md
      answer-policy.md
    assets/
```

本项目对 skill 的“管理”采用“目录即来源”的原则：

1. 以 skill 目录本身作为事实来源
2. 不维护人工手写的全局触发表
3. 不维护人工手写的 skill 索引表
4. skill 索引由 loader 在运行时自动构建

推荐的 skill 元信息最小集合如下：

1. `name`
2. `description`

推荐的项目内扩展元信息如下：

1. `status`
   - `active | deprecated`
2. `owner`
   - 责任模块或责任人
3. `domain`
   - 如 `process-workbench`、`collect-workbench`
4. `updated_at`
   - 最近更新时间
5. `references`
   - 允许暴露给路由模型选择的 reference 名称列表

实现上可采用两种方式：

1. 直接放在 `SKILL.md` frontmatter
2. 放在同目录的 `skill.meta.json`

Phase 1 推荐优先读取 `SKILL.md` frontmatter，避免引入第二份元数据源。

索引方式必须明确：

1. `skill_loader.py` 启动时扫描 `automation/chat/skills/*/SKILL.md`
2. 解析每个 `SKILL.md` 的 frontmatter
3. 枚举 `references/` 下可用文档
4. 为每个 skill 生成内存中的 `SkillIndexEntry`
5. 将所有 `SkillIndexEntry` 组成 `SkillIndex`

`SkillIndexEntry` 建议至少包含：

1. `skill_name`
2. `skill_dir`
3. `description`
4. `status`
5. `owner`
6. `domain`
7. `references`
8. `mtime`
9. `content_hash`

索引的用途必须区分两层：

1. 路由阶段
   - 只向模型暴露 skill 名称、描述和 reference 名称摘要
2. 执行阶段
   - 仅在模型选中后加载对应 `SKILL.md` 正文和指定 reference 内容

这样可以避免：

1. 每轮都把全部 skill 正文塞进上下文
2. skill 越多，路由成本越失控

维护方式也要明确：

1. 新增 skill
   - 新建目录
   - 新建 `SKILL.md`
   - 按需补 `references/`
   - 补 loader 测试
2. 修改 skill
   - 同步更新 `SKILL.md`
   - 若 reference 口径变更，同步更新 reference 文件
   - 若影响路由结果，同步补回归测试
3. 废弃 skill
   - 不直接物理删除
   - 先标记 `status = deprecated`
   - 在索引阶段默认不暴露给路由模型

建议增加一个运行时只读快照文件，便于排障：

1. `automation/runtime/chat/skill-index.json`

用途如下：

1. 查看当前被发现的 skill 列表
2. 查看每个 skill 的 metadata、references、mtime
3. 快速判断“为什么某个 skill 没被加载”

为避免缓存脏读，索引刷新策略建议如下：

1. API 进程启动时全量构建一次
2. 每次对话路由前，按 `mtime` 快速检查是否有 skill 目录变更
3. 有变更时增量重建 `SkillIndex`
4. 生产环境不做文件监听常驻线程，避免复杂度失控

这意味着：

1. skill 的维护是文件级维护
2. skill 的索引是运行时自动索引
3. skill 的启停由 `status` 和索引暴露策略控制

最终责任边界如下：

1. 文档维护者负责 skill 内容正确
2. `skill_loader.py` 负责索引正确
3. `ChatService` 负责只消费已索引的 skill，不直接扫盘拼 prompt

### 5.3 模型路由，而不是规则路由

后端识别意图的方式，明确调整为：

1. 不做正则命中
2. 不做关键词触发表
3. 单独增加一轮“路由模型调用”

路由模型输入包括：

1. 最新用户问题
2. 最近若干轮会话摘要
3. 可用 skill 的元信息
4. 可用工具的名称、描述、参数 schema

路由模型输出必须是结构化 JSON，而不是自由文本。建议字段如下：

```json
{
  "route": "tool_first" | "direct_answer" | "general_chat",
  "selectedSkills": ["clawcheck-project"],
  "selectedReferences": ["process-workbench"],
  "toolCalls": [
    {
      "name": "get_process_workbench",
      "arguments": {}
    }
  ],
  "answerMode": "templated" | "model_generated",
  "reason": "..."
}
```

说明：

1. 这里的“结构化 JSON”是模型输出 schema，不是规则映射表。
2. 后端只负责校验 schema 和执行被允许的工具。
3. 意图本身由模型根据 skill 和工具描述推断。

### 5.4 有界工具注册，而不是让模型自由猜接口

虽然不采用“触发短语/正则映射表”，但后端仍必须有“可调用工具注册表”。

这不是为了识别意图，而是为了控制执行边界。

建议首批注册工具：

1. `get_process_workbench`
   - 返回处理工作台当前正式快照
2. `get_process_document_detail`
   - 返回单据详情
3. `get_collect_workbench`
   - 返回采集工作台当前正式快照

工具注册内容建议包含：

1. `name`
2. `description`
3. `arguments_schema`
4. `resolver`
5. `source_of_truth`
6. `allowed_in_chat`

关键点：

1. 模型决定“该调用哪个工具”
2. 后端决定“哪些工具允许被调用”
3. 模型不允许直接自行发明新接口、新参数、新字段

### 5.5 后端如何识别意图

这是本次用户追问的核心，这里明确回答。

后端识别意图，不再通过正则，而是通过“模型路由 + skill 上下文 + 工具目录”完成。

完整流程如下：

1. 用户消息进入 `ChatService`
2. `ChatService` 不直接进入主对话执行，而是先进入 `route_turn()`
3. `route_turn()` 读取：
   - 最新用户问题
   - 会话最近几轮摘要
   - skill 元信息
   - 工具目录摘要
4. 路由模型输出结构化决策
5. 后端校验：
   - skill 是否存在
   - reference 是否允许
   - tool 是否已注册
   - 参数是否符合 schema
6. 若是 `tool_first`
   - 加载对应 skill / references
   - 执行正式 resolver
7. 若 `answerMode = templated`
   - 后端直接生成最终回复
8. 若 `answerMode = model_generated`
   - 将 skill 上下文 + 工具结果再交给主模型生成
9. 若 `route = general_chat`
   - 回落到原来的通用对话链路

因此，“后端如何识别意图”在修订方案中已经明确：

1. 识别动作由模型完成
2. 后端做的是模型路由编排与执行边界控制
3. skill 文件负责提供项目语义上下文

### 5.5.1 第四点“路由模型输出结构化决策”的具体实现

为避免第四点停留在概念层，这里明确到工程实现。

建议新增以下文件：

1. `automation/chat/router_models.py`
2. `automation/chat/router_prompt.py`
3. `automation/chat/tool_registry.py`
4. `automation/chat/skill_loader.py`
5. `automation/chat/service.py`
6. `automation/chat/codex_runner.py`

分工如下：

1. `router_models.py`
   - 定义路由决策 schema
2. `router_prompt.py`
   - 生成路由阶段专用 prompt
3. `tool_registry.py`
   - 输出允许模型选择的工具目录摘要
4. `skill_loader.py`
   - 输出 skill 元信息和 reference 片段
5. `service.py`
   - 在 `route_turn()` 中编排调用
6. `codex_runner.py`
   - 新增支持 schema 输出的路由执行函数

推荐的路由输出结构如下：

```json
{
  "route": "tool_first",
  "selectedSkills": ["clawcheck-project"],
  "selectedReferences": ["process-workbench"],
  "toolCalls": [
    {
      "name": "get_process_workbench",
      "arguments": {}
    }
  ],
  "answerMode": "templated",
  "confidence": 0.96,
  "missingInputs": [],
  "clarificationQuestion": "",
  "reason": "用户在询问处理工作台当前待办数量，属于项目内实时结构化查询。"
}
```

推荐用 `Pydantic` 定义两个模型：

1. `RouterToolCall`
2. `RouterDecision`

至少包含以下字段：

1. `route`
   - `tool_first | direct_answer | general_chat`
2. `selectedSkills`
3. `selectedReferences`
4. `toolCalls`
5. `answerMode`
   - `templated | model_generated`
6. `confidence`
   - 0 到 1 的置信度
7. `missingInputs`
   - 当前工具调用缺失的必要输入
8. `clarificationQuestion`
   - 当信息不足时，返回给用户的追问语句
9. `reason`
   - 供日志审计使用

路由阶段输入载荷必须控制在“小而必要”的范围内，建议只包含：

1. 最新用户问题
2. 最近 3 到 5 轮会话摘要
3. skill 元信息
   - 仅 `name + description`
4. 工具目录摘要
   - 仅 `name + description + arguments_schema`

路由 prompt 必须明确约束模型：

1. 你现在不是回答用户
2. 你只能做路由决策
3. 只能从给定 skill 和工具中选择
4. 不允许发明新工具、新 skill、新参数
5. 如果信息不足以安全执行工具，不允许猜测或默认补全
6. 信息不足时，必须返回 `missingInputs` 和 `clarificationQuestion`
7. 只能输出符合 schema 的 JSON

推荐执行流程如下：

1. `ChatService.route_turn()` 调用 `build_router_prompt(...)`
2. `route_turn()` 从 `tool_registry.py` 取工具目录摘要
3. `route_turn()` 从 `skill_loader.py` 取 skill 元信息
4. 调用 `codex_runner.run_router_exec(...)`
5. `run_router_exec(...)` 使用 `codex exec --output-schema <schema-file>`
6. 拿到输出后用 `RouterDecision.model_validate_json(...)` 校验

也就是说，第四点的真正实现不是“让主模型顺便决定一下”，而是：

1. 单独一次路由模型调用
2. 单独一份路由 prompt
3. 单独一份结构化 schema
4. 单独的后端校验与回退

建议在 `codex_runner.py` 中新增：

1. `run_router_exec(...)`

它与当前 `run_codex_exec(...)` 的区别是：

1. 不追求流式 token 输出
2. 只接收结构化结果
3. 必须启用 `--output-schema`
4. 返回值不是自然语言，而是 `RouterDecision`

路由失败时的处理要明确：

1. CLI 调用失败
   - 直接回退 `general_chat`
2. 模型输出不是合法 JSON
   - 直接回退 `general_chat`
3. `route` 值不在枚举中
   - 直接回退 `general_chat`
4. `toolCalls` 中出现未注册工具
   - 拒绝执行并回退 `general_chat`
5. `confidence` 低于阈值
   - 回退 `general_chat`
6. `missingInputs` 非空
   - 不执行任何工具，直接进入追问补齐分支

建议默认阈值：

1. `confidence >= 0.75` 才允许进入 `tool_first`

“单据处理模块有多少条待办”的实际执行例子如下：

1. 用户消息进入 `route_turn()`
2. 路由模型输出：
   - `route = tool_first`
   - `selectedSkills = ["clawcheck-project"]`
   - `selectedReferences = ["process-workbench", "answer-policy"]`
   - `toolCalls = [{"name": "get_process_workbench", "arguments": {}}]`
   - `answerMode = templated`
3. 后端校验通过
4. 执行 `get_process_workbench`
5. 从结果中取 `stats` 里 `待处理单据`
6. 因为 `answerMode = templated`
   - 后端直接生成最终答复

“这张单据现在什么状态”的信息不全例子如下：

1. 用户消息进入 `route_turn()`
2. 路由模型输出：
   - `route = direct_answer`
   - `selectedSkills = ["clawcheck-project"]`
   - `selectedReferences = ["answer-policy"]`
   - `toolCalls = []`
   - `answerMode = model_generated`
   - `missingInputs = ["documentNo"]`
   - `clarificationQuestion = "请提供单据编号，我再帮你查询当前状态。"`
3. 后端检测到 `missingInputs` 非空
4. 本轮不执行任何工具
5. 直接把 `clarificationQuestion` 返回给用户

在本方案中，这一条是强约束：

1. 只要工具执行所需关键参数缺失，就不得默认猜测
2. 必须先追问补齐，再进入工具执行
3. 仅当问题本身允许无歧义默认值时，才允许不追问

允许不追问的例外仅限于：

1. 用户明确询问工作台总览类问题
2. 工具本身允许空参数且语义唯一

因此，第四点在工程上应被定义为：

1. 一次独立的模型路由调用
2. 一份严格 schema 的结构化输出
3. 一套后端可程序化校验的决策对象
4. 一条失败即可安全回退到通用聊天的链路

### 5.6 为什么这比规则表更合适

原因如下：

1. 用户问法天然多样，规则表扩展成本高
2. “这个问题是否属于处理工作台待办数查询”本质是语义判断，不是字符串匹配
3. 本项目未来会继续增加采集工作台、评估分析、主数据同步等域问题，skill 更适合承载域知识
4. 模型路由可以利用会话上下文，而规则表通常只能看当前一句话

## 6. 具体落地方式

### 6.1 新增 Router 阶段

建议在 `automation/chat/service.py` 里新增一层 orchestrator，而不是直接 `_build_prompt() -> run_codex_exec()`。

建议分成三个阶段：

1. `route_turn()`
   - 做模型路由
2. `execute_tool_plan()`
   - 执行后端正式工具
3. `compose_answer()`
   - 模型生成最终答复，或直接模板化答复

### 6.2 新增 Skill Loader

建议新增：

1. `automation/chat/skill_loader.py`

职责：

1. 发现 chat skill 目录
2. 读取 `SKILL.md` frontmatter
3. 按需加载 `references/*.md`
4. 控制最大上下文注入范围

这样后端可以做到：

1. 路由阶段只加载 skill 元信息
2. 选中 skill 后再加载 skill 正文
3. 选中 reference 后再加载对应 reference

### 6.3 路由模型的输出约束

建议路由阶段使用严格 schema 约束输出。

可以考虑两种实现方式：

1. 若继续沿用 `codex exec`
   - 使用 `--output-schema`
2. 若后续改为直接模型 API
   - 使用响应 schema / JSON mode

验收要求：

1. 路由模型不能返回自由散文
2. 后端必须能程序化验证
3. 校验失败要有明确 fallback

### 6.3.1 `codex exec --output-schema` 的落地约定

考虑到当前项目已经通过 `automation/chat/codex_runner.py` 调用 `codex exec`，Phase 1 推荐沿用这条链路，不额外引入新的模型 SDK。

建议实现如下：

1. 在运行时生成临时 schema 文件
   - 例如写入 `automation/runtime/chat/router-decision.schema.json`
2. 在 `run_router_exec(...)` 中调用：
   - `codex exec --json --output-schema <schema-file> --skip-git-repo-check --sandbox read-only --cd <workspace>`
3. stdin 传入路由 prompt，而不是主回答 prompt
4. 只读取最后一个满足 schema 的 assistant 输出

`run_router_exec(...)` 的返回对象建议包含：

1. `decision`
   - `RouterDecision | None`
2. `status`
   - `succeeded | failed | timeout`
3. `raw_output`
   - 保留原始输出用于审计
4. `error_message`
   - 失败原因

后端对该返回对象的处理约定：

1. `status = succeeded` 且 `decision != None`
   - 进入后续执行
2. 其他情况
   - 一律按 `general_chat` 回退

后端拿到合法 `decision` 后，执行顺序必须固定为：

1. 若 `missingInputs` 非空
   - 不执行工具
   - 直接返回 `clarificationQuestion`
2. 若 `route = tool_first`
   - 再进入工具执行
3. 若 `route = direct_answer`
   - 进入直接答复分支
4. 若 `route = general_chat`
   - 回退原有通用链路

这样可以保证：

1. 不需要把“路由”塞进主回答模型的自由文本里
2. 不需要依赖脆弱的字符串解析
3. 路由失败不会阻塞原有通用对话能力
4. 信息不全时一定先追问，而不是猜测参数

### 6.4 结果生成策略

工具执行成功后，答复分两类：

1. 模板化直答
   - 适用于“当前待处理单据多少条”这类极确定问题
2. 模型组织答复
   - 适用于“为什么这些单据待处理”“请对处理工作台做总结”这类需要解释的问题

示例：

1. 若用户问“单据处理模块有多少条待办”
   - 工具执行后直接模板化返回
2. 若用户问“处理工作台整体情况如何，重点风险在哪”
   - 工具返回结构化结果后再交给模型总结

若信息不全，则本节策略暂停，先进入“追问补齐”分支：

1. 优先级高于 `templated`
2. 优先级高于 `model_generated`
3. 只有用户补齐参数后，才重新进入路由与工具执行

### 6.5 失败兜底

需要明确三层失败处理：

1. 路由失败
   - 回落到通用聊天
   - 同时记录路由失败日志
2. 工具执行失败
   - 返回“已尝试调用的正式数据源 + 失败原因 + 下一步动作”
3. 结果生成失败
   - 至少返回工具原始摘要，不丢结果

禁止再出现这种兜底：

1. “你自己去仓库搜 API”
2. “你自己去 curl”

同时增加一条非失败但必须处理的分支：

1. 信息不全
   - 不属于失败
   - 不允许猜
   - 必须追问补齐
   - 追问内容由路由模型通过 `clarificationQuestion` 生成

## 10. 结论

本次问题的根因不是没有实时接口，而是对话工作台没有形成“模型路由 + skill 上下文 + 有界工具”的架构。

修订后的明确结论如下：

1. 后端识别意图，应通过大模型完成，而不是通过正则和触发表完成。
2. 项目业务上下文，应通过 skill 文件提供，而不是堆进通用 prompt。
3. 后端必须维护有界工具目录，供模型选择和调用。
4. 对确定性强的问题，工具执行后允许后端直接模板化答复。

按本方案落地后，`chat-workspace` 才能从“通用聊天入口”升级为“可直接回答本项目实时业务数据”的项目内智能工作台。

# 方案编号005：组织属性查询表设计方案文档

## 1. 设计目标
在现有 `组织列表` 与 `城市所属战区` 两张基础表之上，新增一张面向高频查询场景的结果表：
- `组织属性查询`

该表用于承接以下查询诉求：
- 按组织编码、组织名称、公司、城市、战区、组织单位、万御城市营业部、组织流程层级判断、组织流程层级分类等维度高频查询
- 对 `组织列表.org_full_name` 做稳定、可复用的“组织单位”映射
- 对 `组织列表.city_name` 做“所属战区”映射
- 在 `组织列表` 全量更新后，同步全量刷新结果表

本方案目标不是替代 `组织列表`，而是构建一张：
- 字段稳定
- 口径固定
- 可直接用于业务查询
- 避免每次查询时重复解析路径和关联战区的派生表

## 2. 设计结论

### 2.1 最终选型
本方案建议采用：
- **实体结果表**：`组织属性查询`
- **数据库函数**：用于 `org_full_name -> 组织单位` 映射
- **数据库存储过程**：用于全量重建 `组织属性查询`
- **应用侧导入流程调用刷新过程**：在 `组织列表` 全量写入完成后立即刷新

不建议采用以下方案作为最终实现：
- 普通 `VIEW`
- `MATERIALIZED VIEW`
- `GENERATED COLUMN`
- 行级 `TRIGGER`

原因如下：

1. `VIEW`
   - 每次查询都要重复做路径拆分与战区关联
   - 不适合“将被频繁查询”的目标场景

2. `MATERIALIZED VIEW`
   - 需要额外刷新管理
   - 对中文表名、定制函数、索引和切换控制不如实体表直观
   - 当前项目已存在全量导入逻辑，用实体表更容易和现有流程对齐

3. `GENERATED COLUMN`
   - `所属战区` 依赖 `城市所属战区` 表，生成列不适合跨表查值
   - 组织单位规则后续可能持续扩充，放在函数 + 刷新过程更灵活

4. 行级 `TRIGGER`
   - 当前 `组织列表` 是全量覆盖导入
   - 若逐行维护结果表，复杂度更高，整体效率也未必优于一次性全量重建

### 2.2 总体策略
采用“**源表全量导入 -> 结果表全量重建**”模式：

1. 自动化导出并导入 `组织列表`
2. `组织列表` 落库成功后，调用数据库存储过程
3. 存储过程基于 `组织列表` + `城市所属战区` 全量重建 `组织属性查询`
4. 业务查询统一查询 `组织属性查询`

该方案在当前数据量级下最稳妥：
- `组织列表` 当前约 `32020` 行
- 即使全量重建，成本也可控
- 读取性能优于每次在线计算

## 4. 目标表设计

### 4.1 表名
- `组织属性查询`

### 4.2 字段设计
建议字段如下：

#### 4.2.1 基础字段
- `org_code TEXT PRIMARY KEY`
- `row_no INTEGER`
- `org_name TEXT`
- `org_type TEXT`
- `parent_org_name TEXT`
- `parent_org_code TEXT`
- `company_name TEXT`
- `org_level TEXT`
- `city_name TEXT`
- `physical_level TEXT`
- `dept_subcategory_code TEXT`
- `dept_subcategory_name TEXT`
- `dept_category_code TEXT`
- `dept_category_name TEXT`
- `org_full_name TEXT`

#### 4.2.2 派生字段
- `org_unit_name TEXT`
  - 由 `org_full_name` 计算得到
  - 代表统一口径下的“组织单位”

- `war_zone TEXT`
  - 由 `city_name` 关联 `城市所属战区` 得到
  - 代表“所属战区”

- `wanyu_city_sales_department TEXT`
  - 代表“万御城市营业部”结果列
  - 当 `L4` 后 `5` 个字命中 `城市营业部` 时，取 `L4`
  - 否则取 `NULL`

- `process_level_name_resolved TEXT`
  - 代表“组织流程层级判断”结果列
  - 若 `L2 = 蝶城发展中心` 且 `L3 = 人力资源与行政服务部`，并且 `L4` 为空或不等于 `研选家人力资源组`，则优先取 `蝶发人行部`
  - 若 `L3 = 人力资源与行政服务中心` 且 `L4` 有值，则优先取 `L4`
  - 以上两类特殊路径规则均高于当前组织值和上级递归回溯规则
  - 否则优先取当前组织 `process_level_name`
  - 若当前组织为空，则沿 `parent_org_code` 向上追溯最近一个非空 `process_level_name`
  - 若整条上级链路均为空，则结果为 `NULL`

- `process_level_category TEXT`
  - 代表“组织流程层级分类”结果列
  - 基于 `process_level_name_resolved` 与 `company_name` 进行分类映射
  - 当 `company_name = 人事远程交付中心` 时优先取 `人事远程交付中心`
  - 当 `L3 = 人力资源与行政服务中心` 且 `L4` 有值且不等于 `BG人力资源行政服务中心`，并且组织路径最后 `7` 个字命中 `人事交付服务组` 时，取 `属地服务站`
  - 当 `L3 = 人力资源与行政服务中心` 且 `L4` 有值且不等于 `BG人力资源行政服务中心` 时，取 `战区人行部门`
  - 否则当 `process_level_name_resolved` 为空时取 `NULL`，其余分类结果包括：`万物云本部`、`业务单元本部`、`人事远程交付中心`、`属地服务站`、`战区人行部门`、`属地组织`

- `org_unit_rule TEXT`
  - 记录本行命中的组织单位规则
  - 用于排查映射逻辑、审计、抽样验证

#### 4.2.3 同步审计字段
- `source_import_batch_no TEXT`
  - 来源于 `组织列表.import_batch_no`
- `refreshed_at TIMESTAMPTZ NOT NULL`
  - 本次结果表刷新时间
- `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`
- `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`

### 4.3 主键与唯一性
建议：
- 主键：`org_code`

原因：
- `org_code` 在 `组织列表` 中已是天然唯一编码
- 结果表与源表保持同一唯一键最稳妥

## 5. 组织单位映射口径

### 5.1 总体规则
以 `org_full_name` 为输入，使用 `_` 拆分层级：
- `L1 = split_part(org_full_name, '_', 1)`
- `L2 = split_part(org_full_name, '_', 2)`
- `L3 = split_part(org_full_name, '_', 3)`
- `L4 = split_part(org_full_name, '_', 4)`

采用“**细粒度优先，其次直接命中**”策略。

### 5.2 特殊同名映射

- `万物云` → `万物云`

## 6. 组织流程层级判断口径

### 6.1 字段定义
新增派生字段：
- `process_level_name_resolved`

业务含义：
- 作为“组织流程层级判断”统一口径列
- 用于补齐当前组织 `process_level_name` 为空时的查询口径
- 保留源字段 `process_level_name`，同时提供可直接查询的最终判断值

### 6.2 取值规则
按以下优先级取值：

1. 特殊路径规则优先
- 若 `L2 = 蝶城发展中心` 且 `L3 = 人力资源与行政服务部`，并且 `L4` 为空或不等于 `研选家人力资源组`，则：
  - `process_level_name_resolved = 蝶发人行部`
- 若 `L3 = 人力资源与行政服务中心` 且 `L4` 有值，则：
  - `process_level_name_resolved = L4`
- 以上两类特殊路径规则均高于当前组织值和上级递归回溯规则

2. 当前组织优先
- 若未命中上述规则，且当前行 `process_level_name` 有值，则：
  - `process_level_name_resolved = process_level_name`

3. 上级组织递归回溯
- 若未命中上述规则，且当前行 `process_level_name IS NULL`，则：
  - 取其上级组织的 `process_level_name`
- 若其上级组织的 `process_level_name` 仍为 `NULL`，则：
  - 继续沿 `parent_org_code` 向上追溯
- 一直追溯至某个隔级上级首次出现非空 `process_level_name`
- 命中后：
  - `process_level_name_resolved = 该上级组织的 process_level_name`

4. 全链路均为空
- 若当前组织及全部上级组织的 `process_level_name` 均为空，则：
  - `process_level_name_resolved = NULL`

### 6.3 技术实现建议
建议在 `refresh_组织属性查询()` 中使用递归 CTE 一次性完成整表回溯计算：

1. 从 `组织列表` 每一行作为起点
2. 先判断当前行 `process_level_name`
3. 对空值行沿 `parent_org_code -> org_code` 逐级向上关联
4. 对命中特殊路径规则的组织，优先取规则指定值作为最终值
5. 对其余组织取最近一层非空 `process_level_name` 作为最终值
6. 将结果落入 `process_level_name_resolved`

这样设计的好处：
- 刷新时一次性固化结果，避免查询时递归
- 可覆盖“父级为空、祖父级有值”的隔级继承场景
- 与当前“源表全量导入 -> 结果表全量重建”模式一致

### 6.4 组织流程层级分类口径
新增派生字段：
- `process_level_category`

按以下优先级取值：

1. 公司优先覆盖
- 当 `company_name = 人事远程交付中心` 时：
  - `process_level_category = 人事远程交付中心`
- 该规则高于 `process_level_name_resolved` 为空及其他分类规则

2. 判定为“属地服务站”
- 当 `L3 = 人力资源与行政服务中心` 且 `L4` 有值，且 `L4 <> BG人力资源行政服务中心`，并且组织路径最后 `7` 个字命中 `人事交付服务组` 时：
  - `process_level_category = 属地服务站`
- 该规则优先于“战区人行部门”规则

3. 判定为“战区人行部门”
- 当 `L3 = 人力资源与行政服务中心` 且 `L4` 有值，且 `L4 <> BG人力资源行政服务中心` 时：
  - `process_level_category = 战区人行部门`

4. `process_level_name_resolved` 为空
- 当 `process_level_name_resolved IS NULL` 时：
  - `process_level_category = NULL`

5. 判定为“万物云本部”
- 当 `process_level_name_resolved` 命中以下值时：
  - `万物云本部`
  - `本部部门`
  - `本部二级部门`
  - `本部三级部门`
  - `万科物业`
- 则：
  - `process_level_category = 万物云本部`

6. 判定为“业务单元本部”
- 当 `process_level_name_resolved` 命中以下值时：
  - `本部`
  - `修缮业务本部`
  - `朴邻本部`
  - `万御电梯本部`
  - `福讯信息`
- 则：
  - `process_level_category = 业务单元本部`

7. 其他情况
- 除上述情况外：
  - `process_level_category = 属地组织`

### 6.5 查询使用建议
后续查询口径建议如下：
- 保留 `process_level_name` 作为源数据原值
- 业务查询、筛选、导出优先使用 `process_level_name_resolved`
- 对原“属于人行中心”场景，统一改查 `process_level_name_resolved`
- 若需按组织流程口径分组统计，优先使用 `process_level_category`

## 8. 同步刷新机制设计

### 8.1 刷新触发时机
刷新时机应定义为：
- `组织列表` 全量导入完成后立即刷新

即在以下流程后增加一步：
1. 自动化下载组织列表 Excel
2. 解析并全量写入 `组织列表`
3. 调用 `refresh_组织属性查询()`

### 8.2 刷新策略
采用“**全量重建**”而不是增量更新：

原因：
- `组织列表` 本身就是快照型全量覆盖
- 组织路径或城市发生变化时，增量维护复杂度更高
- 当前数据量较小，全量重建成本可接受

### 8.3 推荐实现方式
建议使用：
- `fn_map_org_unit_name(org_full_name text)`
- `fn_map_org_unit_rule(org_full_name text)`
- `refresh_组织属性查询()`

其中：
- 函数负责组织单位映射
- 递归 CTE 负责 `process_level_name_resolved` 回溯取值
- 存储过程负责整表重建

### 8.4 刷新过程建议步骤
建议刷新过程采用“影子表 + 原子切换”模式：

1. 创建临时构建表，例如：`组织属性查询__staging`
2. 将 `组织列表` 与 `城市所属战区` 关联，并同步完成 `process_level_name_resolved` 递归回溯后全量插入到 staging 表
3. 在 staging 表上补建索引
4. 执行 `ANALYZE`
5. 在事务内完成：
   - 旧表重命名为备份名
   - staging 表重命名为正式表 `组织属性查询`
   - 删除旧备份表

这样设计的好处：
- 避免高频查询时读到半成品数据
- 避免 `TRUNCATE + INSERT` 期间结果表短暂为空
- 更适合后续定时任务场景

### 8.5 与定时任务的关系
后续若存在定时任务定时刷新 `组织列表`：
- 定时任务完成 `组织列表` 全量导入后
- 同一任务链路内继续调用 `refresh_组织属性查询()`

即：
- `组织属性查询` 不直接采集
- 始终由 `组织列表` 派生

### 9.2 索引建议

## 12. 风险与边界

### 12.1 城市字典未覆盖
风险：
- `war_zone` 出现空值

控制方案：
- 允许 `war_zone` 为空
- 定期补齐 `城市所属战区`

### 12.2 组织路径新变体
风险：
- `org_full_name` 后续出现新命名模式
- 可能导致“组织单位”误分配到较粗粒度节点

控制方案：
- 在函数中维护细粒度优先规则
- 每次组织架构变化后抽样评估
- 必要时补充规则并全量刷新

### 12.3 组织层级链路异常
风险：
- 若 `parent_org_code` 存在错误配置、断链或循环引用
- 可能导致 `process_level_name_resolved` 无法按预期回溯

控制方案：
- 递归查询中增加循环保护
- 若链路断裂，则保留当前已找到的最近非空值；若始终未找到则置空
- 建议后续增加组织树巡检 SQL，识别异常父子关系

### 12.4 高频查询下的刷新影响
风险：
- 结果表刷新过程影响在线查询

控制方案：
- 使用 staging 表 + 原子切换
- 避免直接 `TRUNCATE` 正式查询表

## 13. 最终结论
本方案最终建议如下：

1. 新增结果表：`组织属性查询`
2. 数据来源：`组织列表` + `城市所属战区`
3. 组织单位口径：以 `org_full_name` 为准，采用“细粒度优先 + 直接命中 + 特殊同名映射”
4. 组织流程层级判断口径：优先命中特殊路径规则（`L2 = 蝶城发展中心` 且 `L3 = 人力资源与行政服务部` 且 `L4` 为空或不等于 `研选家人力资源组` 时取 `蝶发人行部`；`L3 = 人力资源与行政服务中心` 且 `L4` 有值时取 `L4`），否则取当前组织 `process_level_name`，为空时沿上级组织链路递归回溯最近非空值，落入 `process_level_name_resolved`
5. 组织流程层级分类口径：基于 `process_level_name_resolved`、`company_name` 与组织路径分类；当 `company_name = 人事远程交付中心` 时优先归类为 `人事远程交付中心`，当 `L3 = 人力资源与行政服务中心` 且 `L4` 有值且不等于 `BG人力资源行政服务中心` 且组织路径最后 `7` 个字命中 `人事交付服务组` 时优先归类为 `属地服务站`，否则该组规则归类为 `战区人行部门`，当 `process_level_name_resolved` 为空时结果为 `NULL`，其余分类包括 `万物云本部`、`业务单元本部`、`属地组织`
6. 万御城市营业部口径：当 `L4` 后 `5` 个字命中 `城市营业部` 时，写入 `wanyu_city_sales_department = L4`，否则为 `NULL`
7. 所属战区口径：以 `city_name` 关联 `城市所属战区.war_zone`
8. 同步策略：`组织列表` 每次全量更新后，同步全量刷新 `组织属性查询`
9. 查询策略：业务统一查询 `组织属性查询`，其中组织流程层级优先使用 `process_level_name_resolved`
10. 性能策略：为高频过滤字段建立索引，避免查询时重复计算

在当前数据范围与业务模式下，这是最适合落地、维护成本最低、查询性能最稳定的设计方案。

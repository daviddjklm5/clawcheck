# 设计编号003：权限申请待办单据采集入库技术设计文档

## 1. 设计目标
基于现有 `automation/` 下的 `Python + Playwright` 自动化框架，新增“权限申请待办单据采集入库”能力，实现以下目标：
- 从 iERP 首页进入“我的消息 -> 待办任务”
- 识别 `单据=权限申请` 的待办记录
- 进入详情页后采集基本信息、权限申请明细、审批记录、行政组织详情组织编码
- 将数据落库至 PostgreSQL
- 支持 `--dry-run` 本地 JSON 落盘验证，降低联调风险

## 2. 技术方案概览

### 2.1 运行入口
沿用现有入口文件：`automation/scripts/run.py`

新增动作：
- `collect`

示例命令：
```bash
python automation/scripts/run.py collect --dry-run --document-no QX-260311-00000223 --headless
python automation/scripts/run.py collect --document-no QX-260311-00000223 --headless
```

### 2.2 核心模块划分
- `automation/scripts/run.py`
  - 解析 `collect` 命令参数
  - 处理登录态与运行参数
  - 触发采集流程
  - 负责 JSON 落盘与 PG 写库
- `automation/flows/permission_collect_flow.py`
  - 负责页面导航与数据采集
  - 负责明细表、审批记录、组织编码滚动抓取
- `automation/db/postgres.py`
  - 负责 PostgreSQL 连接与写入逻辑
- `automation/sql/001_permission_apply_collect.sql`
  - 负责建表 DDL

## 3. 页面自动化流程设计

### 3.1 总流程
1. 打开首页。
2. 校验登录态，若失效则重新登录。
3. 点击“我的消息 -> 待办任务”。
4. 将待办列表分页切换为 `1000条/页`。
5. 读取待办列表 grid。
6. 过滤 `单据 = 权限申请`。
7. 根据 `--document-no` 或 `--limit` 确定采集范围。
8. 进入每条目标单据详情。
9. 采集基本信息。
10. 若 `申请单基本信息` 中已存在该 `单据编号`，则先切换“审批记录”页签，过滤空审批步骤，补齐审批人工号，并生成当前单据的最新审批时间。
11. 比较 `申请单基本信息.最新审批时间` 与当前单据审批记录最后一条审批时间：
    - 一致：判定为已采集完成，跳过整单重采；
    - 不一致：判定为单据发生退回、转办后重新提交，继续执行整单重采。
12. 若 `申请单基本信息` 中不存在该 `单据编号`，或审批末条时间发生变化，则继续执行整单采集。
13. 采集权限申请明细。
14. 对每条明细点击“行政组织详情”。
15. 在当前单据页滚动采集组织编码，采完后按 `Esc` 关闭详情框。
16. 当前单据处理完成后，关闭当前申请单据页签，返回“待办任务”列表。
17. 继续进入下一条目标单据，直至本轮采集完成。
18. 输出 JSON。
19. 非 `--dry-run` 时写入 PG。

### 3.2 待办列表定位策略
待办列表页使用 grid 表头识别，而不是依赖动态页面 ID。
目标表头：
- `单据`
- `单据编号`
- `发起人`
- `主题`
- `状态`

进入详情时，点击主题列中的链接文本：
- `单据编号：{document_no}`

分页策略：
- 进入待办页后立即检查分页按钮当前值
- 若不是 `1000条/页`，则点击右上角分页下拉并切换到 `1000条/页`
- 从详情页返回待办页后再次执行同样检查，避免页面恢复默认分页

待办列表行读取策略：
- 待办 grid 为虚拟滚动列表，不能只读取当前 DOM 中可见的 `tbody tr`
- 采集时需在待办 grid 内纵向滚动，按 `单据编号` 去重收集所有已渲染批次
- 若页面显示 `共N条`，则以该数量作为待办列表全量采集的期望值
- 当收集数量达到 `共N条`，或滚动到底部后连续多轮无新增行，视为待办列表读取完成

跨单据返回策略：
- 单据详情采集完成后，不再重新打开首页进入“待办任务”
- 改为关闭当前申请单据所在页签，再回到已打开的 `待办任务` 页签
- 回到待办列表后再次校验分页仍为 `1000条/页`

### 3.3 基本信息采集策略
基本信息区域使用字段容器 `#fs_baseinfo` 作为根节点。
每个字段容器按以下规则解析：
1. 读取字段标题 `.kd-cq-field-title-wrap span`
2. 优先读取 `input/textarea` 的 `value`
3. 若无，则读取下拉选中值 `.kd-cq-combo-selected`
4. 仍无值时，读取 `.kd-cq-field-value-wrap` 文本

该策略避免了依赖输入框固定顺序，较“按 input 序号取值”更稳。

### 3.4 权限申请明细采集策略
权限申请明细区域是标准 grid，按表头匹配：
- `申请类型`
- `角色名称`
- `角色编码`
- `行政组织`
- `行政组织详情`

实现策略：
- 通用读取 `.kd-table-container`
- 匹配包含目标表头的 grid
- 从 `tbody tr -> td` 提取每行数据
- 以表头映射为字典，避免写死列号
- `行政组织详情` 不再直接落库文本，改为提取 `查看详情(n)` 中的 `n` 写入 `org_scope_count`

### 3.5 审批记录采集策略
审批记录区域不是标准表格，而是时间轴卡片结构。
已确认可见容器为：
- `#tabpageap_approvalrecord`

审批记录区可能存在内部纵向滚动或惰性渲染，不能只读取首屏当前已渲染卡片。
处理策略：
- 切换到 `审批记录` 页签后，优先定位 `#tabpageap_approvalrecord`；
- 识别承载审批卡片的纵向滚动容器；
- 从顶部开始循环收集当前可见卡片，并持续下滚直到无新增卡片或触底；
- 以卡片 `raw_text` 为主键去重，保证“驳回后重新提交”等多轮审批记录可完整保留。

每条审批记录卡片解析规则：
- 节点与审批人信息：`h4`
- 时间：`p.zPG-NFNf`
- 审批意见：`p._3aIXYPkW`（当 display 非 none 时）
- 原始展示文本：整张卡片 `innerText`

字符串拆解规则：
- 从 `h4` 中提取：`节点名称 + 审批人|组织职位 + 审批动作`
- 若结构异常，则保留 `raw_text` 兜底

空步骤过滤规则：
- 若卡片 `raw_text` 精确等于 `属地人力资源部负责人 通过规则：全部通过`，则该卡片视为空审批步骤；
- 空审批步骤不写入 `申请单审批记录`；
- `审批记录顺序号` 以过滤后的保留记录重新从 `1` 开始编号。

审批人工号匹配规则：
- 新增审批记录字段：`工号`；
- 若审批人文本为 `姓名(工号)` 或 `姓名（工号）`，直接提取括号中的工号作为审批人工号；
- 若审批人文本不是该格式，则按 `审批人 = 在职花名册表.姓名` 精确匹配，取 `在职花名册表.人员编号`；
- 若未匹配到唯一工号，则审批记录仍保留，但 `工号` 置空并记录日志。

最新审批时间派生规则：
- 新增主表字段：`最新审批时间`；
- 取过滤后最后一条审批记录的 `审批时间` 回填到 `申请单基本信息`；
- 若过滤后无有效审批记录，或最后一条审批记录无时间，则 `最新审批时间 = NULL`。

重采判定规则：
- 对已入库单据，先轻量采集当前单据的审批记录；
- 将过滤后最后一条审批记录时间与 `申请单基本信息.最新审批时间` 比较；
- 若两者一致，则视为单据未发生需要重采的状态变化，本轮直接跳过；
- 若两者不一致，则视为该单据在退回、转办等场景后重新提交，需要整单删除后重采。

### 3.6 行政组织详情采集策略
行政组织详情是虚拟滚动 grid，不是一次性把全部行渲染到 DOM 中。
因此不能只取 `tbody tr` 当前可见行，否则会漏抓。

处理策略：
1. 先在权限申请明细区识别当前单据的目标明细 grid，并在整单采集过程中复用该 grid 选择器。
2. 按行号逐行点击明细行中的 `行政组织详情` 链接。
3. 匹配包含以下表头的组织详情 grid：
   - `组织编码`
   - `组织名称`
   - `所属公司`
   - `组织长名称`
4. 定位其滚动容器：`.kd-table-body.kd-horizontal-scroll-container`
5. 循环执行：
   - 收集当前可见 `tbody tr` 中的 `组织编码`
   - 下滚 `scrollTop`
   - 再次收集并去重
6. 终止条件：
   - 已采集数量达到 `查看详情(n)` 中的期望数量；或
   - 滚动到底部且连续多轮无新增编码
7. 读取完成后，在当前单据页面按 `Esc` 关闭详情框，再继续下一条明细

补充实现约束：
- 不再为每一条明细重新打开单据页面
- 明细 grid 在横向滚动到右侧后，仍复用首次识别出的同一 grid，而不是重新按左侧表头扫描整页
- 对 `不检查组织范围 = TRUE` 的角色，即使页面上仍显示 `查看详情(n)`，采集层也按规则跳过组织详情读取，只保留空组织列表

### 3.7 去重策略
去重分两层：
- 采集层：Python `set()` 去重
- 数据库层：`申请表组织范围(document_no, org_code)` 唯一约束再次防重

## 4. 数据库写入设计

### 4.1 写入模式
采用“采前判重 + 变化后整单删重写”的方式：
- `申请单基本信息` 中不存在该 `单据编号`：直接采集并写入，`采集次数 = 1`
- `申请单基本信息` 中已存在该 `单据编号` 且 `最新审批时间` 未变化：跳过写入
- `申请单基本信息` 中已存在该 `单据编号` 且 `最新审批时间` 已变化：先按 `单据编号` 删除主表及全部子表记录，再整单重写，`采集次数 = 原值 + 1`
- 主表实际落库仍使用 `ON CONFLICT(document_no) DO UPDATE` 兜底，避免并发或重试场景下的重复主键错误

### 4.2 这样设计的原因
- 大量待办单据会重复出现在列表中，先比对审批末条时间可以减少无效全量采集
- 页面状态可能变化（如退回、转办、重新提交后审批记录末条时间变化）
- 发生变化时整单清空后重写最简单、可预测，便于追平最新页面数据并避免新旧子表数据混杂

### 4.3 表设计说明
建表 SQL 文件：`automation/sql/001_permission_apply_collect.sql`

核心表：
- `申请单基本信息`
- `申请单权限列表`
- `申请单审批记录`
- `申请表组织范围`

关联方式：
- 均通过 `document_no` 关联

本次新增字段：
- `申请单基本信息.最新审批时间`
- `申请单基本信息.采集次数`
- `申请单审批记录.工号`

## 7. 异常与兜底设计

### 7.1 登录态失效
- 自动判断登录态
- 若失效则重新登录并刷新 `storage_state`

### 7.2 页面结构波动
- 表格采集按“表头集合”识别，不依赖动态 pageId
- 基本信息按字段标题解析，不依赖输入框固定序号

### 7.3 审批记录结构异常
- 拆解失败时保留 `raw_text`
- 避免因为单条解析异常导致整单失败
- 精确匹配到 `属地人力资源部负责人 通过规则：全部通过` 的空步骤按过滤规则直接跳过，不视为异常

### 7.4 组织编码滚动不完整
- 通过 `查看详情(n)` 识别期望数量
- 滚动过程中持续去重
- 达到期望数量优先终止
- 若未达到期望数量但已无新增，记录日志，便于后续排障

### 7.5 详情框关闭与行间切换
- 组织详情采集完成后，通过键盘 `Escape` 关闭详情框
- 若一次 `Escape` 未关闭，允许再次发送 `Escape` 兜底
- 关闭成功后直接继续下一行，避免反复返回待办页或重新打开单据导致的额外波动

### 7.6 单据页签关闭与跨单据切换
- 单据详情整体采集完成后，优先关闭当前申请单据页签，而不是重新从首页进入“待办任务”
- 关闭动作作用于顶部页签栏中当前激活的申请单据页签
- 若页签关闭失败，可降级为直接点击 `待办任务` 页签兜底，但正常路径应保持“同一待办会话内连续采集”
- 关闭后需等待 `待办任务` grid 恢复可见，并重新校验分页大小与表头状态

### 7.7 审批人工号匹配异常
- 若审批人文本中未携带括号工号，则按 `在职花名册表.姓名` 匹配 `人员编号`
- 若花名册未匹配到唯一人员编号，则 `申请单审批记录.工号` 允许为空
- 单条工号未命中不得导致整单采集失败，但需记录日志便于后续排查

### 7.8 联调中已验证的页面坑与处理方式
坑 1：待办列表是虚拟滚动 grid，只读取当前 DOM 中可见 `tbody tr` 时，会把页面显示 `共43条` 的待办误读成当前可见的 `17` 条。
- 现象：自动化首轮统计只得到 `17` 条待办，而人工在 EHR 页面确认总数为 `43` 条。
- 原因：待办 grid 只渲染当前视口附近的行，未滚动时后续行并不存在于 DOM。
- 处理：待办列表读取改为纵向滚动采集，按 `单据编号` 去重收集所有已渲染批次，并以页面 `共N条` 作为期望值。
- 验证结果：修复后自动化已可稳定读取完整 `43` 条待办。

坑 2：即使已经拿到了完整 `43` 条 `单据编号`，打开详情时如果仍只在当前可见 DOM 中查找链接，滚到列表后半段时会因目标行未渲染而超时。
- 现象：联调时 `RA-20260310-00019844`、`RA-20260310-00019868` 等后半段单据曾出现 `waiting for locator(\"span.link-cell-content\") ... to be visible` 超时。
- 原因：`open_document()` 直接在页面中查找 `单据编号：{document_no}` 链接，没有先把目标行滚动到可见区域。
- 处理：打开单据前，先在待办虚拟列表中按 `单据编号` 滚动定位目标行，待目标行渲染后再点击主题链接。
- 验证结果：修复后已可直接打开列表最后一条 `RA-20260312-00019936`，并完成完整 `43` 条待办采集。

坑 3：全量待办重跑时，已存在单据与新单混跑，若不先清理指定单据，无法验证“重采后重新落库”链路。
- 现象：已入库单据会直接命中“审批末条时间一致”并跳过，不能重新进入写库流程。
- 处理：对指定 `单据编号` 先删除 `申请单基本信息` 主表记录，依赖外键级联清空 `申请单权限列表`、`申请单审批记录`、`申请表组织范围`，再执行全量采集。
- 验证结果：对 4 张指定单据清空后，已重新完成采集并写回数据库。

## 8. 输出与验证设计

### 8.1 本地输出
`collect` 默认落盘 JSON 到：
- `automation/logs/collect_时间戳.json`

并生成截图到：
- `automation/screenshots/collect_result_时间戳.png`

### 8.2 联调建议
建议联调顺序：
1. 执行 `collect --dry-run`，确认 JSON 结构正确
2. 执行 SQL 建表
3. 配置 PG 连接
4. 执行 `collect` 写库
5. 使用 SQL 验证四表数据是否按 `document_no` 关联完整，并核对：
   - 已存在且审批末条时间未变化的单据被正确跳过；
   - 已存在且审批末条时间变化的单据已被整单重采；
   - `申请单基本信息.采集次数` 首次为 `1`，重采后按次累计；
   - `申请单审批记录` 已过滤空审批步骤；
   - `申请单审批记录.工号` 已按括号或花名册匹配回填；
   - `申请单基本信息.最新审批时间` 等于最后一条有效审批记录时间。

## 9. 当前实现边界
- 当前实现按页面现状完成，适用于已核实的 iERP 页面结构
- 若后续系统升级导致 DOM 结构变化，需要重新校准字段选择器和审批记录卡片结构
- 当前默认采集“待办任务”中的权限申请单据，不覆盖“已办”“在办”等列表
- 当前待办列表完整性依赖分页下拉中存在 `1000条/页` 选项；如页面配置下线该选项，需要重新调整策略
- 审批记录空步骤过滤当前仅限定为 `raw_text = 属地人力资源部负责人 通过规则：全部通过` 的精确匹配；若页面文案变化，需要同步调整规则

## 10. `105B` 后补口径

自 `105B` 起，`collect` 成功写库时，除原有主表字段外，还需同步回写：

- `申请单基本信息.待办处理状态 = 待处理`
- `申请单基本信息.待办状态更新时间 = 当前写库时间`

原因：

- `collect` 的来源就是当前账号 EHR `待办任务`
- 因此凡是成功采到的单据，都应被确认仍在当前账号待办中

该补充规则不改变原有：

- `最新审批时间` 对比策略
- `采集次数` 累计策略

也就是说：

- 成功采集会刷新 `待办处理状态`
- 但不会新增 `采集次数` 以外的累计规则

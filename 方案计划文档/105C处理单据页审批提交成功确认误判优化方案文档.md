# 方案编号105C：处理单据页审批提交成功确认误判优化方案文档

## 1. 方案背景

`105 / 105A / 105B` 已分别补齐：

- `处理单据 -> 审批单据` 页签中的批准执行能力
- 审批执行卡住时的超时与日志兜底
- 审批前“当前账号待办是否仍存在”的状态分流

但在 `2026-03-17` 的现场实操中，单据 `RA-20260316-00020062` 出现了新的异常场景：

- 在审批单据页签点击 `批准`
- 现场确认 `EHR` 侧实际上已经处理成功
- 但接口返回失败：
  - `审批执行失败：点击提交后未在预期时间内观察到成功反馈`
  - `last_error=Locator.wait_for: Timeout 20000ms exceeded`
  - `waiting for locator("text=审批记录").first to be visible`

对应日志：

- `automation/logs/approval_20260317_101636_RA-20260316-00020062.json`

这说明当前问题已不再是：

- `105A` 定义的“审批流卡住”
- 也不是 `105B` 定义的“单据已不在当前账号待办”

而是一个新的后置确认问题：

- **审批动作已经发出且 EHR 实际成功，但系统在提交后的成功确认阶段发生误判。**

## 2. 问题定义

本次问题的准确定义应为：

- `提交成功确认误判`

它不是“点击提交失败”，而是“点击提交后，代码没有在有限时间内命中当前定义的成功信号，因此把结果错误归类为失败”。

该问题的风险高于一般超时，原因是：

1. 用户界面会提示“审批失败”，容易引导操作员再次点击 `批准`
2. 实际 EHR 可能已经完成处理，重复提交会带来二次操作风险
3. 当前日志只能看到 `approval_flow_started` 到最终异常，缺少提交后的状态快照
4. `105B` 的“审批成功后回写待办状态 = 已处理”也会因为接口判失败而漏写

因此，`105C` 的目标不是单纯延长等待时间，而是重构“提交后如何判定成功”的工程口径。

## 3. 现状分析

### 3.1 当前提交后确认逻辑

当前实现位于：

- `automation/flows/document_approval_flow.py`

`execute_approve()` 点击 `提交` 后，会进入 `wait_for_submission_confirmation()`。

当前判定顺序是：

1. 轮询是否出现失败类提示文案
2. 轮询是否出现成功类 toast 文案
3. 若没有 toast，则尝试切到 `审批记录` 页签并重新抓取审批记录
4. 若审批记录条数增加，且最新动作为 `同意` 或意见匹配，则视为成功
5. 否则在超时后抛出失败异常

### 3.2 当前逻辑的脆弱点

结合这次日志与现有代码，可以确认当前成功判定过度依赖：

- `审批记录` 页签仍然可见
- 且提交后短时间内可以再次切回该页签
- 且审批记录立即刷新

但真实 EHR 页面在点击 `提交` 后，完全可能出现以下任一变化：

1. 页面直接回到待办列表
2. 当前单据详情区被关闭或重绘
3. `任务处理 / 审批记录` 页签区域短时间不可见
4. 审批记录存在落库或前端刷新延迟
5. 成功 toast 文案短暂出现但未被当前轮询命中

在这些场景下，审批实际上已经成功，但现有代码仍会因为“看不到 `审批记录` 页签”而误报失败。

### 3.3 本次日志体现出的信号

从 `automation/logs/approval_20260317_101636_RA-20260316-00020062.json` 可见：

- 提交前待办状态探测结果仍为 `待处理`
- 审批流已经正常启动
- 最终异常并不是页面错误 toast，而是等待 `审批记录` 页签可见超时
- 异常后截图也未成功保留，说明当前失败现场证据不足

这意味着当前更接近：

- **成功信号未命中**

而不是：

- **EHR 明确返回提交失败**

## 4. 方案目标

`105C` 的目标如下：

1. 将“提交后成功确认”从单一信号改为多信号判定
2. 降低“实际成功但接口返回失败”的误判概率
3. 在无法立即确认成功或失败时，返回“已提交待确认”而不是直接报失败
4. 补齐提交后的日志快照与排查证据
5. 不新增 PostgreSQL 表结构，不改动 `105 / 105A / 105B` 的前置分流口径

## 5. 方案设计

### 5.1 提交后结果判定改为“三段式”

提交后结果统一分为三类：

1. `明确成功`
2. `明确失败`
3. `已提交待确认`

其中：

- `明确成功` 才回写 `待办处理状态 = 已处理`
- `明确失败` 保持现有失败语义
- `已提交待确认` 用于承接“很可能已经提交成功，但当前无法拿到强确认信号”的场景，前端必须提示用户**不要重复点击**

### 5.2 明确失败信号

以下信号任一命中，直接判定为 `明确失败`：

1. 页面出现失败类 toast / notification / alert
2. 页面出现必填校验失败、格式错误、业务异常等阻断文案
3. 点击提交后页面仍停留在原单据任务处理区，且确认目标单据仍在当前账号待办中

这部分继续沿用当前 `_ERROR_KEYWORDS`，但应补充：

- 模态框正文
- 表单内联错误
- 页面级红色告警文本

### 5.3 明确成功信号

提交后成功信号按优先级依次判定：

#### 5.3.1 第一优先级：成功类 toast

若命中成功类提示文案，直接返回成功：

- `confirmationType = toast`

#### 5.3.2 第二优先级：审批记录追加

若仍能进入 `审批记录` 页签，且出现以下任一情况，判定成功：

1. 审批记录条数增加
2. 最新记录动作为 `同意`
3. 最新记录意见与本次提交意见一致

返回：

- `confirmationType = approval_record`

#### 5.3.3 第三优先级：提交后单据已从待办消失

若提交后页面已回到待办列表、详情区关闭、页签区域重绘，导致无法再依赖 `审批记录` 页签，则应增加新的成功确认路径：

1. 判断当前是否已回到待办列表视图
2. 在当前账号待办列表中重新查找目标 `document_no`
3. 若目标单据已不存在于当前待办中，则判定本次提交成功

返回：

- `confirmationType = todo_disappeared`

这个信号与 `105B` 的前置待办分流是互补关系：

- `105B` 负责“提交前就不在待办里”
- `105C` 负责“提交后从待办里消失”

#### 5.3.4 第四优先级：页面结构已切换且不存在失败信号

若出现以下组合特征：

- `提交` 按钮消失
- `任务处理 / 审批记录` 页签不可见
- 当前单据详情区域被关闭或切离
- 没有任何失败类提示

则说明页面状态已经因提交动作发生切换。

若此时再做一次待办重查仍无法稳定命中目标单据，可判定为：

- `已提交待确认`

而不应直接报失败。

### 5.4 提交后轮询顺序优化

建议将 `wait_for_submission_confirmation()` 改造成分阶段轮询：

#### 5.4.1 第一阶段：短窗口监听即时反馈

- 时长建议：`0 ~ 8` 秒
- 重点监听：
  - 成功 toast
  - 失败 toast
  - 模态错误

原因：

- 这一阶段最容易捕获最直接的页面反馈

#### 5.4.2 第二阶段：尝试审批记录确认

- 时长建议：`8 ~ 25` 秒
- 重点动作：
  - 尝试进入 `审批记录` 页签
  - 抓取最新审批记录

注意：

- `审批记录` 页签探测失败时，不应立刻把整次提交视为失败
- 这里只能记为“审批记录路径暂不可用”

#### 5.4.3 第三阶段：回退到待办视角确认

- 时长建议：`25 ~ 45` 秒
- 重点动作：
  - 判断当前是否已回到待办列表
  - 若不在待办列表，主动尝试一次回到待办列表
  - 再次按 `document_no` 搜索目标单据

若此时目标单据已从当前账号待办中消失，则直接视为成功。

### 5.5 新增“提交后状态快照”日志

当前日志只有：

- `approval_flow_started`
- 最终成功或失败

这不足以支撑提交后排查。

`105C` 要求新增以下关键事件：

1. `approval_submit_clicked`
2. `approval_confirmation_probe_started`
3. `approval_confirmation_feedback_detected`
4. `approval_record_probe_failed`
5. `approval_record_probe_hit`
6. `approval_todo_reprobe_started`
7. `approval_todo_reprobe_result`
8. `approval_confirmation_uncertain`

其中 `approval_todo_reprobe_result` 至少记录：

- `documentNo`
- `todoListVisible`
- `documentStillInTodo`
- `submitButtonVisible`
- `taskTabVisible`
- `approvalTabVisible`

要求：

- 日志以“关键状态切换”方式写出
- 不做高频全量刷屏
- 但必须保证现场能看出提交后到底卡在哪个确认分支

### 5.6 截图兜底修正

本次日志中还出现：

- `screenshot_failed: Event loop is closed! Is Playwright already stopped?`

因此 `105C` 还应同步修正失败截图时机：

1. 先判断 `page` 是否仍可用
2. 尽量在 `context.close()` / `browser.close()` 之前截图
3. 若页面已关闭，则记录：
   - `page_already_closed_before_screenshot`

避免再次出现：

- 原始失败现场没有截图
- 同时截图失败信息也不具备排查价值

### 5.7 前端交互口径调整

若后端返回：

- `status = submitted_pending_confirmation`

前端 `审批单据` 页签需展示黄色提示，而不是红色失败提示。

提示口径建议为：

- `提交动作已发出，但当前未拿到强成功回执。请先不要重复点击批准，可先查看最新审批日志或执行一次“同步待办状态”确认。`

这样可以避免操作员因为看到“失败”而重复点击。

若前端本轮暂不支持新增状态，也不应直接沿用纯失败文案，而应至少区分：

- `失败`
- `待确认`

## 6. 代码改动范围

本方案落地时，建议至少涉及以下文件：

- `automation/flows/document_approval_flow.py`

- `webui/src/pages/ProcessDocumentsPage.tsx`

其中：

- `document_approval_flow.py`
  - 重构提交后确认逻辑
- `process_dashboard.py`

  - 补充提交后日志事件
- 前端页面与请求层
  - 衔接新的后端状态展示
- 测试
  - 补齐多分支确认测试

## 7. 验证要求

`105C` 落地后，至少应验证以下场景：

1. 成功 toast 明确出现
   - 应返回 `明确成功`
2. toast 未命中，但审批记录追加
   - 应返回 `明确成功`
3. toast 未命中，页签消失，但单据从待办中消失
   - 应返回 `明确成功`
4. 页面出现明确错误提示
   - 应返回 `明确失败`
5. 页面结构切换，但拿不到失败信号，也暂时查不到明确成功信号
   - 应返回 `已提交待确认`

单元测试建议新增以下覆盖：

- `approval_record` 成功分支
- `todo_disappeared` 成功分支
- `submitted_pending_confirmation` 分支
- `审批记录页签不可见但不应立即失败` 分支

## 8. 与其他方案文档的关系

### 8.1 与 `105` 的关系

`105` 负责定义“审批单据页签 -> EHR 提交”的主链路。

`105C` 不改变主链路，只修正：

- 提交动作发出后的结果判定策略

### 8.2 与 `105A` 的关系

`105A` 解决的是：

- 审批执行时间过长时的超时与日志兜底

`105C` 解决的是：

- 提交实际上成功，但成功回执误判为失败

两者不是替代关系，而是前后衔接关系：

- `105A` 关注“有没有卡住”
- `105C` 关注“提交后有没有被误判”

### 8.3 与 `105B` 的关系

`105B` 负责提交前分流：

- 单据是否还在当前账号待办中

`105C` 负责提交后确认：

- 单据是否已经从当前账号待办中消失

因此 `105B + 105C` 组合后，审批链路可以形成：

1. 提交前先判断单据是否仍可处理
2. 提交后再判断单据是否已成功移出待办

## 9. 最终结论

针对单据 `RA-20260316-00020062` 的这次异常，建议正式定性为：

- **审批提交成功确认误判**

而不是：

- 审批执行失败

`105C` 的核心结论如下：

1. 当前问题根因是“成功确认信号过窄”，不是“点击提交本身失败”
2. 单纯把等待时间从 `20s` 调大，并不能根治该问题
3. 必须把提交后判定改为：
   - `明确成功`
   - `明确失败`
   - `已提交待确认`
4. 成功判定要新增“提交后目标单据从待办消失”的确认路径
5. 提交后日志与截图兜底必须同步增强，否则后续现场仍难以复盘

因此，建议将本次优化正式纳入：

- `方案编号105C`

并作为 `105` 系列对真实审批提交链路的下一步修订方案。

## 10. 2026-03-24 补充口径

- 若提交后已返回待办列表，且连续两次待办复探都满足以下条件，也应判定为 `todo_disappeared` 成功，而不是降级为 `submitted_pending_confirmation`：
  - 两次复探均未在当前账号待办中命中目标单据
  - 两次复探都拿到了有效待办扫描证据，例如扫描到了可见行，或确认当前待办数为 `0`
  - 两次复探都未出现新的待办探测错误
- 该补充口径用于覆盖“页面已回到待办列表、目标单据已稳定消失，但分页器或总数控件暂时未提供强确认信号”的场景。
- 批量审批沿用 `105C` 的提交后确认口径，因此该补充同样适用于 `230` 批量批准与批量驳回链路。

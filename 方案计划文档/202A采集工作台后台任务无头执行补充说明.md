# 方案编号202A：采集工作台后台任务无头执行补充说明

## 1. 背景

`202` 方案已将 `采集单据` 页面接入真实 `collect` 链路，但工作台后台任务原先默认沿用运行配置中的 `browser.headed`。

在生产默认配置 `automation/config/settings.prod.yaml` 中，当前 `browser.headed = true`。这会导致：

- 工作台后台采集任务启动可见浏览器
- 登录阶段可能出现浏览器原生密码/认证对话框
- 无人值守任务在原生对话框出现后阻塞，页面内选择器无法继续执行

## 2. 本次补充口径

自本补充说明起，`采集工作台` 通过 FastAPI 后台触发的 `collect` 任务，默认强制追加：

```text
--headless
```

约束如下：

- 仅限工作台后台采集入口
- 不改变命令行直接运行 `run.py collect` 时显式传 `--headed` 的能力
- 不改变手工登录调试、选择器排查等人工观察场景

## 3. 原因

后台长任务的目标是稳定完成采集，不应依赖桌面焦点和人工干预。

相较于可见浏览器，无头模式更适合以下场景：

- Windows 计划任务或后台 API 线程触发
- 无人值守的正式采集
- 避免浏览器原生弹框阻塞 Playwright

## 4. 实施落点

本次实现落点：

- `automation/api/collect_workbench.py`

实现方式：

- 工作台后台线程组装 `run.py collect` 命令时，固定追加 `--headless`

## 5. 边界说明

若后续需要提供“工作台中手工选择可见浏览器调试”的能力，应作为新的显式参数设计，不应再回退为后台任务默认可见浏览器。

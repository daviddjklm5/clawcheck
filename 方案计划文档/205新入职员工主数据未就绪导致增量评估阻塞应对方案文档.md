# 方案编号205：EHR待办已存在但处理工作台因无评估结果漏展示应对方案文档

## 1. 方案目标

针对 `2026-03-17` 现场确认的问题：

- EHR 当前待办实时存在
- 单据也已采集入库
- 但因没有任何评估结果，无法进入 `处理单据` 工作台

建设一套可长期复用的稳定应对机制。

本方案明确交付以下结果：

- 只要单据已采集入库，且 `待办处理状态 = 待处理`，即使尚无评估结果，也必须进入 `处理单据`
- `处理单据` 不再以 `申请单风险信任评估` 作为唯一入口条件
- 对尚未产出任何评估结果的单据，页面显示正式状态：
  - `待评估`
- 对已经执行过评估但未成功完成的单据，继续支持展示：
  - `阻塞`
  - `异常`
- `在职花名册 / 人员属性查询 / 组织属性查询` 补齐后，可对 `待评估 / 阻塞` 单据自动或手动补评
- 不再出现“EHR 待办里有单据，但处理工作台看不到”的情况

## 2. 问题边界

本方案只解决以下问题：

- EHR 待办实时存在，但处理工作台因“无评估结果”漏展示

本方案**不**解决以下问题：

- 处理工作台仍显示 `待处理`，但 EHR 实际已无该待办

后者已由 `105B / 同步待办状态` 负责处理，不再纳入 `205` 主目标。

## 4. 问题本质

当前 `处理单据` 工作台的主集合来自“逐单最新评估结果”。

这意味着：

1. 单据必须先完成评估
2. 才能进入 `处理单据`

于是出现以下错误耦合：

- `collect` 成功，但 `audit` 尚未执行 -> 单据不可见
- `collect` 成功，但 `audit` 执行中断 -> 单据不可见
- `collect` 成功，但 `audit` 因主数据未就绪未产出结果 -> 单据不可见

这违背了工作台的真实业务目标：

- `处理单据` 首先是“当前账号待处理单据工作台”
- 风险评估结果应是附着信息，而不是单据能否进入工作台的门槛

## 5. 与现有方案边界

### 5.1 与 `203` 的关系

`203` 解决的是：

- `collect -> audit -> 处理工作台` 的增量链路

`205` 补充的是：

- 当 `audit` 尚未执行、未完成、或未产出结果时，单据如何仍然进入处理工作台

### 5.2 与 `105B` 的关系

`105B` 负责：

- 当前账号视角下，单据是否仍在 EHR 待办中

`205` 不再处理这一类“待办实时性”问题。

`205` 只处理：

- 对已经确认属于 `待处理` 的单据，如何保证其在工作台中可见

### 5.3 与 `103` 的关系

`103` 定义的是风险评分模型。

`205` 不改评分矩阵本身，只改：

- 工作台入口条件
- `待评估` 展示口径
- 评估未完成时的可见性

## 6. 设计原则

### 6.1 可见性优先于评估完备性

进入 `处理单据` 的前提应是：

- 单据已采集
- 且当前账号视角仍为 `待处理`

而不是：

- 必须先完成评估

### 6.2 不伪造评估结果

对尚未完成评估的单据，不允许为了让其显示在工作台而伪造：

- `最终信任分`
- `总结论`
- `建议动作`

应明确展示为：

- `待评估`

### 6.3 评估失败不再导致单据消失

无论是：

- `audit` 未执行
- `audit` 阻塞
- `audit` 异常

都不应让单据从工作台中消失。

### 6.4 与待办同步解耦

`处理单据` 是否可见，与单据是否有评估结果相关；

`处理单据` 是否仍在当前账号待办，与 `同步待办状态` 相关。

两者必须分层，不再混为一个问题。

## 7. 备选方案评估

### 7.1 方案 A：继续以评估结果表作为工作台唯一入口

做法：

- `处理单据` 继续从 `申请单风险信任评估` 取主集合
- 通过提高 `audit` 成功率来尽量减少漏展示

优点：

- 查询层改动小

缺点：

- 只要 `audit` 没跑、跑挂、未产出结果，单据仍然不可见
- 不能解决本次现场已经确认的 `EHR 有 11 张、工作台只看见 6 张` 问题

结论：

- 不推荐

### 7.2 方案 B：单据无评估结果时，自动补一条默认评估结果

做法：

- 对尚无评估结果的单据自动写一条“默认评估”

优点：

- 可以沿用现有评估表驱动工作台

缺点：

- 容易把“未评估”伪装成“已评估”
- 会污染正式评估结果口径

结论：

- 不推荐

### 7.3 方案 C：工作台改为“待办主集合 + 最新评估附着”

做法：

- `处理单据` 先从 `申请单基本信息` 中取：
  - `待办处理状态 = 待处理`
- 再 `LEFT JOIN` 最新评估结果
- 若没有评估结果，则展示：
  - `待评估`

优点：

- 直接解决当前漏展示问题
- 不伪造评估结果
- 与未来的 `阻塞 / 异常 / 自动补评` 兼容

缺点：

- 需要改查询层、API 和页面展示

结论：

- **推荐作为 `205` 正式方案**

## 8. 推荐总方案

`205` 推荐采用 **方案 C**：

1. `collect` 仍负责把当前 EHR 待办单据尽快落库
2. `处理单据` 主集合改为：
   - `申请单基本信息` 中 `待办处理状态 = 待处理` 的单据
3. `最新评估结果` 改为附着信息，而不是唯一入口条件
4. 若单据尚无评估结果，则工作台展示：
   - `待评估`
5. 若单据已有最新评估结果，则按评估结果展示：
   - `成功`
   - `阻塞`
   - `异常`
6. 主数据刷新后，对 `待评估 / 阻塞` 单据做自动或手动补评

## 9. 状态模型设计

### 9.1 工作台状态

建议在 `处理单据` 页面引入独立的“工作台状态”：

1. `待评估`
   - 单据已采集
   - `待办处理状态 = 待处理`
   - 但还没有任何评估结果行

2. `成功`
   - 单据存在最新评估结果
   - 且评估执行成功

3. `阻塞`
   - 单据存在最新评估结果
   - 但评估因主数据未就绪或采集数据不完整而未完成

4. `异常`
   - 单据存在最新评估结果
   - 但评估执行发生技术异常

说明：

- `待评估` 是工作台展示态，可以先不落到 `申请单风险信任评估`
- `成功 / 阻塞 / 异常` 则来自最新评估结果

### 9.2 评估状态

若后续继续推进原始 `205` 中的评估状态化，可在 `申请单风险信任评估` 增加：

- `评估状态`
- `阻塞分类`
- `阻塞原因类型`
- `阻塞原因说明`
- `阻塞证据快照`

但这是第二阶段，不是解决本次漏展示问题的前提。

## 10. 后端改造方案

### 10.1 查询层

`fetch_process_workbench()` 需要改为：

1. 从 `申请单基本信息` 读取：
   - `待办处理状态 = 待处理`
2. `LEFT JOIN` 每张单据的最新评估结果
3. 若评估结果为空，则派生：
   - `工作台状态 = 待评估`

`fetch_process_document_detail()` 需要支持：

- 单据已有基础采集信息
- 但暂无评估结果

此时详情页应返回：

- 单据基础信息
- 角色/组织/审批记录
- `工作台状态 = 待评估`
- “尚未产出评估结果”的提示

### 10.2 `audit`

`audit` 的职责需要收敛为：

- 为工作台中的单据补充评估信息
- 而不是决定单据是否能进入工作台

因此即使 `audit` 未执行、失败、阻塞，单据仍然可见。

### 10.3 评估异常隔离

虽然当前主问题不是 `audit` 失败，但仍建议保留以下改造方向：

- 单据级异常隔离
- 主数据未就绪时不再整批中断

原因：

- 这能减少 `待评估` 单据长期不被补评的数量

## 11. 主数据刷新后的补评机制

### 11.1 为什么仍需要补评

把单据展示成 `待评估`，只是解决“看得见”的问题。

要让它真正进入审批判断，还需要后续补评。

### 11.2 补评对象

建议优先对以下两类单据发起补评：

1. 当前 `待办处理状态 = 待处理` 且无评估结果的单据
2. 当前最新评估结果为 `阻塞` 的单据

### 11.3 触发时机

建议在以下任务成功后自动补评：

1. `run.py roster`
   - 刷新 `在职花名册表`
   - 同步刷新 `人员属性查询`

2. `run.py orglist`
   - 刷新 `组织列表`
   - 同步刷新 `组织属性查询`

### 11.4 手动入口

建议在 `处理单据` 或 `同步主数据` 页增加：

- `重评待评估单据`
- `重评阻塞单据`

## 12. 页面与 API 方案

### 12.1 `处理单据` 页

建议新增以下列与筛选项：

- `工作台状态`
- `评估状态`
- `阻塞分类`
- `阻塞原因说明`

交互规则：

- `工作台状态 = 待评估`
  - 允许查看单据详情
  - 不允许直接执行审批
  - 页面提示：该单据已采集，但尚未产出评估结果

- `工作台状态 = 阻塞`
  - 不允许直接执行审批
  - 页面提示：请先补齐主数据并重评

- `工作台状态 = 异常`
  - 不允许直接执行审批
  - 页面提示：请重试评估或查看日志

### 12.2 `采集单据` 页

当前 `collect` 后自动 `audit` 的提示需升级。

建议支持以下文案：

- `已进入待评估：5 张`
- `增量评估完成：成功 3 张，阻塞 1 张，异常 0 张`
- `增量评估未执行：当前仍可在处理工作台查看待评估单据`

页面需新增展示：

- `pendingAssessmentCount`
- `blockedDocumentCount`
- `blockedReasonSummary`

### 12.3 `评估分析` 页

`评估分析` 仍继续按评估批次取数。

说明：

- `待评估` 单据因为尚无评估结果，不属于批次分析对象
- 它们属于处理工作台对象

## 13. 数据库改造建议

第一阶段优先：

- **不新增新的中文业务表**
- **不新增 SQL 迁移**
- 先通过查询层和页面派生 `待评估`

推荐原因：

- 当前最紧急的问题是“漏展示”
- `待评估` 可以通过 `LEFT JOIN` 空值即时派生
- 不必等状态字段迁移完成后再修复主问题

第二阶段若继续推进 `阻塞 / 异常` 状态化，再评估是否新增评估状态字段和索引。

## 14. 发布实施建议

建议分三步实施：

### 14.1 第一步：止血

目标：

- 不再发生“EHR 待办存在，但处理工作台看不到”

内容：

- `fetch_process_workbench()` 改为：
  - `申请单基本信息.待办处理状态 = 待处理` 为主集合
  - `LEFT JOIN` 最新评估结果
- 页面新增 `工作台状态 = 待评估`
- `fetch_process_document_detail()` 支持无评估结果单据

### 14.2 第二步：状态化

目标：

- 区分 `待评估 / 成功 / 阻塞 / 异常`

内容：

- 查询层/API/UI 补齐 `工作台状态`
- 视需要扩展评估状态字段
- `audit` 输出阻塞原因

### 14.3 第三步：补评闭环

目标：

- 主数据刷新后自动恢复

内容：

- `roster / orglist` 成功后自动增量补评
- 提供“重评待评估 / 阻塞单据”手动入口

## 15. 验收标准

至少满足以下验收项：

1. 单据已在 EHR 待办中，且 `collect` 成功落库后：
   - 即使尚无评估结果
   - 也能进入 `处理单据`
   - 页面显示 `工作台状态 = 待评估`

2. `处理单据` 的主列表中，不再只显示“已有评估结果”的单据

3. 主数据刷新后重新触发补评：
   - 原 `待评估` 单据可转为 `成功 / 阻塞 / 异常`
   - 成功时能生成正式 `最终信任分 / 总结论 / 低分明细`

4. `同步待办状态` 仍单独负责：
   - 把已不在 EHR 当前待办中的单据改为 `已处理`

5. `205` 修复后，本次已确认的 `5` 张 EHR 待办单据不会再因“无评估结果”而漏展示

## 16. 开放问题

当前仍需实现层再确认以下事项：

1. `待评估` 单据是否默认排在工作台最前
   - 本方案建议：是

2. `待评估` 单据详情页是否允许直接发起“评估当前单据”
   - 本方案建议：允许

3. `待评估` 单据是否计入顶部统计卡
   - 本方案建议：单独统计，不混入低分/人工干预

4. 第二阶段是否需要把 `阻塞 / 异常` 正式扩展到评估表字段
   - 本方案建议：先修可见性，再做状态化

## 17. 当前建议结论

`205` 的主问题已经明确，不应再继续扩散。

当前最需要解决的是：

- EHR 待办实时存在
- 单据也已采集
- 但因为无评估结果，处理工作台漏展示

因此正式建议是：

- `处理单据` 的主集合改为 `申请单基本信息.待办处理状态 = 待处理`
- `申请单风险信任评估` 只作为附着信息
- 无评估结果时正式展示为 `待评估`

这比继续提高 `audit` 成功率、更符合工作台本质，也能直接解决这次实测暴露出来的 `EHR 11 张、工作台只看见 6 张` 问题。

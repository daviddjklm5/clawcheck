# 方案编号200：可视化页面 UI 建设方案文档

## 1. 方案目标

在现有 `Python + Playwright + PostgreSQL` 项目基础上，建设一套面向运营与治理场景的可视化页面 UI，用于统一承接以下能力：

- 主数据同步执行与结果查看
- 权限申请单据采集执行与数据查询
- 待处理单据查看、风险与信任度评估结果展示
- 后续审批 / 驳回 / 加签等页面操作入口预留
- 运行时常规配置管理

本方案的目标不是重写现有自动化逻辑，而是为现有能力增加一层稳定的 Web 操作与查询界面。

本方案优先解决“可用、可查、可追踪”的问题，不在第一阶段追求过度复杂的管理后台能力。

## 2. 当前项目现状分析

### 2.1 已具备的能力

当前仓库已经具备以下后端基础：

- `automation/scripts/run.py`
  - 已支持 `check / login / run / collect / roster / orglist / rolecatalog / dbinit`
- `automation/flows/`
  - 已实现花名册下载、组织列表下载、权限申请待办单据采集
- `automation/db/postgres.py`
  - 已实现主要业务表建表与写库
- `automation/sql/`
  - 已沉淀主表、迁移、人员属性、风险信任评估结果表等 SQL
- `automation/config/rules/`
  - 已存在 `risk_trust_matrix.yaml` 与 `risk_trust_constants.yaml`

当前已落地或已建表的核心业务对象包括：

- `申请单基本信息`
- `申请单权限列表`
- `申请单审批记录`
- `申请表组织范围`
- `人员属性查询`
- `权限列表`
- `在职花名册表`
- `组织列表`
- `组织属性查询`
- `申请单风险信任评估`
- `申请单风险信任评估明细`

### 2.2 当前缺口

当前项目仍存在以下明显缺口：

1. 还没有前端工程
2. 还没有 HTTP API 层
3. 还没有“长任务执行状态”的统一任务中心
4. `103` 方案中的风险信任评估资产已落地到 YAML 和 SQL，但仓库当前还没有正式 `audit` 执行动作
5. 审批 / 驳回 / 加签等页面写操作尚未形成正式执行链路

### 2.3 对 200 计划的定位判断

因此，`200` 计划的合理定位应为：

- 第一阶段先建设“操作台 + 查询台”
- 先打通“触发任务、查看结果、查看单据、查看评估”的闭环
- 不在第一阶段承诺直接回写 iERP 审批动作

这与 `103` 文档中的阶段结论一致：

- 第一阶段输出的是“审批建议”
- 第一阶段不做页面回写
- 第一阶段不做完整管理后台

## 3. 方案范围

### 3.1 本次纳入范围

本方案纳入以下范围：

- 新增一个独立 Web UI
- 新增一个 Python Web API 层
- 新增统一左侧菜单与页面壳
- 将现有脚本能力映射到 UI 操作入口
- 使用 `MUI + MUI X Data Grid` 作为主要页面组件栈
- 为风险信任评估、单据详情、任务状态提供表格化展示
- 为常规运行配置提供可视化配置页

### 3.2 本次暂不纳入范围

本方案暂不纳入以下范围：

- 在线编辑审核规则 YAML
- 在线维护复杂审批矩阵
- 直接在浏览器里暴露真实账号口令
- 第一阶段直接执行 iERP 的审批 / 驳回 / 加签写回
- 移动端适配

## 4. 技术栈选择结论

### 4.1 总体结论

本项目不建议更换到完全不同的前端栈。

用户指定的 `https://mui.com/x/react-data-grid/` 与本项目“强表格、强检索、强明细、强批量操作”的特点是匹配的，方向正确。

本方案建议的最终选型为：

- 前端框架：`React + TypeScript + Vite`
- UI 组件：`@mui/material`
- 表格组件：`@mui/x-data-grid`
- 当前阶段档位：社区版 `DataGrid`
- 后端 API：`FastAPI`
- 后端数据访问：复用 `automation/db/postgres.py`
- 自动化执行：复用现有 `automation/flows/` 与后续抽离出来的 `services`

### 4.2 为什么不建议改成别的 UI 方案

不建议改成 `Streamlit`、`Gradio` 一类轻应用框架，原因如下：

- 本项目不是单页报表展示，而是持续运营后台
- 页面会长期存在“多表联查 + 行级展开 + 长任务状态 + 批量操作”
- 后续还要承接审批动作、任务历史、日志追踪、权限分层

这类场景更适合标准 React 管理台，而不是以快速原型为目标的轻框架。

### 4.3 对 MUI Data Grid 档位的建议

结合当前明确范围，本方案建议第一阶段直接采用社区版 `DataGrid`，暂不引入 `DataGridPro`。

原因如下：

1. 当前已明确不需要以下 `Pro` 典型能力
   - 列固定
   - 多条件过滤
   - 多列排序
   - 树形展示
   - 顶部 / 底部固定行

2. 当前“明细查看”方式也不依赖 `Pro` 的主从展开
   - 本项目更适合“单据列表 + 单据号联动详情页签 / 右侧抽屉”的结构
   - 即：主表点击后，通过页签展示 `基本信息 / 权限明细 / 审批记录 / 组织范围 / 风险明细`
   - 这种交互用社区版 `DataGrid` 完全可以承接

3. 当前真正必须解决的是“统一接线”，不是“高级表格特性”
   - 服务端分页
   - 服务端排序
   - 基础过滤
   - 行点击详情
   - 列定义统一
   - 空态 / 错误态统一

因此，本阶段更合适的选型是：

- 底层采用社区版 `DataGrid`
- 项目内统一封装 `AppDataGrid`

后续如果需求范围发生变化，再评估是否升级到 `Pro`。

### 4.4 与官方文档的匹配点

MUI 官方文档已经明确提供了以下与本项目相关的能力与路径：

- Data Grid 基础能力与升级体系  
  `https://mui.com/x/react-data-grid/`
- 服务端数据模式与 Data Source  
  `https://mui.com/x/react-data-grid/server-side-data/`

因此本项目不需要重新找“更合适的表格组件”，更合理的决策点是：

- 是否先做统一封装
- 是否从第一天就按服务端表格模式设计

本方案判断：应该按服务端模式设计，并以社区版 `DataGrid + AppDataGrid` 作为当前正式方案。

## 5. 架构设计结论

### 5.1 总体分层

建议新增以下分层：

1. 前端层
   - `webui/`
   - React 页面、路由、Data Grid、表单与抽屉详情

2. API 层
   - `automation/api/`
   - 提供任务触发、任务查询、单据查询、评估查询、配置读取与保存

3. 服务层
   - `automation/services/`
   - 将 `run.py` 里的动作拆为可复用服务函数

4. 自动化与数据层
   - 保持现有 `automation/flows/`
   - 保持现有 `automation/db/`
   - 保持现有 PostgreSQL 表结构

### 5.2 为什么必须新增 API 层

前端不应直接连接 PostgreSQL，也不应直接执行本地脚本命令。

必须有 API 层的原因：

- 统一鉴权与权限控制
- 统一参数校验
- 统一任务状态追踪
- 统一复用现有 Python 逻辑
- 避免把脚本命令细节暴露到前端

### 5.3 对现有 `run.py` 的处理建议

不建议让 UI 直接调用 `python automation/scripts/run.py ...` 作为长期方案。

更合理的做法是：

1. 先把 `run.py` 中每个动作抽成服务函数
2. CLI 入口继续保留，调用服务函数
3. FastAPI 也调用同一批服务函数

这样可以避免：

- 一套逻辑维护两份
- UI 和 CLI 输出口径不一致
- 日志、异常和任务状态不统一

### 5.4 建议目录结构

```text
webui/
  src/
    app/
    components/
    features/
    pages/
    services/
    types/

automation/
  api/
    main.py
    routers/
    schemas/
  services/
    collect_service.py
    roster_service.py
    orglist_service.py
    audit_service.py
    settings_service.py
    jobs_service.py
```

## 6. 页面信息架构设计

### 6.1 总体布局

页面建议采用标准后台布局：

- 左侧：一级菜单导航
- 顶部：环境标识、运行状态、任务入口、用户信息
- 中部：当前模块内容区
- 右侧抽屉：行级详情、任务日志、表单操作

### 6.2 左侧菜单结构

建议左侧菜单固定为四大类：

| 一级菜单 | 子页面建议 | 对应当前能力 |
| --- | --- | --- |
| 同步主数据 | 在职花名册、组织列表、权限列表 | `roster`、`orglist`、`rolecatalog` |
| 采集单据 | 待办采集、采集结果、单据详情 | `collect` |

## 7. 各页面设计

### 7.1 同步主数据

#### 7.1.1 页面目标

用于触发与查看以下任务：

- 在职花名册同步
- 组织列表同步
- 权限列表初始化 / 补种

#### 7.1.2 页面结构

页面建议由三部分组成：

1. 顶部概览卡片
   - 最近同步时间
   - 最近导入批次号
   - 最新数据量
   - 最近任务状态

2. 操作区
   - `同步在职花名册`
   - `同步组织列表`
   - `初始化权限列表`
   - 高级参数折叠区

3. Data Grid 列表区
   - 展示最近任务与最近导入结果

#### 7.1.3 建议操作参数

同步主数据页建议暴露以下参数：

- 浏览器是否可见
- 查询超时秒数
- 下载超时分钟数
- 是否仅查询不导入
- 是否仅导入已有文件

其中：

- `浏览器是否可见` 属于用户明确提出的核心配置

#### 7.1.4 建议列表字段

- `任务ID`
- `任务类型`
- `任务状态`
- `触发时间`
- `完成时间`
- `导入批次号`
- `来源文件名`
- `写入行数`
- `错误摘要`

### 7.2 采集单据

#### 7.2.1 页面目标

用于触发权限申请单据采集，并以结构化方式查看已经采集入库的数据。

#### 7.2.2 页面结构

建议拆成两个视图：

1. `采集任务视图`
   - 执行 `collect`

2. `单据列表视图`
   - 查询已采集入库的单据

- 浏览器是否可见

不建议把所有明细平铺在主页面。

建议采用“主 Grid + 右侧详情抽屉”模式，抽屉内使用页签展示：

- `基本信息`

其中：

这比只看本地数据库数据更接近真实处理场景。

### 7.3 处理单据

#### 7.3.1 页面定位

该页面用于承接“从采集数据到处理建议”的过程，不应在第一阶段就等同于“可直接审批”。

第一阶段建议定位为：

- 待处理单据查询页
- 风险与信任度评估结果页

#### 7.3.2 第一阶段功能

第一阶段建议只做以下内容：

- 待办单据列表展示
- 风险信任评估总表展示
- 风险信任评估明细展示
- 单据明细联查
- 打开 iERP 原始单据

#### 7.3.3 第一阶段不应承诺的功能

以下功能必须标记为“预留”或“暂不可用”：

- 自动审批
- 自动驳回
- 自动加签

原因很明确：

- `103` 当前只定义到评估层
- 仓库当前不存在正式 `audit` 执行动作
- 也不存在正式审批写回服务

#### 7.3.4 处理单据主 Grid 字段建议

当 `申请单风险信任评估` 已有数据时，主 Grid 建议展示：

- `单据编号`
- `权限对象`
- `工号`
- `单据状态`
- `申请日期`
- `最终信任分`
- `总结论`

#### 7.3.5 风险明细展示建议

当用户点开某条单据时，详情区建议提供以下页签：

- `风险总览`

- `申请角色`

其中：

- `风险总览` 展示总结论、建议动作、最低命中维度、低分结论

#### 7.3.6 审批动作预留设计

虽然第一阶段不执行写回，但页面结构应提前预留按钮位：

- `审批通过`
- `驳回`
- `加签`

这些按钮在第一阶段建议：

- 默认禁用
- 显示提示文案：`待 103 评估执行链路与页面写回服务完成后开放`

### 7.4 常规配置

#### 7.4.1 页面目标

用于管理运行时通用配置，而不是管理真实口令。

#### 7.4.2 可配置项建议

建议在页面中允许修改以下非敏感配置：

- 浏览器是否可见
- `slow_mo_ms`
- `timeout_ms`
- `navigation_timeout_ms`
- `retries`
- `retry_wait_sec`
- 下载目录
- 日志目录
- 默认环境标识

#### 7.4.3 敏感信息处理原则

以下信息不建议在页面明文展示或可视化编辑：

- 登录账号
- 登录密码
- PostgreSQL 密码

页面只建议展示：

- 是否已配置
- 来源是本地文件还是环境变量
- 数据库连接摘要：`host / port / dbname / schema`

这与 `AGENTS.md` 中的安全要求保持一致。

## 8. Data Grid 使用规范

### 8.1 统一原则

本项目中的大表页面，统一采用 Data Grid，而不是各页面自行拼表格。

### 8.2 统一封装

建议封装一个统一组件：

- `AppDataGrid`

该组件统一处理：

- 服务端分页
- 服务端排序
- 服务端过滤
- 列显示记忆
- 空态与错误态
- 行点击详情
- 查询参数与 URL 同步
- 通用工具栏与刷新动作

### 8.3 为什么必须按服务端模式设计

项目中的多张表规模都不小，例如：

- `在职花名册表`
- `组织列表`
- `申请表组织范围`
- `申请单风险信任评估明细`

如果前端一次性全量加载，会带来明显问题：

- 首屏慢
- 浏览器内存压力大
- 过滤排序不稳定
- 后续扩展困难

因此 200 计划里必须从第一天就采用服务端 Data Grid 模式。

### 8.4 列名与字段名规范

Data Grid 对用户显示的列名，统一优先使用中文业务名，例如：

- `单据编号`
- `角色编码`
- `角色名称`
- `最终信任分`

API 层内部字段可以保留技术英文键，但前端显示层不应长期向业务暴露英文业务字段名。

这与仓库现有中文字段化原则保持一致。

## 9. API 与任务模型建议

### 9.1 API 路由建议

建议新增以下 API：

- `POST /api/jobs/masterdata/roster`
- `POST /api/jobs/masterdata/orglist`
- `POST /api/jobs/masterdata/rolecatalog`
- `POST /api/jobs/documents/collect`
- `POST /api/jobs/documents/audit`
- `GET /api/jobs`
- `GET /api/jobs/{job_id}`
- `GET /api/documents`
- `GET /api/documents/{document_no}`
- `GET /api/audits`
- `GET /api/audits/{document_no}`
- `GET /api/settings/runtime`
- `PUT /api/settings/runtime`

其中：

- `documents/audit` 在 `103` 执行动作实现后启用
- 审批 / 驳回 / 加签 API 暂不进入第一阶段

### 9.2 任务状态表建议

若要让 UI 具备稳定的“任务中心”，建议新增一张技术运营表：

- `页面运行任务`

建议字段：

- `任务ID`
- `任务类型`
- `任务状态`
- `触发来源`
- `参数快照`
- `日志文件路径`
- `结果摘要`
- `开始时间`
- `结束时间`
- `记录创建时间`
- `记录更新时间`

这样可以避免 UI 只能被动读取磁盘日志文件，提升：

- 查询能力
- 状态一致性
- 排障效率

### 9.3 日志处理建议

现有 `automation/logs/` 继续保留。

UI 任务详情页建议显示：

- 任务摘要
- 关键日志片段
- JSON dump 路径
- 截图路径

但日志全文仍建议保留在文件系统或对象存储中，不建议第一阶段全文入库。

## 10. 分阶段实施路径

### 10.1 第一阶段：页面壳与任务中心

目标：

- 建立基础 UI 壳

## 11. 验收标准建议

满足以下条件时，可认为 200 第一阶段完成：

1. 页面左侧已经稳定展示四类一级菜单
2. `同步主数据` 页可以触发花名册和组织列表任务，并看到任务状态
3. `采集单据` 页可以触发 `collect` 任务，并查询已采集单据
4. `常规配置` 页可以修改运行时非敏感配置
5. 所有核心表格都已经统一接入 Data Grid
6. 页面展示的业务列名与现有中文字段口径保持一致
7. 页面中没有明文暴露真实账号口令或数据库密码

满足以下条件时，可认为 200 第三阶段完成：

1. `处理单据` 页可以查看风险信任评估总表
2. 可以查看风险信任评估明细
3. 可以从页面打开原始单据
4. 页面能明确区分“已评估”和“未评估”

## 12. 风险与注意事项

### 12.1 不要让前端直接依赖脚本命令细节

UI 如果长期直接拼接 CLI 命令，会很快失控。

必须尽快抽服务层。

### 12.2 不要把审批写回放在第一阶段

如果没有完整的审计、回滚、验证与权限边界，直接把页面做成“可审批”很危险。

### 12.3 不要把规则配置编辑器放进第一阶段

当前项目首先缺的是操作台，不是在线规则设计器。

### 12.4 不要忽略任务状态持久化

如果没有任务中心，UI 很容易退化成“点按钮后等日志”的壳，不足以支撑长期运营。

## 13. 方案结论

本方案建议：

- 保持现有 `Python + PostgreSQL` 主体不变
- 新增 `React + TypeScript + MUI` 前端
- 新增 `FastAPI` API 层
- 统一采用 `MUI X Data Grid` 作为大表页面基础组件
- 当前阶段采用社区版 `DataGrid`，并统一封装 `AppDataGrid`
- 第一阶段先做任务触发、数据查询、评估展示
- 审批 / 驳回 / 加签操作在 `103` 评估链路与写回服务成熟后再开放

这条路线与当前仓库实际能力、方案文档现状和后续演进方向是匹配的。

# 方案编号201：处理单据页抽屉化与评估分析页拆分方案文档

## 1. 方案目标

在现有 `200 / 104 / 105` 方案基础上，对 `处理单据` 页面做一次信息架构重整，使其回归“待办处理工作台”定位，并将批次级分析信息拆分到独立页面。

本方案明确交付以下结果：

- `处理单据` 页面以 `待处理单据列表` 作为唯一主工作区，占据整个页面主体
- `单据详情` 改为右侧弹出式抽屉，仅当点击列表中的 `单据编号` 时打开
- 抽屉内页签顺序调整为：
  - `单据概览`
  - `风险总览`
  - `审批单据`
  - `原始低分明细`
  - `申请角色`
  - `目标组织`
  - `审批记录`
- 将以下批次级内容从 `处理单据` 页面剥离，归集到独立 `评估分析` 页面：
  - `最新评估批次`
  - `最近执行日志`
  - `总结论分布`
  - `最终信任分分布`
  - `低分维度分布`
  - `低分规则 Top 10`

本方案不改评分规则、不改 PostgreSQL 表结构、不改 `104` 聚合反馈口径，只改页面承载方式与 API 输出结构。

补充说明：

- 本方案解决的是页面信息架构与交互方式
- 自 `203` 方案起，`处理单据` 的正式数据口径明确为“逐单最新评估结果”，不再以“最新评估批次”作为工作台唯一数据源  
  详见：`方案计划文档/203新采集单据增量评估与处理工作台最新结果口径方案文档.md`

## 2. 背景与问题

截至当前版本，`webui/src/pages/ProcessDocumentsPage.tsx` 已同时承载三类信息：

- 待处理单据列表
- 单据详情页签
- 批次级分析与执行日志

该结构存在以下问题：

1. `处理单据` 页面同时承担“工作台”和“分析台”两种职责，主次不清
2. 右侧详情常驻占位，导致待处理列表无法成为页面唯一主视图
3. 单据详情默认跟随行选择变化，容易打断列表连续处理节奏
4. `单据概览` 混在 `风险总览` 里，基础事实与风险判断没有分层
5. 批次分布、执行日志等批次级内容与单据级处理动作放在同页，干扰操作聚焦

因此，需要通过 `201` 方案把页面拆回符合行业常见模式的结构。

## 3. 与现有方案关系

### 3.1 与 `200` 的关系

`200` 已提出“主 Grid + 右侧详情抽屉”的方向，但当前 `处理单据` 页实际仍是“主表 + 常驻详情 + 批次分析”混合页。

`201` 的作用是把 `200` 中抽屉化思路正式落到 `处理单据` 页，并进一步完成“工作台 / 分析台”拆分。

### 3.2 与 `104` 的关系

`104` 定义的是 `风险总览` 的聚合展示口径。

`201` 不改变 `104` 的聚合规则，只调整其在详情抽屉中的位置：

- `单据概览` 独立为首个页签
- `风险总览` 专注展示结论、聚合反馈、处理提示

### 3.3 与 `105` 的关系

`105` 已在详情区新增 `审批单据` 页签并定义批准写回契约。

`201` 不改变 `105` 的审批动作语义，只将其承载位置从“右侧常驻详情区”调整为“右侧详情抽屉”。

`105A` 已补充审批连通性卡住场景下的超时与日志兜底口径；`201` 需同步接受该交互约束。

实施时需同步回写 `105` 文档中的相关表述：

- `右侧详情区` -> `右侧详情抽屉`

## 4. 设计原则

### 4.1 工作台与分析台分离

按行业常见的风控审核、工单处理、客服审单后台做法：

- `工作台` 负责列表检索、逐单处理、保持上下文连续性
- `分析台` 负责批次概况、分布统计、规则热点、执行日志

两者不应长期混在一个页面。

### 4.2 列表优先，明细按需打开

待处理场景的核心对象是“队列中的多张单据”，不是单张单据的长页面展示。

因此主页面应优先保证：

- 列表可视面积最大
- 过滤、排序、连续切换顺畅
- 明细仅在需要时再打开

### 4.3 基础事实与风险判断分层

`单据概览` 解决“这张单据是谁、申请了什么、当前状态如何”。

`风险总览` 解决“为什么这张单据需要关注、建议如何处理”。

两者必须拆开，避免用户在进入详情第一页时同时接收过多类型信息。

### 4.4 抽屉保持上下文，不跳页

对待处理列表来说，详情打开后不应导致用户离开当前列表上下文。

因此推荐使用右侧大抽屉，而不是：

- 行内展开
- 全页面跳转到详情页
- 页面内永久双栏占位

## 5. 范围与边界

### 5.1 本次纳入范围

- `处理单据` 页面重构为“全宽列表 + 右侧抽屉详情”

- 调整详情页签顺序，新增首个 `单据概览` 页签
- 后端接口按“工作台 / 分析页 / 单据详情”拆分输出
- 左侧导航增加独立菜单入口

### 5.2 本次不纳入范围

- 不修改 `申请单风险信任评估`、`申请单风险信任评估明细` 表结构
- 不修改 `104` 聚合算法与文案规则
- 不新增审批审计表
- 不新增图表引擎或复杂 BI 能力
- 不改变 `105` 已定义的批准执行链路

### 5.3 数据库边界

本方案原则上：

- **不新增 SQL 迁移**
- **不新增数据库字段**
- 仅在 Python 查询层拆分接口与返回结构

## 6. 页面方案

### 6.1 `处理单据` 页定位

页面定位调整为：

- 单据处理工作台
- 以待处理队列为中心
- 面向逐单打开、逐单判断、逐单审批

页面中不再承载批次分布和执行日志区块。

### 6.2 `处理单据` 页结构

页面结构统一为三层：

1. 页面标题区
   - 仅保留模块标题、简要说明
2. 列表工具区
   - 查询条件
   - 刷新
   - 快速筛选
   - 当前结果数
3. 待处理单据主 Grid
   - 占据整个页面主体
   - 不再与固定右侧详情并排分栏

说明：

- 原页面顶部大卡片区不再保留
- 原页面底部 `最新评估批次 / 分布 / 日志` 区不再保留
- 如需展示少量工作台摘要，只允许放在工具栏一行内，不得再次占据独立卡片区域

### 6.3 列表交互规则

主 Grid 保持现有字段主体不变，重点调整交互方式：

- 仅点击 `单据编号` 单元格时打开详情抽屉
- 点击其他单元格时，不自动打开抽屉
- 行选择、排序、分页、筛选状态在抽屉关闭后必须保留
- 抽屉打开期间允许继续看到列表背景，不跳离工作台
- 列表新增 `重新评估` 操作列，可对单张单据触发一次单据级 audit 补跑
- 点击 `重新评估` 后需展示任务状态（排队中 / 运行中 / 已完成 / 失败），并在任务运行期间自动轮询刷新工作台结果，避免“已点击但页面无变化”的误判

默认展示契约：

- `待处理单据列表` 默认每页显示 `20` 项
- 桌面端默认分页口径调整为 `20` 项；列表区域可按当前页面可用高度滚动浏览
- 小屏可按响应式降级，但默认分页口径保持 `20` 项

推荐补充规则：

- `单据编号` 使用明显的链接样式
- 支持键盘 `Enter` 打开抽屉
- 抽屉关闭后焦点返回最近一次点击的 `单据编号`

### 6.4 详情抽屉设计

详情采用右侧大抽屉模式。

桌面端建议尺寸：

- 宽度建议 `min(1568px, 100vw)`，即在既有桌面端抽屉基础上放大约 `40%`
- 当屏宽不足时直接贴合当前视口宽度
- 小屏下退化为全屏抽屉

采用大抽屉而不是窄抽屉，原因如下：

- 详情内存在多个 Data Grid 页签
- `审批单据`、`原始低分明细`、`目标组织` 均需要较宽可视区域
- 窄抽屉会导致再次出现横向压缩和信息换行

### 6.5 详情抽屉头部

抽屉头部建议包含：

- 单据编号
- 权限对象
- 当前总结论
- 最终信任分
- 关闭按钮
- `打开原始单据` 快捷入口

说明：

- 头部只保留高频摘要，不重复堆叠整套明细字段
- 详细字段统一下沉到 `单据概览` 页签

### 6.6 页签顺序

抽屉内页签顺序统一为：

1. `单据概览`
2. `风险总览`
3. `审批单据`
4. `原始低分明细`
5. `申请角色`
6. `目标组织`
7. `审批记录`

该顺序的理由如下：

- 先看基础事实
- 再看风险判断
- 再决定是否审批
- 最后查看证据与展开明细

### 6.7 `单据概览` 页签

该页签承接当前 `overviewFields`。

字段建议包括但不限于：

- `单据编号`
- `申请人`
- `权限对象`
- `单据状态`
- `部门`
- `申请日期`
- `申请人身份`
- `申请人组织单位`
- `最新审批时间`
- `申请人组织流程层级分类`
- `最终信任分`
- `总结论`
- `最低命中维度`
- `低分明细条数`
- `评估批次号`
- `评估时间`

展示要求：

- 使用键值清单组件，优先双列或三列排布
- 不在该页签重复展示聚合风险摘要文本
- 不在该页签展示审批输入框

### 6.8 `风险总览` 页签

该页签继续承接 `104` 聚合反馈，专注展示：

- 当前展示结论
- 风险统计卡片
- 聚合反馈分组摘要
- 处理提示

该页签不再放 `单据概览`。

### 6.9 `审批单据` 页签

该页签继续承接 `105` 已定义的能力：

- 审批意见
- `批准`
- `验证连通性（dry-run）`
- 执行结果回显

仅调整承载容器，不改变后端执行语义。

补充约束：

- 若审批请求超过 `120` 秒未返回，抽屉内不应无限停留在 `验证中...` 或 `批准中...`
- 前端需给出明确提示，引导用户查看：
  - 弹出的 EHR 浏览器
  - `automation/logs` 下最新 `approval_*.json`
- 具体排查与兜底策略以 `105A` 为准

### 6.10 其余页签

以下页签继续保留原有职责：

- `原始低分明细`
- `申请角色`
- `目标组织`
  - 去掉角色编码、角色名称
  - 按组织维度聚合展示组织编码、组织名称、组织单位、所属战区、物理层级、组织流程层级分类、组织路径名称、组织授权级别、聚合行数
  - `组织路径名称` 取自 `组织属性查询.组织长名称`，用于承接组织路径阅读诉求
  - 默认按物理层级数字升序显示，`聚合行数` 表示命中的角色-组织明细条数
- `审批记录`
  - 基于审批人工号带出职位、组织单位、所属战区、组织流程层级分类、组织路径名称
  - 其中职位、组织路径名称取自 `人员属性查询`，组织单位、所属战区、组织流程层级分类取自 `人员属性查询.部门ID -> 组织属性查询`

这些页签均在抽屉内按需加载或在首次打开后缓存当前结果，避免频繁重刷。

列表型页签默认展示契约：

- `原始低分明细` 默认每页显示 `20` 项
- `申请角色` 默认每页显示 `20` 项
- `申请角色` 默认按 `权限级别` 顺序展示：`S1类-限定` -> `S2类-限定` -> `A类-远程` -> `W类-取消` -> `B1类-涉薪` -> `B2类-涉档案绩效` -> `C类-常规` -> `D类-普通`，未识别或空值排最后；同级别内按 `明细行号` 升序稳定展示
- `目标组织` 默认每页显示 `20` 项
- `审批记录` 默认每页显示 `20` 项
- 若后续 `单据概览` 或 `风险总览` 内新增列表型子区块，默认页容量也统一按 `20` 项执行，除非方案文档另有明确说明

说明：

- `201` 统一将“工作台主列表 + 详情抽屉内全部列表页”的默认分页口径收敛为 `20`
- 不再出现主列表与详情列表默认分页不一致的体验

## 7. `评估分析` 独立页面方案

### 7.1 页面命名

新增左侧一级菜单：

- `评估分析`

页面描述建议：

- 批次概况、规则热点与执行日志

说明：

- `处理单据` 强调“逐单处理”
- `评估分析` 强调“批次观察”

这一命名符合行业中 `Workbench / Analytics` 的拆分习惯。

### 7.2 页面承接内容

以下内容从 `处理单据` 页面迁移到 `评估分析` 页面：

- `最新评估批次`
- `最近执行日志`
- `总结论分布`
- `最终信任分分布`
- `低分维度分布`
- `低分规则 Top 10`

### 7.3 页面结构建议

建议按以下结构组织：

1. 查询区
   - 评估批次号选择
   - 版本信息
   - 刷新
2. 批次摘要区
   - `最新评估批次`
   - 单据数
   - 明细数
   - 原始低分明细数
   - 评估时间
3. 分布区
   - `总结论分布`
   - `最终信任分分布`
   - `低分维度分布`
   - `低分规则 Top 10`
4. 执行日志区
   - `最近执行日志`

### 7.4 行业规范说明

之所以将这些信息拆到独立页面，而不是继续挂在 `处理单据` 底部，原因是：

- 这些信息天然是“批次级 / 规则级 / 运行级”信息
- 它们服务于规则治理、运行监控、结果复盘
- 它们不直接服务于操作员逐条处理当前列表中的某一张单据

在审核类系统中，这类内容通常归入：

- 分析页
- 看板页
- 运营监控页

而不是单据处理工作台。

## 8. 接口与数据结构方案

### 8.1 接口拆分原则

后端接口按职责拆分为三类：

1. 工作台列表接口
2. 单据详情接口
3. 评估分析接口

避免继续使用一个大接口同时返回：

- 列表
- 批次摘要
- 分布统计
- 执行日志

### 8.2 建议接口

建议保留现有单据详情与审批接口能力，同时新增或拆分如下：

- `GET /api/documents/process-workbench`
  - 返回工作台列表数据
- `GET /api/documents/process-workbench/{document_no}`
  - 返回单据详情抽屉数据
- `POST /api/documents/process-workbench/{document_no}/approval`
  - 返回审批执行结果
- `GET /api/documents/process-analysis`
  - 返回批次分析页数据

若实施时考虑兼容已有前端，也可采用过渡命名：

- 保留 `/api/documents/process-dashboard/{document_no}`
- 新增 `/api/documents/process-analysis`

但从长期命名看，`workbench / analysis` 比继续沿用 `dashboard` 更清晰。

### 8.3 Python 查询层建议

`automation/db/postgres.py` 建议拆分为：

- `fetch_process_workbench()`
- `fetch_process_document_detail()`
- `fetch_process_analysis_dashboard()`

`fetch_process_document_detail()` 继续复用现有明细查询逻辑。

`fetch_process_analysis_dashboard()` 复用现有：

- `_fetch_process_batch_summary()`
- `_fetch_process_distribution_sections()`
- `_load_execution_logs()` 对应逻辑

### 8.4 前端类型建议

`webui/src/types/dashboard.ts` 建议拆为：

- `ProcessWorkbench`
- `ProcessDocumentDetail`
- `ProcessAnalysisDashboard`

避免继续让 `ProcessDashboard` 同时承担两种页面职责。

### 10.3 第三步：工作台抽屉化

- 移除 `处理单据` 页常驻右侧详情区
- 改为点击 `单据编号` 打开右侧大抽屉
- 调整页签顺序，新增首个 `单据概览`

### 10.4 第四步：联调与回归

- 校验抽屉详情字段、审批动作、原始明细分页
- 校验批次分析页数据与原页面一致
- 校验关闭抽屉后列表位置、分页、筛选未丢失

## 11. 验收标准

满足以下条件时，可认为 `201` 完成：

1. `处理单据` 页面主体只保留待处理列表与工具栏，不再展示批次分布和执行日志
2. 只有点击 `单据编号` 时才会打开详情抽屉
3. `待处理单据列表` 默认每页显示 `20` 项
4. 详情以右侧大抽屉打开，桌面端宽度足以承载多页签表格
5. 详情第一页签为 `单据概览`
6. `风险总览` 中不再混放 `单据概览`
7. 详情抽屉内所有列表型页签默认每页显示 `20` 项
8. `审批单据` 页签仍可正常执行 `105` 方案定义的批准与 dry-run
9. `最新评估批次 / 最近执行日志 / 总结论分布 / 最终信任分分布 / 低分维度分布 / 低分规则 Top 10` 已完整迁移到 `评估分析` 页面
10. 关闭抽屉后，列表筛选、排序、分页状态保持不变

## 12. 风险与注意事项

### 12.1 不要把抽屉做成窄侧栏

如果抽屉宽度过小，`原始低分明细`、`目标组织`、`审批记录` 仍会出现阅读困难，等于只把问题从常驻右栏变成窄抽屉。

### 12.2 不要让整行点击继续打开详情

本次需求已明确要求“仅当点击单据编号时打开抽屉”。

因此实施时必须移除现有整行点击打开详情的行为。

### 12.3 不要把批次分析内容再塞回工作台头部

即使改成折叠卡片，也会再次侵占工作台空间。

`201` 的核心之一就是职责拆分，实施时应坚持边界。

### 12.4 文档同步要求

代码实施前后，需同步更新以下文档中的相关表述：

- `方案计划文档/200可视化页面UI建设方案文档.md`

尤其是：

- `处理单据` 页布局描述
- `审批单据` 页签所在容器描述

## 13. 方案结论

`201` 的结论是：

- `处理单据` 应回归为面向操作员的单据处理工作台
- `单据详情` 应改为按需打开的右侧大抽屉
- `单据概览` 应作为详情首个独立页签
- 批次级分析与执行日志应拆出为独立 `评估分析` 页面

该方案不改变底层评分、审批、数据库契约，只调整页面职责边界与交互形态，属于低业务风险、高清晰度收益的结构优化。

## 14. `105B` 后补口径

自 `105B` 起，`处理单据` 页主列表在 `201` 的结构基础上，再补充以下工作台维度：

- 新增列：`待办处理状态`
- 新增筛选：`待办处理状态`
- 默认视图：仅展示 `待处理`
- 新增工具栏动作：`同步待办状态`

这里的 `待办处理状态` 指：

- 当前账号最近一次待办同步结果

它不等于：

- 业务单据终态
- 风险评估结论

因此 `201` 之后的 `处理单据` 页面，应理解为：

- “逐单最新评估结果”之上的“当前账号待办处理工作台”

而不再只是纯评估结果列表。

# 方案编号007：组织属性查询表城市战区未命中补齐方案文档

## 1. 方案目标
在 `组织属性查询` 已落地的基础上，针对 `war_zone` 未命中的 `city_name` 值进行专项治理。

本方案目标：
- 对 `组织列表.city_name` 当前未命中的城市值进行逐项核实
- 通过联网查询确认其真实行政归属或地理归属
- 结合现有 `城市所属战区` 字典中的战区口径，提出可执行的映射建议
- 形成一套可落地的补齐方案，支撑后续刷新 `组织属性查询`
- 与 `005` 方案保持口径一致，明确 `组织流程层级判断` 列仍按既定递归回溯规则生成

## 3. 分析方法
本次建议分两步形成：

1. **联网核实真实归属**
- 对县级市、县、自治州首府、香港地区名称，确认其所属地级市 / 自治州 / 特别行政区

2. **套用现有战区口径**
- 以现有 `城市所属战区` 中已存在的上级城市或州级名称作为参照
- 若可对应到现有字典城市，则沿用同一 `war_zone`
- 若属于香港地区等现有字典未覆盖区域，则结合当前组织落点与地理邻接关系提出建议战区

#### 4.2.2 不建议直接补字典的 1 个泛化值
- `省直辖县级行政区划`

原因：
- 该值不是稳定、明确的具体城市名称
- 它是统计口径下的泛化行政区描述
- 如果直接写入：
  - `省直辖县级行政区划` → `琼桂战区`
- 将来若同名值出现在其他省份，可能会错误归区

因此本方案建议：
- **不要**把 `省直辖县级行政区划` 作为普适字典值直接写入 `城市所属战区`
- 而是在 `组织属性查询` 刷新逻辑中增加**上下文规则**：
  - 当 `city_name = '省直辖县级行政区划'`
  - 且 `org_full_name` 或组织上下文指向 `海口城市` / `海南`
  - 则映射为 `琼桂战区`

当前这 1 行的实际数据为：
- `city_name = 省直辖县级行政区划`
- `company_name = 广州区域公司`
- `org_full_name = 万物云_万物梁行_广州区域公司_海口城市_海口设施管理一部_Q海南辰腾国际智能物流园`

因此该行当前建议明确归为：
- `琼桂战区`

## 5. 建议依据

### 5.1 结合现有内部战区字典的对照依据
当前 `城市所属战区` 已有以下口径：
- `苏州市` → `苏州战区`
- `泰州市` → `苏州战区`
- `嘉兴市` → `上海战区`
- `长沙市` → `湘赣战区`
- `岳阳市` → `湘赣战区`
- `巴音郭楞蒙古自治州` → `西北战区`
- `恩施土家族苗族自治州` → `鄂豫战区`
- `海口市` → `琼桂战区`
- `深圳市` → `深圳战区`
- `惠州市` → `深圳战区`

因此：
- 县级市 / 县 / 州府可直接沿用其上级地市或自治州的战区
- 香港地区暂无现成字典，需要专项建议

### 5.2 对香港地区映射为 `深圳战区` 的建议依据
对以下值：
- `香港特别行政区`
- `新界`
- `元朗区`
- `大埔区`
- `深水埗区`

本方案建议统一映射到：
- `深圳战区`

这是一个**结合数据现状与现有战区边界做出的推断**，不是来自外部官方“战区”口径本身。

建议理由：
- 当前未命中香港数据全部落在 `香港区域公司`
- 地理上香港与深圳接壤，是大湾区东侧最紧密联动区域
- 现有 `城市所属战区` 中：
  - `深圳市` → `深圳战区`
  - `惠州市` → `深圳战区`
- 相比 `广州战区` 或 `佛山战区`，将香港归入 `深圳战区` 更符合当前东侧区域的业务邻接逻辑

因此这里应标注为：
- **业务映射建议 / 推断性建议**
- 后续如集团明确给出香港区域归属口径，应以集团正式口径覆盖本建议

## 7. 与 005 方案的口径一致性补充

### 7.1 一致性原则
本次 `007` 专项方案的核心目标是补齐 `war_zone` 未命中问题，
**不改变** `005` 方案已定义的 `组织流程层级判断` 口径。

因此 `组织属性查询` 仍应同时保留：
- `process_level_name`
  - 表示源表原始值
- `process_level_name_resolved`
  - 表示“组织流程层级判断”统一口径列

### 7.2 组织流程层级判断规则
`process_level_name_resolved` 的规则与 `005` 方案保持一致：

1. 当前组织优先
- 如果 `process_level_name` 有值，则直接取当前值

2. 上级组织递归回溯
- 如果 `process_level_name` 为 `NULL`
- 则取其上级组织的 `process_level_name`
- 如果其上级组织的 `process_level_name` 也为 `NULL`
- 则继续沿 `parent_org_code` 向上追溯
- 直至某个隔级上级首次出现非空 `process_level_name`
- 命中后取该值作为 `process_level_name_resolved`

3. 全链路均为空
- 如果当前组织及全部上级组织均未取到非空 `process_level_name`
- 则 `process_level_name_resolved = NULL`

### 7.3 本方案落地时的实现要求
本次若继续调整 `refresh_组织属性查询()`，应同时保证以下事项不被破坏：
- 保留 `005` 中 `process_level_name_resolved` 字段
- 保留沿组织树向上递归回溯最近非空 `process_level_name` 的计算逻辑
- 本次 `war_zone` 补齐规则应与 `process_level_name_resolved` 规则并存，不应相互覆盖
- 若后续重构刷新 SQL，需同步回归验证这两个派生口径均正确产出

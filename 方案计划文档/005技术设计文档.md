# 方案编号005：组织属性查询表设计方案文档

## 1. 设计目标
在现有 `组织列表` 与 `城市所属战区` 两张基础表之上，新增一张面向高频查询场景的结果表：
- `组织属性查询`

该表用于承接以下查询诉求：
- 按组织编码、组织名称、公司、城市、战区、组织单位等维度高频查询
- 对 `组织列表.org_full_name` 做稳定、可复用的“组织单位”映射
- 对 `组织列表.city_name` 做“所属战区”映射
- 在 `组织列表` 全量更新后，同步全量刷新结果表

本方案目标不是替代 `组织列表`，而是构建一张：
- 字段稳定
- 口径固定
- 可直接用于业务查询
- 避免每次查询时重复解析路径和关联战区的派生表

## 2. 设计结论

### 2.1 最终选型
本方案建议采用：
- **实体结果表**：`组织属性查询`
- **数据库函数**：用于 `org_full_name -> 组织单位` 映射
- **数据库存储过程**：用于全量重建 `组织属性查询`
- **应用侧导入流程调用刷新过程**：在 `组织列表` 全量写入完成后立即刷新

不建议采用以下方案作为最终实现：
- 普通 `VIEW`
- `MATERIALIZED VIEW`
- `GENERATED COLUMN`
- 行级 `TRIGGER`

原因如下：

1. `VIEW`
   - 每次查询都要重复做路径拆分与战区关联
   - 不适合“将被频繁查询”的目标场景

2. `MATERIALIZED VIEW`
   - 需要额外刷新管理
   - 对中文表名、定制函数、索引和切换控制不如实体表直观
   - 当前项目已存在全量导入逻辑，用实体表更容易和现有流程对齐

3. `GENERATED COLUMN`
   - `所属战区` 依赖 `城市所属战区` 表，生成列不适合跨表查值
   - 组织单位规则后续可能持续扩充，放在函数 + 刷新过程更灵活

4. 行级 `TRIGGER`
   - 当前 `组织列表` 是全量覆盖导入
   - 若逐行维护结果表，复杂度更高，整体效率也未必优于一次性全量重建

### 2.2 总体策略
采用“**源表全量导入 -> 结果表全量重建**”模式：

1. 自动化导出并导入 `组织列表`
2. `组织列表` 落库成功后，调用数据库存储过程
3. 存储过程基于 `组织列表` + `城市所属战区` 全量重建 `组织属性查询`
4. 业务查询统一查询 `组织属性查询`

该方案在当前数据量级下最稳妥：
- `组织列表` 当前约 `32020` 行
- 即使全量重建，成本也可控
- 读取性能优于每次在线计算

## 5. 组织单位映射口径

## 6. 所属战区映射口径

## 7. 同步刷新机制设计

### 7.1 刷新触发时机
刷新时机应定义为：
- `组织列表` 全量导入完成后立即刷新

即在以下流程后增加一步：
1. 自动化下载组织列表 Excel
2. 解析并全量写入 `组织列表`
3. 调用 `refresh_组织属性查询()`

### 7.2 刷新策略
采用“**全量重建**”而不是增量更新：

原因：
- `组织列表` 本身就是快照型全量覆盖
- 组织路径或城市发生变化时，增量维护复杂度更高
- 当前数据量较小，全量重建成本可接受

### 7.3 推荐实现方式
建议使用：
- `fn_map_org_unit_name(org_full_name text)`
- `fn_map_org_unit_rule(org_full_name text)`
- `refresh_组织属性查询()`

其中：
- 函数负责组织单位映射
- 存储过程负责整表重建

### 7.4 刷新过程建议步骤
建议刷新过程采用“影子表 + 原子切换”模式：

1. 创建临时构建表，例如：`组织属性查询__staging`
2. 将 `组织列表` 与 `城市所属战区` 关联结果全量插入到 staging 表
3. 在 staging 表上补建索引
4. 执行 `ANALYZE`
5. 在事务内完成：
   - 旧表重命名为备份名
   - staging 表重命名为正式表 `组织属性查询`
   - 删除旧备份表

这样设计的好处：
- 避免高频查询时读到半成品数据
- 避免 `TRUNCATE + INSERT` 期间结果表短暂为空
- 更适合后续定时任务场景

### 7.5 与定时任务的关系
后续若存在定时任务定时刷新 `组织列表`：
- 定时任务完成 `组织列表` 全量导入后
- 同一任务链路内继续调用 `refresh_组织属性查询()`

即：
- `组织属性查询` 不直接采集
- 始终由 `组织列表` 派生

## 8. 查询性能设计

### 8.1 使用原则
后续业务查询原则上统一查询：
- `组织属性查询`

不建议在业务查询中直接：
- 使用 `组织列表` 现拆 `org_full_name`
- 临时关联 `城市所属战区`

### 8.2 索引建议
建议为 `组织属性查询` 建立以下索引：

1. 主键 / 唯一索引
- `PRIMARY KEY (org_code)`

2. 高频单列索引
- `org_unit_name`
- `war_zone`
- `city_name`
- `company_name`
- `parent_org_code`
- `org_level`
- `dept_category_code`
- `dept_subcategory_code`

3. 高频组合索引（建议首批即建立）
- `(war_zone, city_name)`
- `(org_unit_name, company_name)`

### 8.3 不建议的设计
当前数据规模下，不建议：
- 分区表
- 过度冗余索引
- 每次查询时临时正则拆分路径

## 10. 应用侧改造建议

### 10.1 改造入口
建议在 `PostgresOrganizationListStore.write_rows()` 成功写入 `组织列表` 后，立即执行：
- `SELECT refresh_组织属性查询();`

当前相关代码位置：
- `automation/db/postgres.py:387`

### 10.2 调用时机建议
建议顺序：
1. `ensure_table()`
2. 清空并写入 `组织列表`
3. 提交事务
4. 调用 `refresh_组织属性查询()`

如需保证同一事务内一致性，也可在同一连接中先写源表，再调用刷新过程。

### 10.3 错误处理建议
若刷新过程失败：
- `组织列表` 导入结果仍保留
- 记录明确错误日志
- 标记本次“派生表刷新失败”
- 可支持单独重跑 `refresh_组织属性查询()`

## 11. 风险与边界

### 11.1 城市字典未覆盖
风险：
- `war_zone` 出现空值

控制方案：
- 允许 `war_zone` 为空
- 定期补齐 `城市所属战区`

### 11.2 组织路径新变体
风险：
- `org_full_name` 后续出现新命名模式
- 可能导致“组织单位”误分配到较粗粒度节点

控制方案：
- 在函数中维护细粒度优先规则
- 每次组织架构变化后抽样评估
- 必要时补充规则并全量刷新

### 11.3 高频查询下的刷新影响
风险：
- 结果表刷新过程影响在线查询

控制方案：
- 使用 staging 表 + 原子切换
- 避免直接 `TRUNCATE` 正式查询表

## 12. 最终结论
本方案最终建议如下：

1. 新增结果表：`组织属性查询`
2. 数据来源：`组织列表` + `城市所属战区`
3. 组织单位口径：以 `org_full_name` 为准，采用“细粒度优先 + 直接命中 + 特殊同名映射”
4. 所属战区口径：以 `city_name` 关联 `城市所属战区.war_zone`
5. 同步策略：`组织列表` 每次全量更新后，同步全量刷新 `组织属性查询`
6. 查询策略：业务统一查询 `组织属性查询`
7. 性能策略：为高频过滤字段建立索引，避免查询时重复计算

在当前数据范围与业务模式下，这是最适合落地、维护成本最低、查询性能最稳定的设计方案。

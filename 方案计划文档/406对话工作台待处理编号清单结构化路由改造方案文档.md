# 方案编号406：对话工作台待处理编号清单结构化路由改造方案文档

## 1. 背景

`403`/`404` 已完成对话工作台的“路由 + 正式工具 + 审批计划”主链路。2026-03-22 联调中出现了一个高优先级口径问题：

1. 用户明确要求“列出全部待处理单据编号”。
2. 助手有时只返回数量，或返回不完整编号列表。
3. 在极端情况下，返回内容与“待处理”真实口径不一致。

该问题直接影响对话工作台在处理单据场景下的可信度，需按长期方案修复，不采用关键词临时兜底作为最终形态。

## 2. 问题复盘

### 2.1 线上现象

1. 同一时刻 `stats.待处理单据=5`，但助手输出的待处理编号清单并非 5 条。
2. 个别回复混入 `已处理` 单据编号。

### 2.2 根因

1. 路由在部分“编号清单”问题中返回 `answerMode=templated`。
2. `templated` 分支默认模板只输出数量，不输出编号。
3. 模型生成分支中 `get_process_workbench` 的摘要仅注入 `documents[:10]`，导致“全部编号”在数据量较大时被截断。
4. 缺少“是否必须输出待处理编号清单”的结构化路由信号，导致后端只能依赖文案/关键词推断。

## 3. 方案目标

1. 用结构化意图替代关键词判断，消除临时硬编码。
2. 当用户明确要求“待处理编号清单”时，稳定输出完整清单。
3. 对错误路由结果增加后端强校验，避免 silent wrong answer。
4. 保持 `403/404/405` 主链路兼容，不引入新业务 Tool。

## 4. 范围与边界

### 4.1 纳入范围

1. 路由 Schema 扩展。
2. Router Prompt 约束增强。
3. `ChatService` 决策校验与模板分支调整。
4. 单测/回归测试补齐。

### 4.2 明确排除

1. 不新增数据库表结构。
2. 不新增审批写动作工具。
3. 不改 `process-workbench` 正式接口契约。

## 5. 方案设计

### 5.1 RouterDecision 新增结构化字段

在 `automation/chat/router_models.py` 的 `RouterDecision` 中新增：

- `requiresPendingDocumentList: bool = false`

语义：

1. 仅当用户明确要求“列出全部待处理单据编号”时设置为 `true`。
2. 默认保持 `false`，避免泛化污染。

### 5.2 Router Prompt 增加显式约束

在 `automation/chat/router_prompt.py` 中增加规范：

1. 何时设置 `requiresPendingDocumentList=true`。
2. 当其为 `true` 时，必须：
   - `route=tool_first`
   - 使用 `get_process_workbench`
   - `missingInputs` 为空

### 5.3 ChatService 增加后端强校验

在 `automation/chat/service.py::_validate_router_decision()` 增加规则：

1. `requiresPendingDocumentList=true` 但 `route!=tool_first` -> 拒绝该路由结果。
2. `requiresPendingDocumentList=true` 且 `missingInputs` 非空 -> 拒绝。
3. `requiresPendingDocumentList=true` 但首个工具不是 `get_process_workbench` -> 拒绝。

未通过校验时统一回退 `general_chat`，保持可控降级。

### 5.4 模板分支按结构化意图输出编号清单

在 `compose_answer()` 的 `templated` 分支中：

1. 当 `requiresPendingDocumentList=true` 且工具结果为 `get_process_workbench` 时，
2. 走专用清单回复：按 `todoProcessStatus=待处理` 过滤 `documents`，输出全部 `documentNo`。

### 5.5 工具摘要补充完整待处理集合

在 `_summarize_tool_result(get_process_workbench)` 中新增：

1. `pendingDocumentCount`
2. `pendingDocumentNos`（完整集合）

并保留 `documents[:10]` 作为兼容摘要字段，避免旧逻辑断裂。

## 7. 验收标准

### 7.1 功能验收

1. 用户问“列出全部待处理单据编号，不要只给数量”时，返回完整编号清单。
2. 返回结果仅包含 `todoProcessStatus=待处理` 的单据。
3. 不再出现“统计值=5，清单=1（因截断）”问题。

### 7.2 稳定性验收

1. `requiresPendingDocumentList` 非法组合可被后端拦截并回退。
2. 现有 `403/404` 用例不回归。

### 7.3 测试清单

1. `tests/test_chat_router_models.py`
   - 校验新字段默认值与显式值。
2. `tests/test_chat_service.py`
   - 新增“待处理编号不在前10条也能完整返回”的回归用例。
   - 新增“结构化意图非法组合触发回退”的用例。

## 8. 发布与回滚

### 8.1 发布步骤

3. 联调验证真实 `GET /api/documents/process-workbench` 数据下的编号清单场景。
4. 观察 1 个工作日后再扩大流量。

### 8.2 回滚策略

1. 若出现路由兼容问题，先回滚本方案相关提交。
2. 保持 `tool_first + templated` 原链路可用（仅回到旧行为，不影响服务可用性）。

## 9. 风险与后续

### 9.1 已知风险

1. 路由模型若长期误设 `requiresPendingDocumentList`，可能造成不必要的清单化回复。
2. 前端若对超长清单展示无折叠，可能影响可读性。

### 9.2 后续优化

1. 在路由阶段增加 few-shot，提升 `requiresPendingDocumentList` 触发精度。
2. 在前端增加“编号清单折叠 + 一键复制”。
3. 为“待处理编号清单”增加端到端 smoke case。

## 10. 与既有方案关系

1. 继承 `403` 的“路由 + 工具”框架，不改变工具边界。
2. 与 `404` 审批能力互补：先查准“待处理编号”，再执行审批计划。
3. 与 `405` 执行层升级兼容：本方案在路由契约层，不依赖具体执行后端模式。

# 402A 对话页面重复内容回归补充修复文档

## 1. 问题现象

在 `http://127.0.0.1:5173/chat-workspace` 中，部分 Assistant 消息会出现整段文本重复一次的情况，例如：

- 同一条欢迎语被完整重复
- 同一条查询结果被完整重复
- 重复内容通常发生在单条消息内部，而不是额外新增第二条消息卡片

## 2. 本次新增根因

`402` 已修复以下问题：

- SSE 重连时历史事件重复消费
- `Last-Event-ID` 未参与续传

但本次回归说明还存在另一条重复路径：

1. 前端先调用 `GET /api/chat/sessions/{id}` 获取会话详情
2. 详情接口返回的 `messages` 已经包含完整 Assistant 正文
3. 随后前端仍以 `afterSeq=0` 建立 SSE
4. 服务端重放该会话历史 `token` 事件
5. 前端把历史 `delta` 再次拼接到已经完整的消息正文后面

结果就是：

- 同一条 Assistant 消息内容被再次追加
- 页面上看到“整段复制一遍”的重复效果

## 3. 修复目标

本次补充修复要求如下：

1. 会话详情加载完成后，SSE 必须从当前最新事件游标继续，而不是固定从 `0` 开始。
2. 前端在收到 `done` 事件时，应使用最终完整消息做一次内容对齐，避免 token 级别误差残留到界面。

## 4. 实现方案

### 4.1 后端返回 `lastEventSeq`

文件：

- `automation/chat/service.py`

改动：

- 在 `get_session_detail` 返回值中补充 `lastEventSeq`
- 取值口径为当前会话内存事件流的最新 `seq`

作用：

- 前端可在加载完详情后，从正确游标继续订阅后续 SSE

### 4.2 前端先 hydrate，再按 `lastEventSeq` 建连

文件：

- `webui/src/pages/ChatWorkspacePage.tsx`
- `webui/src/types/chat.ts`

改动：

1. `ChatSessionDetail` 增加 `lastEventSeq`
2. 进入会话时，先加载详情
3. 详情返回后，将 `lastSeqRef` 与 `streamAfterSeq` 设置为 `detail.lastEventSeq`
4. 仅当 `streamAfterSeq` 就绪后，才创建 `EventSource`

效果：

- 历史消息保持以详情接口为准
- SSE 只消费详情之后的新事件
- 不再把历史 token 重新拼接到已存在正文上

### 4.3 `done` 事件做最终正文对齐

文件：

- `webui/src/pages/ChatWorkspacePage.tsx`

改动：

- 收到 `done` 事件时，如果带有 `assistantMessageId + message`，则直接把对应消息内容覆盖为最终全文

作用：

- 即使流式 token 存在边界抖动，最终完成态也以服务端落库全文为准

## 5. 验收标准

修复后需要满足：

1. 打开已有历史会话时，不再出现历史 Assistant 文本被整段重复。
2. 刷新页面后，同一条消息内容保持稳定，不会再被 SSE 历史事件追加一次。
3. 新发起对话时，流式展示正常，完成后正文与数据库最终内容一致。

## 6. 影响范围

本次仅涉及对话工作台：

- 不改数据库表结构
- 不改正式业务工具返回口径
- 不影响审批与查询链路

属于前后端对话事件消费口径补充修复。

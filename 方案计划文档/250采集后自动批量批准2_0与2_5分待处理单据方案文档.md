# 方案编号250：采集后自动批量批准 2.0 / 2.5 分待处理单据方案文档

## 1. 背景

当前链路已具备：

1. 采集完成后自动增量评估（220A）。
2. 处理单据页批量批准能力（230）。

但“采集 -> 评估 -> 自动处理”仍需人工点击批量批准。为缩短低风险待办处理链路，新增“采集后自动批量批准”能力。

## 2. 目标

新增一个可配置开关：开启后，在采集完成（且满足触发条件）时自动调用批量批准能力，自动批准本轮命中的待处理单据，命中条件为：

- 待办处理状态 = `待处理`
- 最终信任分 ∈ `{2.0, 2.5}`

## 3. 配置口径

### 3.1 配置项

- 配置名称：`采集后自动批量批准`
- API 字段：`autoBatchApprove`
- daemon 参数：`-AutoBatchApprove`
- runner 参数：`--auto-batch-approve`

### 3.2 单一事实来源

与定时采集现有口径保持一致，仍以 `automation/config/windows_task_daemon.local.json` 中 `collect.args` 为单一事实来源，通过是否包含 `-AutoBatchApprove` 表示开关状态。

## 4. 执行链路

1. `RuntimeSettingsPage` 保存配置，写入 `autoBatchApprove`。
2. `PUT /api/settings/runtime/collect-schedule` 调用 `update_collect_schedule`，维护 `collect.args`。
3. daemon 到期执行：
   - `run_windows_task.ps1 -Action collect -AutoBatchApprove ...`
4. `run_windows_task.ps1` 透传：
   - `run.py collect --auto-batch-approve ...`
5. `run.py collect` 在采集（及可选自动评估）后筛选候选单据，调用现有 `approve_process_documents_batch(action="approve")` 完成自动批准。

### 4.1 采集工作台手工启动链路

`/collect-documents` 页面手工启动采集时，也必须支持同一开关口径：

1. 页面请求字段：
   - `autoAudit`
   - `autoBatchApprove`
2. `POST /api/documents/collect-workbench/run` 必须将 `autoBatchApprove` 写入采集任务状态。
3. `automation/api/collect_workbench.py` 在采集 runner 成功、且可选增量评估成功后，对本轮成功写库或因审批轨迹未变化而跳过的单据筛选：
   - `todoProcessStatus = 待处理`
   - `finalScore ∈ {2.0, 2.5}`
4. 命中候选后调用 `approve_process_documents_batch(action="approve")`。
5. 任务摘要需返回：
   - `autoBatchApprove`
   - `batchApprovalStatus`
   - `batchApprovalMessage`
   - `batchApprovalLogFile`

该链路与 Windows daemon 链路并列，避免“页面开关可见但后台 runner 未收到自动批准指令”的断点。

## 5. 触发条件与筛选规则

### 5.1 自动批量批准触发条件

需同时满足：

1. `--auto-batch-approve = true`
2. 非 `dry-run`
3. 本轮存在可用于自动审批重试的候选来源单据

### 5.2 与自动评估关系

- 若本轮开启了自动评估且自动评估未成功，则自动批量批准跳过（避免在评估结果不完整时误触发自动审批）。
- 若本轮没有新增成功写库单据，仅存在“审批轨迹未变化”的跳过单据，则允许直接复用库内已有评估结果继续尝试自动批量批准。

### 5.3 候选单据范围

在以下来源单据范围内筛选，且满足：

1. 本轮成功写库单据
2. 本轮因审批轨迹未变化而跳过、但仍保留在当前账号待办中的单据

最终命中单据仍需满足：

1. `todoProcessStatus = 待处理`
2. `finalScore` 归一到 1 位小数后为 `2.0` 或 `2.5`

## 6. 结果与失败口径

### 6.1 结果口径

采集任务消息追加自动批量批准结果摘要，沿用批量审批返回状态：

- `succeeded`
- `partial`
- `failed`
- `skipped`（本轮无命中候选，或因自动评估未成功而跳过）

### 6.2 失败口径

- `partial / failed` 视为采集链路未闭环，本轮采集任务退出码置为非 0。
- `skipped` 不视为失败。

## 8. 验收标准

1. 常规配置页可查看并保存 `采集后自动批量批准` 开关。
2. 保存后 `collect.args` 正确增删 `-AutoBatchApprove`。
3. `run.py collect --auto-batch-approve` 可在采集后触发自动批量批准。
4. 仅 `待处理 + 2.0/2.5` 单据会被自动纳入批准。
5. 若上一轮自动批量批准因网络异常等原因失败，后续采集轮次对“审批轨迹未变化但仍待处理”的 `2.0/2.5` 单据会继续重试自动批量批准。
6. 自动批准结果会写入采集任务消息；`partial/failed` 时任务返回非 0。
7. 采集工作台手工启动采集时，开启“采集后自动批准”后，同样会在自动评估完成后执行批量批准并展示自动批准状态。

# 方案编号405：对话工作台模型执行长会话化升级方案文档

## 1. 背景

`403` 已完成“路由 + 澄清 + Explain + 只读工具编排”主链路，`404` 已在此基础上扩展审批计划能力。当前模型执行层仍采用每次调用拉起一次 `codex exec` 的短进程模式，适合早期快速落地，但不应作为长期形态。

当对话进入高频 turn、并发会话增加、需要更稳定会话态与更低延迟时，必须升级为长生命周期执行架构。

## 2. 现状确认

### 2.1 当前执行模式

1. 路由调用与主回答调用均通过 `subprocess.Popen` 启动 `codex exec`
2. 路由链路为 `run_router_exec(...)`
3. 主回答链路为 `run_codex_exec(...)`

### 2.2 当前可观测问题

1. 每 turn 至少一次模型子进程冷启动，工具问答场景通常还会叠加主回答调用
2. 仅限制“同一 session 不并发 run”，缺少全局并发治理
3. 运行中事件缓冲为进程内内存，API 进程重启后无法恢复该段运行态
4. 技能索引按 turn 触发签名扫描，高频对话下会产生额外 I/O

### 2.3 与 403 设计定位一致性

`403` 明确了 Phase 1 沿用 `codex exec`，本方案作为后续升级阶段，不推翻既有业务契约（路由、澄清、只读工具、审批计划状态机）。

## 3. 根因分析

1. 模型执行抽象层缺失：`service.py` 直接依赖 `run_codex_exec/run_router_exec`
2. 运行生命周期与会话生命周期未解耦：run 结束即进程结束
3. 缺少统一调度器：并发控制、排队、公平性、熔断均分散在业务层
4. 缺少持久化运行态：仅有消息与执行日志落库，没有 run 状态恢复协议

## 4. 方案目标

1. 在不破坏 403/404 业务能力的前提下，引入可切换执行后端
2. 将模型执行从“一轮一进程”升级为“长生命周期优先”
3. 建立全局并发与排队控制，避免高并发下资源抖动
4. 建立 run 状态持久化与重启恢复语义
5. 保留可一键回退到现有 `codex exec` 短进程模式的能力

## 5. 范围与边界

### 5.1 纳入范围

1. 对话执行内核改造（router + answer 两类模型调用）
2. 运行态持久化、并发治理、可观测指标
3. 兼容现有 SSE 事件协议与审批计划链路

### 5.2 明确排除

1. 不改动业务工具口径与权限边界（仍按注册工具执行）
2. 不改动 404 审批状态机语义
3. 不在本期扩展新的业务 Tool

## 6. 目标架构

### 6.1 分层

1. `ChatOrchestrator`：保留现有路由/澄清/工具/审批编排
2. `ModelExecutionAdapter`：统一执行接口，屏蔽后端差异
3. `ExecutionBackend`：
   - `oneshot_exec`（现状保底）
   - `app_server`（Phase 1 主推，起步实现）
   - `persistent_subprocess`（可选过渡/补充）
4. `RunStateStore`：run 生命周期持久化
5. `ExecutionScheduler`：全局并发、队列、超时、取消

### 6.2 核心原则

1. 先抽象后替换：先引入适配层，再切换后端
2. 模式可配置：按环境开关选择后端
3. 失败可降级：后端异常时自动回退 `oneshot_exec`
4. 业务无感：上层 `route_turn/compose_answer` 语义不变

## 7. 执行模式设计

### 7.1 模式 A：`oneshot_exec`（保底）

沿用当前 `codex exec` 短进程实现，作为全程兜底与紧急回滚模式。

### 7.2 模式 B：`persistent_subprocess`（可选过渡）

1. API 进程内维护常驻 Codex 子进程池
2. session 与 worker 做粘性绑定（可配置 TTL 自动回收）
3. router 与 answer 调用复用同一长连接执行通道
4. 支持健康检查、超时中断、异常重建

### 7.3 模式 C：`app_server`（推荐起步落地）

1. 由独立 `codex app-server` 提供模型执行服务
2. API 仅做编排与状态管理，执行层外置
3. 支持横向扩展与更稳定的长会话状态承载

## 8. 并发与会话稳定性设计

### 8.1 调度策略

1. 每 session 单飞行（保持当前语义）
2. 全局并发上限（新增）
3. 超限进入排队，返回排队状态事件
4. 支持取消排队与取消运行中任务

### 8.2 持久化运行态

新增 run 状态存储（建议新增 SQL `031_chat_run_state.sql`），至少包含：

1. `run_id`
2. `session_id`
3. `status`（queued/running/cancel_requested/succeeded/failed/timeout/interrupted）
4. `backend_mode`
5. `worker_id`
6. `started_at/updated_at/finished_at`
7. `error_code/error_message`

### 8.3 重启恢复语义

1. API 启动时扫描 `running/queued` run，统一置为 `interrupted`
2. SSE 向前端回放“上次运行中断”事件
3. 不自动重放旧 run，避免幂等风险

## 9. 延迟优化策略

1. 长连接复用，消除频繁进程冷启动
2. 路由与主回答在同执行通道内串行，减少初始化成本
3. 技能索引改为“按目录变更触发刷新 + 最短刷新间隔”
4. 增加首 token 延迟与总耗时埋点，支撑容量调优

## 10. 与 403/404 的兼容策略

1. `RouterDecision`、`ToolRegistry`、`ApprovalPlanStore` 契约保持不变
2. `execute_tool_plan()` 与审批命令链路不改语义
3. 仅替换模型执行入口，不改变业务决策结构
4. `router_fallback -> general_chat` 兜底策略保持可用

## 11. 配置开关设计

建议新增环境变量：

1. `CLAWCHECK_CHAT_EXEC_MODE=oneshot_exec|persistent_subprocess|app_server`
2. `CLAWCHECK_CHAT_GLOBAL_MAX_CONCURRENT_RUNS=4`
3. `CLAWCHECK_CHAT_RUN_QUEUE_SIZE=200`
4. `CLAWCHECK_CHAT_SESSION_IDLE_TTL_SECONDS=900`
5. `CLAWCHECK_CHAT_APP_SERVER_BASE_URL=...`
6. `CLAWCHECK_CHAT_APP_SERVER_TIMEOUT_SECONDS=...`
7. `CLAWCHECK_CHAT_EXEC_AUTO_FALLBACK=true|false`

## 12. 实施清单

### 12.1 Phase 0：抽象层与基线指标

1. 新增 `ModelExecutionAdapter` 接口与 `oneshot` 适配器
2. `service.py` 改为依赖适配器，不再直接调用 `run_*_exec`
3. 增加耗时与并发指标埋点
4. 验证行为与现网一致

### 12.2 Phase 1：app-server 起步模式

1. 实现 `app_server` backend 与健康探针
2. 新增 `ExecutionScheduler`（全局并发+队列）
3. 新增 run 状态持久化表及读写
4. 先灰度切流到 `app_server`（10% -> 50% -> 100%），异常自动降级到 `oneshot`

### 12.3 Phase 2：稳定性增强与可选补充

1. 完成容量压测与扩缩容策略，固化告警与健康探针阈值
2. 按环境完善 `app_server` 灰度/回滚自动化
3. 必要时按低优先级补充 `persistent_subprocess`，仅用于特定环境过渡
4. 将生产默认模式保持为 `app_server`

## 13. 文件级改造建议

1. `automation/chat/service.py`：改为调用统一执行适配层
2. `automation/chat/codex_runner.py`：保留 `oneshot`，拆分公共协议
3. `automation/chat/` 新增：
   - `execution_adapter.py`
   - `execution_scheduler.py`
   - `persistent_runner.py`（可选）
   - `run_state_store.py`
   - `app_server_runner.py`
4. `automation/sql/031_chat_run_state.sql`：run 状态持久化表
5. `tests/` 新增并发、恢复、降级场景测试

## 14. 测试方案

1. 单元测试：
   - 适配器选择与回退
   - 调度器并发与排队
   - run 状态机迁移
2. 集成测试：
   - 高频 turn 下首 token 延迟
   - 多 session 并发稳定性
   - API 重启后的中断恢复回显
3. 压测：
   - 20/50/100 并发会话阶梯压测
   - 记录 P50/P95 延迟与错误率

## 15. 验收标准

1. 功能一致性：403/404 全量回归通过
2. 性能目标：
   - P50 首 token 延迟下降 >= 30%
   - P95 总耗时下降 >= 20%
3. 稳定性目标：
   - 并发场景错误率不高于基线
   - API 重启后 run 中断可感知、不可误执行
4. 可运维性：
   - 可观测指标齐全
   - 模式切换与回滚可在分钟级完成

## 16. 发布与回滚

### 16.1 发布步骤

1. 先上线抽象层（仍跑 `oneshot_exec`）
2. 灰度开启 `app_server`（10% -> 50% -> 100%）
3. 完成压测与观测验收后，将生产默认切到 `app_server`
4. 仅在必要时补充 `persistent_subprocess` 作为特定环境过渡能力

### 16.2 回滚策略

1. 任一阶段出现异常，立即切回 `CLAWCHECK_CHAT_EXEC_MODE=oneshot_exec`
2. 保留 run 状态表，不做破坏性回退
3. 关闭新 backend 后，业务链路继续沿用 403/404 原能力

## 17. 风险清单与控制

1. 风险：`app_server` 连接池耗尽或健康波动导致请求抖动
   - 控制：健康探针 + 限流/排队 + 熔断与自动降级到 `oneshot_exec`
2. 风险：并发排队导致用户体感变差
   - 控制：显式排队事件 + 超时提示 + 队列上限
3. 风险：app-server 不稳定导致大面积失败
   - 控制：自动降级到 `oneshot_exec`
4. 风险：升级影响 404 审批链路
   - 控制：审批路径专项回归与开关隔离

## 18. 结论

`405` 的解决方案是：保留短进程链路作为保底，分阶段升级到长生命周期执行后端，并以 `app_server` 作为起步落地与默认目标形态；`persistent_subprocess` 仅作为可选过渡补充。这样可以在不破坏 403/404 业务契约的前提下，优先获得执行层解耦与横向扩展能力，并系统性解决延迟、并发、高频 turn 与会话稳定性问题。

## 4. 自动化实现方案

### 4.1 凭据与登录态方案
新增正式环境凭据文件：
- `automation/config/credentials.prod.local.yaml`

建议格式：
- `auth.username`
- `auth.password`

实现要求：
- 凭据仅保存在本地文件，不提交到 Git；
- 自动化运行时优先读取正式环境凭据文件；
- 首次登录成功后保存正式环境登录态，例如：
  - `automation/state/auth_prod.json`
- 后续优先复用登录态，减少重复登录。

针对“可能出现多个密码框、且不是同一个框”的处理策略：
- 每次进入认证页时，扫描当前页面全部可见 `input[type='password']`；
- 对每个可见密码框都尝试填入密码；
- 若登录存在多阶段跳转，则每个阶段都重复上述逻辑；
- 若检测到仍停留在认证页，则继续下一轮登录处理；
- 若检测到错误提示（如认证失败），立即终止并记录截图。

### 4.6 下载等待方案
由于后台导出通常需要 `5~10 分钟`，不建议始终阻塞在前台页面轮询。

建议采用以下方案：
- Playwright 监听浏览器下载事件；
- 下载目录固定到项目本地目录，例如：
  - `automation/downloads/`
- 后台执行后，轮询下载目录：
  - 是否出现新的 `.xlsx` / `.xls` 文件
  - 文件大小是否稳定
  - 文件最后修改时间是否更新完成
- 建议最大等待时长：`15 分钟`
- 若超时未下载成功：
  - 记录失败日志
  - 保留截图
  - 不进行入库

### 4.7 文件清洗与表头处理方案
需求中已指出“表头不规范，需要做处理无用行”。结合实际导出类报表经验，建议按以下规则处理：

#### 识别真正表头
以首个包含核心列名的行作为正式表头，建议识别关键列：
- `公司`
- `公司ID`
- `部门`
- `部门ID`
- `人员编号`
- `姓名`
- `入职日期`

#### 删除无用行
在正式表头之前，删除以下无效内容：
- 空白行
- 报表标题行
- 查询条件说明行
- 导出时间说明行
- 合并单元格形成的说明头
- 重复表头行

#### 字段标准化
建议统一处理：
- 去除列名前后空格
- 去除换行符、制表符
- 将全角括号/特殊空格标准化
- 将 `#` 列删除或转为序号字段
- 对重复列名自动重命名或映射到标准字段名

#### 数据清洗
建议处理：
- 全空行删除
- 纯说明文本行删除
- 日期列转标准日期格式
- 编码列统一转字符串，避免前导零丢失
- 对 `是否关键岗位` 等枚举值做标准化

#### 字段落库原则
花名册导出文件中的业务字段，方案上应采用“全部物理列”方式落库：
- 导出表头中需要保留的业务列，均在正式表中建立对应物理字段；
- 不再使用 `JSON` / `JSONB` 字段承载多列业务数据；
- 不再将“公司/组织/岗位/任职”等多组字段合并后存入单个扩展字段；
- 如后续报表显示方案新增列，应同步补充正式表字段与导入映射；
- 原始下载文件保留在文件层做追溯，不再依赖表内 `raw_row_json` 一类字段兜底。

## 5. 数据库入库方案
目标表：`在职花名册表`

建议采用两段式入库：
1. 原始文件解析到 DataFrame / staging 临时表；
2. 清洗标准化后再写入正式表。

正式表设计原则：
- 花名册需要保留的列全部使用物理字段，不做多列合并；
- 正式表不使用 `JSON` / `JSONB` 保存花名册业务列；
- staging 临时处理阶段可保留中间结构，但正式入库时必须全部展开。

建议入库策略：
- 以“查询日期 + 人员编号 + 部门ID + 职位编码”等业务键做幂等控制；
- 如需保留历史快照，建议按“查询日期”分区或保留批次号；
- 保留源文件名、下载时间、导入批次号等审计字段；
- 原始追溯以下载文件归档为主，不在正式表内额外保存多列业务 JSON。

建议附加字段：
- `query_date`
- `import_batch_no`
- `source_file_name`
- `downloaded_at`
- `imported_at`

## 6. 异常处理要求
### 6.1 登录异常
- 认证失败：终止任务并截图
- 登录页结构变化：记录当前页面 DOM 文本快照
- 多次密码框变化：按“扫描所有可见密码框”重试

### 6.2 页面异常
- 左上菜单未展开：重试点击并截图
- `最近使用` 未出现：重试 2~3 次
- `在职人员花名册` 未出现：记录页面快照后失败退出

### 6.3 查询异常
- 查询 60 秒后仍处于 `加载中`：失败退出
- 工具栏 `引出` 未出现：视为查询未成功

### 6.4 导出异常
- `引出中` 弹窗未出现：重试点击 `引出`
- `转后台执行` 未出现：记录页面快照
- 15 分钟内未下载文件：记录失败并终止入库

### 6.5 入库异常
- 表头识别失败：将原文件转存到异常目录
- 字段缺失：写入异常日志并保留原始文件
- 数据库连接失败：不删除下载文件，等待人工补导

以下内容为本次从“方案确认”到“真实下载并入库”过程中，实际踩到的坑和对应处理方式总结，后续维护该流程时应重点关注。

### 10.1 环境与凭据坑
#### 坑点 1：本地已有凭据文件是 UAT，不是正式环境
现象：
- 仓库里原先已有本地凭据文件，但对应的是 UAT；
- 本次目标系统是正式环境 `hr.onewo.com`，账号密码完全不同。

处理方式：
- 单独新增正式环境配置与凭据文件；
- 正式环境登录态也单独保存，避免与 UAT 混用；
- 运行 `roster` 动作时默认切换到正式环境配置。

结论：
- UAT 与正式环境必须物理隔离配置，不能只靠命令行传参临时区分。

#### 坑点 2：统一认证页可能出现多个密码框，且不一定是同一个框
现象：
- 登录过程不是单页单密码框模型；
- 需求方已提前说明“会涉及多次输入密码框，且不是同一个框”。

处理方式：
- 登录逻辑改成“每轮扫描当前页面所有可见密码框并全部填值”；
- 如果仍停留在认证页，则继续下一轮；
- 不再依赖单一固定密码框选择器。

结论：
- 登录页自动化必须按“多阶段、多密码框”思路实现，不能写死一个 selector。

### 10.2 页面识别与导航坑
#### 坑点 3：需求文案与正式环境实际文案不一致
现象：
- 用户最初描述是 `在职花名册`；
- 正式环境实际菜单名是 `在职人员花名册`；
- 用户最初描述按钮是 `转后台运行`；
- 正式环境实际按钮是 `转后台执行`。

处理方式：
- 方案文档和代码均以实测文案为准；
- 自动化定位时增加兼容兜底，允许老文案与新文案并存；
- 但主流程日志、文档、选择器说明全部统一成实测名称。

结论：
- 此类 ERP 页面上，“差两个字”就足以导致定位失败，必须以实测页面为准。

#### 坑点 4：`报表显示方案`、`任职类型` 不是普通下拉框
现象：
- 初看像条件框，但实际是 F7 弹窗选值；
- 若按普通输入框或下拉框处理，会直接失败。

处理方式：
- 点击字段右侧 F7 图标；
- 在弹窗表格中选中目标行；
- 点击 `确定` 并回读字段值做校验。

结论：
- 该类金蝶页面的“条件字段”经常是弹窗型控件，不能从外观猜交互方式。

### 10.3 导出与下载链路坑
#### 坑点 5：点击 `转后台执行` 后，不会直接自动落地到本地下载目录
现象：
- 最初按常规 Playwright 下载处理思路，监听 `download` 事件；
- 但后台执行后，页面并不会立刻触发浏览器下载；
- 用户现场补充说明：不会自动落地，需要再点蓝色悬浮圈或浏览器下载入口。

处理方式：
- 放弃“点击后台执行后被动等下载”的单一方案；
- 改成“后台执行 -> 跟踪后台任务 -> 找到任务详情 -> 主动点下载文件”；
- 下载逻辑不再依赖页面自动触发。

结论：
- 此流程的核心难点不在查询，而在后台导出后的二次下载动作。

#### 坑点 6：右侧蓝色悬浮圈是页面内 DOM，不是普通文本按钮
现象：
- 页面右下出现一个蓝色圆形浮圈，显示类似 `3/4`、`3/5`；
- 它没有可读文本标签，不容易通过常规 `text=` 方式定位；
- 初期抓 DOM 时容易被表格右侧大量元素噪声淹没。

处理方式：
- 通过页面截图先确认浮圈真实位置；
- 再用 `elementsFromPoint` + 右侧区域筛选，识别 `数字/数字` 文案的小圆形元素；
- 统一改成“按右侧小尺寸浮圈特征点击”，而不是靠固定 class 名硬编码。

结论：
- 这种浮圈类控件必须结合截图和几何位置定位，单靠文本或类名都不稳定。

#### 坑点 7：展开浮圈后，列表里会混入历史任务，不能直接点第一个“已完成”
现象：
- 浮圈展开后会显示后台运行列表；
- 其中会有历史 `报表导出 - 在职人员花名册` 已完成任务；
- 如果直接点第一个已完成项，可能下载的是旧文件，不是本次任务。

处理方式：
- 在首次展开后台面板时，先记录“已存在的历史完成任务 ID”；
- 同时跟踪当前活跃导出任务 ID；
- 只在“当前任务完成”或“新出现的完成任务 ID”时才打开任务详情。

结论：
- 后台任务面板是一个任务池，不是只展示当前导出任务，必须做任务去重和关联。

#### 坑点 8：列表中的“已完成”项本身不是最终下载入口
现象：
- 展开浮圈后，列表里只能看到任务状态；
- 继续点击任务项右侧功能图标，才会打开 `报表导出文件下载` 弹窗；
- 最终还要在弹窗里点击 `下载文件` 才会真正触发下载。

处理方式：
- 下载流程改成三级动作：
  1. 点击右下浮圈；
  2. 点击任务项右侧功能图标；
  3. 在 `报表导出文件下载` 弹窗中点击 `下载文件`；
- 自动化中增加如下稳定定位：
  - 后台执行按钮：`#dialogShow [data-btn-key="btnok"]`
  - 任务详情下载按钮：`#dialogShow [data-page-id$="_btnexport"]`

结论：
- 本流程的真实下载入口不在导出进度弹窗，也不在后台列表项文本里，而在任务详情弹窗的 `下载文件`。

#### 坑点 9：同一下载动作可能触发多次 `download` 事件
现象：
- 在调试时，点击 `下载文件` 后曾看到同名下载事件出现 2 次；
- 如果代码不做收敛处理，可能误判或重复保存。

处理方式：
- 流程中只消费首个有效 `download` 对象并落盘；
- 文件名统一追加时间戳，避免同名覆盖；
- 日志中保留原始 `suggested_filename` 便于追查。

结论：
- 下载事件要按“至少一次”处理，而不是假定“恰好一次”。

### 10.4 文件解析与入库坑
#### 坑点 10：正式环境导出的 `xlsx` 文件维度信息错误，`openpyxl` 只读模式会误判成只有 `A1`
现象：
- 文件真实大小约 `99MB`，XML 中数据完整；
- 但工作表 `<dimension ref="A1"/>` 写错；
- 导致 `openpyxl` 只读模式下 `max_row=1`、`max_col=1`，解析器只能看到一行 `在职人员花名册`，进而报错：`Unable to identify roster header row from workbook`。

处理方式：
- 在 `xlsx` 读取逻辑中增加兜底：
  1. 先按只读模式读取；
  2. 若发现只有 `1x1`，则调用 `sheet.reset_dimensions()` 重新展开；
  3. 若仍异常，则回退到非只读模式再读取一次；
- 解析器修复后，已可正确识别：
  - 表头数 `98`
  - 数据行数 `154849`

结论：
- 生产导出文件不一定是“标准 Excel”，解析器必须兼容工作表元数据不规范的情况。

#### 坑点 11：表头与正文之间存在多行说明/噪声行
现象：
- 前几行不是正式表头，而是：
  - 报表标题行
  - 查询条件摘要行
  - 真正表头行
- 如果不先识别表头位置，直接按首行解析会全部错位。

处理方式：
- 通过候选表头关键字组合识别真正表头行；
- 跳过标题、摘要、重复表头、空行、说明行；
- 仅从正式表头下方开始抽取数据行。

结论：
- 这类报表导出不能按“第一行就是 header”假设处理。

### 10.5 运行策略上的经验
#### 坑点 12：真实全流程比“导出成功”更容易卡在“导出后下载”或“下载后解析”
现象：
- 查询本身很快完成；
- 真正难的是：
  - 后台任务下载链路复杂；
  - 正式文件格式又不完全规范；
- 若只验证到“出现引出中弹窗”，并不代表整体流程可用。

处理方式：
- 本次实施坚持做到“真实下载 + 真实解析 + 真实入库 + DB 行数核验”四步闭环；
- 最终确认数据库中 `2026-03-11` 共入库 `154849` 行。

结论：
- 此需求的验收标准必须是“数据入库成功”，不能只停留在页面点击成功。

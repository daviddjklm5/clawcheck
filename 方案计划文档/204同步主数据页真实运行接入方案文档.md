# 方案编号204：同步主数据页真实运行接入方案文档

## 1. 方案目标

在 `200` 可视化页面骨架基础上，把 `同步主数据` 页面从 mock 展示页升级为可真实运行的主数据工作台。

本方案明确交付以下结果：

- `同步主数据` 页面不再读取 `automation/api/mock_data.py` 中的示意数据
- 页面可真实触发 `在职花名册`、`组织列表`、`权限列表` 三类主数据任务
- 页面可展示当前运行任务、最近任务结果与最新主数据摘要
- 后端统一由 FastAPI 承接任务启动、状态输出与结果摘要
- 前端不直接执行命令行，不直接读取 PostgreSQL

本方案解决的是“主数据同步页如何正式接入真实运行链路”，不改既有业务字段口径，不重写已有 Playwright 流程。

## 2. 背景与问题

截至 `2026-03-16`，仓库中与 `同步主数据` 页相关的现状如下：

1. `webui/src/pages/MasterDataPage.tsx` 仍是骨架页  
   - 页面按钮没有接入真实执行  
   - 页面提示文案仍明确写着“当前为 UI 骨架版本，按钮与表格展示 mock 接口数据”

2. `GET /api/jobs/master-data` 仍返回 mock 数据  
   - `automation/api/routers/jobs.py` 直接调用 `automation/api/mock_data.py:get_master_data_dashboard()`
   - 页面中的统计卡、任务卡、最近任务表格都不是实时结果

3. 三条主数据链路本身已经存在  
   - `run.py roster`：下载在职花名册并写入 `在职花名册表`，随后刷新 `人员属性查询`
   - `run.py orglist`：下载组织快速维护清单并写入 `组织列表`，随后刷新 `组织属性查询`
   - `run.py rolecatalog`：承接权限主数据初始化入口

4. 当前缺的是“页面工作台层”，不是“主数据业务自动化本体”  
   - 没有主数据任务的统一摘要文件
   - 没有主数据页的真实统计查询
   - 没有页面轮询中的运行状态输出

5. `rolecatalog` 当前存在一个真实接入前必须先澄清的问题  
   - `run.py rolecatalog` 当前调用 `PostgresPermissionCatalogStore.seed_catalog()`
   - 但 `seed_catalog()` 在表结构已就绪时只返回统计，不会重新执行 `009_permission_catalog.sql` 中的 seed upsert
   - 这意味着如果直接把当前 `rolecatalog` 接到页面按钮，用户点击“初始化权限主数据”后，可能只是读取当前表统计，而不是真正刷新主数据

因此，`204` 不能只做“按钮接线”，而必须把主数据页正式升级为真实运行工作台，并补齐 `rolecatalog` 的真实刷新语义。

## 3. 与现有方案关系

### 3.1 与 `200` 的关系

`200` 解决的是页面壳、路由、MUI Data Grid 选型与整体信息架构。

`204` 的作用是把 `200` 中仍停留在 mock 阶段的 `同步主数据` 页推进到真实运行阶段，并保持页面职责仍然是“主数据同步工作台”。

### 3.2 与 `202` 的关系

`202` 已经验证了一条可复用的正式接线模式：

- 前端调用 FastAPI
- FastAPI 后台线程启动 `run.py`
- 子进程执行真实 runner
- 结果写入 JSON 摘要文件
- 页面轮询当前任务与最近任务

`204` 直接复用这套模式，不再单独发明第二套任务机制。

### 3.3 与 `003 / 004 / 009` 的关系

`204` 不改以下方案的业务口径，只承接其运行入口：

- `003在职花名册下载入库方案文档.md`
- `004组织快速维护列表下载入库方案文档.md`
- `009权限列表主数据建设方案文档.md`

也就是说：

- 花名册仍按既有下载、导入、刷新 `人员属性查询` 的链路执行
- 组织列表仍按既有下载、导入、刷新 `组织属性查询` 的链路执行
- 权限列表仍沿用 `009` 的主数据口径与中文字段体系

## 4. 设计原则

### 4.1 页面只做工作台，不重写业务流程

主数据页的职责是：

- 触发同步
- 查看当前运行状态
- 查看最近执行结果
- 查看最新主数据摘要

页面不直接处理 Excel，不直接连接数据库，不在浏览器中暴露账号口令。

### 4.2 三类任务共用一套任务工作台模型

虽然 `roster`、`orglist`、`rolecatalog` 的执行链路不同，但页面层统一抽象为：

- 一个启动入口
- 一个运行中任务
- 一组最近任务记录
- 一组当前主数据摘要卡片

避免主数据页内部再拆成三套彼此不兼容的页面逻辑。

### 4.3 任务状态与数据状态分离

页面需要同时展示两类信息：

1. 任务状态  
   - 当前是否正在运行
   - 最近一次是否成功
   - 错误摘要是什么

2. 数据状态  
   - 当前表里最新批次是什么
   - 当前表有多少行
   - 最近更新时间是什么

不能继续用 mock 卡片把二者混写。

### 4.4 真实运行优先，但不扩大边界

本方案优先把真实运行接入打通，但仍保持以下边界：

- 不新增通用任务中心
- 不实现任务取消
- 不处理多用户并发调度
- 不在页面支持本地文件上传
- 不修改主数据表中文命名规范

### 4.5 `rolecatalog` 必须具备真实刷新语义

`权限列表` 页面按钮不能停留在“读取统计”或“仅在新库首建时有效”的语义。

本方案要求：

- 明确拆分“确保表结构存在”和“执行 seed/upsert 刷新”
- `rolecatalog` 页面动作必须真实执行 seed/upsert
- 执行完成后返回最新总行数与按 `权限级别` 聚合统计

## 5. 范围与边界

### 5.1 本次纳入范围

- `同步主数据` 页改为真实工作台
- 新增主数据工作台查询 API
- 新增主数据任务启动 API
- 新增主数据任务摘要文件输出
- 新增 PostgreSQL 主数据摘要查询
- 修正 `rolecatalog` 的真实刷新语义

### 5.2 本次不纳入范围

- 不新增业务表
- 不新增中文业务字段
- 不修改 `在职花名册表`、`组织列表`、`权限列表` 的业务口径
- 不做页面内文件上传
- 不做运行中任务取消
- 不做多用户并发任务排队

### 5.3 数据库边界

本方案原则上：

- **不新增 SQL 迁移**
- **不新增中文业务表**
- **不新增业务字段**

主数据工作台所需摘要优先通过现有表实时查询与任务摘要文件组合得到。

## 6. 页面方案

### 6.1 页面结构

`同步主数据` 页统一为三层结构：

1. 页面说明与实时摘要卡片
2. 操作区
   - 选择任务类型
   - 配置可选参数
   - 启动同步
   - 手动刷新
   - 当前任务提示
3. 最近任务 Grid
   - 展示最近主数据任务执行结果

### 6.2 实时摘要卡片

顶部卡片不再展示 mock 文案，统一切到真实口径，建议至少包括：

1. `在职花名册`
   - 最新导入时间
   - 最新导入批次号
   - 最新行数
   - 是否已刷新 `人员属性查询`

2. `组织列表`
   - 最新导入时间
   - 最新导入批次号
   - 最新行数
   - 是否已刷新 `组织属性查询`

3. `权限列表`
   - 当前总行数
   - 最新更新时间
   - 当前权限级别分布摘要

4. `最近主数据任务`
   - 最近任务状态
   - 最近成功时间
   - 最近失败摘要

### 6.3 操作区能力

主数据页的真实执行入口统一收敛在操作区，不再使用静态卡片按钮直接展示命令字符串。

建议页面支持以下任务类型：

- `同步在职花名册`
- `同步组织列表`
- `刷新权限主数据`

### 6.4 页面参数口径

建议参数按任务适配显示，而不是所有任务共用一组无差别参数。

#### 6.4.1 通用参数

- `浏览器是否可见`
- `dry-run`
- `输入文件路径`

说明：

- `输入文件路径` 仅表示服务端可访问的本机路径，不是浏览器上传文件
- `dry-run` 仅用于不落库验证；若任务类型不支持则禁用

#### 6.4.2 花名册任务参数

- `报表方案名称`
- `任职类型`
- `查询超时秒数`
- `下载超时分钟数`
- `仅查询不导入`

#### 6.4.3 组织列表任务参数

- `查询超时秒数`
- `下载超时分钟数`
- `仅查询不导入`

#### 6.4.4 权限主数据任务参数

- 是否强制重种 seed 数据

说明：

- 建议默认即执行真实 seed/upsert
- 第一阶段不开放在线编辑权限级别字典

### 6.5 最近任务 Grid

最近任务列表建议字段：

- `任务ID`
- `任务类型`
- `状态`
- `触发时间`
- `开始时间`
- `完成时间`
- `导入批次号`
- `来源文件名`
- `写入行数`
- `日志文件`
- `摘要文件`
- `结果说明`

### 6.6 运行中交互规则

- 任一时刻仅允许一个主数据任务处于 `queued / running`
- 运行中页面继续允许刷新，但不允许再次启动新的主数据任务
- 任务运行期间页面自动轮询当前任务
- 任务结束后自动刷新摘要卡片与最近任务表格

## 7. 接口与数据结构方案

### 7.1 接口拆分

后端接口按职责拆分为两类：

1. `主数据工作台查询`
2. `主数据任务启动`

### 7.2 建议接口

- `GET /api/jobs/master-data`
  - 继续沿用当前路径，但返回真实工作台数据
- `POST /api/jobs/master-data/run`
  - 启动主数据任务

说明：

- `GET /api/jobs/master-data` 不再返回 `mock_data`
- 为减少前端改动，保留现有 `GET` 路径不变

### 7.3 `GET /api/jobs/master-data` 返回结构

建议统一返回：

- `stats`
  - 页面顶部实时摘要卡片
- `actions`
  - 页面可执行任务定义与默认参数
- `currentTask`
  - 当前运行中的主数据任务，没有则为 `null`
- `recentRuns`
  - 最近任务列表

不再返回当前 mock 结构中的静态 `command` 文案作为主要内容。

### 7.4 `POST /api/jobs/master-data/run` 入参建议

建议统一为：

- `taskType`
  - `roster | orglist | rolecatalog`
- `headed`
- `dryRun`
- `inputFile`
- `skipExport`
- `skipImport`
- `queryTimeoutSeconds`
- `downloadTimeoutMinutes`
- `scheme`
- `employmentType`
- `forceRefresh`

其中：

- `scheme`、`employmentType` 仅对 `roster` 有效
- `forceRefresh` 仅对 `rolecatalog` 有效
- 无效参数由后端忽略或校验拒绝，不应由前端自行拼命令

### 7.5 主数据任务摘要文件

为支撑页面展示最近任务，本方案要求每次任务完成后输出一份摘要文件，至少包含：

- `taskId`
- `taskType`
- `status`
- `requestedAt`
- `startedAt`
- `finishedAt`
- `headed`
- `dryRun`
- `inputFile`
- `skipExport`
- `skipImport`
- `queryTimeoutSeconds`
- `downloadTimeoutMinutes`
- `importBatchNo`
- `sourceFileName`
- `insertedCount`
- `tableName`
- `logFile`
- `summaryFile`
- `message`
- `outputTail`

对 `rolecatalog` 补充：

- `totalRows`
- `countsByPermissionLevel`

## 8. 后端实现要求

### 8.1 新增主数据工作台服务模块

建议新增：

- `automation/api/master_data_workbench.py`

职责包括：

- 汇总 PostgreSQL 主数据摘要
- 读取主数据任务摘要文件
- 管理主数据任务的内存态
- 启动后台线程执行真实 runner

实现方式直接复用 `collect_workbench.py` / `audit_workbench.py` 的模式。

### 8.2 任务执行方式

三类任务统一通过 `subprocess.run()` 调起 `automation/scripts/run.py`：

- `run.py roster`
- `run.py orglist`
- `run.py rolecatalog`

统一要求：

- FastAPI 只负责参数校验与任务编排
- 真实执行逻辑仍落在既有 CLI / flow / store 中
- 命令输出需截取 `outputTail` 供页面排障

### 8.3 主数据摘要查询

建议在 `automation/db/postgres.py` 中新增主数据工作台查询方法，例如：

- `fetch_master_data_workbench()`

至少聚合以下信息：

1. `在职花名册表`
   - 最新导入批次号
   - 最新导入时间
   - 最新查询日期
   - 当前总行数

2. `人员属性查询`
   - 当前总行数
   - 最近刷新时间

3. `组织列表`
   - 最新导入批次号
   - 最新导入时间
   - 当前总行数

4. `组织属性查询`
   - 当前总行数
   - 最近刷新时间

5. `权限列表`
   - 当前总行数
   - 最近更新时间
   - `权限级别` 分布

### 8.4 `rolecatalog` 刷新语义修正

本方案要求补齐一个明确改造点：

- 当前 `PostgresPermissionCatalogStore.seed_catalog()` 需要拆成：
  - `ensure_table()` 或 `ensure_schema_ready()`
  - `seed_catalog()` 或 `refresh_catalog()`

目标语义：

1. 表不存在或结构不完整时，先补齐结构
2. 每次执行 `rolecatalog` 时，真实执行 seed/upsert
3. 执行完成后返回最新统计摘要

否则页面接入后会出现“看似执行成功，实际上没有刷新权限主数据”的假闭环。

### 8.5 日志与摘要文件命名建议

建议主数据任务摘要统一命名为：

- `master_data_summary_{timestamp}_{taskId}.json`

若希望区分任务类型，也可采用：

- `master_data_roster_summary_{timestamp}_{taskId}.json`
- `master_data_orglist_summary_{timestamp}_{taskId}.json`
- `master_data_rolecatalog_summary_{timestamp}_{taskId}.json`

要求：

- 文件统一落在 `settings.runtime.logs_dir`
- 页面展示时统一转成 repo 相对路径

## 11. 风险与注意事项

### 11.1 FastAPI 进程重启后的运行态丢失

当前 `collect / audit` 也是以内存态保存运行中任务。

因此 `204` 延续同样边界：

- FastAPI 进程重启后，运行中任务内存态会丢失
- 但已落盘的摘要文件仍可用于展示最近完成任务

这属于当前阶段可接受限制，不单独在 `204` 内扩展为完整任务中心。

### 11.2 `inputFile` 不是上传能力

页面如支持输入文件路径，必须明确：

- 路径是 API 所在机器可访问的本地路径
- 前端不会直接把用户电脑文件上传到服务端

否则用户容易误解为浏览器上传文件能力。

### 11.3 `rolecatalog` 是本方案唯一需要先修语义再接页面的动作

`roster` 与 `orglist` 已具备真实执行和真实入库语义。

`rolecatalog` 若不先补齐“真实重种”语义，页面接线后会形成误导，因此必须纳入同一方案处理。

## 12. 实施顺序建议

建议按以下顺序落地：

1. 补齐 `rolecatalog` 的真实 seed/upsert 语义
2. 新增 `master_data_workbench.py` 与 `GET/POST /api/jobs/master-data*`
3. 在 `postgres.py` 中补齐主数据摘要查询
4. 改造 `MasterDataPage.tsx` 为真实工作台
5. 补齐单测与页面联调

## 13. 结论

`204` 的本质不是“再做一个主数据页”，而是把已经存在的三条主数据链路正式收口到统一工作台中。

其中：

- `roster` 与 `orglist` 主要是页面与 API 接线问题
- `rolecatalog` 同时包含一个必须先修正的真实刷新语义问题

完成 `204` 后，`同步主数据` 页将与 `202` 的 `采集单据` 页保持同一工程模式：

- 页面只调 API
- API 只做编排
- CLI / Flow / PostgreSQL 继续承接真实业务逻辑

这也是当前仓库最稳妥、最符合既有实现方向的落地路径。

# 方案编号220：定时采集单据实现方案文档

## 1. 背景

当前仓库已经具备以下基础能力：

1. `202 / 202A` 已将 `采集单据` 页面接入真实 `collect` 链路，并明确“后台采集默认无头执行”的口径。
2. `automation/scripts/run.py collect` 已是正式采集入口，负责登录、待办筛选、单据采集、落库与 dump 输出。
3. `automation/scripts/run_windows_task.ps1` 已对 `collect` 动作做了任务脚本封装。
4. `automation/scripts/windows_task_daemon.py` 已支持按分钟间隔执行 `collect` 任务，且 `automation/config/windows_task_daemon.example.json` 中已有 `collect` 任务样例。

但现状仍有 4 个缺口：

1. 现有 daemon 的间隔计算基于 `lastStartedAt`，不满足“下一批次从上一批次结束时间起算”。
2. `常规配置` 页当前仅做只读摘要展示，无法在页面内配置定时采集频率。
3. daemon 配置当前只在启动时读取一次，页面修改后不会自动生效。
4. 采集工作台入口与 daemon 入口之间没有统一的跨进程互斥，存在并发撞车风险。

因此需要形成 `220` 方案，在不重写采集业务的前提下，实现“定时采集单据”能力。

## 2. 目标

本方案目标如下：

1. 每隔固定 `X` 分钟自动执行一次 `采集单据`。
2. 下一批次开始时间按“上一批次实际采集结束时间 + X 分钟”计算。
3. `X` 在 `常规配置` 模块中配置，单位为分钟。
4. 自动采集默认固定为无头执行，不显示浏览器。
5. 复用当前正式 `collect` 采集能力，不另写一套采集逻辑。
6. 当 EHR 当前没有待办权限申请单据时，采集任务应快速结束，并以“正常空批次”口径展示，而不是误报失败。

## 3. 现状评估与复用结论

### 3.1 可直接复用的现有能力

1. `automation/scripts/run.py collect`
   - 已具备登录、进入待办、筛选 `单据=权限申请`、采集详情、写库与 dump 输出能力。
2. `automation/flows/permission_collect_flow.py`
   - 已是待办列表读取与单据详情采集的核心流程，不应在 `220` 中重写。
3. `automation/scripts/run_windows_task.ps1`
   - 已支持 `-Action collect -Headless -Limit 100` 这类正式任务调用方式。
4. `automation/scripts/windows_task_daemon.py`
   - 已具备本地守护、按任务配置循环调度、记录 `lastStartedAt / lastFinishedAt / lastExitCode` 的基础框架。

### 3.2 已确认的可复用“无待办快速结束”行为

当前 `run.py collect` 已包含如下逻辑：

1. 先读取待办列表并筛选 `单据=权限申请`。
2. 若没有目标单据，则记录日志：
   - `No permission application documents found in todo list; continuing with empty result`
3. 不进入逐单据采集循环。
4. 不执行 PostgreSQL 写入。
5. 仍以成功退出并输出空结果 dump。

因此，`220` 不需要重新设计“无待办快速结束”主逻辑，只需要：

1. 保留这条现有链路；
2. 补齐 daemon / 配置页的状态文案，让“空批次”被识别为正常结果。

### 3.3 不宜直接复用的部分

以下现有行为不能直接照搬：

1. `register_windows_task.ps1` 的固定重复触发
   - 它更适合“按时钟时间重复”，不适合“从上一批结束时间起算”。
2. `windows_task_daemon.py` 当前的 `lastStartedAt + intervalMinutes`
   - 与本次需求冲突，需要调整为“按完成时间计算”。
3. `RuntimeSettingsPage.tsx` 当前只读摘要页
   - 与“在常规配置模块中配置频率”的需求冲突，需要扩成可编辑配置页。

## 4. 方案设计

### 4.1 调度载体

本方案继续使用现有 `task daemon`，不新建第二套调度器。

调度职责拆分如下：

1. Windows Task Scheduler / 开机自启动
   - 仅负责拉起 `API` 与 `task daemon` 进程；
   - 不直接承担“每 X 分钟执行 collect”的业务频率语义。
2. `windows_task_daemon.py`
   - 负责真正的分钟级循环调度；
   - 负责读取 `collect` 定时配置；
   - 负责根据“上一批结束时间”判断是否到期。

这样做的原因是：

1. 仓库中已经存在 daemon 能力，复用成本最低。
2. 需求要求“从结束时间起算”，daemon 比计划任务重复触发更适合承接。
3. 该方式与 `300 / 301` 的 Windows 原生正式运行口径一致。

### 4.2 配置来源与常规配置模块

本方案将“定时采集单据”的单一事实来源收敛为：

- `automation/config/windows_task_daemon.local.json` 中 `name=collect` 的任务配置

原因：

1. 现有 daemon 已直接消费该文件；
2. 不需要再把同一套调度参数同时落到 `settings*.yaml` 与 daemon JSON，避免双写多源；
3. `常规配置` 页只作为管理入口，不再引入第二份调度配置。

`常规配置` 页新增一个卡片区：`定时采集单据`，至少包含以下字段：

1. `启用定时采集`
2. `采集频率（分钟）`
3. `执行模式`
   - 固定展示 `无头浏览器`
4. `最近开始时间`
5. `最近结束时间`
6. `下一次计划时间`
7. `最近一次结果`

页面保存后由后端写回 `windows_task_daemon.local.json` 中 `collect` 任务的以下字段：

1. `enabled`
2. `intervalMinutes`

`collect` 任务的 `args` 维持为正式后台口径，例如：

```json
{
  "name": "collect",
  "enabled": true,
  "script": "automation/scripts/run_windows_task.ps1",
  "args": ["-Action", "collect", "-Headless", "-Limit", "100"],
  "intervalMinutes": 15,
  "dailyTimes": [],
  "runOnStartup": false
}
```

本次只把“是否启用 + 频率分钟数”纳入页面编辑范围；`Limit` 等参数继续沿用现有 daemon 配置，不在 `220` 中扩成新的 UI 配置项。

### 4.3 配置热生效

当前 `windows_task_daemon.py` 在启动时只读取一次配置文件。  
这与页面在线修改配置的目标不匹配。

因此本方案要求：

1. daemon 每个轮询周期重新读取 `windows_task_daemon.local.json`
2. 配置文件变化后，无需手工重启 daemon
3. 页面保存后，新的 `enabled / intervalMinutes` 最迟在下一次 daemon 轮询时生效

说明：

1. daemon 配置文件体积很小，按轮询重载即可，不需要做复杂的 watch 机制。
2. 实际生效误差受 `pollSeconds` 影响，默认不超过一个轮询周期。

### 4.4 结束时间起算口径

`collect` 的到期判断改为：

```text
最近一次实际完成时间 + intervalMinutes <= now
```

具体规则如下：

1. 若 `collect` 当前正在运行，则不启动下一批次。
2. 若 `collect` 当前未运行，且 `lastFinishedAt + intervalMinutes <= now`，则启动新批次。
3. 首次启用定时采集、且历史上没有完成记录时：
   - 以“保存配置成功时间”作为基准；
   - 第一批次在 `保存时间 + intervalMinutes` 后触发。
4. 若用户在采集运行中修改频率：
   - 新频率不打断当前批次；
   - 下一批次从“当前批次实际结束时间”开始按新频率起算。

### 4.5 共享运行状态

为了保证“人工采集”和“定时采集”对同一套节奏认知一致，本方案要求引入统一的 `collect` 运行状态写入点。

建议继续复用现有：

- `automation/state/windows_task_daemon_state.json`

其中 `tasks.collect` 至少维护以下字段：

1. `lastStartedAt`
2. `lastFinishedAt`
3. `lastExitCode`
4. `lastMessage`
5. `lastLogPath`

关键要求：

1. daemon 启动的 `collect` 任务会更新这组状态。
2. 采集工作台触发的人工 `collect` 任务也必须更新同一组状态。
3. daemon 的“是否到期”判断以这组共享状态中的 `lastFinishedAt` 为准。
4. 若 `collect` 子进程已退出，但 `run.py collect` 未能完整写回结束状态：
   - daemon 必须在检测到子进程退出后的下一轮轮询中补写 `lastFinishedAt / lastExitCode / lastMessage`
   - 页面展示与下一次计划时间计算应基于补写后的状态继续推进，不能长期停留在“采集任务运行中”

这样可以避免以下错误场景：

1. 用户手工触发了一次采集；
2. daemon 不知道这次人工采集已经在跑或刚刚结束；
3. 于是错误地按旧时间再次拉起一批采集。

### 4.6 执行链路

定时采集执行链路如下：

1. daemon 读取 `collect` 配置
2. 到期后执行：
   - `powershell.exe -File automation/scripts/run_windows_task.ps1 -Action collect -Headless -Limit 100`
3. `run_windows_task.ps1` 继续复用现有 runner 封装
4. 最终落到 `automation/scripts/run.py collect`
5. `run.py collect` 继续复用现有 `PermissionCollectFlow`

自动采集的浏览器口径固定为：

1. 强制 `-Headless`
2. 不受 `browser.headed` 默认配置影响
3. 不受 `采集单据` 页面手工“可见浏览器”开关影响

手工采集能力保持不变：

1. 页面仍可按当前能力手工勾选 `可见浏览器`
2. 但该开关仅影响手工触发，不影响定时任务

### 4.7 跨进程互斥

当前 API 工作台入口与 daemon 入口之间只有进程内锁，没有跨进程互斥。  
这会导致以下风险：

1. 用户在页面点击“启动采集”；
2. daemon 同时判断到期并拉起 `collect`；
3. 两条 `collect` 链路同时打开浏览器、同时读写同一批待办。

因此本方案要求新增统一的 `collect` 跨进程锁。

建议做法：

1. 在 `run.py collect` 入口增加全局锁文件，例如：
   - `automation/state/collect_task.lock`
2. 任何来源触发 `collect` 前，都必须先尝试获取该锁
3. 锁已被占用时：
   - 工作台手工触发：直接返回冲突提示
   - daemon 定时触发：记录“本轮跳过，原因=已有采集运行中”，不把它视为失败批次

这样可保证：

1. 同一时刻只会有一条真实 `collect` 在跑
2. daemon 不会与人工采集撞车
3. 共享 `lastFinishedAt` 仍以“真正完成的一批采集”为准

### 4.8 无待办快速结束口径

本方案明确复用当前 `run.py collect` 的空批次处理能力，不重写主逻辑。

需要补齐的是“状态展示口径”：

1. 若本轮未命中任何 `权限申请` 待办：
   - 任务状态记为成功结束
   - 最近结果展示为：`本轮无待办，已快速结束`
2. 不把这类批次展示为失败
3. 不触发额外告警
4. 下一次计划时间仍按“本轮结束时间 + X 分钟”继续推进

说明：

1. “无待办快速结束”不代表跳过 EHR 登录与待办列表检查；
2. 它指的是在完成待办探测后，不进入逐单据详情采集与写库阶段。

## 7. 接口建议

### 7.1 查询接口

继续保留：

- `GET /api/settings/runtime`

但返回结构需补充 `collectSchedule`，例如：

```json
{
  "collectSchedule": {
    "enabled": true,
    "intervalMinutes": 15,
    "mode": "headless",
    "lastStartedAt": "2026-03-23 10:00:00",
    "lastFinishedAt": "2026-03-23 10:03:12",
    "nextPlannedAt": "2026-03-23 10:18:12",
    "lastExitCode": 0,
    "lastMessage": "本轮无待办，已快速结束"
  }
}
```

### 7.2 保存接口

建议新增：

- `PUT /api/settings/runtime/collect-schedule`

请求体建议如下：

```json
{
  "enabled": true,
  "intervalMinutes": 15
}
```

校验规则：

1. `enabled=true` 时，`intervalMinutes` 必须为正整数
2. 单位固定为分钟
3. 不允许页面写入 `headed=true`

保存动作需要同步完成：

1. 写回 `windows_task_daemon.local.json`
2. 必要时重置 `collect` 的调度基准时间
3. 返回最新调度摘要

2. `collect` 由“开始时间起算”改为“结束时间起算”的到期判断测试
3. 首次启用后按“保存时间 + X 分钟”触发的测试
4. 运行中修改频率后按“本轮结束时间 + 新频率”触发的测试
5. 跨进程锁冲突测试
6. “无待办快速结束”状态文案测试
7. 设置接口保存与读取回显测试

### 8.2 验收标准

满足以下条件可认为 `220` 完成：

1. 用户可在 `常规配置` 页启用/关闭定时采集，并设置分钟频率。
2. 保存后无需手工重启 daemon，配置可自动生效。
3. 自动采集固定无头执行，不显示浏览器。
4. 若上一批次结束于 `10:03:12`，频率为 `15` 分钟，则下一批次最早开始时间为 `10:18:12`。
5. 当 EHR 中没有待办权限申请单据时，本轮可快速结束，并在页面中显示“本轮无待办，已快速结束”。
6. 人工采集与定时采集不会并发执行同一条 `collect` 链路。
7. 定时采集仍复用当前 `run.py collect`，没有出现第二套采集业务实现。

## 9. 风险与注意事项

### 9.1 daemon 未运行

即使页面已保存定时配置，若 `task daemon` 进程未运行，定时采集也不会真正执行。  
因此实现时建议在 `常规配置` 页明确展示 daemon 运行状态或给出提醒文案。

### 9.2 轮询精度

daemon 属于轮询调度，不是秒级实时调度。  
实际开始时间允许存在一个 `pollSeconds` 量级的误差，但不应破坏“按上一批结束时间顺延”的主口径。

### 9.3 配置单一事实来源

`220` 落地后，不应再在 `settings*.yaml` 中新增第二份 `collect interval` 配置；  
定时采集频率统一由 daemon 配置承载，页面只做管理入口。

## 10. 结论

`220` 的核心不是重写采集，而是把现有 `collect + run_windows_task + windows_task_daemon` 串成一条正式的“可配置、可热生效、按结束时间顺延”的后台采集链路。  
本方案落地后，可以在不改数据库结构、不改采集口径的前提下，为项目补齐稳定的定时采集能力。

# 方案编号005C：组织流程层级分类口径拆分方案文档

## 1. 方案目标
将 `005组织属性查询表设计方案文档` 中原本关于 `process_level_category` 的字段定义、分类规则与查询使用建议独立拆分为单独方案，形成可单独维护、单独引用的规则文档。

本方案目标：
- 明确 `process_level_category` 的业务含义与适用范围
- 固化当前“公司优先覆盖 + 特殊路径优先 + 基于 process_level_name_resolved 映射”的统一分类口径
- 为 `005A` 等后续增强方案提供稳定的分类规则来源
- 降低 `005` 主方案中组织流程相关规则的耦合度，便于后续独立迭代

## 2. 背景与拆分原因

### 2.1 当前现状
在 `组织属性查询` 中，`process_level_category` 已作为“组织流程层级分类”的统一结果列落地，并被以下场景复用：
- 组织流程类分组统计
- 高流量查询中的流程口径筛选
- `005A` 中组织授权级别规则的辅助判断

当前相关实现文件：
- `方案计划文档/005组织属性查询表设计方案文档`
- `方案计划文档/005B组织流程层级判断口径拆分方案文档`
- `automation/sql/006_organization_attribute_query.sql`

### 2.2 拆分原因
继续将 `process_level_category` 详细规则保留在 `005` 主方案中，会产生以下问题：
1. `005` 同时承载结果表总体设计、组织单位、战区、流程判断、流程分类等多类规则，文档过重
2. 后续若仅调整“组织流程层级分类”规则，容易连带修改 `005` 全文，维护边界不清
3. `005A` 等后续方案引用该字段时，缺少单点、稳定的规则来源

因此，本方案判断：
- 将 `process_level_category` 相关规则整体拆出为 `005C`
- `005` 继续保留结果表总体方案
- 所有涉及“组织流程层级分类”详细规则的后续方案，统一引用 `005C`

## 3. 适用范围
本方案仅覆盖以下内容：
- 字段：`process_level_category`
- 中文含义：组织流程层级分类
- 数据来源：`组织列表` 路径、`company_name` 与 `process_level_name_resolved`
- 落地点：`组织属性查询`

本方案不覆盖：
- `process_level_name_resolved` 的生成规则
- `org_unit_name`、`war_zone`、`wanyu_city_sales_department` 等其他派生字段规则

## 4. 字段定义

### 4.1 字段定义
新增派生字段：
- `process_level_category TEXT`

业务含义：
- 作为“组织流程层级分类”统一口径列
- 用于承接基于组织路径、公司名称及 `process_level_name_resolved` 的分类结果
- 用于面向业务查询、筛选、分组统计输出更稳定的流程分类值

### 4.2 与 005B 的关系
本字段依赖：
- `process_level_name_resolved`

该输入字段的详细规则统一见：
- `方案计划文档/005B组织流程层级判断口径拆分方案文档`

## 5. 分类规则

### 5.1 总体优先级
按以下优先级取值：

1. 公司优先覆盖
2. 特殊路径归类为“BG人行中心与学社”
3. 特殊路径归类为“业务单元本部”
4. 判定为“属地服务站”
5. 判定为“战区人行部门”
6. `process_level_name_resolved` 为空返回 `NULL`
7. 判定为“万物云本部”
8. 判定为“业务单元本部”
9. 其他情况归为“属地组织”

### 5.2 公司优先覆盖
- 当 `company_name = 人事远程交付中心` 时：
  - `process_level_category = 人事远程交付中心`
- 该规则高于 `process_level_name_resolved` 为空及其他分类规则

### 5.3 判定为“BG人行中心与学社”
- 当 `org_full_name` 包含 `BG人力资源行政服务中心` 或包含 `知之学社` 时：
  - `process_level_category = BG人行中心与学社`
- 该规则高于“属地服务站”“战区人行部门”及按 `process_level_name_resolved` 的归类规则

### 5.4 特殊路径归类为“业务单元本部”
- 当 `L2 = 蝶城发展中心` 且 `L3 = 人力资源与行政服务部`，并且 `L4` 为空或不等于 `研选家人力资源组` 时：
  - `process_level_category = 业务单元本部`
- 该规则用于承接原先被 `蝶发人行部` 表达的业务分类语义，但不再改写 `process_level_name_resolved`

### 5.5 判定为“属地服务站”
- 当 `L3 = 人力资源与行政服务中心` 且 `L4` 有值，且 `L4 <> BG人力资源行政服务中心`，并且组织路径最后 `7` 个字命中 `人事交付服务组` 时：
  - `process_level_category = 属地服务站`
- 该规则优先于“战区人行部门”规则

### 5.6 判定为“战区人行部门”
- 当 `L3 = 人力资源与行政服务中心` 且 `L4` 有值，且 `L4 <> BG人力资源行政服务中心` 时：
  - `process_level_category = 战区人行部门`

### 5.7 `process_level_name_resolved` 为空
- 当 `process_level_name_resolved IS NULL` 时：
  - `process_level_category = NULL`

### 5.8 判定为“万物云本部”
- 当 `process_level_name_resolved` 命中以下值时：
  - `万物云本部`
  - `本部部门`
  - `本部二级部门`
  - `本部三级部门`
  - `万科物业`
- 则：
  - `process_level_category = 万物云本部`

### 5.9 判定为“业务单元本部”
- 当 `process_level_name_resolved` 命中以下值时：
  - `本部`
  - `修缮业务本部`
  - `朴邻本部`
  - `万御电梯本部`
  - `福讯信息`
- 则：
  - `process_level_category = 业务单元本部`

### 5.10 其他情况
- 除上述情况外：
  - `process_level_category = 属地组织`

## 6. 查询使用建议
后续查询口径建议如下：
- 保留 `process_level_name` 作为源数据原值
- 业务查询、筛选、导出优先使用 `process_level_name_resolved`
- 若需按组织流程口径分组统计，优先使用 `process_level_category`

## 7. 与其他方案关系
- `005组织属性查询表设计方案文档`：保留结果表总体设计；`process_level_category` 详细规则统一引用本方案
- `005B组织流程层级判断口径拆分方案文档`：定义 `process_level_name_resolved` 的输入口径，本方案基于该字段继续分类
- `005A组织属性查询表增强方案文档`：涉及 `process_level_category` 时，应以本方案作为基础口径来源

## 8. 最终结论
本方案最终建议如下：

1. 将 `process_level_category` 的详细口径从 `005` 主方案独立拆分到 `005C`
2. `process_level_category` 继续作为 `组织属性查询` 的正式派生字段保留
3. 字段规则统一采用“公司优先覆盖 + 特殊路径优先 + 基于 process_level_name_resolved 映射”
4. 具体实现继续落在 `automation/sql/006_organization_attribute_query.sql`
5. `005`、`005A` 等后续方案引用该字段时，统一以本方案为准

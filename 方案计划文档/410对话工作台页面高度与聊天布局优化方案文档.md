# 410 对话工作台页面高度与聊天布局优化方案文档

## 1. 背景

`http://127.0.0.1:5173/chat-workspace` 当前采用左右两栏布局：

- 左侧为对话记录列表
- 右侧为当前会话聊天区

现状中，左右两栏处于同一个 `grid` 行高体系下，导致左侧会话记录条目持续增多时，会把整行高度一起拉高。结果是：

- 右侧聊天区会被动拉成异常高的长面板
- 聊天消息区不像常规对话框，而更像一整块被撑长的内容页
- 输入框会被推到过深位置，影响连续对话效率

本次优化按方案编号 `410` 收口，仅处理前端页面布局与交互表现，不改动后端对话协议。

## 2. 目标

本次优化目标如下：

1. 右侧聊天区高度应以“工作区高度”控制，而不是被左侧对话记录数量驱动。
2. 左侧对话记录应独立滚动，不再继续拉高右侧聊天面板。
3. 右侧页面应更符合聊天框设计：
   - 顶部信息区固定
   - 中间消息区独立滚动
   - 底部输入区固定在面板底部
4. 新消息持续流式追加时，消息区默认保持贴底；用户手动上滑查看历史后，不强制抢回滚动位置。
5. 页面首屏不应被健康检查接口阻塞，会话列表与聊天区应优先可用。

## 3. 实现范围

本次仅涉及：

- `webui/src/pages/ChatWorkspacePage.tsx`

本次不涉及：

- 后端 SSE 协议变更
- `chatApi` 接口契约变更
- 路由改造
- 数据库、采集、审批链路改动

## 4. 方案设计

### 4.1 工作区高度改为视口驱动

对话主区域改为桌面端固定工作区高度，使用类似：

- `clamp(620px, calc(100vh - 290px), 820px)`

作为聊天工作区高度边界，保证：

- 低分辨率下仍有可用最小高度
- 常规桌面分辨率下高度随视口变化
- 超高屏下不会无限拉长聊天面板

### 4.2 左右两栏各自独立滚动

左侧会话栏与右侧聊天栏均设置为：

- 卡片容器固定高度
- 内部列表区域 `overflowY: auto`

这样左侧记录增多时，只会在左栏内部滚动，不再把右侧聊天区整体撑高。

### 4.3 右侧聊天面板改为三段式

右侧聊天面板拆为三段：

1. 顶部信息头
   - 当前会话标题
   - 工作目录
   - 运行状态与消息数摘要
2. 中部消息区
   - 独立滚动
   - 用户消息靠右、助手消息靠左
   - 使用更接近聊天气泡的视觉样式
3. 底部输入区
   - 输入框与发送按钮固定在面板底部
   - 不因消息数量增长而被继续推远

### 4.4 消息贴底策略

新增消息区滚动状态判断：

- 若用户当前仍处在接近底部的位置，则新消息到达后自动贴底
- 若用户主动向上滚动查看历史，则暂停自动贴底

该策略适用于：

- 首次进入会话
- 新建会话
- 正常发送消息
- SSE token 持续追加

### 4.5 健康检查改为后台加载

`chat-workspace` 首屏原先同时等待：

- `sessions`
- `config-summary`
- `health`

但 `health` 在 `app_server` 模式下会探测远端模型服务，接口耗时明显高于会话列表和配置摘要，容易让聊天区长时间停留在“加载中”。

因此本次口径调整为：

1. 首屏仅等待 `sessions + config-summary`
2. `health` 改为页面挂载后的后台异步请求
3. 健康状态未返回前，状态栏显示 `Checking`

这样可以保证：

- 页面框架先出现
- 会话列表与消息区先可用
- 健康信息作为补充状态延后刷新

## 5. 代码落点

核心修改文件：

- `webui/src/pages/ChatWorkspacePage.tsx`

主要改动点：

1. 增加桌面端工作区高度常量
2. 为消息区增加滚动容器引用与“是否贴底”状态引用
3. 重构左右栏卡片样式，使其固定在统一工作区内
4. 将右侧聊天区改为“头部 + 消息区 + 输入区”结构
5. 调整消息样式，强化聊天窗口视觉层次

## 6. 验收标准

完成后需满足以下验收：

1. 左侧存在大量对话记录时，右侧聊天面板高度不再继续被拉高。
2. 桌面端右侧聊天区始终保持稳定、合理的聊天框高度。
3. 左侧对话记录可独立滚动。
4. 右侧消息区可独立滚动，输入区固定在底部。
5. 新消息流式返回时，在用户未主动离开底部的前提下，消息区持续贴底。

## 7. 发布影响

本次为纯前端展示层优化：

- 不改数据库
- 不改接口
- 不影响既有审批或查询能力

发布方式沿用常规 `webui` 前端构建与替换流程即可。

# 方案编号107：处理单据模块耗时长问题排查与审批时序优化方案文档

## 1. 背景

`处理单据` 模块在现场使用中出现“点击批准后，浏览器页面打开了，但在页面里停留很久”的体感问题。

本次排查明确只关注 `处理单据` 模块，不讨论 `采集单据` 模块的耗时。

本次以真实审批单据 `RA-20260317-00020103` 作为样本，目标是：

1. 记录 `处理单据` 模块关键步骤耗时
2. 判断耗时长的主要原因
3. 给出可执行的优化建议

## 2. 排查范围

本次排查范围限定为以下链路：

1. `处理单据` 列表加载
2. `处理单据` 详情抽屉加载
3. `审批单据 -> 批准` 触发后的后端审批执行链路
4. 浏览器打开后到审批完成返回的页面停留耗时

不纳入本次范围：

1. `采集单据` 页面任务执行耗时
2. 增量 `audit` 耗时
3. 主数据同步耗时

## 6. 关键结论

### 6.1 这次“处理单据耗时长”主要不是数据库查询慢

从接口结果看：

- 列表接口约 `0.1s`
- 详情接口约 `2.0s`

虽然详情接口仍有优化空间，但与 `61.8s` 的真实审批总耗时相比，不是主要矛盾。

### 6.2 主要耗时集中在浏览器页面内的两个 `20s` 级等待

本次真实审批中，浏览器页面内最重的两段耗时分别是：

1. `打开待办并等待列表就绪 = 22.21s`
2. `提交后回待办复查 = 20.20s`

二者合计：

- `42.41s`
- 占总耗时 `68.6%`

### 6.3 再加上一个固定 `8s` 的提交后确认窗口，三段合计已接近全链路耗时的大头

额外固定等待：

- `点击提交后首轮确认窗口 = 8.13s`

三段合计：

- `50.54s`
- 占总耗时 `81.8%`

结论：

- 当前“浏览器页面停留很久”的核心原因，确实在浏览器页面内部等待逻辑，而不是前端首包或数据库查询

## 7. 根因分析

### 7.1 根因一：审批流复用了采集流的待办列表就绪逻辑，并强依赖 `1000条/页`

当前审批流在以下两个位置都会进入：

- 打开待办列表准备找单据
- 提交后回到待办列表复查单据是否消失

其内部复用了：

- `automation/flows/permission_collect_flow.py`
- `_wait_for_todo_list_ready()`

而该逻辑会先执行：

- `_ensure_todo_page_size(1000)`

问题在于：

1. 它会等待分页控件 `1000条/页` 可见
2. 若控件不可见或渲染不稳定，会一直等到全局超时
3. 当前超时时间是 `timeout_ms = 20000`
4. 超时后虽然只是 `warning` 并继续执行，但 `20s` 已经被消耗掉

现场日志已直接出现以下证据：

- `Todo page size selector not ready, continue with current pagination`
- `Timeout 20000ms exceeded`

也就是说：

- 这不是功能性失败
- 而是“明知可以继续执行，但在继续前先白等了 `20s`”

### 7.2 根因二：提交后确认逻辑先固定等待约 `8s` 再进入下一步

当前提交后确认逻辑中，第一阶段会先做一个短窗口反馈监听。

在本次样本中：

- 页面没有立即命中强成功 toast
- 但系统仍先消耗了约 `8.13s`
- 然后才进入后续待办复查

这会带来一个问题：

- 对于“提交后直接回到待办列表”的页面行为，`8s` 等待大概率是无效等待

### 7.3 根因三：审批成功确认阶段再次触发待办列表就绪逻辑，导致第二次 `20s` 级白等

提交后，本次审批的成功确认方式是：

- `todo_disappeared`

也就是通过“回到待办列表后复查目标单据已消失”来确认成功。

但当前待办复查前，又会再次进入待办列表就绪逻辑，因此再次触发：

- 分页控件可见性等待
- `20s` 级白等

这就是第二个 `20s` 大头来源。

### 7.4 次级原因：审批链路每次都新开浏览器和首页

本次中以下固定成本较稳定：

- 浏览器启动并创建上下文：`1.94s`
- 打开首页并等待首页就绪：`2.61s`

两者合计约：

- `4.55s`

这不是主因，但它属于每张单据都要重复支付的固定成本。

### 7.5 次级原因：详情接口一次性返回全部页签数据

`处理单据` 详情接口当前一次性返回：

- 概览
- 风险明细
- 审批记录
- 角色
- 目标组织
- 备注提示

对于本次样本单据，单是：

- `orgScopes = 1120`

就使详情接口达到：

- 约 `290KB`
- 约 `2s`

这会拉高“打开抽屉”的体感，但不是本次审批 `61.8s` 的主因。

### 7.6 与本次问题相关的其他固定等待/超时行为

除本次已经量化出的两段 `20s` 和一段 `8s` 外，当前 `处理单据 -> 审批单据` 链路还存在以下相关等待行为。

这些行为并不一定都在本次样本中成为主瓶颈，但它们与“浏览器页面停留时间长”的体感属于同一类问题，需要一并评估。

#### 7.6.1 全局元素等待超时：`20s`

来源：

- `automation/config/settings.prod.yaml`
- `browser.timeout_ms = 20000`

它会影响审批链路中大量 `wait_for(..., timeout=self.timeout_ms)` 行为，例如：

- 等待 `待办任务` 入口可见
- 等待 `任务处理 / 审批记录` 页签可见
- 等待 `审批意见`、`审批决策` 字段可见
- 等待待办目标行、目标链接、Grid 头部就绪

引入原因：

- EHR 页面存在异步渲染、抽屉重绘、虚拟滚动和偶发卡顿
- 历史上如果等待时间过短，容易把“只是慢”误判为“页面失效”

现状判断：

- 这个 `20s` 作为全局容错上限可以理解
- 但不应直接拿来做审批链路中的“常规路径等待”
- 本次问题里真正触发的是分页控件等待，但同类隐患并不只这一处

是否可优化：

- 可以
- 方式不是简单删除全局 `20s`
- 而是将审批链路改为“场景化短等待 + 失败后再升级到长等待”

#### 7.6.2 提交后即时反馈窗口：`0 ~ 8s`

来源：

- `automation/flows/document_approval_flow.py`
- `feedback_deadline = min(total_timeout_ms, 8000)`

引入原因：

- `105C` 为避免“EHR 实际成功但系统误判失败”，专门增加了第一阶段的 toast / modal 监听窗口
- 目的是优先抓住最直接的成功/失败提示

现状判断：

- 设计目的合理
- 但当前实现是串行等待
- 如果页面已经直接切回待办列表，这 `8s` 就会退化成空等

是否可优化：

- 可以
- 建议改成：
  - 页面结构判断与 toast 监听并行
  - 一旦确认已回待办，立即进入 `todo_disappeared` 复查

#### 7.6.3 审批记录确认窗口：`8 ~ 25s`

来源：

- `automation/flows/document_approval_flow.py`
- `approval_record_deadline = min(total_timeout_ms, 25000)`

引入原因：

- `105C` 认为部分场景不会立即出现 toast
- 但审批记录页可能在稍后刷新出最新“同意”动作

现状判断：

- 这个窗口属于“成功确认兜底”
- 它不是本次样本的主要耗时来源
- 但在其他单据上，如果页面没有直接切回待办，而系统又持续尝试抓审批记录，就可能成为新的长等待来源

是否可优化：

- 可以
- 建议把它从串行大窗口改为：
  - 更短的审批记录快速探测
  - 若页面已切离详情区，则直接放弃审批记录路径

#### 7.6.4 总确认窗口：`25 ~ 45s`

来源：

- `automation/flows/document_approval_flow.py`
- `wait_timeout_ms = max(self.timeout_ms, 45000)`

引入原因：

- `105C` 需要保留“已提交待确认”这类中间状态
- 因此不能在较短时间内就把请求直接判定为失败

现状判断：

- `45s` 的目标是避免误报失败，不是追求性能
- 它本身是一个安全兜底，不是本次 `61.8s` 的直接根因
- 但如果前面几个阶段仍然串行空等，那么这个 `45s` 上限会继续放大最坏体感

是否可优化：

- 可以，但应在前序确认路径优化后再调
- 不建议在现阶段直接把 `45s` 砍短
- 否则可能重新引入 `105C` 当时要避免的“成功误判失败”

#### 7.6.5 前端审批接口超时：`120s`

来源：

- `webui/src/services/api.ts`
- `timeoutMs = 120000`

引入原因：

- `105A` 为解决“前端一直显示批准中/验证中，没有退出”问题而加入
- 它的作用是前端兜底，不是后端性能策略

现状判断：

- 这个 `120s` 不是当前页面停留 `61.8s` 的主因
- 它只是避免页面无限等待

是否可优化：

- 当前不建议优先动
- 在后端耗时压缩前，贸然缩短只会更容易把长链路打断

#### 7.6.6 多处固定 sleep：`600ms / 500ms / 300ms / 200ms / 800ms / 3000ms`

典型位置：

- 切换 `任务处理 / 审批记录` 页签后等待 `600ms`
- 下拉选择审批决策后等待 `500ms`
- 写入审批意见后等待 `300ms`
- 若审批记录页暂不可见，轮询间隔 `800ms`
- 第二次待办复查前固定再等 `3000ms`

引入原因：

- EHR 页面经常在点击后发生延迟重绘
- 历史上完全依赖事件状态时，容易在 DOM 尚未稳定时过早读取旧值

现状判断：

- 单次 sleep 体量不大
- 但会造成“系统像是在发呆”的体感
- 其中 `3000ms` 二次复查前等待尤其明显

是否可优化：

- 可以
- 建议逐步改为：
  - 等待具体元素状态变化
  - 等待旧节点消失 / 新节点出现
  - 等待文案变化或表单值确认完成

## 8. 优化建议

### 8.1 第一优先级：审批链路不要再为 `1000条/页` 阻塞 `20s`

建议：

1. 将审批流与采集流的待办就绪逻辑分离
2. 审批流不要强制执行 `_ensure_todo_page_size(1000)`
3. 若仍需保留分页控件探测，只允许：
   - 最多等待 `1s ~ 2s`
   - 超时立即降级继续

原因：

- 审批是单据级动作，不是全量采集
- 它的目标是“找到当前这张单据并完成提交”，不需要为“大页容量”付出 `20s` 级等待

预期收益：

- 可直接消除两段 `20s` 级白等中的绝大部分
- 单次审批预计可减少约 `35s ~ 40s`

### 8.2 第一优先级：提交后确认逻辑应更早进入“待办消失”确认，不要先固定空等 `8s`

建议：

1. 点击 `提交` 后，先快速判断页面结构是否已经切回待办列表
2. 若已切回待办列表，立即开始按 `document_no` 复查
3. toast 监听应改为并行补充信号，而不是串行硬等待

原因：

- 本次样本最终成功确认靠的是 `todo_disappeared`
- 说明页面实际上并不依赖成功 toast 也可以确认成功
- 先空等 `8s` 再复查，纯属放大体感延迟

预期收益：

- 可再减少约 `6s ~ 8s`

### 8.3 第一优先级：把确认阶段改成“并行信号 + 情景化短超时”，而不是串行大窗口

建议：

1. 将当前 `0~8s -> 8~25s -> 25~45s` 的串行分段改为并行判定
2. 对不同确认路径使用不同短超时：
   - toast / modal：`1s ~ 3s`
   - 审批记录快速探测：`3s ~ 5s`
   - 待办消失复查：`3s ~ 5s`
3. 只有在上述路径都未命中时，才进入长兜底

原因：

- 当前长耗时并不是因为 EHR 一定慢，而是因为代码把多个兜底阶段串起来顺序跑
- 优先级合理，但执行方式过于保守

预期收益：

- 可进一步压缩提交后的确认等待
- 且不必牺牲 `105C` 的误判保护能力

### 8.4 第二优先级：审批链路复用浏览器上下文，不要每单都重开 Chromium

建议：

1. 在 `处理单据` 模块引入“会话级浏览器上下文”
2. 同一操作员在短时间内连续审批多张单据时，复用：
   - browser
   - context
   - 已登录页面

预期收益：

- 每单减少约 `4s ~ 5s`

截至 `2026-03-17` 状态：

- 已实现首版
- 当前策略为：
  - `处理单据` 审批 API 在进程内复用 `browser / context / page`
  - 同一时刻通过互斥锁只允许一个审批任务占用该会话
  - 空闲超过 `5 分钟` 自动回收，避免长期挂着陈旧会话
  - 若浏览器断开、页签被手工关闭或运行参数变化，则自动重建会话
- 已通过本地烟测验证：
  - 第一次调用创建会话
  - 第二次调用成功复用同一个 `browser / context / page`

### 8.5 第二优先级：详情接口改为按页签懒加载

建议：

1. `overview / feedbackOverview` 作为首次加载主数据
2. `orgScopes / approvals / riskDetails` 改为按页签请求
3. 对 `orgScopes` 这种大表格支持分页或延迟加载

原因：

- 当前详情接口一次性返回 `1120` 条组织范围
- 会放大抽屉打开耗时和前端渲染压力

预期收益：

- 抽屉打开可从约 `2s` 继续压缩
- 也能降低前端切换页签前的无效数据传输

### 8.6 第二优先级：将审批流中的关键等待参数拆为独立配置

建议：

不要继续让以下行为共用同一个 `browser.timeout_ms`：

- 等待待办分页控件
- 等待页签出现
- 等待表单字段可见
- 等待审批确认完成

建议拆分成独立参数，例如：

- `todo_ready_quick_timeout_ms`
- `todo_ready_fallback_timeout_ms`
- `approval_feedback_timeout_ms`
- `approval_record_probe_timeout_ms`
- `approval_todo_reprobe_timeout_ms`

原因：

- 现在 `20s` 的本质问题不是值本身，而是“所有等待都吃同一把尺子”

### 8.7 第三优先级：减少固定 `wait_for_timeout(...)`

当前审批流中仍存在多处固定等待，例如：

- `600ms`
- `500ms`
- `300ms`

建议：

1. 用“元素状态已就绪”替代固定睡眠
2. 对必须保留的固定等待，尽量缩短到最小值

原因：

- 单次看起来不大
- 但会在高频审批场景中形成累计延迟

### 8.8 第三优先级：保留 `120s` 前端兜底和 `45s` 后端兜底，但不要让常规链路依赖它们

建议：

1. `120s` 前端超时先保留
2. `45s` 后端总确认上限先保留
3. 先优化常规路径，让大多数审批在 `10s ~ 20s` 内结束
4. 待稳定后，再评估是否收缩兜底上限

原因：

- 这两项本质是风险控制，不是性能优化手段
- 直接压短，容易把“慢但成功”重新打成“失败”

## 9. 优先级排序后的实施建议

建议按以下顺序实施：

1. `P0`：审批流去掉 `1000条/页` 强等待
2. `P0`：提交后确认逻辑提前进入 `todo_disappeared` 复查
3. `P0`：把确认阶段改成并行信号 + 情景化短超时
4. `P1`：浏览器上下文复用
5. `P1`：详情接口按页签懒加载
6. `P1`：将审批等待参数拆成独立配置
7. `P2`：清理固定睡眠
8. `P2`：在前述优化稳定后，再评估是否收缩 `45s / 120s` 兜底上限

   - 打开待办不再白等 `20s`

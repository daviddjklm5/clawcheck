# 方案编号005A：组织授权级别规则方案文档

## 1. 方案目标
在 `005组织属性查询表设计方案文档` 已完成的基础上，对结果表 `组织属性查询` 继续做增强，逐步沉淀更多可直接面向业务查询的统一口径字段。

本次 `005A` 的目标聚焦于首个增补规则：
- 在 `组织属性查询` 中新增“组织授权级别”字段
- 给出可稳定复用、可全量刷新、可持续扩展的首版取值规则
- 保持 `005`、`007` 既有字段与口径不被破坏

本方案的重点不是改造源表 `组织列表`，而是在派生结果表中继续承接业务规则，保证：
- 查询字段持续增补但出口统一
- 规则集中维护，避免业务查询各自重复判断
- 新规则可随 `组织列表` 全量导入后自动重建

## 2. 背景与问题判断

### 2.1 当前现状
当前 `组织属性查询` 已沉淀以下业务字段：
- `org_unit_name`
- `org_unit_rule`
- `war_zone`
- `wanyu_city_sales_department`
- `process_level_name_resolved`
- `process_level_category`

现有方案见：
- `方案计划文档/005组织属性查询表设计方案文档`
- `方案计划文档/005B组织流程层级判断口径拆分方案文档`
- `方案计划文档/005C组织流程层级分类口径拆分方案文档`
- `方案计划文档/005D万御城市营业部口径拆分方案文档`
- `方案计划文档/007组织属性查询表城市战区未命中补齐方案文档`
- `automation/sql/006_organization_attribute_query.sql`

但当前结果表仍缺少一列可直接表达“该组织对应授权口径处于哪个层级”的统一字段。

### 2.2 当前问题
若业务要判断某组织属于哪一档授权层级，当前只能临时基于以下字段自行拼规则：
- `org_level`
- `physical_level`
- `org_full_name`

这种做法存在明显问题：
1. 各处 SQL / 报表容易各写一套口径
2. 规则迭代后难以统一回溯历史查询结果
3. 无法直接作为高频筛选列复用
4. 后续新增第 2、第 3 个增强字段时，也会继续出现重复判断问题

### 2.3 设计判断
本方案判断：
- “组织授权级别”应作为 `组织属性查询` 的**派生字段**落表
- 首版规则应优先复用当前已稳定落库的层级字段，不引入新的采集入口
- 规则设计应支持后续继续追加更多增强字段，而不是仅为单一字段写一次性逻辑

## 3. 方案结论

### 3.1 最终选型
`005A` 建议继续沿用 `005` 的总体模式：
- **实体结果表增强**：直接扩充 `组织属性查询` 字段
- **刷新过程增强**：在 `refresh_组织属性查询()` 中同步计算新字段
- **规则函数化**：将“组织授权级别”口径封装为可维护规则

### 3.2 本次新增字段
建议新增以下字段：
- `org_auth_level TEXT`
  - 中文含义：组织授权级别
- `org_auth_level_rule TEXT`
  - 中文含义：组织授权级别命中规则
  - 用于审计、抽样验证、排查规则命中路径

### 3.3 首版规则策略
首版“组织授权级别”采用：
- **特殊组织优先**
- **组织层级优先**
- **物理层级兜底**
- **无法判断则归入未分类**

即：
1. 先判断是否命中特殊组织路径
2. 未命中时，优先依据 `org_level` 标准取值映射
3. 若 `org_level` 为空或出现新值，则使用 `physical_level` 兜底映射
4. 若仍无法判断，则 `org_auth_level = 未分类`

## 5. 字段设计

### 5.1 新增字段定义
建议在 `组织属性查询` 中新增：

- `org_auth_level TEXT`
  - 表示标准化后的“组织授权级别”
  - 建议取值：`1级授权`、`2级授权`、`3级授权`、`4级授权`、`五级授权`

- `org_auth_level_rule TEXT`
  - 记录命中的规则标识
  - 便于后续核对口径、排查边界数据

### 5.2 字段用途
新增字段主要用于：
- 授权范围分层查询
- 授权规则配置前置筛选
- 审批路由、审计规则、报表统计的统一维度
- 后续自动审核规则引擎的组织层级输入

## 6. 组织授权级别口径

### 6.1 总体优先级
首版规则按以下优先级执行：

1. 特殊组织编码与路径规则
2. `org_level` 标准映射规则
3. `physical_level` 兜底映射规则
4. 空值返回规则

### 6.2 特殊组织编码与路径规则
以下组织优先判定为最高授权层级，即 `1级授权`：

1. 按 `org_code` 强规则命中
- 当 `org_code` 命中以下组织编码时：
  - `50907939`：`万物云本部`
  - `50802218`：`管理层`
  - `50802206`：`人力资源与行政服务中心`
  - `50915927`：`知之学社`
  - `50802202`：`规划发展中心`
  - `55002557`：`证券与公司治理部`
  - `50802204`：`财务与资金管理中心`
  - `50907940`：`数据与信息技术中心`
  - `50916400`：`BG人力资源行政服务中心`
  - `99999998`：`集团其他BGBU`
  - `50907621`：`投资与创新发展中心`
  - `50916021`：`政府与企业客户服务中心`
- 则：
  - `org_auth_level = 1级授权`
  - `org_auth_level_rule = org_code:l1_specified`

2. 按组织路径名称强规则命中
- 当 `org_full_name` 包含 `BG人力资源行政服务中心` 时：
  - `org_auth_level = 1级授权`
  - `org_auth_level_rule = org_full_name:contains_BG人力资源行政服务中心`

说明：
- 本节规则为当前已明确确认的一类强规则，优先级最高
- 该规则优先于后续 `org_level` / `physical_level` 映射
- 若某条记录同时命中组织编码和路径包含规则，统一按1级授权处理即可

### 6.3 2级授权规则
当前正式定义的2级授权规则如下：

1. 按 `org_code` 强规则命中
- 当 `org_code` 命中以下组织编码时：
  - `50939479`：`福建伯恩`
  - `50921744`：`重庆天骄`
- 则：
  - `org_auth_level = 2级授权`
  - `org_auth_level_rule = org_code:l2_specified`

2. 按 `org_unit_name` 前缀 + `physical_level` 命中
- 当 `org_unit_name` 以 `（作废）` 开头，且 `physical_level` 解析为数值后等于 `2` 时：
  - `org_auth_level = 2级授权`
  - `org_auth_level_rule = org_unit_name:startswith_（作废）_and_physical_level:eq_2`

3. 按 `physical_level` 命中
- 当 `physical_level` 解析为数值后等于 `2` 时：
  - `org_auth_level = 2级授权`
  - `org_auth_level_rule = physical_level:eq_2`

4. 按流程层级分类命中
- 当 `process_level_category = 业务单元本部`、`org_unit_name = 万睿科技`，且 `physical_level` 解析为数值后在 `1~4`（含）时：
  - `org_auth_level = 3级授权`
  - `org_auth_level_rule = process_level_category:业务单元本部_and_org_unit_name:万睿科技_and_physical_level:lte_4`
- 该规则为 `万睿科技` 组织单位例外规则，优先于通用“业务单元本部且 physical_level <= 3 归入2级授权”的规则
- 当 `process_level_category = 业务单元本部`，且 `physical_level` 解析为数值后在 `1~3`（含）时：
  - `org_auth_level = 2级授权`
  - `org_auth_level_rule = process_level_category:业务单元本部_and_physical_level:lte_3`
- 当 `process_level_category = 蝶发人行部` 时：
  - `org_auth_level = 2级授权`
  - `org_auth_level_rule = process_level_category:蝶发人行部`

5. 优先级说明
- 2级授权规则低于 `6.2` 的1级授权强规则
- 即使某条记录同时满足2级授权条件，只要先命中1级授权规则，仍应判定为 `1级授权`
- 若同一条记录命中 `org_code` 二级强规则，则优先记录 `org_code:l2_specified`
- 若同一条记录同时命中“作废前缀且 `physical_level = 2`”与流程层级分类二级规则，当前优先记录 `org_unit_name:startswith_（作废）_and_physical_level:eq_2`
- 若同一条记录同时命中 `physical_level = 2` 与 `process_level_category = 蝶发人行部`，当前优先记录 `process_level_category:蝶发人行部`
- 若同一条记录满足 `process_level_category = 业务单元本部 and physical_level <= 4`，但 `org_unit_name = 万睿科技`，当前按 `3级授权` 处理，不再落入2级授权
- 若同一条记录同时命中 `physical_level = 2` 与 `process_level_category = 业务单元本部`（且 `physical_level <= 3`），当前优先记录 `process_level_category:业务单元本部_and_physical_level:lte_3`

### 6.4 3级授权规则
当前正式定义的3级授权规则如下：

1. 按流程层级分类 + 组织单位 + 物理层级命中（万睿科技例外）
- 当 `process_level_category = 业务单元本部`、`org_unit_name = 万睿科技`，且 `physical_level` 解析为数值后在 `1~4`（含）时：
  - `org_auth_level = 3级授权`
  - `org_auth_level_rule = process_level_category:业务单元本部_and_org_unit_name:万睿科技_and_physical_level:lte_4`

2. 按流程层级分类 + 物理层级命中（业务单元本部降级）
- 当 `process_level_category = 业务单元本部`，且 `physical_level` 解析为数值后大于等于 `4` 时：
  - `org_auth_level = 3级授权`
  - `org_auth_level_rule = process_level_category:业务单元本部_and_physical_level:gte_4`

3. 按 `physical_level` 命中
- 当记录未命中 `6.2` 的1级授权规则、也未命中 `6.3` 的2级授权规则，且 `physical_level` 解析为数值后等于 `3` 时：
  - `org_auth_level = 3级授权`
  - `org_auth_level_rule = physical_level:eq_3`

4. 适用范围说明
- 第 1 条适用于 `万睿科技` 组织单位在 `process_level_category = 业务单元本部 and physical_level <= 4` 的例外场景
- 第 2 条适用于 `process_level_category = 业务单元本部 and physical_level >= 4` 的降级场景
- 第 3 条适用于“物理层级为 3 的其他组织”
- 也就是说，只要某条记录已经命中一级或2级授权，就不再落入3级授权

### 6.5 4级授权规则
当前正式定义的4级授权规则如下：

1. 按流程层级分类 + 物理层级命中
- 当 `process_level_category = 属地组织`，且 `physical_level >= 4` 时：
  - `org_auth_level = 4级授权`
  - `org_auth_level_rule = process_level_category:属地组织_and_physical_level:gte_4`

2. 按流程层级分类为空 + 物理层级命中
- 当 `process_level_category IS NULL`，且 `physical_level >= 6` 时：
  - `org_auth_level = 4级授权`
  - `org_auth_level_rule = process_level_category:<NULL>_and_physical_level:gte_6`

3. 适用范围说明
- 4级授权规则低于 1级、2级、3级授权规则
- 仅当记录未命中前述一、二、三级规则时，才落入4级授权

### 6.6 当前未分类记录的处理方式
当记录既未命中 `6.2` 的1级授权强规则，也未命中 `6.3` 的2级授权规则、`6.4` 的3级授权规则、`6.5` 的4级授权规则时，当前阶段统一按“未分类”处理。

即：
- `org_auth_level = 未分类`
- `org_auth_level_rule = unresolved`

说明：
- 当前阶段不再使用 `org_level` 直接映射1级授权
- 当前阶段使用“`org_unit_name` 作废前缀且 `physical_level = 2`”、`physical_level = 2`、`process_level_category = 蝶发人行部`、`process_level_category = 业务单元本部 and physical_level <= 3` 识别2级授权
- 当前阶段使用 `physical_level = 3`，以及 `process_level_category = 业务单元本部 and physical_level >= 4` 识别3级授权
- 当前阶段使用 `process_level_category = 属地组织 and physical_level >= 4`、`process_level_category IS NULL and physical_level >= 6` 识别4级授权
- 五级授权及以下规则仍待后续业务输入后再补充

## 7. 实现建议

### 7.1 与 005 / 007 的关系
本次增强应与既有规则并存，不应覆盖下列口径：
- `org_unit_name`
- `org_unit_rule`
- `war_zone`
- `wanyu_city_sales_department`
- `process_level_name_resolved`
- `process_level_category`

也就是说，`005A` 是在 `005` 的结果表结构上继续增补字段，而不是重写 `005` 的既有计算逻辑。

### 7.2 已落地实现方式
本方案现已在 `automation/sql/006_organization_attribute_query.sql` 中落地实现：
- `fn_map_org_auth_level(...)`
- `fn_map_org_auth_level_rule(...)`
- `refresh_组织属性查询()` 中的目标列插入逻辑

当前函数入参包括：
- `p_org_code TEXT`
- `p_org_level TEXT`
- `p_physical_level TEXT`
- `p_org_full_name TEXT`

### 7.3 建议 DDL 增补
建议将 `组织属性查询` 表增补为包含：

```sql
ALTER TABLE "组织属性查询"
    ADD COLUMN IF NOT EXISTS org_auth_level TEXT,
    ADD COLUMN IF NOT EXISTS org_auth_level_rule TEXT;
```

若仍采用 staging 表全量切换，则在 staging 建表语句中直接补充上述两列。

### 7.4 刷新过程建议
建议在 `refresh_组织属性查询()` 中按以下顺序处理：
1. 从 `组织列表` 读取基础列
2. 保持 `005` 既有组织单位、战区、流程层级等计算逻辑不变
3. 同步计算 `org_auth_level`
4. 同步写入 `org_auth_level_rule`
5. 完成 staging 表原子切换

## 8. 查询与索引建议

### 8.1 查询原则
后续凡涉及组织授权层级相关业务判断，统一使用：
- `组织属性查询.org_auth_level`

不建议业务侧再次临时按 `org_level` 或 `physical_level` 自行判断。

### 8.2 索引建议
若后续查询中 `org_auth_level` 会作为高频筛选条件，建议增加索引：

```sql
CREATE INDEX IF NOT EXISTS idx_组织属性查询_org_auth_level
ON "组织属性查询" (org_auth_level);
```

若后续常与组织单位联合过滤，也可考虑：
- `(org_auth_level, org_unit_name)`

### 9.3 1级授权明细校验 SQL
建议刷新完成后直接执行以下 SQL 校验 `1级授权` 明细：

```sql
SELECT
    org_code,
    org_name,
    org_level,
    physical_level,
    org_auth_level,
    org_auth_level_rule,
    org_full_name

ORDER BY
    CASE org_auth_level_rule
        WHEN 'org_code:l1_specified' THEN 1
```

## 10. 风险与边界

### 10.1 新层级值持续出现
风险：
- 后续 `org_level` 可能出现当前未分类的新值

控制方案：
- 将规则命中写入 `org_auth_level_rule`
- 定期对 `unresolved` 记录做抽样复核
- 待2级授权及以下规则确认后，再补齐后续口径

### 10.2 首版授权口径仍可能继续细化
风险：
- 后续业务可能要求某些具体组织路径跳级授权

控制方案：
- 保留“特殊组织编码与路径优先”机制
- 后续补充2级授权及以下规则时，继续遵循强规则优先原则

### 10.3 业务直接依赖源字段自行判断
风险：
- 业务方继续绕过结果表直接使用 `组织列表` 自行计算

控制方案：
- 明确约定授权相关查询统一查 `组织属性查询`
- 将 `org_auth_level` 作为标准出口字段对外提供

## 11. 最终结论
`005A` 最终建议如下：

1. 继续沿用 `005` 的“源表全量导入 -> 结果表全量重建”模式
2. 在 `组织属性查询` 中新增 `org_auth_level` 与 `org_auth_level_rule`
3. 当前“组织授权级别”正式定义四档规则：1级授权强规则、2级授权 `physical_level = 2 / 蝶发人行部 / 业务单元本部且 physical_level <= 3（排除万睿科技） / 指定编码` 规则、3级授权 `physical_level = 3 / 业务单元本部且 physical_level >= 4 / 万睿科技且业务单元本部且 physical_level <= 4` 规则、4级授权 `属地组织 and physical_level >= 4 / process_level_category is null and physical_level >= 6` 规则
4. 以下组织编码直接归入 `1级授权`：`50907939`、`50802218`、`50802206`、`50915927`、`50802202`、`55002557`、`50802204`、`50907940`、`50916400`、`99999998`、`50907621`、`50916021`
5. 当 `org_full_name` 包含 `BG人力资源行政服务中心` 时，直接归入 `1级授权`
6. 2级授权规则为：未命中1级授权时，命中2级授权指定编码的组织，或 `org_unit_name` 以 `（作废）` 开头且 `physical_level = 2` 的组织，或 `physical_level = 2` 的组织，或 `process_level_category = 蝶发人行部` 的组织，或 `process_level_category = 业务单元本部 and physical_level <= 3 and org_unit_name != 万睿科技` 的组织，归入 `2级授权`
7. 3级授权规则为：未命中一级、2级授权时，`physical_level = 3` 的其他组织，以及 `process_level_category = 业务单元本部 and physical_level >= 4` 的组织，以及 `org_unit_name = 万睿科技 and process_level_category = 业务单元本部 and physical_level <= 4` 的组织归入 `3级授权`
8. 原基于 `org_level` 的1级授权映射已取消，避免与强规则并行
9. 五级授权及以下规则暂不定义，未命中记录当前统一记为 `未分类 / unresolved`
10. 刷新逻辑仍统一收敛在 `refresh_组织属性查询()`，不改动源表采集链路
11. 当前实现文件为 `automation/sql/006_organization_attribute_query.sql`，可通过执行 `SELECT refresh_组织属性查询();` 生效
12. 后续若新增第 4、第 5 个增强字段，可继续按 `005A` 的增强模式在结果表中扩展

在当前项目阶段，这是一种实现成本低、口径集中、并且便于后续继续增补业务规则的增强方案。

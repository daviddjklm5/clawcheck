# 方案编号008：权限申请单据直达链接生成与打开方案文档

## 1. 方案目标
在现有 `003` 单据采集能力基础上，新增“**为某个具体单据生成完整链接，并可通过该链接打开对应单据详情**”能力。

本方案目标：
- 为指定 `document_no` 生成一个可访问的完整链接
- 支持通过该链接直接打开对应的权限申请单据详情页
- 已登录状态下可直接进入目标单据
- 未登录状态下完成登录后仍可回到目标单据
- 将链接能力与 `003` 的采集能力解耦，便于独立建设与复用

本方案聚焦“**打开具体单据**”这一目标，不涉及单据审批、回写、提交等写操作。

## 2. 背景与问题判断

### 2.1 当前现状
`003` 方案当前仅支持：
1. 登录首页
2. 进入 `我的消息 -> 待办任务`
3. 在列表中查找目标单据
4. 点击主题列链接进入详情页

现有实现见：
- `方案计划文档/003权限申请待办单据采集入库技术设计文档.md`
- `automation/flows/permission_collect_flow.py`

### 2.2 当前问题
当前系统尚未沉淀“可分享、可复用、可直接打开指定单据”的稳定链接能力。

也就是说，现状是：
- **人可以通过页面点击打开单据**
- **程序也可以通过页面点击打开单据**
- **但不能直接给业务一个完整链接去打开某个具体单据**

### 2.3 对 iERP 原生 URL 的判断
结合现有自动化探测结果：
- 打开首页后 URL 为：`https://thr.onewo.com:8443/ierp/?formId=home_page`
- 点击待办任务后，浏览器地址仍停留在 `home_page`
- 点击业务页面后，浏览器地址也未体现出稳定的单据编号参数

这说明当前系统大概率采用：
- 单页应用（SPA）
- 页签分屏承载
- 页面内部路由 / pageId / taskId 驱动

因此本方案判断：
- **不能在当前阶段仅凭 `document_no` 直接硬编码拼出可靠的 iERP 原生单据 URL**
- 若强行拼接，很可能出现：
  - 打开后回到首页
  - 打开空白页签
  - 找不到具体单据
  - 登录后丢失目标上下文

## 3. 方案结论

### 3.1 最终选型
本方案建议采用：
- **中转链接方案** 作为正式落地方案
- **原生路由采集方案** 作为增强能力逐步补齐

### 3.2 方案含义
#### 3.2.1 中转链接方案
由我方系统生成一条完整链接，例如：
- `https://<your-domain>/ierp/document/open?documentNo=QX-260311-00000223`

用户访问该链接后，由中转服务负责：
1. 校验访问身份
2. 建立或复用 iERP 登录态
3. 自动进入 `我的消息 -> 待办任务`
4. 定位 `document_no`
5. 打开对应详情页

#### 3.2.2 原生路由采集方案
在打开单据详情的过程中，额外抓取系统真实路由参数，例如：
- `pageId`
- `taskId`
- `billId`
- `formId`
- 其他内部跳转参数

若后续确认存在稳定原生 URL，则可升级为：
- 直接返回 iERP 原生可访问链接

### 3.3 为什么优先选中转链接
原因如下：
1. 当前证据不足以证明 iERP 暴露了稳定、可复用、可拼接的单据详情 URL
2. 中转链接不依赖页面真实地址是否变化，更适配 SPA 系统
3. 中转链接可以统一处理登录态失效问题
4. 中转链接可以兼容后续系统升级，不影响对外链接格式
5. 中转链接最符合“提供一个完整链接，可以打开某个具体单据”的目标

## 4. 总体设计

### 4.1 总体架构
建议拆分为两层：

1. **链接生成层**
- 输入：`document_no`
- 输出：完整中转链接

2. **链接执行层**
- 接收中转链接请求
- 自动登录 / 复用登录态
- 打开指定单据
- 成功后停留在详情页

### 4.2 与 003 的关系
`003` 负责：
- 从待办列表采集单据数据
- 入库

`008` 负责：
- 生成单据访问链接
- 通过链接打开具体单据

二者关系建议为：
- `003` 可复用 `008` 的“打开单据”能力
- `008` 不依赖 `003` 完成采集后才能工作
- 若数据库中已存在 `document_no`，可直接生成链接

### 4.3 输出对象
本方案建议新增“单据链接对象”，字段如下：
- `document_no`
- `open_url`
- `open_type`
- `route_payload`
- `generated_at`
- `source_system`

字段含义：
- `open_url`：最终提供给用户访问的完整链接
- `open_type`：`redirect` 或 `native`
- `route_payload`：原生路由参数快照，便于后续升级和排障

## 5. 链接设计

### 5.1 对外链接格式
建议正式格式：
- `https://<your-domain>/ierp/document/open?documentNo=<document_no>`

示例：
- `https://ops.example.com/ierp/document/open?documentNo=QX-260311-00000223`

如需增强安全性，也可使用签名参数：
- `https://ops.example.com/ierp/document/open?documentNo=QX-260311-00000223&ts=1773302400&sign=<signature>`

### 5.2 链接返回方式
中转链接访问后有两种返回方式：

1. **浏览器自动打开模式**
- 中转服务直接驱动浏览器打开目标单据
- 适合内部运维、自动化、值守场景

2. **页面跳转模式**
- 中转服务完成鉴权后，跳到一个前端承载页
- 承载页内再调用自动化或嵌入式打开逻辑
- 适合内部门户统一入口场景

当前推荐：
- 优先采用 **浏览器自动打开模式**
- 实现路径更短，验证成本更低

### 5.3 open_type 约定
建议统一约定：
- `redirect`：通过我方中转链接打开
- `native`：直接使用 iERP 原生详情链接打开

当前默认：
- `open_type = redirect`

## 6. 打开单据的执行流程设计

### 6.1 主流程
1. 接收 `document_no`
2. 生成中转链接
3. 用户点击链接
4. 服务端校验参数合法性
5. 校验当前 iERP 登录态
6. 若失效则自动登录
7. 打开首页
8. 进入 `我的消息 -> 待办任务`
9. 在列表中匹配目标 `document_no`
10. 点击 `主题` 列链接进入详情页
11. 校验详情页基本信息中 `单据编号` 与入参一致
12. 返回成功结果

### 6.2 打开成功判定
满足以下条件可视为成功：
- `#fs_baseinfo` 可见
- 页面存在 `权限申请明细`
- 基本信息中的 `单据编号 = document_no`

### 6.3 打开失败判定
出现以下任一情况视为失败：
- 登录失败
- 待办列表中不存在该 `document_no`
- 点击后未进入详情区域
- 详情页 `单据编号` 与请求值不一致
- 页面结构变化导致无法定位目标元素

## 7. 数据与存储设计

### 7.1 是否需要入库
建议：**需要入库**。

原因：
- 便于业务侧查询和复制链接
- 便于后续复核链接是否失效
- 便于逐步从 `redirect` 升级到 `native`

### 7.2 建议新增字段
若复用 `申请单基本信息` 主表，建议新增：
- `document_open_url TEXT`
- `document_open_type VARCHAR(32)`
- `document_route_payload JSONB`
- `document_open_url_updated_at TIMESTAMP`

### 7.3 route_payload 示例
```json
{
  "document_no": "QX-260311-00000223",
  "native_url": null,
  "page_id": null,
  "task_id": null,
  "bill_id": null,
  "open_strategy": "redirect_v1"
}
```

说明：
- 在当前阶段允许 `native_url`、`page_id`、`task_id`、`bill_id` 为空
- 先保证链接可用，再逐步补齐原生参数

## 8. 接口设计

### 8.1 生成链接接口
建议新增接口：
- `GET /api/documents/{document_no}/open-link`

返回示例：
```json
{
  "document_no": "QX-260311-00000223",
  "open_url": "https://ops.example.com/ierp/document/open?documentNo=QX-260311-00000223",
  "open_type": "redirect"
}
```

### 8.2 执行打开接口
建议新增入口：
- `GET /ierp/document/open?documentNo=QX-260311-00000223`

处理职责：
- 参数校验
- 会话校验
- 自动打开单据
- 成功后停留在详情页

## 9. 原生链接增强设计

### 9.1 增强目标
在后续联调阶段，可增加“原生链接提取”能力。

增强动作：
1. 成功打开单据详情后
2. 记录以下信息：
   - `page.url()`
   - 当前页签节点属性
   - 页面请求 URL
   - 请求体中的业务参数
   - 响应中的路由字段
3. 分析是否存在稳定可复用的详情页地址模板

### 9.2 增强成功标准
若确认以下条件同时成立，则可升级为 `native`：
- 单据详情存在稳定 URL 模板
- URL 可携带业务主键或流程实例主键
- 退出登录后重新登录仍能回到原单据
- 跨时间多次打开仍有效

### 9.3 升级策略
升级时不修改对外链接格式，只调整后台解析策略：
- 对外仍保持：`/ierp/document/open?documentNo=...`
- 服务端内部若已存在 `native_url`，则优先跳转 `native_url`
- 若 `native_url` 不可用，则回退到 `redirect` 打开逻辑

这样可以保证：
- 历史链接不失效
- 用户使用方式不变
- 系统可平滑演进

## 10. 安全与权限设计

### 10.1 访问控制
虽然目标是“给出完整链接”，但不建议做成匿名可访问。

建议要求：
- 链接使用方必须是内网用户或已登录用户
- 中转服务需校验访问人权限
- 仅允许查看其本身有权访问的单据

### 10.2 参数防篡改
建议至少支持以下一种：
- 登录态校验
- 服务端二次鉴权
- 链接签名
- 访问时效控制

### 10.3 审计日志
建议记录：
- 访问人
- 访问时间
- `document_no`
- 打开结果
- 失败原因
- 实际采用的打开策略：`redirect` / `native`

## 11. 异常与兜底设计

### 11.1 单据不存在
返回明确错误：
- `document_no not found in todo list`

### 11.2 单据已不在待办
当前阶段中转逻辑依赖 `待办任务`，因此若单据已流转出待办，可能无法打开。

建议分两步处理：
1. 当前版本明确边界：仅保证打开“待办中的权限申请单据”
2. 后续版本扩展到：
   - `已办任务`
   - `在办申请`
   - 更通用的流程中心检索

### 11.3 登录态失效
- 自动重新登录
- 登录成功后重新执行打开流程
- 失败则返回登录异常

### 11.4 页面结构变化
- 保留截图
- 保留原始路由快照
- 返回结构变更错误
- 便于后续快速修复

## 12. 验收标准

### 12.1 基础验收
以下场景应通过：
1. 输入 `QX-260311-00000223`，可生成完整链接
2. 点击链接后可进入该单据详情
3. 详情页显示的 `单据编号` 与入参一致
4. 链接重复打开多次仍可用

### 12.2 登录态验收
以下场景应通过：
1. 已登录时点击链接，直接打开单据
2. 未登录时点击链接，登录后仍能打开同一单据

### 12.3 失败场景验收
以下场景应有明确结果：
1. 单据号不存在
2. 当前用户无权限访问
3. 页面结构变化
4. 系统超时

## 13. 实施顺序建议
建议分三步实施：

1. **第一阶段：中转链接可用**
- 完成链接生成
- 完成链接打开
- 完成基础校验与日志

2. **第二阶段：链接入库与接口化**
- 将 `open_url` 写入数据库
- 提供查询接口
- 支持批量获取单据链接

3. **第三阶段：原生链接增强**
- 补采真实路由参数
- 验证是否可形成稳定原生 URL
- 若可行，切换 `open_type=native`

## 14. 当前实施边界
- 当前方案默认以 `权限申请` 单据为目标对象
- 当前优先支持“待办任务”中的单据
- 当前不承诺已经获得 iERP 原生稳定详情 URL
- 当前正式落地建议以 `redirect` 方式实现“完整链接打开具体单据”目标

## 15. 最终结论
本次 `008` 方案建议明确命名并聚焦于：
- **权限申请单据直达链接生成与打开**

最终落地建议为：
- 对外提供稳定不变的完整中转链接
- 对内封装登录、导航、打开、校验逻辑
- 将 iERP 原生 URL 识别作为增强项，而不是首版落地前提

这样最符合当前目标：
- **真正提供一个完整链接，可以打开某个具体的单据**

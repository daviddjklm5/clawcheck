# 方案编号105B：处理单据页待办状态同步与已处理标记方案文档

## 1. 方案背景

`105 / 105A` 已为 `处理单据 -> 单据详情抽屉 -> 审批单据` 页签补齐：

- `批准`
- `验证连通性（dry-run）`
- EHR `任务处理` 页签中的 `审批决策 = 同意`
- `审批意见` 写入与提交
- 审批过程日志与超时兜底

但在 `2026-03-16` 的现场使用中，样例单据 `RA-20260312-00019920` 出现了新的失败场景：

- 在 `处理单据` 页点击 `审批单据 -> 验证连通性`
- 后端进入 `PermissionCollectFlow.open_document(document_no)`
- Playwright 需先从 EHR `待办任务` 列表中定位该单据
- 实际该单据已不在当前账号待办中
- 最终报错：
  - `Todo document link not ready. document_no='RA-20260312-00019920'`

现场日志文件：

- `automation/logs/approval_20260316_163239_RA-20260312-00019920.json`

这说明当前系统仍缺少一个关键能力：

- `处理单据` 工作台虽然能展示“项目内待处理单据”
- 但并不知道这些单据是否**仍然存在于当前账号的 EHR 待办列表**

因此会出现：

- 页面上看起来还可处理
- 实际 EHR 侧已经不在待办
- 用户点击审批或验证连通性后，只能看到技术报错，而不是业务可理解的“该单据已处理”

## 2. 问题定义

本次问题不是 `审批单据` 页签失效，而是**工作台待办视图与 EHR 实际待办状态不同步**。

当前缺口有三层：

1. PostgreSQL 中没有“当前账号视角下，该单据是否仍在待办”的独立字段
2. `处理单据` 页面列表只展示风险评估结果，未展示“待办处理状态”
3. 审批 API 在执行前不会先判断该单据最近一次待办同步结果，仍直接尝试从 EHR 待办中打开

因此，像 `RA-20260312-00019920` 这种“已从当前账号待办消失”的单据，会被误当成“仍可处理单据”。

## 3. 现状分析

### 3.1 当前失败链路

当前审批链路为：

1. 前端调用  
   `POST /api/documents/process-workbench/{document_no}/approval`
2. 后端进入  
   `automation.api.process_dashboard.approve_process_document()`
3. `DocumentApprovalFlow.execute_approve()` 内部调用  
   `PermissionCollectFlow.open_document(document_no)`
4. `open_document()` 强依赖从 EHR `待办任务` grid 中找到目标单据链接
5. 若单据已不在待办，则抛出：
   `Todo document link not ready`

结论：

- 当前失败点发生在“打开待办单据”之前
- 失败根因是“待办已不存在”，不是“审批页签无法操作”

### 3.2 当前 `处理单据` 页数据口径

根据 `201 / 203` 方案，当前 `处理单据` 页主列表来自：

- `逐单最新评估结果`

当前查询入口：

- `automation.db.postgres.PostgresRiskTrustStore.fetch_process_workbench()`

当前列表字段包含：

- `单据编号`
- `权限对象`
- `申请人`
- `工号`
- `部门`
- `单据状态`
- `最终信任分`
- `总结论`
- `建议动作`

但不包含：

- “当前账号视角的待办处理状态”
- “最近一次待办状态同步时间”

因此，页面上的“待处理单据列表”实际更接近：

- “项目内已有评估结果的单据列表”

而不是真正意义上的：

- “当前账号仍可在 EHR 待办中处理的单据列表”

### 3.3 为什么不能直接复用 `单据状态`

`申请单基本信息.单据状态` 来自业务单据自身页面字段，表示业务单据状态。

它不能直接承载“当前账号是否仍在待办”，原因是：

- 同一业务单据状态下，不同账号的待办归属可能不同
- 单据可能业务上仍是 `已提交`，但对当前审批人来说已经处理完
- `已处理` 在本方案中的含义是：
  - **当前账号待办列表未命中**
  - 不是“业务单据已完结”

因此必须新增独立字段，不能覆写原 `单据状态`。

### 3.4 为什么不能影响 `采集次数`

根据 `003 / 017` 方案，`采集次数` 的设计语义是：

- 首采 `= 1`
- 当 `最新审批时间` 变化时，整单重采并 `原值 + 1`
- 当 `最新审批时间` 未变化时，不累加

本次“待办状态同步”只是在当前账号视角下补充：

- 该单据是否仍位于 EHR 待办中

它不是一次新的单据页面重采，也不是一次审批轨迹变化。

因此：

- `待办处理状态` 的更新**不得**触发 `采集次数` 变化
- `待办处理状态 = 已处理` 后，如果该单据因驳回重提再次进入待办，也**不影响**原有“采集次数可累计”的设计

## 4. 方案目标

`105B` 的目标是在 `处理单据` 模块补齐“待办同步层”，实现以下结果：

1. 为单据增加独立的“待办处理状态”字段，明确区分：
   - `待处理`
   - `已处理`
2. 在 `处理单据` 页面中增加：
   - `待办处理状态` 列
   - 状态筛选
   - 手工触发的“待办状态同步”功能
3. 对于已经不在 EHR 待办中的单据，不再让用户点击审批后才看到技术错误
4. 当单据因驳回重提再次进入待办时，状态可以从 `已处理` 重新切回 `待处理`
5. 全程不破坏 `采集次数` 既有语义

## 5. 方案设计

### 5.1 新增字段设计

建议在 `申请单基本信息` 增加以下字段：

1. `待办处理状态`
   - 类型：`VARCHAR(32)`
   - 非空
   - 默认值：`待处理`
   - 取值口径：
     - `待处理`：最近一次确认时，单据仍在当前账号 EHR 待办中
     - `已处理`：最近一次确认时，单据已不在当前账号 EHR 待办中

2. `待办状态更新时间`
   - 类型：`TIMESTAMP`
   - 可空
   - 含义：最近一次写入 `待办处理状态` 的时间

字段归属说明：

- 这两个字段属于“单据主表上的派生状态字段”
- 不属于风险评估结果字段
- 不写入 `申请单风险信任评估` 表

推荐原因：

- `待办处理状态` 用于直接驱动页面展示和审批前判断
- `待办状态更新时间` 用于提示该状态的新鲜度，避免用户把旧状态当成实时结果

### 5.2 字段语义约束

`待办处理状态` 必须明确限定为：

- **当前登录账号**
- **当前一次待办同步结果**
- **工作台视角**

它不等于：

- 单据最终审批结论
- 单据业务完结状态
- 风险评估结论

也就是说，`已处理` 的真实含义是：

- “该单据最近一次未在当前账号 EHR 待办中命中”

因此允许出现以下回切：

1. 某单据先被同步为 `已处理`
2. 后续因驳回、转办、重新提交再次进入当前账号待办
3. 下一次同步或下一次成功采集时，再改回 `待处理`

### 5.3 状态变更规则

本方案定义以下四类写入场景：

1. `collect` 成功采集单据
   - 来源就是 EHR 当前待办
   - 写库时直接将 `待办处理状态 = 待处理`
   - 同步刷新 `待办状态更新时间`

2. `处理单据 -> 同步待办状态` 任务执行
   - 若项目单据存在于当前 EHR 待办列表：
     - 更新为 `待处理`
   - 若项目单据不存在于当前 EHR 待办列表：
     - 更新为 `已处理`
   - 同步刷新 `待办状态更新时间`

3. `审批单据 -> 批准` 成功且非 `dry-run`
   - 提交成功后，可立即将当前单据更新为 `已处理`
   - 目的是让工作台尽快反映当前账号已完成该单据处理
   - 若后续该单据重新进入待办，仍允许被下一次同步或采集改回 `待处理`

4. `审批单据 -> 验证连通性（dry-run）`
   - 只做连通性验证
   - 不修改 `待办处理状态`

### 5.4 待办状态同步任务

#### 5.4.1 交互形态

建议在 `处理单据` 页工具栏增加按钮：

- `同步待办状态`

该按钮不是简单刷新 PostgreSQL 查询，而是触发一次真实 EHR 待办扫描任务。

不建议在页面加载时自动执行，原因是：

- 当前 `处理单据` 页已有明确的 PostgreSQL 实时查询口径
- 自动打开 EHR 会让一个轻量查询页变成重型自动化入口
- 用户需要可控地决定何时消耗浏览器与登录态资源

#### 5.4.2 同步范围

同步任务的比对范围定义为：

- 项目侧：当前 `处理单据` 工作台可见的“逐单最新评估结果”单据集
- EHR 侧：当前账号 `待办任务` 中，`单据 = 权限申请` 的单据编号集合

对比规则：

- `项目侧 ∩ EHR 待办` -> `待处理`
- `项目侧 - EHR 待办` -> `已处理`

说明：

- 对于 `EHR 待办` 中存在但项目工作台不存在的单据，本方案只在同步结果中计数提示，不在 `105B` 内顺带触发采集
- 新单据进入工作台，仍由 `202 / 203` 的 `collect + audit` 链路负责

#### 5.4.3 任务执行方式

建议复用现有 `collect / audit` 工作台任务模式，新增独立任务封装。

建议新增：

- 后端任务入口：`automation/api/process_todo_sync.py`
- 脚本入口：`python automation/scripts/run.py sync-todo-status`
- 日志文件：`automation/logs/todo_sync_YYYYMMDD_HHMMSS.json`
- 摘要文件：`automation/logs/todo_sync_summary_*.json`

最低任务输出字段建议包含：

- `taskId`
- `status`
- `requestedAt`
- `startedAt`
- `finishedAt`
- `projectDocumentCount`
- `ehrTodoCount`
- `pendingCount`
- `processedCount`
- `unchangedCount`
- `extraEhrTodoCount`
- `message`
- `logFile`

### 5.5 后端查询与写库改造

#### 5.5.1 `automation/db/postgres.py`

需同步补齐以下能力：

1. `BASIC_INFO_COLUMNS` 增加：
   - `待办处理状态`
   - `待办状态更新时间`

2. `PostgresPermissionStore`
   - 建表 SQL 与升级 SQL 支持新字段
   - `write_documents()` 的主表写入逻辑在成功采集时覆盖写入：
     - `待办处理状态 = 待处理`
     - `待办状态更新时间 = NOW()`
   - 新增批量更新方法，例如：
     - `update_todo_process_status(document_nos, status, updated_at)`
     - 或等价批量 diff 更新方法

3. `PostgresRiskTrustStore`
   - `fetch_process_workbench()` / `fetch_process_document_detail()` 查询时带出：
     - `待办处理状态`
     - `待办状态更新时间`
   - `stats` 口径补充“已处理单据”或“最近待办同步时间”

注意：

- `PostgresRiskTrustStore` 与 `PostgresPermissionStore` 当前各自维护 `ensure_table()`
- 若 `处理单据` 查询要依赖新字段，则对应升级 SQL 需要同时被 `RiskTrustStore` 感知
- 否则会出现：
  - 主表已写入新字段
  - 但处理页查询侧未确保列存在

#### 5.5.2 SQL 迁移

建议新增升级 SQL，例如：

- `automation/sql/024_process_workbench_todo_status.sql`

最小内容：

- `申请单基本信息` 增加 `待办处理状态`
- `申请单基本信息` 增加 `待办状态更新时间`
- 为历史记录回填：
  - `待办处理状态 = 待处理`
  - `待办状态更新时间 = 记录更新时间`
- 视查询需要增加索引，例如：
  - `idx_apply_form_basic_info_todo_process_status`

发布注意：

- 新字段上线后，历史记录只具备“最后一次已知状态”的默认回填值
- 要得到真实当前结果，发布后必须至少执行一次全量 `同步待办状态`

### 5.6 审批前置拦截

为避免用户再次看到：

- `Todo document link not ready`

建议在审批 API 进入 Playwright 前，增加一次本地前置判断。

规则建议如下：

1. 查询该单据最近一次 `待办处理状态`
2. 若为 `已处理`，则优先返回业务提示，而不是直接启动审批流

返回文案建议：

- `该单据最近一次待办同步结果为“已处理”，当前账号待办列表中未找到。若怀疑单据已驳回后重新提交，请先点击“同步待办状态”后再重试。`

这样可以把当前技术错误改造成业务可理解反馈。

说明：

- 这里不建议把 `已处理` 设计成永久硬拦截
- 一旦用户重新执行待办同步，或该单据重新被成功采集，就可以重新回到 `待处理`

  - `全部`
  - `待处理`
  - `已处理`

#### 5.7.2 详情抽屉

## 6. 与“采集次数累计设计”的兼容规则

### 6.1 不新增采集次数触发条件

以下动作不计入 `采集次数`：

- 点击 `同步待办状态`
- 批量把单据更新为 `已处理`
- 批量把单据更新为 `待处理`
- 审批成功后仅回写 `待办处理状态`

### 6.2 重新进入待办时的处理

若单据已经被标记为 `已处理`，后续又因驳回、转办、重新提交再次进入当前账号待办：

1. 下一次 `同步待办状态` 会把它改回 `待处理`
2. 若随后对该单据重新执行 `collect`：
   - 且 `最新审批时间` 已变化，则继续按原规则：
     - 整单重采
     - `采集次数 = 原值 + 1`
   - 若 `最新审批时间` 未变化，则仍不累加

结论：

- `待办处理状态` 是“工作台状态字段”
- `采集次数` 是“采集版本累积字段”
- 两者必须解耦

  - 明确“逐单最新评估结果”之上新增“待办处理状态”维度
- `方案计划文档/003权限申请待办单据采集入库技术设计文档.md`
  - 补充 `collect` 成功时同步回写 `待办处理状态 = 待处理`

## 10. 最终结论

`105B` 的核心结论如下：

1. 当前问题的本质不是审批流失效，而是工作台缺少“待办同步层”
2. `已处理` 应作为独立字段落在 `申请单基本信息`，不能覆写 `单据状态`
3. `处理单据` 页面应增加手工触发的“待办状态同步”功能，而不是只依赖数据库静态结果
4. `已处理` 的语义是“当前账号 EHR 待办未命中”，不是“业务单据终态”
5. 该状态必须允许回切为 `待处理`，以兼容驳回后重新提交再次进入待办的场景
6. `待办处理状态` 与 `采集次数` 必须解耦，不能破坏既有“采集次数可累计”的设计

## 11. 2026-03-25 补充（EHR 命中但未入库单据可见化）

为减少“EHR 已在待办、工作台暂时不可见”的窗口期，处理工作台补充如下口径：

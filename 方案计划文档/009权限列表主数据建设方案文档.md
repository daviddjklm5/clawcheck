# 方案编号009：权限列表主数据建设方案文档

## 1. 方案目标
在现有权限申请采集与后续规则引擎设计基础上，新增一张稳定、可查询、可维护的“权限列表”主数据表，用于统一管理角色编码、角色名称、权限级别及启停状态。

本方案目标：
- 建立统一的权限目录主数据表
- 将本次提供的 96 条权限数据一次性初始化入库
- 为后续规则引擎提供稳定的角色主数据来源
- 明确停用角色与远程角色的识别口径

本方案聚焦“**权限主数据落库**”这一目标，不覆盖页面采集、审批回写或管理后台建设。

## 2. 背景与问题判断

### 2.1 当前现状
当前仓库中已有：
- `automation/sql/001_permission_apply_collect.sql`
  - 负责权限申请单据主表与明细表
- `方案计划文档/102权限自动审核规则引擎技术设计.md`
  - 已预留权限目录主数据作为后续规则输入来源

但当前系统仍缺少一张独立的角色目录表，导致以下问题：
- 规则引擎无法稳定识别某个角色是否为远程角色
- 无法直接区分已取消角色与仍有效角色
- 权限明细表中的 `role_code`、`role_name` 只能保留采集时快照，缺少标准主数据维表

### 2.2 问题结论
因此需要新增一张“权限列表”主数据表，作为：
- 权限申请明细的标准参照表
- 规则引擎事实计算的角色目录来源
- 后续人工维护、新旧角色切换的统一入口

## 3. 方案结论

### 3.1 最终选型
本方案建议新增表：
- `权限列表`

并通过以下方式初始化：
- 新增 SQL 文件 `automation/sql/009_permission_catalog.sql`
- 在 SQL 中直接内置本次提供的 96 条权限清单

### 3.2 采用中文表名的原因
本项目现有主数据与结果表已大量采用中文表名，例如：
- `在职花名册表`
- `组织列表`

本方案采用 `权限列表` 作为物理表名，原因如下：
- 与现有仓库命名风格保持一致
- 更符合业务侧直接查表与维护习惯
- 与本次方案目标“创建权限列表”完全一致

## 4. 表结构设计

### 4.1 建议字段
建议表结构如下：
- `role_code`
  - 角色编码，主键
- `role_name`
  - 角色名称
- `permission_level`
  - 原始权限级别，如 `B1类-涉薪`
- `role_group`
  - 归一化分组，如 `A`、`B1`、`B2`、`C`、`S1`、`S2`、`W`
- `is_remote_role`
  - 是否远程角色
- `不检查组织范围`
  - 是否跳过组织范围校验
- `is_deprecated`
  - 是否已取消角色
- `is_active`
  - 是否有效
- `source_system`
  - 数据来源，当前默认 `manual_seed`
- `raw_payload`
  - 原始快照，便于回溯
- `created_at`
  - 创建时间
- `updated_at`
  - 更新时间

### 4.2 字段口径
字段计算口径建议统一如下：
- `role_group`
  - 从 `permission_level` 归一化提取
- `is_remote_role`
  - `permission_level = A类-远程` 时为 `true`
- `不检查组织范围`
  - `role_code IN ('RLHMD001', 'DTX009')` 时为 `true`
- `is_deprecated`
  - `permission_level = W类-取消` 时为 `true`
- `is_active`
  - `is_deprecated = false`

这样设计的原因：
- 保留原始业务口径 `permission_level`
- 同时产出规则引擎更易消费的布尔标记
- 减少后续 facts 计算重复解析字符串

## 5. 初始化数据设计

### 5.1 数据来源
本次初始化数据直接采用需求中提供的权限清单，共 `96` 条，字段包括：
- 权限编码
- 权限名称
- 权限级别

### 5.2 分类分布
初始化数据覆盖以下类别：
- `A类-远程`
- `B1类-涉薪`
- `B2类-涉档案绩效`
- `C类-常规`
- `S1类-限定`
- `S2类-限定`
- `W类-取消`

### 5.3 写入策略
建议采用：
- `INSERT ... ON CONFLICT (role_code) DO UPDATE`

原因：
- 可重复执行
- 方便后续增补或修正角色名称
- 避免重复导入造成脏数据

## 6. 与现有系统的关系

### 6.1 与权限申请采集表的关系
`申请单权限列表.role_code`、`申请单权限列表.role_name` 继续保留采集快照。

后续使用时建议：
- 以 `申请单权限列表.role_code = "权限列表".role_code` 关联
- 若单据明细无编码，则仍以原始明细兜底，不强依赖主数据表

### 6.2 与规则引擎的关系
该表可直接支持 `102` 中以下 facts 计算：
- `detail.role.group`
- `detail.role.is_remote_role`
- `detail.role.is_legacy`

当前版本中：
- `W类-取消` 可直接视为 `is_deprecated = true`
- 是否高权限、是否老角色的更复杂规则，可在后续方案中继续扩展字段

## 7. 实施步骤
1. 执行 `automation/sql/009_permission_catalog.sql`
2. 校验 `权限列表` 建表成功
3. 校验初始化行数为 `96`
4. 抽查 `A类-远程`、`W类-取消`、`B1类-涉薪` 等分类字段是否符合预期

## 8. 验证建议

### 8.1 建议校验 SQL
```sql
SELECT COUNT(*) FROM "权限列表";

SELECT permission_level, COUNT(*)
FROM "权限列表"
GROUP BY permission_level
ORDER BY permission_level;

SELECT role_code, role_name, role_group, is_remote_role, "不检查组织范围", is_deprecated, is_active
FROM "权限列表"
WHERE role_code IN ('QF24', 'WY32', 'HMC003', 'RLHMD001', 'DTX009');
```

### 8.2 验收标准
满足以下条件可视为方案完成：
- `权限列表` 表创建成功
- 初始化数据共 `96` 条
- `A类-远程` 角色被正确标记为 `is_remote_role = true`
- `RLHMD001`、`DTX009` 被正确标记为 `"不检查组织范围" = true`
- `W类-取消` 角色被正确标记为 `is_deprecated = true` 且 `is_active = false`
- 其他分类角色正常入库，可按 `role_code` 查询

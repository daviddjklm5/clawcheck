# 方案编号010A：人员属性查询发布与回滚步骤

## 1. 目标
本清单用于承接 `010申请单申请人HR与疑似HR识别方案文档`，把“申请人 HR 判定从 `申请单基本信息` 解耦，改为由花名册更新触发全量刷新 `人员属性查询`”落成可执行发布步骤。

本清单重点回答 4 个问题：

1. 哪些代码和 SQL 必须一起发布
2. 正确发布顺序是什么
3. 哪些情况必须中止
4. 出现问题时如何回滚

## 2. 当前结论
`010` 本次不是单纯新增一张表，而是一次**落库职责迁移**。

本次变更同时涉及：
- 表结构新增
- 花名册导入链路调整
- 权限申请主表写入逻辑调整
- 文档口径调整

因此不能只执行 `019_person_attributes.sql` 后继续跑旧代码，也不能只发新代码不建 `人员属性查询`。

正确顺序必须是：
- 先准备回滚点
- 再建 `人员属性查询`
- 再发布新代码
- 再执行一次花名册全量导入触发刷新
- 再清理 `申请单基本信息` 历史遗留 HR 列
- 再执行验收 SQL

## 3. 必改文件清单

### 3.1 数据库结构
必须同步：
- `automation/sql/001_permission_apply_collect.sql`
- `automation/sql/019_person_attributes.sql`
- `automation/sql/021_drop_applicant_hr_columns_from_basic_info.sql`

关键点：
- `申请单基本信息` 新初始化结构不再包含 HR 判定列
- 新增 `人员属性查询`
- 对历史库补做 `申请单基本信息` 旧 HR 列删除

### 3.2 入库层
必须同步：
- `automation/db/postgres.py`

关键点：
- 权限申请写库不再向 `申请单基本信息` 写 HR 判定字段
- 花名册导入完成后，必须全量刷新 `人员属性查询`
- `组织列表` 导入不再直接刷新 `人员属性查询`

### 3.3 运行入口
必须同步：
- `automation/scripts/run.py`
- `automation/README.md`

关键点：
- `dbinit` 需纳入 `人员属性查询`
- 运行说明要明确花名册导入会刷新 `人员属性查询`

### 3.4 文档与验收
至少同步：
- `方案计划文档/003在职花名册下载入库方案文档.md`
- `方案计划文档/010申请单申请人HR与疑似HR识别方案文档.md`
- `方案计划文档/010A人员属性查询发布与回滚步骤.md`
- `方案计划文档/010B人员属性查询下游依赖排查与改造清单.md`
- `方案计划文档/011A九张表字段映射台账.md`
- `automation/sql/021_drop_applicant_hr_columns_from_basic_info.sql`
- `automation/sql/020_person_attributes_acceptance.sql`

## 4. 发布前准备

### 4.1 冻结范围
发布窗口前建议冻结以下范围：
- 花名册导入链路
- 申请人 HR 判定口径
- `人员属性查询` 字段设计

### 4.2 执行前盘点
正式执行前至少确认：

1. 当前库中是否已存在 `人员属性查询`
2. 当前应用代码是否仍向 `申请单基本信息` 写 HR 判定字段
3. 当前 `在职花名册表`、`组织属性查询`、`组织列表` 可正常查询
4. 目标库是否存在历史遗留的 `申请单基本信息` HR 列

### 4.3 备份要求
本次建议先备份以下对象：
- `申请单基本信息`
- `在职花名册表`
- `组织列表`
- `人员属性查询`（若目标库已存在）

原因：
- 本次会调整 HR 判定写入位置
- 花名册导入后会触发 `人员属性查询` 全量刷新

## 5. 正式发布顺序

### 5.1 发布前停写
1. 暂停在职花名册导入任务
2. 暂停任何会并发刷新 `人员属性查询` 的脚本

### 5.2 数据备份
1. 备份目标库
2. 至少单独备份：
   - `申请单基本信息`
   - `在职花名册表`
   - `组织列表`
   - `人员属性查询`（若已有）

### 5.3 执行数据库迁移
执行：
- `automation/sql/019_person_attributes.sql`
- 若目标库已存在旧版 `人员属性查询` 且仅需补齐本次字段，也可单独执行 `automation/sql/032_person_attributes_add_employee_group_fields.sql`
- 若目标库已存在旧版 `人员属性查询` 且仅需补齐 `组织单位` 字段，也可单独执行 `automation/sql/033_person_attributes_add_org_unit_name.sql`

该步骤完成后应满足：
- `人员属性查询` 建表成功
- 主键为 `工号`
- `组织单位`、`员工组`、`员工子组`、`申请人HR类型`、`HR判定原因` 等结果字段已存在
- 索引创建成功

### 5.4 发布应用代码
发布包含以下改造的代码版本：
- `automation/db/postgres.py`
- `automation/scripts/run.py`
- `automation/sql/001_permission_apply_collect.sql`
- `automation/sql/019_person_attributes.sql`
- `automation/sql/032_person_attributes_add_employee_group_fields.sql`
- `automation/sql/033_person_attributes_add_org_unit_name.sql`

### 5.5 触发一次花名册全量刷新
执行一次正式的花名册导入：
- `python automation/scripts/run.py roster ...`

该步骤完成后应满足：
- `在职花名册表` 被最新批次全量刷新
- `人员属性查询` 被同步全量刷新

### 5.6 清理历史遗留旧列
执行：
- `automation/sql/021_drop_applicant_hr_columns_from_basic_info.sql`

该步骤完成后应满足：
- `申请单基本信息` 中历史遗留 HR 判定列已删除

### 5.7 最小回归
发布后至少检查：

1. 权限申请采集正常写入 `申请单基本信息`
2. `申请单基本信息` 不再依赖 HR 判定字段
3. `人员属性查询` 行数与花名册人数一致
4. 随机抽样申请单可通过 `工号` 关联到 `人员属性查询`
5. 下游报表、规则 SQL 不再读取 `申请单基本信息` 旧 HR 列

### 5.8 验收 SQL
执行：
- `automation/sql/020_person_attributes_acceptance.sql`

## 6. 中止条件
出现以下任一情况，必须停止继续发布并评估回滚：

1. `019_person_attributes.sql` 执行报错
2. 花名册导入后 `人员属性查询` 未刷新
3. `人员属性查询` 行数明显小于花名册人数
4. `申请单基本信息` 写入因缺少旧 HR 列报错
5. 申请单通过 `工号` 关联 `人员属性查询` 命中率异常偏低
6. `021` 执行后权限申请采集报列不存在

## 7. 回滚原则
本次回滚以：
- 代码版本回退
- 数据库对象保留或从备份恢复

为准。

不建议在发布窗口内强行删除 `人员属性查询` 或反向重构 `申请单基本信息` 旧 HR 列，原因如下：

1. 旧 HR 列已从新代码写入路径移除
2. 历史库即使仍残留旧列，也不影响新结构运行
3. 本次核心风险在于“新表未刷新”或“代码仍写旧口径”，而不是旧列物理存在

## 8. 回滚顺序
若需回滚，推荐顺序：

1. 停止花名册导入与相关写入任务
2. 回退应用代码到本次发布前版本
3. 如有必要，从备份恢复：
   - `在职花名册表`
   - `人员属性查询`
4. 重新执行最小回归，确认旧链路恢复正常
5. 再决定是否恢复业务写入

## 9. 验收顺序
建议按以下顺序验收：

1. `申请单基本信息` 表结构验收
2. `人员属性查询` 表结构与索引验收
3. 花名册与 `人员属性查询` 行数验收
4. HR 分类与布尔派生一致性验收
5. 申请单关联抽样验收

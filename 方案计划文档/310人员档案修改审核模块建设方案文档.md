# 方案编号 310：人员档案修改审核模块建设方案文档

## 1. 方案目标

在现有 `Python + Playwright + PostgreSQL + React/Vite` 工程内，新增“人员档案修改审核”模块，目标是：

1. 复用现有 EHR 登录态、首页打开和浏览器上下文能力。
2. 从正确入口进入 `人员档案信息变更申请` 页面。
3. 打开 `我的审批进度` 列表，并统一切换到 `100条/页`。
4. 默认按列表当前排序自上而下采集前 `20` 单。
5. 点击单据编号进入详情页，按区段采集表单信息。
6. 若详情存在附件信息区段，则识别附件文件名，并在正式采集阶段下载到指定目录。
7. 将采样结果沉淀为稳定采集方案、字段口径和落库设计。

## 5. 页面结构规律

### 5.1 列表页规律

已确认列表字段为：

1. `#`
2. `单据编号`
3. `单据名称`
4. `单据状态`
5. `提交时间`
6. `创建人.姓名`
7. `创建人.工号`

采集口径：

1. 业务主键使用 `单据编号`
2. 默认按当前列表顺序采集，不额外改排序
3. 首版支持 `--limit`，默认 `20`

### 5.2 详情页头部规律

详情页头部稳定出现以下信息：

1. 标题：`变更申请单`
2. `单据编号`
3. `变更人`
4. `提交变更时间`
5. 单据状态
6. 变更人摘要信息行

变更人摘要信息行通常包含：

1. 姓名
2. 工号
3. 司龄
4. 用工属性
5. 任职类型
6. 在岗状态
7. 组织或岗位相关信息

### 5.3 详情页区段规律

本轮 20 单中，详情区段呈现如下规律：

固定高频区段：

1. `人员信息`：20/20
2. 区段说明行 `注：列表中标黄色的字为用户重新修改后提交的信息`：20/20

可选业务区段：

1. `证件信息`：9/20
2. `教育信息`：8/20
3. `附件信息`：8/20
4. `教育信息1 + 附件信息1 + 教育信息2 + 附件信息2`：2/20
5. `教育信息1 + 附件信息1 + 教育信息2 + 附件信息2 + 证件信息`：1/20
6. `基本信息 + 个人信息`：1/20
7. `基本信息 + 证件信息`：1/20

明确结论：

1. 详情页区段不是固定单层结构。
2. 同一种业务区段可能出现编号后缀，例如 `教育信息1`、`教育信息2`。
3. 附件区段与业务区段成对出现，最典型的是 `教育信息N` 对应 `附件信息N`。

## 6. 表格规律

本轮样本中已确认两类稳定表头：

### 6.1 变更明细表

高频表头：

`# | 操作类型 | 变更项 | 变更前 | 变更后 | 不通过 | 理由`

出现次数：

`34`

说明：

1. 该表头是详情页最核心的业务变更表。
2. `操作类型` 典型值包括：`新增`、`修改`、`删除`。
3. `变更项` 为业务字段名。
4. `变更前`、`变更后` 为字段值对照。

### 6.2 附件明细表

高频表头：

`# | 变更项 | 变更前 | 变更后 | 不通过 | 理由`

出现次数：

`14`

说明：

1. 该表头主要出现在 `附件信息` 区段。
2. 附件文件名一般出现在 `变更后` 列。
3. 当同一行有多个附件时，文件名会在同一单元格中以逗号拼接。

## 7. 附件规律

### 7.1 可见性结论

本轮 20 单中，已识别到 `11` 单存在可见附件文件名。

典型示例：

1. `AM-2026041000026` -> `IMG_20260410_115305.jpg`
2. `AM-2026041000023` -> `mmexport1775791948547.jpg`
3. `AM-2026041000022` -> `e5c9d0c099ab77679024d3e29262f8b4.png`
4. `AM-2026041000014` -> 同一单包含多文件，含 `pdf` 与 `jpg`
5. `AM-2026041000012` -> 同一单包含两个附件文件

### 7.2 附件特征

1. 附件主要出现在 `附件信息` 区段。
2. 文件类型以 `jpg/jpeg/png/pdf` 为主。
3. 单据可能无附件、单附件、或多附件。
4. 多附件可能表现为一个单元格内逗号拼接多个文件名。
5. 当前采样已确认“文件名可见”，但未把附件下载动作纳入本轮采样脚本。

### 7.3 正式下载方案

正式采集阶段的附件下载口径如下：

1. 进入详情后，优先识别 `附件信息` 区段。
2. 对附件表逐行读取 `变更项`、`变更前`、`变更后`。
3. 从 `变更后` 中拆出 1 到 N 个文件名。
4. 在对应行内优先查找真实下载控件或文件预览入口。
5. 优先使用 Playwright `expect_download()` 捕获下载。
6. 若点击后进入预览页，再从预览页识别真实下载按钮。
7. 下载目录固定为：

```text
automation/downloads/personnel_profile_change_audit/<单据编号>/
```

8. 同步记录附件元数据，不因个别附件下载失败而中断整单主数据采集。

## 8. 正式采集方案

### 8.1 采集流程

正式模块建议流程：

1. 打开首页并复用现有登录态
2. 点击左上角九宫格菜单
3. 打开 `最近使用`
4. 点击 `人员档案信息变更申请`
5. 等待 `我的审批进度` 列表就绪
6. 切换分页到 `100条/页`
7. 读取前 `N` 条列表
8. 逐条按 `单据编号` 进入详情
9. 采集头部信息
10. 采集全部区段
11. 若识别到附件区段，则执行附件下载与元数据记录
12. 写入 PostgreSQL

### 8.2 详情采集策略

首版必须采用“区段级采集”，不要把字段硬编码成固定单据结构。

建议结构：

1. 先识别区段标题及顺序
2. 对每个区段识别其类型：
   - 普通字段区
   - 变更明细表区
   - 附件区
3. 普通字段区按 `区段 + 字段名 + 字段值` 采集
4. 表格区按 `区段 + 表头 + 行数据` 采集
5. 附件区按 `区段 + 行号 + 文件名 + 下载结果` 采集

### 8.3 重复区段处理

由于存在 `教育信息1/2`、`附件信息1/2`，正式采集必须保留：

1. `区段序号`
2. `区段名称`
3. `子区段序号`
4. `子区段名称`

不能只用区段名称做唯一键。

## 12. 风险与应对

### 12.1 最近使用菜单改版

风险：

1. 左上角菜单或 `最近使用` DOM 结构调整

应对：

1. 选择器分层设计为“九宫格按钮 -> 最近使用组 -> 应用项”
2. 保留文本兜底与 DOM 兜底

### 12.2 详情区段动态变化

风险：

1. 同一业务对象可能出现 `信息`、`信息1/2` 等多段

应对：

1. 按区段序号采集
2. 不首版硬编码业务字段宽表

### 12.3 附件下载链路不唯一

风险：

1. 附件可能是直接下载
2. 也可能先预览再下载

应对：

1. 优先 `expect_download()`
2. 预览页兜底
3. 下载失败不阻断主数据
